## Dónde estamos

En M1.1 definimos qué son los microservicios, de dónde vienen y cuándo tiene sentido usarlos. Conocimos a TechShop y su problema de crecimiento.

Ahora vamos a profundizar en la comparativa. No como una tabla de pros y contras que hayas visto mil veces — sino analizando aspecto por aspecto, con honestidad, usando el monolito de TechShop como ejemplo real.

Al final de esta sesión tendrás claro: cuándo un monolito es la mejor opción, cuándo el monolito modular es suficiente, y cuándo merece la pena dar el salto a microservicios. Y si decides migrar, cómo hacerlo sin reescribir todo.

## El monolito de TechShop: anatomía completa

Vamos a abrir el capó del monolito de TechShop. Un proyecto ASP.NET Core con Controllers, Entity Framework Core y SQL Server. Todo en una solution de Visual Studio.

La solution tiene esta estructura:

```
TechShop.sln
├── TechShop.API              (Controllers, middleware, Program.cs)
├── TechShop.Application      (Services, DTOs, interfaces)
├── TechShop.Domain            (Entities, value objects, reglas)
├── TechShop.Infrastructure    (EF Core, repositorios, emails)
└── TechShop.Tests             (Tests unitarios e integración)
```

Un proyecto por capa. Clean Architecture aplicada correctamente. EF Core con un solo DbContext apuntando a una base de datos SQL Server con 120 tablas. Todo desplegado como un solo App Service en Azure.

No está mal diseñado. De hecho, es mejor que el 80% de los monolitos que se encuentran en la industria. El problema no es el diseño — es la escala.

---

## La base de datos del monolito

Una sola base de datos SQL Server con 120 tablas. Un solo DbContext llamado `TechShopDbContext` que registra las 120 entidades.

Las tablas más relevantes:

```
Orders          (Id, CustomerId, Status, Total, CreatedAt, ...)  → 90 columnas
OrderLines      (Id, OrderId, ProductId, Quantity, Price, ...)
Products        (Id, Name, Description, Price, Stock, ...)
Categories      (Id, Name, ParentId, ...)
Customers       (Id, Name, Email, Address, ...)
Payments        (Id, OrderId, Amount, Status, Provider, ...)
Shipments       (Id, OrderId, TrackingNumber, Status, ...)
Invoices        (Id, OrderId, Number, TaxAmount, ...)
Notifications   (Id, UserId, Type, SentAt, ...)
AuditLogs       (Id, EntityType, EntityId, Action, UserId, ...)
```

La tabla Orders tiene 90 columnas porque a lo largo de 8 años se le han ido añadiendo campos: datos de envío, datos de facturación, flags de estados intermedios, campos legacy que nadie se atreve a borrar.

Un solo `SaveChanges()` puede tocar 5 tablas en una transacción. Cómodo, pero acopla todo.

---

## El despliegue del monolito

Todo TechShop se despliega como un solo artefacto:

```
dotnet publish → TechShop.API.dll → Azure App Service
```

Un App Service, un deployment slot, un pipeline de CI/CD. Un botón para desplegar, un botón para hacer rollback.

Suena simple. Y lo es — hasta que la simplicidad se convierte en rigidez:

- Cualquier cambio, por pequeño que sea, despliega TODO. Un fix en la lógica de notificaciones obliga a desplegar también la lógica de pedidos, pagos y envíos.
- El deploy tarda 12 minutos entre build, tests y publicación. Los 18 developers comparten el mismo pipeline.
- Si el deploy falla, hay que hacer rollback de TODO — incluyendo cambios de otros equipos que sí funcionaban.

---

## Ventajas reales del monolito: simplicidad de desarrollo

Vamos a ser justos. El monolito tiene ventajas genuinas que los microservicios no pueden replicar.

**Un solo proyecto en el IDE.** Abres Visual Studio, cargas la solution, y tienes todo el código de TechShop delante. Ctrl+Shift+F y encuentras cualquier referencia en todo el sistema. No necesitas saber "en qué servicio estaba eso".

**Navegación de código directa.** F12 (Go to Definition) te lleva de un Controller a un Service a un Repository sin salir del IDE. Sin saltos entre proyectos, sin buscar en otro repositorio, sin preguntarte "¿cómo se llama el endpoint que expone esto?".

---

## Ventajas del monolito: refactoring con confianza

**Renombrar, mover, cambiar interfaces — el compilador te cubre.** Cambias el nombre de `OrderService` a `OrderCommandService` y Visual Studio te muestra todos los puntos donde se usa. Un clic y todo actualizado.

Cambias la firma de un método — añades un parámetro, cambias un tipo — y el compilador te lista inmediatamente todos los sitios que necesitan adaptarse. Rojo en el editor, cero en producción.

En microservicios, cambiar un contrato entre servicios no es un "Rename" de Visual Studio. Es:
1. Añadir el nuevo campo al productor (sin quitar el viejo)
2. Desplegar el productor
3. Actualizar todos los consumidores para usar el nuevo campo
4. Desplegar los consumidores
5. Eliminar el campo viejo del productor
6. Desplegar de nuevo

Un cambio que en el monolito tarda 5 minutos puede tardar una semana en microservicios.

---

## Ventajas del monolito: debugging directo

**Un solo proceso, un solo breakpoint.** Pones un breakpoint en el Controller, otro en el Service, otro en el Repository. Pulsas F5 y el debugger para en cada punto. Ves las variables, el stack trace completo, el estado de todo.

**Stack trace lineal.** Cuando algo falla, el stack trace te dice exactamente la cadena de llamadas: `OrderController.Create → OrderService.CreateOrder → OrderRepository.Save → DbContext.SaveChanges`. Todo en un proceso, todo en memoria.

**Watch window con todo el estado.** Puedes inspeccionar cualquier variable, cualquier objeto, cualquier colección en cualquier punto del flujo. Sin saltar entre servicios, sin buscar en logs de otro proceso.

En microservicios, un breakpoint solo ve un servicio. Para seguir el flujo completo necesitas correlation IDs, distributed tracing y herramientas como Application Insights. Funciona — pero nunca es tan inmediato como F5 en un monolito.

---

## Ventajas del monolito: transacciones ACID

Esta es posiblemente la ventaja más potente del monolito. Una sola transacción de base de datos cubre todo.

En el monolito de TechShop, crear un pedido implica:

```csharp
using var transaction = await _dbContext.Database.BeginTransactionAsync();

var order = new Order(customerId, items);
_dbContext.Orders.Add(order);

var payment = new Payment(order.Id, order.Total);
_dbContext.Payments.Add(payment);

product.ReserveStock(quantity);
_dbContext.Products.Update(product);

await _dbContext.SaveChangesAsync();
await transaction.CommitAsync();
```

Si el pago falla, todo hace rollback automáticamente. El stock no se reserva, el pedido no se crea. Consistencia garantizada.

En microservicios, este mismo flujo requiere una Saga con compensaciones, consistencia eventual, y manejo de fallos parciales. Mucho más código, mucha más complejidad mental.

---

## Ventajas reales del monolito: testing integrado

**WebApplicationFactory en toda su potencia.** Levantas todo TechShop en memoria, lanzas peticiones HTTP y verificas el resultado. Todo el flujo, de principio a fin, en un solo test.

```csharp
var client = _factory.CreateClient();
var response = await client.PostAsJsonAsync("/api/orders", newOrder);
Assert.Equal(HttpStatusCode.Created, response.StatusCode);

// Verificar que el stock se actualizó
var product = await client.GetFromJsonAsync<Product>($"/api/products/{productId}");
Assert.Equal(expectedStock, product.Stock);
```

Un test cubre pedidos, stock y pagos. No necesitas levantar 4 servicios, configurar message brokers, ni esperar a que los eventos se propaguen.

En microservicios, este mismo test requiere que los 4 servicios estén arrancados, RabbitMQ esté funcionando, y los eventos se hayan procesado. Es más lento, más frágil y más difícil de mantener.

---

## Ventajas reales del monolito: operaciones simples

**Un solo deploy.** Un pipeline, un artefacto, un App Service. Si funciona, funciona todo. Si falla, rollback de todo.

**Un solo log.** Todos los logs del sistema en un fichero (o en un sink de Serilog). No necesitas correlation IDs ni tracing distribuido para seguir una petición — el stack trace te lo dice todo.

**Un solo health check.** `/health` responde 200 y todo el sistema está bien. No necesitas verificar la salud de 4 servicios, 4 bases de datos y un message broker.

**Backup simple.** Una base de datos SQL Server. Un backup, un restore. No hay 4 bases de datos que sincronizar ni consistencia eventual entre backups.

**Monitoring simple.** Application Insights muestra un Application Map con una sola burbuja. En microservicios, el mapa tiene 4 burbujas conectadas con líneas que a veces se ponen rojas.

---

## Entonces, ¿por qué no quedarse con el monolito?

Con todas estas ventajas, ¿por qué alguien querría microservicios? Porque las ventajas del monolito tienen un límite. Y ese límite lo marca la escala.

Cuando el equipo es pequeño (3-8 personas), estas ventajas compensan cualquier limitación. Pero cuando creces, cada ventaja empieza a erosionarse:

- "Un solo proyecto" → con 200.000 líneas, Intellisense tarda 10 segundos en responder.
- "Debugging directo" → con 18 developers, solo uno puede hacer debug a la vez sin conflictos.
- "ACID" → funciona, pero la tabla de 90 columnas es síntoma de que el modelo necesita partirse.
- "Testing integrado" → sí, pero la suite tarda 40 minutos.
- "Operaciones simples" → sí, pero cada deploy es un todo-o-nada que da miedo.

El monolito no tiene un problema intrínseco. Tiene un problema de escala.

---

## Las señales de que el monolito duele

Señales técnicas de que necesitas hacer algo (no necesariamente microservicios — quizá un monolito modular basta):

**El build tarda más de 5 minutos.** El equipo pierde productividad esperando. La solución incremental es optimizar el build. La solución estructural es partir.

**La suite de tests tarda más de 15 minutos.** Nadie la ejecuta en local. Los bugs se descubren en el pipeline — o en producción.

**Los merge conflicts son diarios.** Tres personas necesitan tocar el mismo fichero para cambios que no tienen relación entre sí.

**El deploy da miedo.** "No desplegamos los viernes" es un síntoma clásico. Cada deploy es una apuesta: funciona o no funciona, pero no hay término medio.

**Hay zonas del código que nadie toca.** El "módulo intocable" que acumula deuda técnica porque "la última vez que lo tocamos, se cayó producción".

---

## Más señales de que el monolito duele

**El escalado es todo-o-nada.** La búsqueda de productos necesita 10 instancias en Black Friday. La administración sigue con 1. Pero escalas todo junto — pagas por 10 instancias completas.

**Los equipos se bloquean entre sí.** El equipo de pagos necesita cambiar una interfaz compartida. El equipo de pedidos está a mitad de un sprint y no puede absorber el cambio. Resultado: el cambio de pagos espera 3 semanas.

**Las releases son eventos coordinados.** "Antes de desplegar, verificar que el cambio de envíos es compatible con el cambio de facturación que subió ayer." Cada deploy necesita una reunión de coordinación.

**La base de datos es un cuello de botella.** 120 tablas, un solo DbContext, conexiones compartidas. Las queries de reporting bloquean las de escritura. Los deadlocks aparecen en horas punta.

**El onboarding es lento.** Un developer nuevo tarda 3 meses en sentirse productivo. Nadie conoce el 80% de la base de código. La documentación está desactualizada.

## Comparativa aspecto por aspecto: Desarrollo

**Velocidad de desarrollo inicial:**
- Monolito: MÁS RÁPIDO. Un solo proyecto, un solo IDE, refactoring directo. Cero infraestructura distribuida.
- Microservicios: MÁS LENTO al principio. Necesitas configurar comunicación, gestionar múltiples proyectos, entender consistencia eventual.

**Velocidad de desarrollo a largo plazo (equipo >10 personas):**
- Monolito: SE DEGRADA. Builds lentos, conflictos, coordinación constante.
- Microservicios: SE MANTIENE. Cada equipo trabaja de forma independiente en su servicio.

**El punto de cruce:** Para equipos de menos de 8 personas, el monolito es más rápido siempre. Para equipos de más de 15, los microservicios empiezan a compensar. Entre 8 y 15, depende de la complejidad del dominio.

---

## Comparativa: Despliegue y release management

**Frecuencia de deploy:**
- Monolito: Limitada. Cada deploy es todo-o-nada. El riesgo aumenta con cada cambio acumulado entre releases.
- Microservicios: Cada servicio despliega de forma independiente. Puedes desplegar 5 veces al día servicios diferentes sin coordinación.

**Riesgo de deploy:**
- Monolito: ALTO. Un bug en cualquier módulo afecta a todo. Rollback también es todo-o-nada.
- Microservicios: CONTENIDO. Un bug en Pagos no afecta a Productos. Rollback individual.

---

## Comparativa: Tiempos de deploy y coordinación

**Tiempo de deploy:**
- Monolito TechShop: 12 minutos (build + test + publish). 18 developers comparten el pipeline.
- Microservicio individual: 3-4 minutos (menos código, menos tests, pipeline propio).

**Coordinación:**
- Monolito: "¿Tu cambio es compatible con el mío? ¿Podemos desplegar juntos? ¿Quién va primero?" Reuniones de release, ventanas de mantenimiento, coordinación entre equipos.
- Microservicios: "Despliega cuando quieras. Si no rompes el contrato, no afectas a nadie." Contract testing lo verifica automáticamente.

**Rollback:**
- Monolito: Vuelves a la versión anterior. Pero esa versión no tiene los cambios de OTROS equipos que sí funcionaban. Rollback colateral.
- Microservicios: Solo rollback del servicio con problemas. El resto no se toca.

---

## Comparativa: Escalado

**Monolito: escalado vertical u horizontal uniforme.**

Vertical: más CPU, más RAM al App Service. Simple pero tiene techo. Un App Service Plan tiene límites.

Horizontal: más instancias del mismo App Service. Pero escalas TODO — el módulo de administración (5 usuarios) escala igual que el de búsqueda (50.000 peticiones/hora).

**Microservicios: escalado selectivo.**

Cada servicio escala de forma independiente. Orders necesita 3 instancias. Products necesita 10 en Black Friday. Notifications necesita 1. Payments necesita 2.

El ahorro en infraestructura puede ser significativo. Si el 80% del tráfico es búsqueda de productos y solo el 5% es administración, pagar por 10 instancias completas del monolito cuando solo necesitas 10 de Products es un desperdicio.

En Azure App Services, cada servicio tiene su propio plan con auto-scale rules independientes.

---

## Comparativa: Consistencia de datos

**Monolito: ACID nativo.**

Una transacción cubre todo. `BeginTransaction → SaveChanges → Commit`. Si algo falla, rollback automático. El desarrollador no necesita pensar en compensaciones ni en estados intermedios.

El modelo mental es simple: "guardo en la base de datos y está ahí". Inmediatamente. Para todo el mundo. Sin excepciones.

**Microservicios: consistencia eventual.**

Cada servicio tiene su propia base de datos. No hay transacción que cruce servicios. Si Orders guarda el pedido y Payments falla al cobrar, hay un pedido sin pago. Necesitas una Saga para compensar — cancelar el pedido, liberar el stock.

El modelo mental cambia: "publico un evento y confío en que el sistema convergerá al estado correcto". No inmediatamente — eventualmente. Puede tardar milisegundos o segundos.

**Esto no es un bug.** Es una decisión de diseño. Amazon te confirma el pedido al instante, pero el cargo en tu banco puede tardar minutos. La experiencia del usuario no sufre — el sistema piensa diferente.

---

## Comparativa: Debugging y diagnóstico

**Monolito: debugging lineal.**

Stack trace → breakpoint → inspección de variables → solución. Todo en un proceso, todo visible, todo en Visual Studio.

Tiempo medio para diagnosticar un bug en TechShop monolito: 30 minutos a 2 horas.

**Microservicios: debugging distribuido.**

Un error en la creación de pedido puede originarse en Orders (validación), Products (stock insuficiente), Payments (timeout al proveedor) o Notifications (formato de email incorrecto).

Necesitas: Correlation IDs que viajen por toda la cadena, distributed tracing con OpenTelemetry, logs estructurados con Serilog que se puedan filtrar por correlación, y herramientas como Application Insights que conecten las piezas.

Tiempo medio para diagnosticar un bug: 15 minutos si tienes buena observabilidad. 4 horas si no la tienes.

La observabilidad no es opcional en microservicios. Es la diferencia entre "lo diagnostico en 15 minutos" y "llevo toda la mañana buscando qué se rompió".

---

## Comparativa: Testing

**Monolito: la pirámide clásica.**

Unitarios (rápidos, del dominio), integración (WebApplicationFactory, todo en memoria), E2E (contra el App Service desplegado). Un solo proceso, un solo setup.

Cubrir el flujo completo de creación de pedido: un test. Levantas la app, lanzas la petición, verificas todo — pedido, stock, pago.

**Microservicios: la pirámide extendida.**

Unitarios (igual), integración interna (WebApplicationFactory por servicio), contract testing (Pact — ¿el contrato entre Orders y Products sigue siendo válido?), integración entre servicios (necesitas los 4 arrancados + RabbitMQ), E2E (todo desplegado).

Cubrir el mismo flujo: 4 tests de integración internos + 3 contract tests + 1 test E2E. Más tests, más infraestructura, más tiempo.

La capa de contract testing es nueva y fundamental. Sin ella, cambias un endpoint en Products y rompes Orders sin enterarte. Se descubre en producción, el viernes a las 18:00.

---

## Comparativa: Coste de infraestructura

**Monolito: simple y predecible.**

Un App Service (Plan B1 en Azure: ~13€/mes para dev, ~50€/mes para producción). Una base de datos SQL Server. Un Application Insights. Coste mensual estimado para TechShop: ~80-100€/mes.

**Microservicios: más recursos, más coste base.**

4 App Services + 1 Gateway = 5 App Services. 4 bases de datos SQL Server (o elastic pool). Azure Service Bus. Application Insights. Coste mensual estimado para OrderFlow: ~200-350€/mes.

---

## Comparativa: el matiz del escalado en costes

El coste base de microservicios es mayor. Pero el coste de escalado es menor.

**Ejemplo Black Friday:**
- Monolito: necesitas 10x capacidad. Escalar el App Service de B1 a P3 con 10 instancias: ~500€ por el día.
- Microservicios: solo Products necesita 10x. Escalar Products de B1 a B3 con 10 instancias + el resto sin cambios: ~200€ por el día.

Para cargas de trabajo constantes, el monolito gana en coste. Para cargas variables con picos en funcionalidades específicas, microservicios son más económicos.

Y hay un coste que no aparece en la factura de Azure: el coste del equipo. Si 18 developers pierden 2 horas al día en coordinación del monolito, esos 360.000€/año superan con creces la diferencia en infraestructura.

---

## Comparativa: Onboarding de nuevos developers

**Monolito: todo o nada.**

El developer nuevo tiene que entender (al menos superficialmente) todo el sistema. 200.000 líneas, 120 tablas, módulos con dependencias cruzadas. "¿Qué hace esta clase?" → "Depende del contexto — se usa en pedidos, en envíos y en facturación."

Tiempo hasta la primera contribución productiva: 2-4 semanas para un cambio simple. 2-3 meses para sentirse cómodo con el sistema.

**Microservicios: uno a la vez.**

El developer nuevo se centra en UN servicio. Orders tiene ~10.000 líneas, su propio dominio, sus propios tests. Puede entender el servicio completo en 1-2 semanas y hacer contribuciones productivas al tercer día.

Entender cómo los servicios se comunican entre sí lleva más tiempo. Pero para el 80% del trabajo diario, basta con conocer tu servicio.

## Comparativa: resumen visual

La mejor forma de verlo no es una tabla — es una curva.

En los primeros meses, el monolito es más productivo. Menos infraestructura, menos decisiones, desarrollo más rápido. La curva de microservicios arranca por debajo.

Conforme el equipo crece (8-15 personas), las curvas se cruzan. El monolito se ralentiza (builds, conflictos, coordinación). Los microservicios mantienen la velocidad porque los equipos trabajan de forma independiente.

Con equipos grandes (+15 personas), la diferencia se amplía. El monolito se convierte en un cuello de botella. Los microservicios escalan linealmente con el número de equipos.

El punto de cruce depende de tu contexto: complejidad del dominio, madurez operativa, experiencia del equipo. No hay una respuesta universal.

## El monolito modular: la opción que nadie menciona

Entre el monolito clásico y los microservicios hay una alternativa que merece mucha más atención: el monolito modular.

La idea: mantienes un solo deployable, pero internamente el código está organizado en módulos con fronteras estrictas. Cada módulo tiene su propio dominio, su propia interfaz pública, y — idealmente — su propio esquema en la base de datos.

Es como vivir en un edificio de apartamentos: cada apartamento es independiente, tiene sus propias llaves, su propia distribución. Pero comparten cimentación, electricidad y fontanería.

---

## Las reglas del monolito modular

**1. Cada módulo expone una interfaz pública y esconde su implementación.** El módulo de Pedidos expone `IOrderService` con métodos como `CreateOrder()`, `GetOrder()`. Su implementación interna (entidades, repositorios, lógica) es invisible para el resto.

**2. Los módulos se comunican solo a través de interfaces.** Nunca accediendo directamente a las clases internas de otro módulo. Si Pagos necesita datos de Pedidos, llama a `IOrderService`, no importa `Order.cs` directamente.

**3. Cada módulo tiene su propio DbContext.** `OrderDbContext` solo conoce las tablas de pedidos. `PaymentDbContext` solo las de pagos. Esquemas separados dentro de la misma base de datos SQL Server.

**4. Las dependencias entre módulos son explícitas y unidireccionales.** Si el módulo A depende del B, se declara en el proyecto. Dependencias circulares están prohibidas. Un módulo compartido contiene las interfaces comunes.

---

## Monolito modular en .NET 10: la estructura

Así se vería TechShop como monolito modular:

```
TechShop.sln
├── TechShop.API                     (Program.cs, composición)
├── TechShop.Shared                  (Interfaces comunes, DTOs compartidos)
│
├── Modules/
│   ├── TechShop.Orders.API          (Controllers de pedidos)
│   ├── TechShop.Orders.Application  (Servicios, handlers)
│   ├── TechShop.Orders.Domain       (Entidades, reglas)
│   ├── TechShop.Orders.Infrastructure (OrderDbContext, repos)
│   │
│   ├── TechShop.Products.API        (Endpoints de productos)
│   ├── TechShop.Products.Application
│   ├── TechShop.Products.Domain
│   ├── TechShop.Products.Infrastructure (ProductDbContext)
│   │
│   ├── TechShop.Payments.API
│   ├── TechShop.Payments.Application
│   ├── TechShop.Payments.Domain
│   ├── TechShop.Payments.Infrastructure (PaymentDbContext)
│   │
│   └── TechShop.Notifications.API
│       └── ... (mismo patrón)
│
└── TechShop.Tests
```

Un solo `dotnet run` arranca todo. Pero internamente, los módulos están separados como si fueran servicios independientes.

---

## Monolito modular: DbContext separados en SQL Server

El truco técnico del monolito modular en .NET: cada módulo tiene su propio DbContext que solo conoce sus tablas.

```csharp
// Módulo Orders
public class OrderDbContext : DbContext
{
    public DbSet<Order> Orders { get; set; }
    public DbSet<OrderLine> OrderLines { get; set; }
    // Solo tablas de Orders — no sabe de Products ni Payments
}

// Módulo Products
public class ProductDbContext : DbContext
{
    public DbSet<Product> Products { get; set; }
    public DbSet<Category> Categories { get; set; }
    // Solo tablas de Products
}
```

Ambos DbContexts apuntan a la misma base de datos SQL Server, pero usan esquemas diferentes:

```
[orders].Orders, [orders].OrderLines
[products].Products, [products].Categories
[payments].Payments, [payments].Transactions
```

Esto marca las costuras. El día que extraigas Products como microservicio, su DbContext ya está aislado — solo necesitas cambiar la connection string a una nueva base de datos.

---

## Monolito modular: comunicación entre módulos

¿Cómo se comunican los módulos dentro del monolito modular? A través de interfaces, no de acceso directo.

```csharp
// En TechShop.Shared — la interfaz pública del módulo Products
public interface IProductService
{
    Task<ProductDto> GetByIdAsync(int id);
    Task<bool> ReserveStockAsync(int productId, int quantity);
}

// En el módulo Orders — consume el módulo Products vía interfaz
public class CreateOrderHandler
{
    private readonly IProductService _products;
    
    public async Task Handle(CreateOrderCommand cmd)
    {
        var available = await _products.ReserveStockAsync(cmd.ProductId, cmd.Qty);
        if (!available) throw new InsufficientStockException();
        // ... crear pedido
    }
}
```

Esto es una llamada en memoria — nanosegundos, sin red, sin serialización. Pero la frontera está clara: Orders no importa `Product.cs`, solo conoce `IProductService` y `ProductDto`.

Cuando extraigas Products como microservicio, reemplazas la implementación de `IProductService`: de una llamada en memoria a una llamada HTTP. La interfaz no cambia. El código de Orders no cambia.

---

## Monolito modular: las reglas que hacen que funcione

Sin disciplina, el monolito modular degenera en monolito clásico en semanas. Estas reglas son innegociables:

**Regla 1: No importar clases internas de otro módulo.** Si necesitas un dato de Products, llamas a `IProductService`. Nunca importas `Product.cs` directamente. El analizador de código debería fallar si alguien lo intenta.

**Regla 2: No compartir DbContext.** Nunca un query que haga JOIN entre tablas de módulos diferentes. Si necesitas datos cruzados, pide al módulo correspondiente por su interfaz.

**Regla 3: No dependencias circulares.** Si Orders depende de Products, Products NO puede depender de Orders. Si ambos necesitan comunicarse, usa un módulo de eventos compartidos.

**Regla 4: Migrations independientes.** Cada módulo tiene sus propias migrations. El esquema de Orders no se mezcla con el de Products.

**Regla 5: Revisar en code review.** Antes de aprobar un PR, verificar que no se violan las fronteras entre módulos. Es el punto donde la disciplina se aplica.

---

## Monolito modular: los beneficios que conservas Un deploy, un proceso, un log, debugging directo con F5. Todo lo que vimos como ventajas del monolito sigue aplicando.

**Estructura de microservicios:** Las fronteras entre módulos son claras. Las dependencias son explícitas. Cada módulo tiene su dominio y su DbContext.

**Costuras marcadas para el futuro:** Si mañana necesitas extraer Orders como microservicio, el esfuerzo es mucho menor. El código ya está separado, los contratos definidos, el DbContext aislado. Solo tienes que ponerlo en su propio proyecto y exponerlo como API.

**Equipos con autonomía parcial:** Cada equipo trabaja en su módulo sin pisar al resto. Los merge conflicts se reducen drásticamente porque cada equipo toca sus carpetas.

**Sin infraestructura distribuida:** No necesitas message broker, ni service discovery, ni distributed tracing, ni circuit breakers. Todo funciona con llamadas en memoria.

---

## Monolito modular: las limitaciones

**Despliegue sigue siendo todo-o-nada.** Un cambio en Notificaciones obliga a redesplegar todo. Menos riesgo que el monolito clásico (los módulos están separados), pero sigue siendo un solo artefacto.

**Escalado sigue siendo uniforme.** No puedes escalar Productos independientemente de Pedidos. Escalas todo el App Service.

**La tentación del atajo.** "Solo esta vez voy a importar directamente esa clase de otro módulo." Sin disciplina, las fronteras se erosionan y acabas con un monolito clásico otra vez.

**Base de datos compartida.** Aunque cada módulo tiene su DbContext, comparten la misma instancia de SQL Server. Migrations pueden interferir. Cambios de esquema en un módulo pueden afectar al rendimiento de otro.

**No es suficiente para TechShop.** Con 18 developers y la necesidad de escalar búsqueda de forma independiente, el monolito modular se queda corto. Pero para un equipo de 8-12, probablemente sería la opción correcta.

---

## ¿Cuándo el monolito modular es suficiente?

**Sí es suficiente cuando:**
- Tu equipo tiene entre 5 y 12 developers
- No necesitas escalar partes del sistema de forma independiente
- El dominio es moderadamente complejo
- Quieres estructura sin complejidad operativa
- Estás empezando y no sabes dónde estarán los bounded contexts exactos

**No es suficiente cuando:**
- Tienes más de 15 developers con necesidad de autonomía real
- Necesitas escalar funcionalidades de forma independiente
- Los ciclos de despliegue de diferentes módulos son muy diferentes
- Un módulo necesita tecnología diferente (aunque con SQL Server en todo, esto es menos frecuente)

**La mejor estrategia:** Empieza con monolito modular. Si un módulo crece, lo extraes como microservicio. Es el camino de mínimo riesgo.

---

## De monolito modular a microservicio: las señales

¿Cómo sabes que un módulo debería convertirse en microservicio? Estas señales:

**Necesita escalar independientemente.** El módulo de Products recibe 10x más tráfico que el resto en campañas. Escalar todo el monolito por él es caro.

**Su ciclo de despliegue es muy diferente.** Products necesita ir a producción 3 veces al día con cambios de catálogo. El resto despliega una vez a la semana.

**Un equipo dedicado lo mantiene.** Si hay un equipo de 5 personas que solo trabaja en Products, ya tienen la autonomía organizativa — solo les falta la autonomía técnica.

**Tiene requisitos no funcionales únicos.** Products necesita caché agresiva con Redis. El resto no. Dentro del monolito, añadir Redis complica la infraestructura para todos.

Cuando ves dos o más de estas señales en el mismo módulo, es candidato a extracción.

---

## Strangler Fig Pattern: migrar sin reescribir

El nombre viene de las higueras estranguladoras: una planta que crece alrededor de un árbol existente, lo va envolviendo y eventualmente lo reemplaza. Sin que el árbol original deje de funcionar durante el proceso.

Aplicado a software: en vez de reescribir el monolito de cero (proyecto de años con alto riesgo de fracaso), lo vas "estrangulando" servicio a servicio. Cada funcionalidad que extraes reduce el monolito y añade un microservicio. El monolito y los microservicios coexisten durante la migración.

Es el patrón más seguro para migrar. Gradual, reversible, con valor en cada paso.

---

## Strangler Fig: los pasos

**Paso 1 — Interceptar.** Pones un reverse proxy (YARP) delante del monolito. Todo el tráfico pasa por el proxy. El proxy reenvía todo al monolito — cero cambio funcional. Solo estás poniendo la pieza que te permitirá redirigir tráfico después.

**Paso 2 — Extraer.** Implementas una funcionalidad como microservicio nuevo. Por ejemplo, Products.API. El proxy redirige las peticiones de `/api/products/*` al nuevo servicio. El resto sigue yendo al monolito.

**Paso 3 — Verificar.** Ambos coexisten. Puedes comparar respuestas, hacer canary testing (enviar el 10% del tráfico al nuevo servicio y el 90% al monolito). Si algo falla, el proxy vuelve a dirigir todo al monolito. Riesgo mínimo.

**Paso 4 — Migrar datos.** Los datos de productos se mueven del monolito a la base de datos del nuevo servicio. El monolito deja de tener la tabla Products y llama al microservicio cuando necesita datos de productos.

**Paso 5 — Eliminar.** Cuando el microservicio es estable y probado, eliminas esa funcionalidad del monolito.

**Paso 6 — Repetir.** Eliges el siguiente módulo con más dolor y repites.

---

## Strangler Fig: TechShop paso a paso

Veamos cómo TechShop aplicaría el patrón:

**Estado inicial:** Monolito en un App Service. 120 tablas en SQL Server.

**Mes 1-2:** Poner YARP como reverse proxy delante del monolito. Todo el tráfico pasa por YARP. Verificar que nada cambia.

**Mes 3-4:** Extraer Products como primer microservicio. Crear Products.API (Minimal APIs + SQL Server con su propia base de datos). Migrar datos de productos. YARP redirige `/api/products/*` al nuevo servicio.

**Mes 5-6:** Extraer Notifications. El módulo más independiente — solo envía emails. Mínima interacción con el resto.

**Mes 7-9:** Extraer Payments. Necesita coordinarse con Orders para el flujo de cobro. Primera implementación de mensajería asíncrona.

**Mes 10-12:** Extraer Orders. El último y más complejo. Requiere implementar la Saga de creación de pedidos.

---

## Strangler Fig: el reto de los datos

La parte más delicada de cualquier migración es mover los datos. No es solo copiar tablas — es cambiar quién es "dueño" de esos datos.

**Al extraer Products:**
1. Crear la nueva base de datos ProductsDb en SQL Server
2. Copiar las tablas Products y Categories del monolito a ProductsDb
3. El monolito cambia: donde antes hacía `_dbContext.Products.Find(id)` ahora llama a `http://products-api/api/products/{id}`
4. Las tablas de productos en la base de datos del monolito se marcan como deprecated
5. Verificar durante semanas que todo funciona
6. Eliminar las tablas del monolito

**El periodo de dual-write es el más peligroso.** Durante la migración, puede haber un momento en que tanto el monolito como el nuevo servicio necesiten escribir en las tablas de productos. Esto se gestiona con el Expand and Contract pattern que veremos en M6.4.

---

## Strangler Fig: por qué funciona

**Siempre tienes un sistema funcionando.** En ningún momento detienes el monolito. Los usuarios no notan la migración.

**Puedes pausar cuando quieras.** Si las prioridades de negocio cambian, pausas la migración. El sistema funciona con un mix de monolito y microservicios.

**Cada paso aporta valor.** No esperas 12 meses para ver resultados. Desde el mes 3, Products ya escala de forma independiente.

**El riesgo es incremental.** Si un microservicio no funciona, vuelves a redirigir al monolito. No has perdido nada.

**Aprendes sobre la marcha.** El primer microservicio que extraes te enseña lecciones que aplicas al segundo. El segundo te enseña lecciones para el tercero. Cada extracción es más fácil que la anterior.

---

## Strangler Fig: el primer servicio a extraer

¿Por cuál empezar? La decisión importa porque el primer microservicio establece los patrones que seguirán los demás.

**Criterios para elegir el primero:**

**Bajo riesgo, alto aprendizaje.** No empieces por el módulo más crítico (pedidos). Empieza por uno donde un fallo no sea catastrófico. Notifications es buen candidato — si falla, los emails se retrasan pero nadie pierde dinero.

**Pocas dependencias de entrada.** El módulo ideal tiene pocas conexiones con el resto. Notifications consume eventos pero no los produce. Products expone datos pero no llama a otros módulos.

**Valor demostrable.** El primer microservicio tiene que demostrar al equipo (y a management) que la migración vale la pena. Si Products escalando independientemente reduce la factura de Azure un 30% en Black Friday, la conversación para extraer el siguiente es más fácil.

Para TechShop, el orden sería: Products → Notifications → Payments → Orders. De menos a más complejo, de menos a más acoplado.

---

## Strangler Fig: errores comunes en la migración

**Error 1: "Big bang rewrite".** Reescribir todo el monolito desde cero. Proyecto de 18 meses con un equipo dedicado que no aporta valor al negocio hasta que termine — si termina. La historia de la industria está llena de reescrituras fallidas.

**Error 2: Extraer sin desacoplar primero.** Sacas el módulo de Payments del monolito pero sigue leyendo de la tabla Orders. Has creado un servicio con dependencia directa de la base de datos del monolito. Monolito distribuido instantáneo.

**Error 3: Migrar todo a la vez.** "Vamos a extraer los 4 servicios este trimestre." Demasiado cambio a la vez, demasiado riesgo, demasiada complejidad. Uno a uno, con meses entre cada extracción.

**Error 4: No invertir en el proxy.** Hacer la migración sin un reverse proxy (YARP). Sin el proxy, no puedes redirigir tráfico gradualmente ni hacer rollback fácilmente.

**Error 5: Olvidar los datos.** Extraes el servicio pero los datos siguen en la base de datos del monolito. Necesitas migrar los datos Y cambiar el monolito para que llame al nuevo servicio en vez de a sus propias tablas.

---

## Strangler Fig: el proxy con YARP

YARP (Yet Another Reverse Proxy) es el reverse proxy de Microsoft para .NET. Es la pieza que hace posible el Strangler Fig en nuestro ecosistema.

La idea es simple: YARP recibe todas las peticiones y decide a dónde enviarlas basándose en rutas configuradas.

Configuración durante la migración:

```
/api/products/*    → Products.API (microservicio nuevo)
/api/notifications/* → Notifications.API (microservicio nuevo)
/api/*             → Monolito TechShop (todo lo demás)
```

Conforme extraes servicios, añades rutas. El monolito se va encogiendo. Un día, la ruta por defecto al monolito ya no recibe tráfico — y lo apagas.

Profundizaremos en YARP en el módulo M4.4. Por ahora, el concepto es lo que importa.

## La línea temporal realista de una migración

Si TechShop hiciera la migración real con Strangler Fig:

**Mes 1-2:** Preparación. Poner YARP, configurar CI/CD, formar al equipo en patrones distribuidos. Sin cambios funcionales.

**Mes 3-5:** Products.API como primer microservicio. Migrar datos, redirigir tráfico, verificar en producción. Aprender de los errores.

**Mes 8-10:** Payments.API. Más complejo por la coordinación con Orders. Primera implementación de comunicación asíncrona.

Total: 12-15 meses para un equipo dedicado. Cada mes con valor entregado.

## Objeciones comunes y respuestas honestas

**"Es demasiado complejo para nuestro equipo."** Puede ser cierto ahora. El monolito modular es un paso intermedio válido. Pero si tu equipo no se forma en sistemas distribuidos, el problema solo se pospone.

**"No tenemos presupuesto para la infraestructura."** Con Aspire para desarrollo y Azure App Services para producción, el coste de infraestructura es moderado. El coste real es el tiempo del equipo — y ese se recupera en velocidad de entrega.

**"¿Y si falla?"** Strangler Fig es reversible en cada paso. Si el primer microservicio no funciona, vuelves a redirigir al monolito. Riesgo acotado.

**"Nuestro monolito funciona bien."** Si funciona bien, no lo toques. En serio. No migres por moda. Solo cuando el dolor sea mayor que el coste de la migración.

**"No queremos usar Docker/Kubernetes."** No hace falta. Con Aspire y Azure App Services puedes hacer microservicios sin contenedores. Es exactamente el enfoque de este curso.

## Lo que haremos diferente en OrderFlow

En el curso no vamos a migrar — vamos a construir desde cero. Pero aplicando las lecciones de migraciones reales:

**Database per service desde el primer día.** Cada servicio tiene su base de datos SQL Server desde el inicio. Sin tentaciones de compartir.

**Contratos explícitos.** Los eventos y APIs se definen como contratos compartidos desde el principio. Contract testing desde el Módulo 2.

**Observabilidad desde el primer servicio.** No "ya la pondremos después". OpenTelemetry y Serilog en cada servicio desde M2.3.

**Dos enfoques de API coexistiendo.** Orders con Controllers (servicio complejo). Products, Payments y Notifications con Minimal APIs (servicios ligeros). El alumno ve ambos y elige con criterio.

**Infraestructura simple.** Aspire para desarrollo. Azure App Services para producción. Sin Docker, sin Kubernetes. Foco en el código, no en la infraestructura.

## El monolito distribuido: el antipatrón que debemos evitar

Hemos hablado de esto brevemente en M1.1. Ahora profundizamos, porque es el error más frecuente y el más dañino.

Un monolito distribuido es lo que ocurre cuando partes un monolito en servicios pero mantienes las dependencias del monolito. Tienes toda la complejidad operativa de los microservicios sin ninguno de sus beneficios.

---

## Señales de un monolito distribuido

**Despliegues coordinados.** "Antes de subir Orders, asegúrate de que Products ya tiene la nueva versión." Si no puedes desplegar un servicio sin desplegar otros simultáneamente, no tienes microservicios.

**Base de datos compartida.** Varios servicios leen y escriben en las mismas tablas. Un ALTER TABLE puede romper a tres servicios a la vez.

**Cadenas síncronas largas.** Una petición cruza 5-6 servicios de forma síncrona. Si un eslabón falla o va lento, todo falla. Es el monolito pero con latencia de red añadida.

**Librerías compartidas con lógica de negocio.** Un NuGet package interno que contiene entidades, reglas de negocio y DTOs compartidos. Actualizar el package requiere actualizar todos los servicios que lo usan — acoplamiento disfrazado.

**Cambios en cascada.** Modificar un campo en la API de Orders requiere cambios simultáneos en Products, Payments y Notifications. Si un servicio no se actualiza, el sistema se rompe.

---

## Cómo evitar el monolito distribuido

**Database per service de verdad.** Cada servicio tiene su propia base de datos SQL Server. Punto. No esquemas compartidos, no tablas compartidas, no vistas cruzadas.

**Comunicación asíncrona por defecto.** Si no necesitas una respuesta inmediata, usa mensajería. Orders publica un evento, Notifications lo consume cuando pueda. Sin cadenas síncronas frágiles.

**Contract testing obligatorio.** Antes de desplegar, verificar automáticamente que los contratos entre servicios se cumplen. Si tu cambio rompe un contrato, el pipeline te lo dice.

**Librerías compartidas mínimas.** Solo interfaces de contratos de eventos (DTOs de eventos, no entidades de dominio). Cada servicio define sus propias entidades internamente.

**Tolerancia a versiones.** Tu servicio debe funcionar correctamente aunque el servicio con el que se comunica esté en una versión anterior. Backward compatibility siempre.

---

## Cómo se crea un monolito distribuido sin darse cuenta

Suele ocurrir cuando un equipo migra el monolito servicio por servicio pero mantiene hábitos del monolito:

**Hábito 1: "Ya que tenemos la base de datos ahí, que todos lean de ella."** Parece eficiente. Es acoplamiento letal.

**Hábito 2: "Vamos a hacer un paquete NuGet con todas las entidades compartidas."** Parece ordenado. Es una dependencia que acopla a todos.

**Hábito 3: "Orders llama a Products, que llama a Inventory, que llama a Warehouse, que llama a Shipping."** Parece lógico. Es una cadena que se romperá.

**Hábito 4: "Desplegamos todos los servicios juntos cada jueves."** Parece seguro. Es un deploy coordinado que elimina la independencia.

Cada uno de estos hábitos, individualmente, parece razonable. Juntos, construyen un monolito distribuido. La disciplina de mantener las fronteras es tan importante como la arquitectura en sí.
