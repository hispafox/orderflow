## Controllers: el principio del Controller delgado

Un Controller bien diseñado hace exactamente una cosa: traducir HTTP.

```csharp
// ❌ Controller gordo — tiene lógica de negocio
[HttpPost]
public async Task<IActionResult> Create([FromBody] CreateOrderRequest request)
{
    // ❌ Lógica de negocio en el Controller
    if (request.Items.Count == 0)
        return BadRequest("Items required");

    var order = Order.Create(request.CustomerId, request.Items);
    await _dbContext.Orders.AddAsync(order);
    await _dbContext.SaveChangesAsync();

    // ❌ Publicar el evento aquí
    await _publisher.PublishAsync(new OrderCreated(order.Id));

    return Created($"/api/orders/{order.Id}", order);
}

// ✅ Controller delgado — solo traduce
[HttpPost]
public async Task<ActionResult<OrderDto>> Create([FromBody] CreateOrderRequest request)
{
    var command = new CreateOrderCommand(
        request.CustomerId, request.Items, request.ShippingAddress);

    var result = await _mediator.Send(command);

    return CreatedAtAction(nameof(GetById), new { id = result.Id }, result);
}
```

## La estructura del OrdersController

```csharp
// Orders.API/API/Controllers/OrdersController.cs

using MediatR;
using Microsoft.AspNetCore.Mvc;
using Orders.API.Application.Commands;
using Orders.API.Application.Queries;
using Orders.API.API.DTOs;

namespace Orders.API.API.Controllers;

[ApiController]
[Route("api/[controller]")]
[Produces("application/json")]
public class OrdersController : ControllerBase
{
    private readonly IMediator _mediator;
    private readonly ILogger<OrdersController> _logger;

    public OrdersController(IMediator mediator, ILogger<OrdersController> logger)
    {
        _mediator = mediator;
        _logger = logger;
    }

    // Los endpoints van aquí — los veremos uno a uno
}
```

**`[ApiController]`:** Activa model validation automática, inferencia de `[FromBody]`/`[FromRoute]`, y Problem Details para errores.

**`ControllerBase`:** Base para APIs — sin vistas. No usamos `Controller` (que incluye soporte para Razor).

---

## Por qué el Controller thin pattern es sostenible

Mantener los Controllers delgados tiene un beneficio que se nota con el tiempo:

**Cambias el transporte sin tocar la lógica.** Si mañana necesitas exponer los mismos casos de uso como gRPC, GraphQL o WebSockets — solo añades un nuevo adaptador. Los handlers de MediatR no cambian.

**Los tests de unidad testean la lógica.** Los handlers tienen tests unitarios completos (M2.4). Los Controllers solo necesitan integration tests que verifican el mapeo HTTP.

**Múltiples versiones de API.** Si necesitas `/api/v2/orders` con un formato de request diferente, creas un `OrdersV2Controller` que mapea al mismo `CreateOrderCommand`. La lógica se reutiliza.

**Onboarding claro.** Un developer nuevo pregunta "¿dónde está la lógica de crear un pedido?" La respuesta es siempre la misma: en el Handler. No en el Controller, no en el Repository, no en el Middleware.

Este patrón no es religión — es pragmatismo. Con 18 developers en TechShop, la predictibilidad importa tanto como la funcionalidad.

## GET /api/orders/{id} — detalle de un pedido

```csharp
/// <summary>Obtiene el detalle de un pedido.</summary>
/// <param name="id">ID del pedido (GUID)</param>
[HttpGet("{id:guid}")]
[ProducesResponseType(typeof(OrderDto), StatusCodes.Status200OK)]
[ProducesResponseType(typeof(ProblemDetails), StatusCodes.Status404NotFound)]
public async Task<ActionResult<OrderDto>> GetById(
    Guid id,
    CancellationToken ct = default)
{
    var query = new GetOrderByIdQuery(id);
    var order = await _mediator.Send(query, ct);

    if (order is null)
    {
        _logger.LogWarning("Order {OrderId} not found", id);
        return NotFound();
    }

    return Ok(order);
}
```

`{id:guid}` — route constraint. Si el ID no es un GUID válido, devuelve 400 automáticamente sin llegar al handler.

## POST /api/orders — crear pedido

```csharp
/// <summary>Crea un nuevo pedido.</summary>
[HttpPost]
[ProducesResponseType(typeof(OrderDto), StatusCodes.Status201Created)]
[ProducesResponseType(typeof(ValidationProblemDetails), StatusCodes.Status422UnprocessableEntity)]
public async Task<ActionResult<OrderDto>> Create(
    [FromBody] CreateOrderRequest request,
    CancellationToken ct = default)
{
    var command = request.ToCommand(); // Extension method de mapeo

    var result = await _mediator.Send(command, ct);

    _logger.LogInformation(
        "Order {OrderId} created for customer {CustomerId}",
        result.Id, result.CustomerId);

    // CreatedAtAction genera el header Location: /api/orders/{id}
    return CreatedAtAction(
        nameof(GetById),
        new { id = result.Id },
        result);
}
```

## ActionResult<T>: el tipo de retorno moderno

En .NET 10, `ActionResult<T>` es el tipo de retorno recomendado para endpoints que devuelven datos:

```csharp
// Retornos posibles con ActionResult<OrderDto>
return Ok(orderDto);           // 200 + el OrderDto en el body
return Created(url, orderDto); // 201 + Location header + OrderDto
return NotFound();             // 404 + ProblemDetails automático
return BadRequest(errors);     // 400 + ValidationProblemDetails
return NoContent();            // 204 sin body
return Conflict();             // 409 + ProblemDetails
return UnprocessableEntity(new { ... }); // 422 + body
```

**Por qué `ActionResult<T>` y no `IActionResult`:**
- OpenAPI sabe el tipo exacto que devuelves → spec más precisa
- El compilador puede ayudarte con conversiones implícitas
- `[ProducesResponseType(typeof(T), ...)]` no es necesario en el tipo principal

---

## IActionResult vs ActionResult<T>: cuándo usar cada uno

```csharp
// IActionResult — cuando el tipo de retorno varía mucho
[HttpGet("{id:guid}")]
public async Task<IActionResult> GetById(Guid id)
{
    var order = await _mediator.Send(new GetOrderByIdQuery(id));

    if (order is null) return NotFound();           // No devuelve OrderDto
    if (!order.IsVisibleToCurrentUser) return Forbid(); // No devuelve OrderDto

    return Ok(order); // Aquí sí devuelve OrderDto
}

// ActionResult<T> — cuando el tipo principal es conocido
[HttpGet("{id:guid}")]
public async Task<ActionResult<OrderDto>> GetById(Guid id)
{
    var order = await _mediator.Send(new GetOrderByIdQuery(id));
    return order is null ? NotFound() : Ok(order);
}
// OpenAPI sabe que el 200 devuelve OrderDto → spec más precisa
```

**Regla práctica:** usa `ActionResult<T>` cuando el tipo de respuesta exitosa es único y conocido. Usa `IActionResult` cuando el endpoint puede devolver tipos muy diferentes.

## Problem Details: el estándar de errores HTTP

RFC 7807 define el formato `ProblemDetails` para errores en APIs. ASP.NET Core 10 lo implementa automáticamente:

```json
// Respuesta de error estándar
{
    "type": "https://tools.ietf.org/html/rfc7807",
    "title": "One or more validation errors occurred.",
    "status": 422,
    "traceId": "00-4bf92f3577b34da6a3ce929d0e0e4736-00f067aa0ba902b7-00",
    "errors": {
        "Items": ["Order must have at least one item"],
        "ShippingAddress.Country": ["Country must be a 2-letter ISO code"]
    }
}
```

El `traceId` es el TraceId de OpenTelemetry — correlacionas el error con la traza. Con esto, cuando un cliente reporta un error, le pides el `traceId` y encuentras toda la información en Application Insights.

## Exception Middleware: convertir DomainException en HTTP

Necesitamos que cuando el dominio lance `DomainException`, el cliente reciba un 422 con Problem Details:

```csharp
// Orders.API/API/Middleware/DomainExceptionMiddleware.cs

namespace Orders.API.API.Middleware;

public class DomainExceptionMiddleware
{
    private readonly RequestDelegate _next;
    private readonly ILogger<DomainExceptionMiddleware> _logger;

    public DomainExceptionMiddleware(
        RequestDelegate next,
        ILogger<DomainExceptionMiddleware> logger)
    {
        _next = next;
        _logger = logger;
    }

    public async Task InvokeAsync(HttpContext context)
    {
        try
        {
            await _next(context);
        }
        catch (DomainException ex)
        {
            _logger.LogWarning(ex,
                "Domain rule violation: {Message}", ex.Message);

            context.Response.StatusCode = StatusCodes.Status422UnprocessableEntity;

            var problem = new ProblemDetails
            {
                Status = StatusCodes.Status422UnprocessableEntity,
                Title = "Business rule violation",
                Detail = ex.Message,
                Type = "https://orderflow.api/errors/domain-rule-violation"
            };

            await context.Response.WriteAsJsonAsync(problem);
        }
    }
}
```

## Registrar el Exception Middleware

```csharp
// Program.cs

// El middleware de excepciones SIEMPRE es el primero
app.UseMiddleware<DomainExceptionMiddleware>();
app.UseMiddleware<CorrelationIdMiddleware>();

// Resto del pipeline...
app.UseHttpsRedirection();
app.UseAuthorization();
app.MapControllers();
app.MapDefaultEndpoints();

app.Run();
```

**El orden importa.** `DomainExceptionMiddleware` antes de todo — si falla en cualquier punto del pipeline, lo captura. Sin este middleware, la excepción llegaría al cliente como un 500.

## Middleware de excepción global vs IExceptionHandler

En .NET 10 hay dos formas de manejar excepciones globalmente. La que usamos y la alternativa:

**Nuestro enfoque — middleware custom:**
```csharp
// Captura excepciones de cualquier tipo que configuremos
app.UseMiddleware<DomainExceptionMiddleware>();
```

**Alternativa — IExceptionHandler (.NET 8+):**
```csharp
public class DomainExceptionHandler : IExceptionHandler
{
    public async ValueTask<bool> TryHandleAsync(
        HttpContext ctx, Exception ex, CancellationToken ct)
    {
        if (ex is not DomainException domainEx) return false;

        ctx.Response.StatusCode = 422;
        await ctx.Response.WriteAsJsonAsync(new ProblemDetails
        {
            Status = 422,
            Title = "Business rule violation",
            Detail = domainEx.Message
        }, ct);

        return true; // Handled
    }
}

// Registro
builder.Services.AddExceptionHandler<DomainExceptionHandler>();
app.UseExceptionHandler();
```

`IExceptionHandler` es más composable — puedes registrar múltiples handlers encadenados. Para OrderFlow, el middleware es suficientemente simple. En proyectos más grandes, IExceptionHandler es más flexible.

## La diferencia entre 400, 422 y 500

En Orders.API distinguimos tres tipos de error:

**400 Bad Request — entrada malformada:**
El JSON no se puede parsear, un tipo de dato no coincide. El cliente mandó algo incorrecto a nivel de formato.
```json
{ "status": 400, "title": "Bad Request",
  "detail": "The JSON value could not be converted to System.Guid" }
```

**422 Unprocessable Entity — regla de negocio o validación:**
El formato es correcto pero no cumple las reglas. Pedido sin items, cantidad negativa, estado de transición inválido.
```json
{ "status": 422, "title": "Business rule violation",
  "detail": "Cannot confirm order in 'Cancelled' status" }
```

**500 Internal Server Error — error inesperado:**
El servidor falló por una razón no anticipada. Bug, fallo de infraestructura. El cliente no puede hacer nada — es un problema del servidor.

Con el `DomainExceptionMiddleware`, los errores de dominio van siempre a 422. Los 500 solo para lo genuinamente inesperado.

## HTTP semántico: usar los códigos correctos

Los códigos de estado HTTP no son arbitrarios. Usarlos bien hace que la API sea predecible:

| Código | Cuándo usarlo en Orders.API |
|---|---|
| 200 OK | GET exitoso con datos |
| 201 Created | POST exitoso — nuevo recurso creado |
| 204 No Content | POST confirm/cancel exitoso — sin body |
| 400 Bad Request | JSON malformado, tipo de dato incorrecto |
| 401 Unauthorized | Sin token JWT (en M5.1) |
| 403 Forbidden | Token válido pero sin permisos |
| 404 Not Found | Pedido no existe |
| 409 Conflict | Conflicto de estado (raro en Orders) |
| 422 Unprocessable Entity | Regla de negocio violada |
| 429 Too Many Requests | Rate limit excedido |
| 500 Internal Server Error | Error inesperado del servidor |

La distinción entre 400 y 422 es fundamental: 400 es "no entiendo lo que me mandas", 422 es "entiendo lo que me mandas pero las reglas de negocio no lo permiten".

## Model Binding: cómo ASP.NET Core lee los datos

Model binding es el proceso de extraer datos de la petición HTTP y construir los parámetros del action:

**`[FromRoute]`:** Datos de la URL. `/api/orders/3fa85f64`.
**`[FromQuery]`:** Query string. `/api/orders?status=pending&page=2`.
**`[FromBody]`:** Cuerpo de la petición (JSON). Solo uno por action.
**`[FromHeader]`:** Headers HTTP.
**`[FromForm]`:** Form data (multipart).

Con `[ApiController]`, la inferencia es automática:
- Tipos simples (Guid, int, string) → `[FromRoute]` si coinciden con un parámetro de ruta
- Tipos complejos (records, clases) → `[FromBody]`
- Si no está en la ruta → `[FromQuery]`

## Routing avanzado: atributos y convenciones

ASP.NET Core soporta dos estilos de routing:

**Attribute routing** (lo que usamos con `[ApiController]`):
```csharp
[Route("api/[controller]")]     // api/orders
[HttpGet("{id:guid}")]          // api/orders/{id}
[HttpPost("{id:guid}/confirm")] // api/orders/{id}/confirm
```

**Convenciones de naming para REST:**
- `GET /api/orders` — listar (GetAll, List, Index)
- `GET /api/orders/{id}` — detalle (Get, GetById, Details)
- `POST /api/orders` — crear (Create, Post)
- `PUT /api/orders/{id}` — reemplazar completo
- `PATCH /api/orders/{id}` — actualizar parcialmente
- `DELETE /api/orders/{id}` — eliminar

Para acciones de negocio (que no son CRUD), usamos POST con subruta:
- `POST /api/orders/{id}/confirm` — confirmar
- `POST /api/orders/{id}/cancel` — cancelar
- `POST /api/orders/{id}/ship` — marcar como enviado

## Route Constraints: validación de URLs

Los route constraints validan el formato de los parámetros de ruta antes de llegar al action:

```csharp
// Guid — solo acepta GUIDs válidos
[HttpGet("{id:guid}")]
public async Task<ActionResult<OrderDto>> GetById(Guid id) { }

// Int con rango
[HttpGet("{page:int:min(1)}")]
public async Task<IActionResult> GetPage(int page) { }

// Regex
[HttpGet("{status:regex(^(pending|confirmed|cancelled)$)}")]
public async Task<IActionResult> GetByStatus(string status) { }
```

Si el valor no cumple el constraint, devuelve **404** (no 400) — el endpoint no existe para esa URL.

Para Orders.API usamos principalmente `:guid`. Los demás datos de filtro van por query string.

## Parámetros complejos en query string

Para filtros de listado, podemos agrupar los parámetros en un objeto:

```csharp
public record OrderFilterParams
```

ASP.NET Core mapea automáticamente `?status=pending&page=2&pageSize=10` al objeto `OrderFilterParams`.

## Validación con DataAnnotations

Con `[ApiController]`, la validación de DataAnnotations es automática — si el modelo no es válido, devuelve 422 antes de llegar al action:

```csharp
// DTO con validaciones
public record CreateOrderRequest
{
    [Required(ErrorMessage = "CustomerId is required")]
    public Guid CustomerId { get; init; }

    [Required]
    [MinLength(1, ErrorMessage = "Order must have at least one item")]
    [MaxLength(50, ErrorMessage = "Order cannot have more than 50 items")]
    public IList<CreateOrderItemRequest> Items { get; init; } = [];

    [Required]
    public CreateOrderShippingRequest ShippingAddress { get; init; } = null!;
}

public record CreateOrderItemRequest
{
    [Required]
    public Guid ProductId { get; init; }

    [Required]
    [MaxLength(200)]
```

## Respuestas con headers personalizados

Algunos casos necesitan añadir headers a la respuesta:

```csharp
[HttpGet]
public async Task<ActionResult<PagedResult<OrderSummaryDto>>> GetAll(
    [FromQuery] OrderFilterParams filters,
    CancellationToken ct = default)
{
    var result = await _mediator.Send(new ListOrdersQuery(filters), ct);

    // Headers de paginación — estilo GitHub API
    Response.Headers["X-Total-Count"] = result.TotalCount.ToString();
    Response.Headers["X-Page"] = result.Page.ToString();
    Response.Headers["X-Page-Size"] = result.PageSize.ToString();
    Response.Headers["X-Total-Pages"] = result.TotalPages.ToString();

    // Link header con URLs de navegación (estilo RFC 8288)
    if (result.HasNextPage)
    {
        var nextUrl = Url.Action(nameof(GetAll),
            new { page = filters.Page + 1, pageSize = filters.PageSize });
        Response.Headers["Link"] = $"<{nextUrl}>; rel=\"next\"";
    }

    return Ok(result);
}
```

```csharp
// Orders.API/API/DTOs/PagedResult.cs

public record PagedResult<T>
{
    public IList<T> Items { get; init; } = [];
    public int TotalCount { get; init; }
    public int Page { get; init; }
    public int PageSize { get; init; }
    public int TotalPages => (int)Math.Ceiling((double)TotalCount / PageSize);
    public bool HasNextPage => Page < TotalPages;
    public bool HasPreviousPage => Page > 1;
}
```

## Action Filters: lógica transversal en Controllers

```csharp
    {
        // Antes del action — validaciones, autenticación adicional

        }
    }

    public void OnActionExecuted(ActionExecutedContext context)
    {
        // Después del action — logging, modificar respuesta
        if (context.Exception is not null)
        {
            // El middleware de excepciones lo capturará
        }
    }
}

// Registrar globalmente
builder.Services.AddControllers(options =>
{
```

## Convenciones de naming para la API REST de Orders

Convenciones que seguimos en el OrdersController para que la API sea predecible:

**URLs:** kebab-case para recursos compuestos. `/api/order-lines` si existiera.

**Query params:** camelCase. `?customerId=...&pageSize=20`.

**Body JSON:** camelCase. `{"customerId": "...", "items": [...]}`.

**Headers:** PascalCase con guiones. `X-Correlation-Id`, `X-Total-Count`.

**HTTP Methods para acciones de negocio:** POST con subruta descriptiva.
- `POST /api/orders/{id}/confirm` → no `PATCH /api/orders/{id}` con status en el body
- `POST /api/orders/{id}/cancel` → más semántico y REST-friendly

**Ids:** siempre GUIDs. No auto-incrementales — los GUIDs permiten generarlos en el cliente sin round-trip al servidor.

## Comparativa final: Controllers vs Minimal APIs

Después de implementar Orders.API con Controllers, podemos hacer la comparativa con criterio:

| Aspecto | Controllers | Minimal APIs |
|---|---|---|
| Líneas de código | Más | Menos (~30-40% menos) |
| Organización | Por Controller (clase) | Por Feature (vertical slice) |
| Filtros | Action Filters (class) | Endpoint Filters (delegates) |
| Model binding | Automático con convenciones | Explícito, más control |
| OpenAPI | Automático por convenciones | Necesita configuración explícita |
| Testing | WebApplicationFactory igual | WebApplicationFactory igual |
| Performance | Similar | Marginalmente mejor (menos middleware) |
| Cuando usar | APIs complejas, muchos endpoints | APIs ligeras, microservicios simples |

Para Orders: Controllers. Para Products, Payments, Notifications: Minimal APIs. No es religión — es pragmatismo.

## Troubleshooting: errores comunes con Controllers

**Error: "The action name is ambiguous"**
```csharp
// ❌ Dos actions con la misma ruta y método
[HttpGet] public async Task<IActionResult> GetAll() { }
[HttpGet] public async Task<IActionResult> GetPending() { }

[HttpGet] public async Task<IActionResult> GetAll() { }
```

**Error: "Cannot bind parameter"**
```csharp
// ❌ ASP.NET Core no sabe de dónde leer el Guid
public async Task<IActionResult> GetById(Guid id) { }

// ✅ Explícito en la ruta o en el atributo
[HttpGet("{id:guid}")]
public async Task<IActionResult> GetById(Guid id) { }
```

**Error: "415 Unsupported Media Type"**
El cliente no manda `Content-Type: application/json`. Con `[ApiController]`, el request body solo acepta JSON por defecto.
