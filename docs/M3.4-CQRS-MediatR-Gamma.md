## ¿Qué problema resuelve CQRS?

CQRS — Command Query Responsibility Segregation — separa las operaciones de escritura (Commands) de las de lectura (Queries).

**El problema sin CQRS:**

```csharp
// Un servicio que hace de todo
public class OrderService
{
    // Operación de escritura — con transacción, validación, eventos
    public async Task<Order> CreateAsync(CreateOrderDto dto) { }

    // Operación de lectura — con joins, proyecciones, paginación
    public async Task<PagedResult<OrderSummary>> ListAsync(OrderFilter filter) { }

    // ¿Mismo modelo para las dos? ¿Mismas reglas de validación?
    // ¿El mismo repositorio? La complejidad crece junta
}
```

**Con CQRS:**
- `CreateOrderCommand` → usa el modelo de escritura (Order aggregate, validaciones de negocio)
- `ListOrdersQuery` → usa el modelo de lectura (DTOs planos, queries optimizadas)

Cada uno optimizado para su propósito.

## CQRS: el principio de separación

```
       WRITE SIDE                    READ SIDE
    (Commands)                     (Queries)

CreateOrderCommand          GetOrderByIdQuery
ConfirmOrderCommand         ListOrdersQuery
CancelOrderCommand          GetOrderHistoryQuery

       ↓                             ↓
   Aggregate                   Read Models
   (Domain)               (DTOs optimizados)

       ↓                             ↓
   OrdersDb                     OrdersDb
  (EF Core)                     (Dapper)
```

En nuestra implementación inicial (misma BD), la separación es conceptual y de código — Commands tocan el aggregate, Queries leen directamente de la BD con proyecciones. En M6.3 llevamos esto más lejos con BDs separadas.

## MediatR: el mediador entre la API y la lógica

MediatR implementa el patrón Mediator. En vez de que el Controller llame directamente al servicio, envía un mensaje al mediador. El mediador enruta al handler correcto.

```csharp
// Sin MediatR — Controller conoce el servicio
public class OrdersController
{
    private readonly IOrderService _service; // Dependencia directa

    public async Task<IActionResult> Create(CreateOrderRequest req)
        => Ok(await _service.CreateAsync(req.ToDto()));
}

// Con MediatR — Controller solo conoce al mediador
public class OrdersController
{
    private readonly IMediator _mediator; // Una sola dependencia

    public async Task<IActionResult> Create(CreateOrderRequest req)
        => Ok(await _mediator.Send(req.ToCommand())); // El mediador enruta
}
```

El Controller no sabe qué handler procesa el comando. No le importa. Solo sabe que existe un mediador que lo resuelve.

---

## MediatR: el patrón Mediator explicado

El patrón Mediator reduce las dependencias directas entre componentes. En vez de que A conozca a B y C, todos conocen solo al Mediator:

```
SIN MEDIATOR:
OrdersController → OrderService
OrdersController → NotificationService
OrdersController → AuditService
(El Controller tiene 3 dependencias)

CON MEDIATOR (MediatR):
OrdersController → IMediator (1 dependencia)
IMediator → CreateOrderHandler
IMediator → OrderCreatedNotificationHandler
IMediator → AuditHandler
(El Controller solo conoce al mediador)
```

**La ventaja:** añadir un nuevo handler que reacciona a `CreateOrderCommand` no toca el Controller. No toca los otros handlers. Solo añades una clase nueva que implementa `IRequestHandler` o `INotificationHandler`.

## CancellationToken: el parámetro que no puedes ignorar

Todos los handlers de MediatR reciben un `CancellationToken`. Antes de escribir el primero, entiende qué es y por qué importa.

**¿De dónde viene?**
ASP.NET Core inyecta automáticamente `HttpContext.RequestAborted` cuando el cliente cancela la petición: cierra el navegador, hace timeout, o cancela desde el móvil. MediatR lo propaga a cada handler de la cadena.

```csharp
// ASP.NET Core extrae el CT del HttpContext y lo pasa a MediatR
[HttpPost]
public async Task<IActionResult> Create(
    [FromBody] CreateOrderRequest request,
    CancellationToken ct)  // ← viene de HttpContext.RequestAborted
{
    var result = await _mediator.Send(
        new CreateOrderCommand(...), ct);  // ← se propaga a todos los handlers
    return Created(...);
}
```

**Propagar siempre — el patrón correcto:**
```csharp
public class CreateOrderHandler : IRequestHandler<CreateOrderCommand, OrderDto>
{
    public async Task<OrderDto> Handle(
        CreateOrderCommand command,
        CancellationToken ct)
    {
        // ✅ Correcto — ct llega a todas las operaciones async
        var product = await _productsClient.GetByIdAsync(command.ProductId, ct);
        await _repository.SaveAsync(order, ct);
        await _publisher.PublishAsync(new OrderCreated(...), ct);

        // ❌ Incorrecto — operación continúa aunque el cliente ya se fue
        var product = await _productsClient.GetByIdAsync(command.ProductId);
    }
}
```

**¿Qué pasa cuando se cancela?**
EF Core, HttpClient y MassTransit lanzan `OperationCanceledException`. El middleware global de excepciones debe manejarla devolviendo 499 (client closed request) o simplemente ignorarla:

```csharp
// Exception middleware — distinguir cancelación de error real
app.Use(async (ctx, next) =>
{
    try
    {
        await next();
    }
    catch (OperationCanceledException) when (ctx.RequestAborted.IsCancellationRequested)
    {
        // El cliente canceló — no es un error, no loggear como error
        ctx.Response.StatusCode = 499;
    }
});
```

**CancellationToken.None en los tests — y por qué:**
```csharp
// En tests usamos CancellationToken.None porque:
// 1. No hay HttpContext que lo cancele
// 2. Queremos que el test se complete siempre
var result = await _handler.Handle(command, CancellationToken.None);

// Si necesitas testear la cancelación explícitamente:
var cts = new CancellationTokenSource();
cts.Cancel();  // Cancelar inmediatamente
await Assert.ThrowsAsync<OperationCanceledException>(
    () => _handler.Handle(command, cts.Token));
```

**Timeout combinado con cancelación:**
```csharp
// Combinar timeout de la operación + cancelación del cliente
using var timeoutCts = new CancellationTokenSource(TimeSpan.FromSeconds(30));
using var linkedCts = CancellationTokenSource.CreateLinkedTokenSource(
    ct,             // cancelación del cliente
    timeoutCts.Token); // timeout propio de la operación

var result = await _repository.GetByIdAsync(id, linkedCts.Token);
// Se cancela si: cliente cierra conexión O pasan 30 segundos
```

**La regla:** `CancellationToken` siempre fluye hacia abajo. Nunca lo ignores, nunca uses `CancellationToken.None` en producción.

## Commands: representar intenciones de escritura

Un Command es un record inmutable que representa la intención de cambiar algo:

```csharp
// Orders.API/Application/Commands/CreateOrderCommand.cs

namespace Orders.API.Application.Commands;

/// <summary>
/// Comando para crear un nuevo pedido.
/// Contiene todos los datos necesarios — sin referencias al dominio.
/// </summary>
public record CreateOrderCommand(
    Guid CustomerId,
    IReadOnlyList<CreateOrderItemDto> Items,
    CreateOrderAddressDto ShippingAddress)
    : IRequest<OrderDto>;  // El handler devuelve OrderDto

public record CreateOrderItemDto(
    Guid ProductId,
    string ProductName,
    int Quantity,
    decimal UnitPrice,
    string Currency);

public record CreateOrderAddressDto(
    string Street,
    string City,
    string ZipCode,
    string Country);

// Otros Commands
public record ConfirmOrderCommand(Guid OrderId) : IRequest;      // Sin retorno
public record CancelOrderCommand(Guid OrderId, string Reason) : IRequest; // Sin retorno
```

`IRequest<TResponse>` indica el tipo de retorno. `IRequest` (sin genérico) para commands sin retorno.

## Command Handler: la lógica de escritura

```csharp
// Orders.API/Application/Commands/CreateOrderHandler.cs

namespace Orders.API.Application.Commands;

public class CreateOrderHandler : IRequestHandler<CreateOrderCommand, OrderDto>
{
    private readonly IOrderRepository _repository;
    private readonly IEventPublisher _publisher;
    private readonly ILogger<CreateOrderHandler> _logger;

    public CreateOrderHandler(
        IOrderRepository repository,
        IEventPublisher publisher,
        ILogger<CreateOrderHandler> logger)
    {
        _repository = repository;
        _publisher = publisher;
        _logger = logger;
    }

    public async Task<OrderDto> Handle(
        CreateOrderCommand command,
        CancellationToken ct)
    {
        _logger.LogInformation(
            "Creating order for customer {CustomerId}",
            command.CustomerId);

        // 1. Construir los items del dominio
        var items = command.Items.Select(i =>
            (i.ProductId, i.ProductName, i.Quantity,
             new Money(i.UnitPrice, i.Currency)));

        var address = new Address(
            command.ShippingAddress.Street,
            command.ShippingAddress.City,
            command.ShippingAddress.ZipCode,
            command.ShippingAddress.Country);

        // 2. Crear el aggregate (reglas de negocio en el dominio)
        var order = Order.Create(command.CustomerId, items, address);

        // 3. Persistir
        await _repository.SaveAsync(order, ct);

        // 4. Publicar domain events como integration events
        foreach (var domainEvent in order.DomainEvents)
        {
            await PublishDomainEventAsync(domainEvent, ct);
        }
        order.ClearDomainEvents();

        _logger.LogInformation(
            "Order {OrderId} created. Total: {Total}",
            order.Id, order.Total);

        return order.ToDto();
    }

    private async Task PublishDomainEventAsync(IDomainEvent domainEvent, CancellationToken ct)
    {
        // En M4.2, esto publicará a RabbitMQ vía MassTransit
        // Por ahora, log de que se publicaría
        _logger.LogInformation(
            "Publishing domain event: {EventType}",
            domainEvent.GetType().Name);
    }
}
```

## Separar la lectura de la escritura: el read model

Una ventaja de CQRS que ya empezamos a aprovechar: las Queries pueden usar un modelo diferente al del aggregate.

**El aggregate (write model):** `Order` con todas sus invariantes, domain events, lógica de estado. Pesado pero correcto.

**El read model (query model):** un DTO plano, desnormalizado, optimizado para la query específica.

```csharp
// Query handler — puede usar Dapper directamente en vez de EF Core
public class ListOrdersHandler : IRequestHandler<ListOrdersQuery, PagedResult<OrderSummaryDto>>
{
    private readonly IDbConnection _connection; // Dapper — más rápido para reads

    public async Task<PagedResult<OrderSummaryDto>> Handle(
        ListOrdersQuery query, CancellationToken ct)
    {
        var sql = """
            SELECT o.Id, o.CustomerId, o.Status, o.CreatedAt,
                   SUM(ol.Quantity * ol.UnitPriceAmount) AS Total
            FROM orders.Orders o
            LEFT JOIN orders.OrderLines ol ON ol.OrderId = o.Id
            WHERE (@Status IS NULL OR o.Status = @Status)
            GROUP BY o.Id, o.CustomerId, o.Status, o.CreatedAt
            ORDER BY o.CreatedAt DESC
            OFFSET @Offset ROWS FETCH NEXT @PageSize ROWS ONLY
            """;

        var orders = await _connection.QueryAsync<OrderSummaryDto>(sql,
            new { Status = query.Status,
                  Offset = (query.Page - 1) * query.PageSize,
                  PageSize = query.PageSize });

        return new PagedResult<OrderSummaryDto>
        {
            Items = orders.ToList(),
            TotalCount = await _connection.ExecuteScalarAsync<int>(...),
            Page = query.Page,
            PageSize = query.PageSize
        };
    }
}
```

La query SQL directa es mucho más rápida que cargar todo el aggregate con EF Core y transformarlo. Lo desarrollaremos en profundidad en M6.3.

## El Pipeline de MediatR

La potencia de MediatR no está solo en el enrutamiento — está en el pipeline de Behaviors. Cada mensaje pasa por una cadena antes de llegar al handler:

```
Send(CreateOrderCommand)
    ↓
LoggingBehavior      ← loggea inicio + fin + duración
    ↓
ValidationBehavior   ← valida el command con FluentValidation
    ↓
TransactionBehavior  ← abre transacción de BD
    ↓
CreateOrderHandler   ← lógica de negocio
    ↓
TransactionBehavior  ← commit (o rollback si hay excepción)
    ↓
LoggingBehavior      ← loggea resultado
    ↓
OrderDto (resultado)
```

Los Behaviors son como middleware — cada uno tiene la responsabilidad de llamar al siguiente o no.

---

## LoggingBehavior: medir la duración de cada operación

```csharp
// Orders.API/Application/Behaviors/LoggingBehavior.cs

namespace Orders.API.Application.Behaviors;

public class LoggingBehavior<TRequest, TResponse>
    : IPipelineBehavior<TRequest, TResponse>
    where TRequest : notnull
{
    private readonly ILogger<LoggingBehavior<TRequest, TResponse>> _logger;

    public LoggingBehavior(
        ILogger<LoggingBehavior<TRequest, TResponse>> logger)
        => _logger = logger;

    public async Task<TResponse> Handle(
        TRequest request,
        RequestHandlerDelegate<TResponse> next,
        CancellationToken ct)
    {
        var requestName = typeof(TRequest).Name;

        _logger.LogInformation(
            "Handling {RequestName}: {@Request}",
            requestName, request);

        var stopwatch = Stopwatch.StartNew();

        try
        {
            var response = await next(); // Llamar al siguiente en el pipeline
            stopwatch.Stop();

            _logger.LogInformation(
                "Handled {RequestName} in {ElapsedMs}ms",
                requestName, stopwatch.ElapsedMilliseconds);

            return response;
        }
        catch (Exception ex)
        {
            stopwatch.Stop();
            _logger.LogError(ex,
                "Error handling {RequestName} after {ElapsedMs}ms",
                requestName, stopwatch.ElapsedMilliseconds);
            throw;
        }
    }
}
```

## ValidationBehavior: validar antes de llegar al handler

```csharp
// Orders.API/Application/Behaviors/ValidationBehavior.cs

namespace Orders.API.Application.Behaviors;

public class ValidationBehavior<TRequest, TResponse>
    : IPipelineBehavior<TRequest, TResponse>
    where TRequest : notnull
{
    private readonly IEnumerable<IValidator<TRequest>> _validators;

    public ValidationBehavior(IEnumerable<IValidator<TRequest>> validators)
        => _validators = validators;

    public async Task<TResponse> Handle(
        TRequest request,
        RequestHandlerDelegate<TResponse> next,
        CancellationToken ct)
    {
        // Si no hay validators registrados para este tipo, continuar
        if (!_validators.Any())
            return await next();

        // Ejecutar todos los validators
        var context = new ValidationContext<TRequest>(request);
        var results = await Task.WhenAll(
            _validators.Select(v => v.ValidateAsync(context, ct)));

        var failures = results
            .SelectMany(r => r.Errors)
            .Where(f => f is not null)
            .ToList();

        if (failures.Count != 0)
            throw new ValidationException(failures);

        return await next();
    }
}
```

## Validator para CreateOrderCommand

```csharp
// Orders.API/Application/Commands/CreateOrderCommandValidator.cs

namespace Orders.API.Application.Commands;

public class CreateOrderCommandValidator : AbstractValidator<CreateOrderCommand>
{
    public CreateOrderCommandValidator()
    {
        RuleFor(x => x.CustomerId)
            .NotEmpty().WithMessage("CustomerId is required");

        RuleFor(x => x.Items)
            .NotEmpty().WithMessage("Order must have at least one item")
            .Must(items => items.Count <= 50)
            .WithMessage("Order cannot exceed 50 items");

        RuleForEach(x => x.Items).ChildRules(item =>
        {
            item.RuleFor(i => i.ProductId)
                .NotEmpty().WithMessage("ProductId is required");

            item.RuleFor(i => i.ProductName)
                .NotEmpty().WithMessage("ProductName is required")
                .MaximumLength(200);

            item.RuleFor(i => i.Quantity)
                .GreaterThan(0).WithMessage("Quantity must be positive")
                .LessThanOrEqualTo(1000);

            item.RuleFor(i => i.UnitPrice)
                .GreaterThan(0).WithMessage("Unit price must be positive");

            item.RuleFor(i => i.Currency)
                .NotEmpty().Length(3)
                .WithMessage("Currency must be 3-letter ISO code");
        });

        RuleFor(x => x.ShippingAddress).NotNull()
            .WithMessage("Shipping address is required");

        RuleFor(x => x.ShippingAddress.Country)
            .Length(2).WithMessage("Country must be 2-letter ISO code")
            .When(x => x.ShippingAddress is not null);
    }
}
```

## TransactionBehavior: envolver Commands en transacciones

Las Queries no necesitan transacción. Los Commands sí:

```csharp
// Orders.API/Application/Behaviors/TransactionBehavior.cs

namespace Orders.API.Application.Behaviors;

public class TransactionBehavior<TRequest, TResponse>
    : IPipelineBehavior<TRequest, TResponse>
    where TRequest : ITransactional  // Solo Commands marcados como transaccionales
{
    private readonly OrderDbContext _dbContext;
    private readonly ILogger<TransactionBehavior<TRequest, TResponse>> _logger;

    public TransactionBehavior(
        OrderDbContext dbContext,
        ILogger<TransactionBehavior<TRequest, TResponse>> logger)
    {
        _dbContext = dbContext;
        _logger = logger;
    }

    public async Task<TResponse> Handle(
        TRequest request,
        RequestHandlerDelegate<TResponse> next,
        CancellationToken ct)
    {
        await using var transaction =
            await _dbContext.Database.BeginTransactionAsync(ct);

        try
        {
            var response = await next();
            await transaction.CommitAsync(ct);

            _logger.LogInformation("Transaction committed for {Request}",
                typeof(TRequest).Name);

            return response;
        }
        catch
        {
            await transaction.RollbackAsync(ct);
            _logger.LogWarning("Transaction rolled back for {Request}",
                typeof(TRequest).Name);
            throw;
        }
    }
}
```

## Marcar Commands como transaccionales

```csharp
// Orders.API/Application/Behaviors/ITransactional.cs

/// <summary>
/// Marker interface para Commands que necesitan transacción de BD.
/// Las Queries no implementan esta interfaz.
/// </summary>
public interface ITransactional { }

// Los Commands que modifican estado implementan ITransactional
public record CreateOrderCommand(...) : IRequest<OrderDto>, ITransactional;
public record ConfirmOrderCommand(Guid OrderId) : IRequest, ITransactional;
public record CancelOrderCommand(Guid OrderId, string Reason) : IRequest, ITransactional;

// Las Queries NO implementan ITransactional — no necesitan transacción
public record GetOrderByIdQuery(Guid OrderId) : IRequest<OrderDto?>;
public record ListOrdersQuery(...) : IRequest<PagedResult<OrderSummaryDto>>;
```

El `TransactionBehavior` solo se activa para Commands que implementan `ITransactional`. Las Queries pasan directamente al handler sin overhead de transacción.

## Actualizar el OrdersController con MediatR

Con los Commands y Queries definidos, el Controller se simplifica:

```csharp
// Orders.API/API/Controllers/OrdersController.cs — actualizado

[ApiController]
[Route("api/[controller]")]
public class OrdersController : ControllerBase
{
    private readonly IMediator _mediator;
    private readonly ILogger<OrdersController> _logger;

    public OrdersController(IMediator mediator, ILogger<OrdersController> logger)
    {
        _mediator = mediator;
        _logger = logger;
    }

    [HttpGet]
    public async Task<ActionResult<PagedResult<OrderSummaryDto>>> GetAll(
        [FromQuery] string? status, [FromQuery] int page = 1,
        [FromQuery] int pageSize = 20, CancellationToken ct = default)
        => Ok(await _mediator.Send(new ListOrdersQuery(status, page, pageSize), ct));

    [HttpGet("{id:guid}")]
    public async Task<ActionResult<OrderDto>> GetById(Guid id, CancellationToken ct = default)
    {
        var order = await _mediator.Send(new GetOrderByIdQuery(id), ct);
        return order is null ? NotFound() : Ok(order);
    }

    [HttpPost]
    public async Task<ActionResult<OrderDto>> Create(
        [FromBody] CreateOrderRequest request, CancellationToken ct = default)
    {
        var result = await _mediator.Send(request.ToCommand(), ct);
        return CreatedAtAction(nameof(GetById), new { id = result.Id }, result);
    }

    [HttpPost("{id:guid}/confirm")]
    public async Task<IActionResult> Confirm(Guid id, CancellationToken ct = default)
    {
        await _mediator.Send(new ConfirmOrderCommand(id), ct);
        return NoContent();
    }

    [HttpPost("{id:guid}/cancel")]
    public async Task<IActionResult> Cancel(Guid id,
```

## La ValidationException y cómo convertirla en 422

MediatR lanza `ValidationException` de FluentValidation cuando la validación falla. Necesitamos que el cliente reciba un 422:

```csharp
// Actualizar DomainExceptionMiddleware
catch (ValidationException ex)
{
    context.Response.StatusCode = 422;

    var errors = ex.Errors
        .GroupBy(e => e.PropertyName)
        .ToDictionary(
            g => g.Key,
            g => g.Select(e => e.ErrorMessage).ToArray());

    await context.Response.WriteAsJsonAsync(new ValidationProblemDetails(errors)
    {
        Status = 422,
        Title = "Validation failed",
        Type = "https://orderflow.api/errors/validation"
    });
}
```

Ahora el flujo es:
1. Controller envía `CreateOrderCommand` → MediatR
2. `LoggingBehavior` → inicia el timer
3. `ValidationBehavior` → detecta errores en la request → lanza `ValidationException`
4. `DomainExceptionMiddleware` → captura → devuelve 422 con detalles de cada campo

## Notifications vs Requests: la diferencia

MediatR tiene dos tipos de mensajes:

**IRequest<TResponse>:** Un solo handler procesa el mensaje y devuelve una respuesta. Se usa para Commands y Queries.

```csharp
// Un Command → Un Handler → Una respuesta
var orderDto = await _mediator.Send(new CreateOrderCommand(...));
```

**INotification:** Múltiples handlers pueden procesar el mismo evento. Sin respuesta.

```csharp
// Un Notification → Múltiples Handlers → Sin respuesta directa
await _mediator.Publish(new OrderCreatedEvent(order.Id, order.CustomerId));
// Todos los handlers de OrderCreatedEvent se ejecutan
```

En Orders.API usamos `IRequest` para Commands y Queries (proceso en el mismo request). Para eventos que cruzan servicios, usamos Integration Events con MassTransit (M4.2).

## CQRS en Products.API: ¿lo aplicamos?

Products.API usa Minimal APIs. ¿Añadimos MediatR también?

**Para el curso: no.** Products.API es más simple. Los endpoints llaman directamente al repositorio con una capa mínima. Añadir MediatR no aporta suficiente valor para compensar la complejidad.

**Cuándo sí añadir CQRS a un Minimal API service:**
- Si hay más de 5-6 casos de uso con lógica compleja
- Si necesitas Pipeline Behaviors (validación, transacciones) en múltiples endpoints
- Si el servicio crece y los handlers directos se vuelven difíciles de mantener

**La regla:** CQRS agrega valor cuando hay complejidad que justifica la separación. Products.API no la tiene todavía.

## El contrato de un handler bien diseñado

Un handler de MediatR bien diseñado sigue estas reglas:

**1. Una responsabilidad:** `CreateOrderHandler` crea pedidos. No envía emails, no genera facturas, no actualiza el inventario directamente.

**2. Orquesta, no implementa:** el handler llama al aggregate (dominio), al repositorio (infraestructura), y al publisher (mensajería). No contiene lógica de negocio propia.

**3. No conoce HTTP:** no usa `HttpContext`, no conoce status codes, no sabe nada del protocolo.

**4. Siempre async:** todas las operaciones I/O son `await`.

**5. Propaga el CancellationToken:** todas las llamadas async reciben el `ct`.

**6. Loggea lo relevante:** inicio, fin, IDs importantes. No loggea datos sensibles.

Si un handler empieza a crecer más de ~50 líneas, probablemente está haciendo demasiado.

## Compatibilidad hacia atrás: migrar de servicio a CQRS

Si tienes un proyecto existente con `IOrderService`, la migración a CQRS es gradual:

**Paso 1:** Añadir MediatR. El IOrderService sigue existiendo.

**Paso 2:** Para cada nuevo caso de uso, crear Command/Query en vez de método del servicio.

**Paso 3:** Gradualmente, mover métodos del servicio a Handlers cuando los toques por otras razones.

**Paso 4:** Cuando `IOrderService` esté vacío, eliminarlo.

No necesitas migrar todo de golpe. Los nuevos endpoints usan CQRS, los viejos siguen con el servicio. Coexisten sin problema.

## Ventajas del pipeline de MediatR que se notan en producción

**Logging centralizado.** Cada Command y Query loggea automáticamente su duración. En Application Insights, puedes ver cuánto tarda cada operación sin tocar el handler.

**Métricas automáticas.** Con una pequeña modificación al LoggingBehavior, puedes registrar métricas de OpenTelemetry para cada Command/Query — sin código en los handlers.

**Validación consistente.** Todos los Commands pasan por el mismo ValidationBehavior. Añadir un validator es añadir una clase — nada más cambia.

**Transacciones automáticas.** Los Commands marcados como `ITransactional` siempre tienen transacción. Nunca olvidas `BeginTransaction` en un nuevo handler.

**Testabilidad.** Cada handler es una clase con una responsabilidad. El unit test es trivial.

## Streams con MediatR: IStreamRequest

Para conjuntos de datos grandes, MediatR soporta streaming:

```csharp
// Query que devuelve datos en stream
public record StreamAllOrdersQuery(DateTime From) : IStreamRequest<OrderSummaryDto>;

// Handler que devuelve IAsyncEnumerable
public class StreamAllOrdersHandler : IStreamRequestHandler<StreamAllOrdersQuery, OrderSummaryDto>
{
    private readonly IOrderRepository _repository;

    public async IAsyncEnumerable<OrderSummaryDto> Handle(
        StreamAllOrdersQuery request,
        [EnumeratorCancellation] CancellationToken ct)
    {
        await foreach (var order in _repository.StreamAllAsync(request.From, ct))
        {
            yield return order.ToSummaryDto();
        }
    }
}

// Uso en el Controller
[HttpGet("export")]
public async IAsyncEnumerable<OrderSummaryDto> Export(
    [FromQuery] DateTime from,
    [EnumeratorCancellation] CancellationToken ct)
{
    await foreach (var order in _mediator.CreateStream(
        new StreamAllOrdersQuery(from), ct))
    {
        yield return order;
    }
}
```

Para exports y reportes grandes, el streaming evita cargar todo en memoria.

## Idempotencia: protegerse contra duplicados

Si un cliente envía el mismo Command dos veces (retry de red), podemos crear dos pedidos duplicados.

La solución: idempotency keys.

```csharp
// Command con idempotency key
public record CreateOrderCommand(
    Guid IdempotencyKey,  // ← NUEVO: clave de idempotencia
    Guid CustomerId,
    IReadOnlyList<CreateOrderItemDto> Items,
    CreateOrderAddressDto ShippingAddress)
    : IRequest<OrderDto>, ITransactional;

// En el handler, verificar antes de crear
public async Task<OrderDto> Handle(CreateOrderCommand command, CancellationToken ct)
{
    // Verificar si ya existe un pedido con esta clave
    var existing = await _repository
        .GetByIdempotencyKeyAsync(command.IdempotencyKey, ct);

    if (existing is not null)
    {
        _logger.LogInformation(
            "Duplicate request detected. Returning existing order {OrderId}",
            existing.Id);
        return existing.ToDto();
    }

    // Crear nuevo pedido...
}
```

El cliente genera el `IdempotencyKey` (un GUID). Si la red falla y reintenta, la segunda petición devuelve el mismo pedido sin crearlo de nuevo.

## Performance del pipeline de MediatR

¿Cuánto overhead añade MediatR al pipeline?

En benchmarks, MediatR añade ~1-5 microsegundos de overhead por mensaje. Para un request HTTP típico de 50-200ms, es ruido.

**Cuándo sí importa:**
- Endpoints que necesitan procesar 100K+ req/s
- Flujos con docenas de Behaviors encadenados
- Hot paths donde cada microsegundo cuenta

**Para OrderFlow:** irrelevante. El bottleneck será siempre la base de datos (ms) o la red (ms), no MediatR (µs).

Si en el futuro se convirtiera en un problema, la solución es eliminar los Behaviors del hot path o usar Minimal APIs directas para esos endpoints críticos.

## Métricas de negocio con el LoggingBehavior

El LoggingBehavior puede hacer más que loggear — puede registrar métricas:

```csharp
public class LoggingBehavior<TRequest, TResponse>
    : IPipelineBehavior<TRequest, TResponse>
{
    private readonly ILogger<LoggingBehavior<TRequest, TResponse>> _logger;
    private readonly OrdersMetrics _metrics;

    public async Task<TResponse> Handle(
        TRequest request, RequestHandlerDelegate<TResponse> next, CancellationToken ct)
    {
        var requestName = typeof(TRequest).Name;
        var stopwatch = Stopwatch.StartNew();

        try
        {
            var response = await next();
            stopwatch.Stop();

            // Registrar métrica de duración
            _metrics.RecordCommandDuration(
                requestName, stopwatch.ElapsedMilliseconds);

            return response;
        }
        catch (Exception ex)
        {
            stopwatch.Stop();

            // Registrar fallo
            _metrics.RecordCommandFailure(requestName, ex.GetType().Name);
            throw;
        }
    }
}
```

Con esto, Application Insights muestra el histograma de duración de cada Command. Si `CreateOrderCommand` empieza a tardar más de lo normal, la alerta salta antes de que el usuario se queje.

## El rol de CQRS en la arquitectura de OrderFlow

CQRS no es solo un patrón de código — define cómo escala el sistema:

**El write side (Commands) puede escalar independientemente del read side (Queries):**

Si Orders.API tiene mucho tráfico de lectura (listados de pedidos), escalas las instancias que procesan Queries. Si hay mucho tráfico de escritura (creación de pedidos en Black Friday), escalas las instancias que procesan Commands.

**En Azure App Services:** la misma instancia procesa Commands y Queries (nuestra implementación actual). En M6.3, con read model separado, podrías tener instancias dedicadas a lectura (escaladas según tráfico de lectura).

## Preguntas frecuentes sobre CQRS y MediatR

**¿Necesito MediatR para implementar CQRS?**
No. CQRS es un patrón conceptual. MediatR es una librería que facilita su implementación. Podrías tener Commands y Queries sin MediatR — pero el pipeline de Behaviors es difícil de replicar sin él.

**¿Puedo usar el mismo handler para Command y Query?**
Técnicamente sí. Pero va contra el principio de CQRS. Mantén la separación — si el handler empieza a mezclar lectura y escritura, es señal de que necesita dividirse.

**¿Los Behaviors se aplican a Notifications también?**
Sí, pero con `INotificationHandler` en vez de `IRequestHandler`. Puedes crear behaviors específicos para notificaciones.

**¿MediatR tiene versión compatible con .NET 10?**
Sí. MediatR 12.x es compatible con .NET 10. Para registrar servicios: `builder.Services.AddMediatR(cfg => cfg.RegisterServicesFromAssembly(...))`.

**¿Es un anti-pattern usar MediatR para todo?**
Puede serlo si se usa en exceso. Para microservicios pequeños como Products.API, añadir MediatR solo para tener "CQRS" introduce complejidad sin beneficio. La regla: usa MediatR donde aporte valor real (complejidad de lógica, múltiples behaviors).
