## ¿Por qué Minimal APIs para Products, Payments y Notifications?

La decisión de M3.2: Controllers para Orders, Minimal APIs para el resto. Ahora el motivo con datos:

**Orders.API tiene:**
- 5 endpoints con lógica compleja
- CQRS con MediatR (muchos handlers)
- Saga orchestrator
- Action Filters cross-cutting
- Múltiples tipos de respuesta por endpoint

**Products.API tiene:**
- 5-6 endpoints directos
- Cada endpoint es independiente y sencillo
- Filtros específicos de pocos endpoints
- Sin saga, sin orquestación

**La regla:** cuando los endpoints son pocos, directos y sin mucha lógica transversal — Minimal APIs es menos ceremony, más código de negocio, menos código de framework.

## Minimal APIs vs Controllers: la diferencia central

**Controllers** organizan endpoints en clases. La clase es el punto de cohesión:

```csharp
public class OrdersController : ControllerBase
{
    [HttpGet] public Task<IActionResult> GetAll() { }
    [HttpPost] public Task<IActionResult> Create() { }
    // Todos los endpoints de Orders en la misma clase
}
```

**Minimal APIs** registran endpoints directamente en el pipeline. Sin clases, sin herencia:

```csharp
app.MapGet("/api/products", GetAll);
app.MapPost("/api/products", Create);
// Los endpoints son funciones, no métodos de clase
```

La organización no es por clase — es por fichero o por feature. El vertical slice pattern encaja perfectamente.

## Route Groups: el equivalente a los Controllers

Los Route Groups agrupan endpoints con prefijo de ruta y configuración común — sin necesidad de una clase Controller:

```csharp
// Products.API/Features/ProductEndpoints.cs

public static class ProductEndpoints
{
    public static IEndpointRouteBuilder MapProductEndpoints(
        this IEndpointRouteBuilder app)
    {
        var group = app.MapGroup("/api/products")
            .WithTags("Products")           // Tag en OpenAPI
            .WithOpenApi()                  // Habilitar OpenAPI para el grupo
            .RequireAuthorization();        // Autenticación (M5.1)

        // Endpoints del grupo
        group.MapGet("/", ListProductsEndpoint.Handle)
             .WithName("ListProducts");

        group.MapGet("/{id:guid}", GetProductEndpoint.Handle)
             .WithName("GetProduct");

        group.MapPost("/", CreateProductEndpoint.Handle)
             .WithName("CreateProduct");

        group.MapPut("/{id:guid}/stock", UpdateStockEndpoint.Handle)
             .WithName("UpdateStock");

        group.MapPost("/{id:guid}/reserve", ReserveStockEndpoint.Handle)
             .WithName("ReserveStock");

        return app;
    }
}
```

## Endpoint: GET /api/products/{id} — detalle de producto

```csharp
// Products.API/Features/GetProduct/GetProductEndpoint.cs

namespace Products.API.Features.GetProduct;

public static class GetProductEndpoint
{
    public static async Task<Results<Ok<ProductResponse>, NotFound>> Handle(
        Guid id,
        IProductRepository repository,
        CancellationToken ct = default)
    {
        var product = await repository.GetByIdAsync(id, ct);

        if (product is null)
            return TypedResults.NotFound();

        return TypedResults.Ok(product.ToResponse());
    }
}
```

`Results<Ok<ProductResponse>, NotFound>` — unión de tipos que dice exactamente qué puede devolver. OpenAPI genera la spec completa automáticamente con los dos posibles tipos de respuesta.

## TypedResults: el retorno fuertemente tipado

La diferencia entre `IResult` y `TypedResults`:

```csharp
// IResult — sin información de tipo para OpenAPI
static IResult GetProduct(Guid id)
{
    var product = // ...
    return product is null ? Results.NotFound() : Results.Ok(product);
    // OpenAPI no sabe qué tipo devuelve el 200
}

// TypedResults — OpenAPI tiene toda la información
static Results<Ok<ProductResponse>, NotFound> GetProduct(Guid id)
{
    var product = // ...
    return product is null
        ? TypedResults.NotFound()
        : TypedResults.Ok(product.ToResponse());
    // OpenAPI sabe: 200 → ProductResponse, 404 → sin body
}
```

Con `TypedResults`, OpenAPI genera automáticamente los schemas de respuesta correctos para cada código de estado. Sin `[ProducesResponseType]`, sin atributos extra.

## Endpoint: POST /api/products — crear producto

```csharp
// Products.API/Features/CreateProduct/CreateProductEndpoint.cs

namespace Products.API.Features.CreateProduct;

public static class CreateProductEndpoint
{
    public static async Task<Results<Created<ProductResponse>, ValidationProblem>> Handle(
        CreateProductRequest request,
        IValidator<CreateProductRequest> validator,
        IProductRepository repository,
        LinkGenerator links,
        CancellationToken ct = default)
    {
        // Validación con FluentValidation
        var validation = await validator.ValidateAsync(request, ct);
        if (!validation.IsValid)
            return TypedResults.ValidationProblem(validation.ToDictionary());

        // Crear el producto
        var product = Product.Create(
            request.Name, request.Description,
            request.Price, request.Currency,
            request.InitialStock, request.CategoryId);

        await repository.SaveAsync(product, ct);

        var location = links.GetPathByName("GetProduct",
            new { id = product.Id });

        return TypedResults.Created(location!, product.ToResponse());
    }
}
```

## FluentValidation: validaciones ricas

FluentValidation en Minimal APIs es tan potente como en Controllers. Lo registramos desde el DI:

```csharp
// Products.API/Features/CreateProduct/CreateProductValidator.cs

namespace Products.API.Features.CreateProduct;

public class CreateProductValidator : AbstractValidator<CreateProductRequest>
{
    public CreateProductValidator()
    {
        RuleFor(x => x.Name)
            .NotEmpty().WithMessage("Product name is required")
            .MaximumLength(200).WithMessage("Name cannot exceed 200 characters")
            .MinimumLength(3).WithMessage("Name must be at least 3 characters");

        RuleFor(x => x.Price)
            .GreaterThan(0).WithMessage("Price must be positive")
            .LessThanOrEqualTo(99999.99m).WithMessage("Price exceeds maximum");

        RuleFor(x => x.Currency)
            .NotEmpty()
            .Length(3).WithMessage("Currency must be a 3-letter ISO code")
            .Must(c => new[] { "EUR", "USD", "GBP" }.Contains(c))
            .WithMessage("Currency must be EUR, USD, or GBP");

        RuleFor(x => x.InitialStock)
            .GreaterThanOrEqualTo(0).WithMessage("Stock cannot be negative")
            .LessThanOrEqualTo(100000);

        RuleFor(x => x.CategoryId)
            .NotEmpty().WithMessage("Category is required");
    }
}
```

## Endpoint Filters: la alternativa a Action Filters

Los Endpoint Filters en Minimal APIs son el equivalente a los Action Filters en Controllers. Se ejecutan antes y después del handler:

```csharp
// Products.API/Filters/ValidationFilter.cs

public class ValidationFilter<TRequest> : IEndpointFilter
{
    private readonly IValidator<TRequest> _validator;

    public ValidationFilter(IValidator<TRequest> validator)
        => _validator = validator;

    public async ValueTask<object?> InvokeAsync(
        EndpointFilterInvocationContext context,
        EndpointFilterDelegate next)
    {
        // Buscar el parámetro del tipo TRequest
        var request = context.Arguments
            .OfType<TRequest>()
            .FirstOrDefault();

        if (request is not null)
        {
            var result = await _validator.ValidateAsync(request);
            if (!result.IsValid)
                return TypedResults.ValidationProblem(result.ToDictionary());
        }

        // Si la validación pasa, continuar con el handler
        return await next(context);
    }
}
```

---

## Aplicar el Endpoint Filter

```csharp
// En la definición del endpoint
group.MapPost("/", CreateProductEndpoint.Handle)
     .WithName("CreateProduct")
     .AddEndpointFilter<ValidationFilter<CreateProductRequest>>();

// O directamente en el handler con una lambda
group.MapPost("/", CreateProductEndpoint.Handle)
     .WithName("CreateProduct")
     .AddEndpointFilter(async (ctx, next) =>
     {
         var validator = ctx.HttpContext.RequestServices
             .GetRequiredService<IValidator<CreateProductRequest>>();
         var request = ctx.GetArgument<CreateProductRequest>(0);
         var result = await validator.ValidateAsync(request);
         if (!result.IsValid)
             return TypedResults.ValidationProblem(result.ToDictionary());
         return await next(ctx);
     });
```

Con el Filter registrado, el endpoint automáticamente valida el request antes de llegar al handler. Sin repetir código de validación en cada endpoint.

## Endpoint: PUT /api/products/{id}/stock — actualizar stock

```csharp
// Products.API/Features/UpdateStock/UpdateStockEndpoint.cs

namespace Products.API.Features.UpdateStock;

public static class UpdateStockEndpoint
{
    public static async Task<Results<NoContent, NotFound, ValidationProblem>> Handle(
        Guid id,
        UpdateStockRequest request,
        IValidator<UpdateStockRequest> validator,
        IProductRepository repository,
        ILogger<UpdateStockEndpoint> logger,
        CancellationToken ct = default)
    {
        var validation = await validator.ValidateAsync(request, ct);
        if (!validation.IsValid)
            return TypedResults.ValidationProblem(validation.ToDictionary());

        var product = await repository.GetByIdAsync(id, ct);
        if (product is null)
            return TypedResults.NotFound();

        product.UpdateStock(request.NewStock);
        await repository.SaveAsync(product, ct);

        logger.LogInformation(
            "Stock updated for product {ProductId}. New stock: {Stock}",
            id, request.NewStock);

        return TypedResults.NoContent();
    }
}
```

## Endpoint: POST /api/products/{id}/reserve — reservar stock

Este endpoint es especialmente importante — lo llama la Saga de Orders cuando se crea un pedido:

```csharp
// Products.API/Features/ReserveStock/ReserveStockEndpoint.cs

namespace Products.API.Features.ReserveStock;

public record ReserveStockRequest(Guid OrderId, int Quantity);

public static class ReserveStockEndpoint
{
    public static async Task<Results<Ok<ReserveStockResponse>, NotFound,
        UnprocessableEntity<ProblemDetails>>> Handle(
        Guid id,
        ReserveStockRequest request,
        IProductRepository repository,
        ILogger<ReserveStockEndpoint> logger,
        CancellationToken ct = default)
    {
        var product = await repository.GetByIdAsync(id, ct);
        if (product is null)
            return TypedResults.NotFound();

        if (!product.HasSufficientStock(request.Quantity))
            return TypedResults.UnprocessableEntity(new ProblemDetails
            {
                Title = "Insufficient stock",
                Detail = $"Product has {product.Stock} units, requested {request.Quantity}",
                Status = 422
            });

        product.ReserveStock(request.Quantity, request.OrderId);
        await repository.SaveAsync(product, ct);

        logger.LogInformation(
            "Stock reserved: {Quantity} units of {ProductId} for order {OrderId}",
            request.Quantity, id, request.OrderId);

        return TypedResults.Ok(new ReserveStockResponse(
            ProductId: id, ReservedQuantity: request.Quantity,
            RemainingStock: product.Stock));
    }
}
```

## El Domain de Products: simple y enfocado

A diferencia de Orders, el dominio de Products es más simple — no tiene saga, no tiene estado complejo:

```csharp
// Products.API/Domain/Product.cs

namespace Products.API.Domain;

public class Product
{
    public Guid Id { get; private set; }
    public string Name { get; private set; }
    public string Description { get; private set; }
    public decimal Price { get; private set; }
    public string Currency { get; private set; }
    public int Stock { get; private set; }
    public bool IsActive { get; private set; }
    public Guid CategoryId { get; private set; }
    public DateTime CreatedAt { get; private set; }
    public DateTime UpdatedAt { get; private set; }

    private Product() { }

    public static Product Create(string name, string description,
        decimal price, string currency, int initialStock, Guid categoryId)
    {
        if (string.IsNullOrWhiteSpace(name))
            throw new ArgumentException("Name is required");
        if (price <= 0)
            throw new ArgumentException("Price must be positive");
        if (initialStock < 0)
            throw new ArgumentException("Stock cannot be negative");

        return new Product { Id = Guid.NewGuid(), Name = name,
            Description = description, Price = price, Currency = currency,
            Stock = initialStock, CategoryId = categoryId,
            IsActive = true, CreatedAt = DateTime.UtcNow,
            UpdatedAt = DateTime.UtcNow };
    }

    public bool HasSufficientStock(int quantity) => Stock >= quantity && IsActive;

    public void ReserveStock(int quantity, Guid orderId)
    {
        if (!HasSufficientStock(quantity))
            throw new InvalidOperationException(
                $"Insufficient stock: {Stock} available, {quantity} requested");
        Stock -= quantity;
        UpdatedAt = DateTime.UtcNow;
    }

    public void UpdateStock(int newStock)
    {
        if (newStock < 0)
            throw new ArgumentException("Stock cannot be negative");
        Stock = newStock;
        UpdatedAt = DateTime.UtcNow;
    }
}
```

```csharp
// Products.API/Infrastructure/ProductDbContext.cs

namespace Products.API.Infrastructure;

public class ProductDbContext : DbContext
{
    public ProductDbContext(DbContextOptions<ProductDbContext> options)
        : base(options) { }

    public DbSet<Product> Products { get; set; }
    public DbSet<Category> Categories { get; set; }

    protected override void OnModelCreating(ModelBuilder modelBuilder)
    {
        modelBuilder.HasDefaultSchema("products");

        modelBuilder.Entity<Product>(entity =>
        {
            entity.ToTable("Products");
            entity.HasKey(p => p.Id);
            entity.Property(p => p.Name).HasMaxLength(200).IsRequired();
            entity.Property(p => p.Price).HasPrecision(18, 2);
            entity.Property(p => p.Currency).HasMaxLength(3).IsRequired();
            entity.HasIndex(p => p.CategoryId);
            entity.HasIndex(p => p.Name);
        });
    }
}
```

```csharp
// Orders.API/Infrastructure/Http/ProductsClient.cs

public class ProductsClient
{
    private readonly HttpClient _httpClient;

    public ProductsClient(HttpClient httpClient)
        => _httpClient = httpClient;

    public async Task<ProductPriceDto?> GetProductPriceAsync(
        Guid productId, CancellationToken ct = default)
    {
        var response = await _httpClient.GetAsync(
            $"/api/products/{productId}", ct);

        if (response.StatusCode == HttpStatusCode.NotFound)
            return null;

        response.EnsureSuccessStatusCode();

        return await response.Content
            .ReadFromJsonAsync<ProductPriceDto>(ct);
    }
}

// Registrar en Orders.API Program.cs

    client.BaseAddress = new Uri("http://products-api")); // Aspire resuelve la URL
```

## Minimal APIs: inyección de dependencias en el handler

Los parámetros del handler pueden ser de tres tipos:

```csharp
// 1. Parámetros de ruta/query/body — ASP.NET Core los extrae del request
static async Task<Ok<ProductResponse>> Handle(
    Guid id,                    // ← route param {id:guid}
    [FromQuery] string? color,  // ← query string ?color=blue
    UpdateRequest request,      // ← body (inferido por tipo complejo)

// 2. Servicios del DI container — se inyectan automáticamente
    IProductRepository repo,        // ← del DI container
    ILogger<UpdateStockEndpoint> logger, // ← del DI container
    LinkGenerator links,            // ← del DI container

// 3. Contexto HTTP — acceso directo si se necesita
    HttpContext httpContext,        // ← el HttpContext completo
    CancellationToken ct)          // ← cancellation token
```

ASP.NET Core distingue entre parámetros de request y servicios del DI basándose en si el tipo está registrado en el container o no.

## Minimal APIs: handlers como clases vs funciones estáticas

Hay tres formas de organizar los handlers:

**Función estática (nuestro enfoque — Products.API):**
```csharp
public static class GetProductEndpoint
{
    public static async Task<Results<Ok<ProductResponse>, NotFound>> Handle(
        Guid id, IProductRepository repo, CancellationToken ct)
    {
        var product = await repo.GetByIdAsync(id, ct);
        return product is null ? TypedResults.NotFound() : TypedResults.Ok(product.ToResponse());
    }
}
// Registro: group.MapGet("/{id:guid}", GetProductEndpoint.Handle);
```

**Lambda inline (para endpoints muy simples):**
```csharp
group.MapGet("/{id:guid}", async (Guid id, IProductRepository repo)
    => await repo.GetByIdAsync(id) is Product p
        ? TypedResults.Ok(p.ToResponse())
        : TypedResults.NotFound());
```

**Método de instancia (si necesitas estado):**
```csharp
public class ProductsEndpointHandler(IProductRepository repo)
{
    public async Task<Ok<ProductResponse>> GetById(Guid id)
        => TypedResults.Ok((await repo.GetByIdAsync(id))!.ToResponse());
}
// Registro: group.MapGet("/{id:guid}", new ProductsEndpointHandler(repo).GetById);
```

Nuestro enfoque con clases estáticas es el más testeable y el más explícito.

## Error Handling en Minimal APIs: el enfoque de Products.API

Products.API tiene errores de dominio simples (no tiene DomainException tan rica). Su middleware de errores:

```csharp
// Products.API/Middleware/ProductsExceptionMiddleware.cs

public class ProductsExceptionMiddleware
{
    private readonly RequestDelegate _next;
    private readonly ILogger<ProductsExceptionMiddleware> _logger;

    public async Task InvokeAsync(HttpContext context)
    {
        try
        {
            await _next(context);
        }
        catch (ArgumentException ex)
        {
            // Errores de dominio básicos
            context.Response.StatusCode = 422;
            await context.Response.WriteAsJsonAsync(new ProblemDetails
            {
                Status = 422,
                Title = "Invalid operation",
                Detail = ex.Message
            });
        }
        catch (InvalidOperationException ex) when (ex.Message.Contains("Insufficient stock"))
        {
            // Stock insuficiente — error de negocio específico
            context.Response.StatusCode = 422;
            await context.Response.WriteAsJsonAsync(new ProblemDetails
            {
                Status = 422,
                Title = "Insufficient stock",
                Detail = ex.Message
            });
        }
    }
}
```

## Patrones adicionales: Response wrapping

En algunos proyectos, todas las respuestas siguen un formato estándar:

```csharp
// Patrón de respuesta envuelta — opcional
public record ApiResponse<T>
{
    public T? Data { get; init; }
    public bool Success { get; init; }
    public string? ErrorMessage { get; init; }
    public DateTime Timestamp { get; init; } = DateTime.UtcNow;

    public static ApiResponse<T> Ok(T data) => new() { Data = data, Success = true };
    public static ApiResponse<T> Error(string message) =>
        new() { Success = false, ErrorMessage = message };
}

// Uso en endpoint
group.MapGet("/{id:guid}", async (Guid id, IProductRepository repo) =>
{
    var product = await repo.GetByIdAsync(id);
    return product is null
        ? Results.NotFound(ApiResponse<ProductResponse>.Error("Product not found"))
        : Results.Ok(ApiResponse<ProductResponse>.Ok(product.ToResponse()));
});
```

**Para OrderFlow no usamos este patrón.** Problem Details es el estándar de la industria y es lo que los clientes esperan en una API REST moderna. El response wrapping custom añade complejidad sin beneficio real.

```csharp
    .Build();

```

## Comparing performance: Minimal APIs vs Controllers

Los benchmarks de Microsoft muestran diferencias de rendimiento:

**BenchmarkDotNet en .NET 10 (peticiones/segundo):**

| Escenario | Controllers | Minimal APIs | Diferencia |
|---|---|---|---|
| Hello World | 6.8M req/s | 7.2M req/s | +6% |
| JSON response | 4.1M req/s | 4.4M req/s | +7% |
| With DB query | 280K req/s | 295K req/s | +5% |
| With validation | 520K req/s | 550K req/s | +6% |

La diferencia (~5-7%) viene de:
- Menos overhead de reflexión (Controllers lo usa para descubrir actions)
- Startup más rápido (no hay scanning de Controllers)
- Pipeline más directo

**Para contexto:** 5% más de throughput en un servicio que procesa 10.000 req/s es 500 req/s más. Significativo a escala, irrelevante en baja carga.

## Cuándo elegir Minimal APIs vs Controllers: la guía definitiva

Después de haber construido con ambos, la guía definitiva:

**Elige Minimal APIs cuando:**
- El servicio tiene pocos endpoints (< 10)
- Los endpoints son simples y directos
- Poco estado compartido entre endpoints
- Prefieres organización por feature (vertical slice)
- Quieres máxima visibilidad del código de framework vs código de negocio

**Elige Controllers cuando:**
- Muchos endpoints con lógica transversal compartida
- Necesitas Action Filters sofisticados en múltiples actions
- El equipo tiene mayor experiencia con MVC
- El servicio tiene convenciones API complejas

**En OrderFlow:**
- Orders → Controllers (complejidad: CQRS, saga, muchos endpoints)
- Products, Payments, Notifications → Minimal APIs (ligeros, directos)
