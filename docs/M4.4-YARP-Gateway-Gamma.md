## Dónde estamos

El Módulo 4 tiene cuatro servicios comunicándose: Orders, Products, Payments y Notifications. El cliente necesita saber las URLs de cada uno:

- `https://localhost:7153` → Orders.API
- `https://localhost:7200` → Products.API
- `https://localhost:7300` → Notifications.API
- `https://localhost:7400` → Payments.API

En producción: cuatro dominios diferentes, cuatro configuraciones de seguridad, cuatro lugares donde añadir rate limiting.

La solución: un **API Gateway** que actúa como punto de entrada único. El cliente habla con el gateway. El gateway enruta al servicio correcto.

Al terminar M4.4, todo el tráfico de OrderFlow pasa por YARP.

---

## ¿Qué hace un API Gateway?

Un API Gateway es el portero de tu sistema de microservicios. En un solo punto gestiona:

**Routing:** `GET /api/orders` → Orders.API. `GET /api/products` → Products.API.

**Autenticación centralizada:** Valida el JWT una vez, en el gateway. Los servicios internos confían en el gateway.

**Rate Limiting:** Limita peticiones por cliente, por endpoint, por segundo.

**Load Balancing:** Distribuye el tráfico entre múltiples instancias del mismo servicio.

**SSL Termination:** El cliente habla HTTPS con el gateway. Los servicios internos pueden usar HTTP.

**Request/Response Transforms:** Modificar headers, paths, bodies antes de reenviar.

**Logging y Observabilidad:** Un punto central para loggear todas las peticiones.

## Por qué YARP

**YARP** (Yet Another Reverse Proxy) es el proxy reverso desarrollado por Microsoft, escrito en .NET y diseñado para integrarse nativamente con ASP.NET Core.

**Por qué YARP y no otras alternativas:**

| | YARP | Ocelot | Azure API Management |
|---|---|---|---|
| Lenguaje | .NET nativo | .NET | Cloud managed |
| Configuración | JSON + C# | JSON | Portal / Terraform |
| Performance | Excelente | Buena | Excelente |
| Coste | Gratuito | Gratuito | Desde 50€/mes |
| Personalización | Total (código C#) | Limitada | Limitada |
| Integración Aspire | Nativa | No oficial | No |

Para OrderFlow: YARP es gratuito, nativo .NET, con IntelliSense y soporte completo de Aspire.

## La configuración de YARP: Routes y Clusters

YARP se configura con dos conceptos:

**Route:** Una regla que dice "si la URL coincide con este patrón, usa este cluster".

**Cluster:** Un conjunto de destinos (instancias de servicio) que reciben el tráfico.

```json
// appsettings.json
{
  "ReverseProxy": {
    "Routes": {
      "orders-route": {
        "ClusterId": "orders-cluster",
        "Match": {
          "Path": "/api/orders/{**catch-all}"
        }
      },
      "products-route": {
        "ClusterId": "products-cluster",
        "Match": {
          "Path": "/api/products/{**catch-all}"
        }
      }
    },
    "Clusters": {
      "orders-cluster": {
        "Destinations": {
          "orders-api-1": {
            "Address": "http://orders-api"
          }
        }
      },
      "products-cluster": {
        "Destinations": {
          "products-api-1": {
            "Address": "http://products-api"
          }
        }
      }
    }
  }
}
```

## YARP: configuración de timeouts

El gateway puede tener sus propios timeouts, independientes de los timeouts de Polly en los servicios:

```json
"orders-cluster": {
  "HttpRequest": {
    "Timeout": "00:00:30",              // Timeout total: 30 segundos
    "Version": "2",                     // HTTP/2 para mejor rendimiento
    "VersionPolicy": "RequestVersionOrLower"
  },
  "Destinations": {
    "primary": { "Address": "http://orders-api" }
  }
}
```

**Layered timeouts:**
- YARP timeout (gateway) → 30 segundos
- Polly timeout (en orders-api para llamadas a products-api) → 5 segundos

Son independientes. YARP controla el tiempo que el cliente espera al gateway. Polly controla el tiempo que Orders espera a Products.

## El Program.cs del Gateway

```csharp
// Gateway.API/Program.cs

var builder = WebApplication.CreateBuilder(args);

builder.AddServiceDefaults();

builder.Host.UseSerilog((ctx, svc, cfg) => cfg
    .ReadFrom.Configuration(ctx.Configuration)
    .ReadFrom.Services(svc)
    .Enrich.WithProperty("Service", "gateway-api"));

// Leer la configuración de YARP desde appsettings.json
builder.Services
    .AddReverseProxy()
    .LoadFromConfig(builder.Configuration.GetSection("ReverseProxy"));

// Rate Limiting (configurado a continuación)
builder.Services.AddRateLimiter(/* ... */);

var app = builder.Build();

app.UseMiddleware<CorrelationIdMiddleware>();
app.UseSerilogRequestLogging();

// Rate Limiting middleware — ANTES de los endpoints
app.UseRateLimiter();

// El gateway
app.MapReverseProxy();

app.MapDefaultEndpoints(); // /health y /alive del propio gateway

app.Run();
```

      // En Azure: apunta a la URL del App Service
    }
  }
}

## Transforms: modificar requests y responses

YARP permite transformar los mensajes antes de enviarlos o recibirlos:

```json
"orders-route": {
  "ClusterId": "orders-cluster",
  "Match": { "Path": "/api/orders/{**catch-all}" },
  "Transforms": [
    // Copiar todos los headers originales
    { "RequestHeadersCopy": "true" },

    // Añadir header custom para identificar peticiones del gateway
    { "RequestHeader": "X-Gateway-Source", "Set": "orderflow-gateway" },

    // Reescribir el path (quitar prefijo si fuera necesario)
    // { "PathRemovePrefix": "/gateway" },

    // Añadir header en la respuesta
    { "ResponseHeader": "X-Served-By", "Set": "OrderFlow Gateway" }
  ]
}
```

## Rate Limiting: proteger los servicios

Con el gateway, el rate limiting es centralizado — no necesitas configurarlo en cada servicio:

```csharp
// Gateway.API/Program.cs

builder.Services.AddRateLimiter(options =>
{
    // Rate limit por IP: 100 peticiones por minuto
    options.AddSlidingWindowLimiter("PerIp", config =>
    {
        config.Window = TimeSpan.FromMinutes(1);
        config.PermitLimit = 100;
        config.SegmentsPerWindow = 4;
        config.QueueLimit = 20;
        config.QueueProcessingOrder = QueueProcessingOrder.OldestFirst;
    });

    // Rate limit estricto para creación de pedidos: 10 por minuto
    options.AddFixedWindowLimiter("CreateOrder", config =>
    {
        config.Window = TimeSpan.FromMinutes(1);
        config.PermitLimit = 10;
        config.QueueLimit = 0; // Sin cola — rechaza directamente con 429
    });

    // Cuando se supera el límite, devolver 429 con el header Retry-After
    options.RejectionStatusCode = StatusCodes.Status429TooManyRequests;
    options.OnRejected = async (context, ct) =>
    {
        context.HttpContext.Response.Headers.RetryAfter = "60";
        await context.HttpContext.Response.WriteAsync(
            "Too many requests. Please try again later.", ct);
    };
});
```

## Load Balancing: múltiples instancias

En producción con Azure App Services escalado, podemos tener múltiples instancias del mismo servicio:

```json
"orders-cluster": {
  "LoadBalancingPolicy": "RoundRobin",
  "Destinations": {
    "orders-api-1": {
      "Address": "https://orders-api-1.azurewebsites.net"
    },
    "orders-api-2": {
      "Address": "https://orders-api-2.azurewebsites.net"
    },
    "orders-api-3": {
      "Address": "https://orders-api-3.azurewebsites.net"
    }
  }
}
```

YARP distribuye las peticiones entre las instancias. Las políticas disponibles son:
- `RoundRobin` — distribución equitativa por turnos
- `LeastRequests` — al que menos peticiones activas tiene
- `Random` — aleatoriamente
- `PowerOfTwoChoices` — elige entre dos opciones aleatorias la menos cargada

```json
"orders-cluster": {
  "HealthCheck": {
    "Passive": {
      "Enabled": true,
      "Policy": "TransportFailureRate",
      "ReactivationPeriod": "00:02:00"
    },
    "Active": {
      "Enabled": true,
      "Interval": "00:00:30",
      "Timeout": "00:00:05",
      "Policy": "ConsecutiveFailures",
      "Path": "/health"
    }
  },
  "Destinations": {
    "primary": { "Address": "http://orders-api" }
  }
}
```

Si un destino empieza a fallar (health check activo: `GET /health` cada 30s), YARP lo marca como unhealthy y deja de enrutarle tráfico hasta que se recupere.

## Circuit Breaker en el Gateway

YARP no implementa Circuit Breaker nativo, pero se puede añadir con un middleware custom:

```csharp
// Gateway.API/Middleware/GatewayResilienceMiddleware.cs

public class GatewayResilienceMiddleware
{
    private readonly RequestDelegate _next;
    private readonly ResiliencePipeline<HttpResponseMessage> _pipeline;

    public async Task InvokeAsync(HttpContext context)
    {
        // Aplica el pipeline de resiliencia a la petición proxy
        var outcome = await _pipeline.ExecuteAsync(async ct =>
        {
            await _next(context);
            return new HttpResponseMessage(
                (HttpStatusCode)context.Response.StatusCode);
        });
    }
}
```

En la práctica, para OrderFlow el circuit breaker vive en cada servicio (con Polly, como en M3.5). El gateway se encarga del routing y rate limiting, los servicios se encargan de su propia resiliencia.

## Configuración dinámica de YARP

YARP puede recargar la configuración sin reiniciar. Útil para añadir nuevas rutas o cambiar destinos en caliente:

```csharp
// Configuración desde código C# (en vez de appsettings.json)
builder.Services
    .AddReverseProxy()
    .LoadFromConfig(builder.Configuration.GetSection("ReverseProxy"))
    .AddConfigFilter<CustomConfigFilter>(); // Filtro custom

// O configuración completamente programática
builder.Services
    .AddReverseProxy()
    .LoadFromMemory(
        routes: new[]
        {
            new RouteConfig
            {
                RouteId = "orders-route",
                ClusterId = "orders-cluster",
                Match = new RouteMatch { Path = "/api/orders/{**catch-all}" }
            }
        },
        clusters: new[]
        {
            new ClusterConfig
            {
                ClusterId = "orders-cluster",
                Destinations = new Dictionary<string, DestinationConfig>
                {
                    ["primary"] = new DestinationConfig
                    {
                        Address = "http://orders-api"
                    }
                }
            }
        });
```

## JWT en el Gateway: preview de M5.1

En M5.1 añadiremos autenticación completa. En el gateway, la validación del JWT será:

```csharp
// Preview de cómo quedará en M5.1

builder.Services.AddAuthentication(JwtBearerDefaults.AuthenticationScheme)
    .AddJwtBearer(options =>
    {
        options.Authority = builder.Configuration["Jwt:Issuer"]; // ASP.NET Core Identity / Azure AD
        options.Audience = "orderflow-gateway";
        options.TokenValidationParameters = new TokenValidationParameters
        {
            ValidateIssuer = true,
            ValidateAudience = true,
            ValidateLifetime = true,
            ClockSkew = TimeSpan.Zero
        };
    });

builder.Services.AddAuthorization();

// En el pipeline
app.UseAuthentication();
app.UseAuthorization();
app.MapReverseProxy();
```

Con esto, el gateway valida el JWT una vez. Los servicios internos reciben el header `Authorization` (por el `RequestHeadersCopy: true` del transform) y pueden extraer los claims sin validar de nuevo.

## YARP con ASP.NET Core Identity: el flujo de autenticación preview

Para que el alumno entienda hacia dónde vamos en M5.1:

```
1. Usuario accede: GET /api/orders (sin token)
2. Gateway devuelve 401 Unauthorized
3. Cliente redirige al Gateway: GET /account/login
4. Usuario hace login (ASP.NET Core Identity UI)
5. Gateway genera JWT firmado
6. Cliente intercambia code por token: POST /auth/.../token
7. Gateway devuelve JWT (access_token + refresh_token)
8. Cliente guarda el JWT y lo envía en cada request:
   Authorization: Bearer eyJhbGciOiJSUzI1NiIsInR...
9. Gateway valida el JWT con la clave de firma (HS256/RS256)
10. Si válido: YARP reenvía el request al servicio con el header Authorization
11. El servicio extrae los claims del token (userId, roles, etc.)
```

En M5.1 implementamos cada paso de este flujo con ASP.NET Core Identity embebido en el Gateway.

## Seguridad del Gateway: qué NO hacer

Errores comunes de seguridad en API Gateways:

```json
// ❌ Exponer servicios internos directamente también
// Los servicios deben ser SOLO accesibles desde el gateway en producción
// App Services: configurar "Access Restrictions"

// ❌ Olvidar el rate limiting
// Sin rate limiting, un atacante puede saturar tus servicios

// ❌ Hacer toda la lógica de negocio en el gateway
// El gateway debe ser un proxy, no un servicio de negocio
// Si el gateway cae, TODO cae

// ❌ JWT con validación en el gateway pero sin propagación
// Si el gateway valida el JWT pero no lo reenvía a los servicios,
// los servicios no pueden saber quién hace la petición
```

---

## Por qué el Gateway simplifica la seguridad

Sin gateway, cada servicio gestiona su propia autenticación:

```
Cliente → Orders.API (valida JWT)
Cliente → Products.API (valida JWT — configura igual)
Cliente → Payments.API (valida JWT — configura igual)
// 3 configuraciones de seguridad idénticas, 3 lugares donde equivocarse
```

Con gateway:

```
Cliente → Gateway (valida JWT una vez)
Gateway → Orders.API (confía en el gateway, sin validar)
Gateway → Products.API (confía en el gateway, sin validar)
Gateway → Payments.API (confía en el gateway, sin validar)
// 1 configuración de seguridad, 1 lugar donde mantener
```

El gateway es la capa de seguridad de la red perimetral. Los servicios internos asumen que si la petición llegó, ya fue autenticada. Esta confianza es posible porque los servicios solo aceptan peticiones del gateway (configurado en Azure App Services con Access Restrictions).

## Middleware custom en el Gateway

El gateway es un proyecto ASP.NET Core completo — puedes añadir cualquier middleware:

```csharp
// Middleware para añadir logging detallado en el gateway
public class GatewayLoggingMiddleware
{
    private readonly RequestDelegate _next;
    private readonly ILogger<GatewayLoggingMiddleware> _logger;

    public async Task InvokeAsync(HttpContext context)
    {
        var startTime = DateTime.UtcNow;
        var correlationId = context.Items["CorrelationId"]?.ToString();
        var path = context.Request.Path;
        var method = context.Request.Method;

        await _next(context);

        var elapsed = (DateTime.UtcNow - startTime).TotalMilliseconds;
        var status = context.Response.StatusCode;

        _logger.LogInformation(
            "Gateway: {Method} {Path} → {Status} ({Elapsed}ms) [{CorrelationId}]",
            method, path, status, elapsed, correlationId);
    }
}
```

Este middleware registra todas las peticiones que pasan por el gateway con su destino, status y duración.

## Diferencia entre YARP y Azure API Management (APIM)

¿Cuándo usaríamos Azure API Management en lugar de YARP?

**YARP es mejor cuando:**
- Control total sobre el código del gateway
- Necesitas lógica custom compleja (middleware .NET)
- Costes: YARP es gratuito (solo el App Service donde corre)
- Equipos de .NET que prefieren C# sobre YAML/Portal

**Azure APIM es mejor cuando:**

- Políticas XML ya existentes
- Necesitas el portal de desarrolladores de APIM (catálogo de APIs)
- Integración con Azure Developer Portal y Azure AD

**Para TechShop con OrderFlow:** YARP. El equipo es .NET, el sistema es homogéneo, APIM sería sobredimensionado y costoso para empezar.

## BFF: Backend for Frontend

Un patrón avanzado que YARP facilita: el BFF (Backend for Frontend).

En vez de un gateway genérico para todos los clientes, creas un gateway específico por tipo de cliente:

```
Gateway Mobile (BFF) → datos compactos, sin imágenes grandes
Gateway Web (BFF) → datos completos, con metadatos de UI
Gateway Admin (BFF) → datos de administración, endpoints internos
```

Con YARP, cada BFF es un proyecto separado con sus propias reglas de routing y transforms:

```json
// Gateway para la app móvil — respuestas compactas
"orders-mobile-route": {
  "ClusterId": "orders-cluster",
  "Match": { "Path": "/api/orders/{**catch-all}" },
  "Transforms": [
    // Añadir header para que Orders.API sepa que es mobile
    { "RequestHeader": "X-Client-Type", "Set": "mobile" }
  ]
}
```

Para OrderFlow del curso, usamos un gateway único. El BFF es un patrón avanzado para cuando los clientes tienen necesidades muy diferentes.

  }

  "ReverseProxy": {
    "Clusters": {
      "orders-cluster": {
        "Destinations": {

En desarrollo, Aspire inyecta las URLs automáticamente sobreescribiendo el appsettings. En producción, el App Service del gateway tiene configuradas las URLs de los servicios como variables de entorno.

## Preguntas frecuentes sobre el Gateway

**¿El gateway es un Single Point of Failure?**
Sí — si el gateway cae, ningún cliente puede acceder. La solución: múltiples instancias del gateway con un load balancer delante (Azure Load Balancer o Azure Front Door). En Azure App Services, el auto-scaling gestiona esto.

**¿Puedo tener la lógica de negocio en el gateway?**
No deberías. El gateway debe ser stateless y simple. La lógica de negocio en el gateway crea un monolito distribuido disfrazado de gateway.

**¿YARP soporta WebSockets?**
Sí. `builder.Services.AddReverseProxy()` incluye soporte de WebSockets por defecto.

**¿Puedo tener múltiples gateways?**
Sí. El patrón BFF (Backend for Frontend) usa un gateway por tipo de cliente. Para OrderFlow es excesivo, pero en sistemas grandes es habitual.
