## El problema: autenticación en microservicios

Con un monolito, la autenticación es sencilla: el usuario hace login, la sesión vive en memoria, cada request lleva una cookie.

Con microservicios tienes un problema distinto. **¿Cómo sabe Orders.API quién es el usuario si la petición llega desde el Gateway?** ¿Y Products.API? ¿Y Payments.API? Cada servicio necesita saber el userId, los roles y los permisos — sin hacer una llamada a una base de datos central en cada request.

La respuesta es JWT. Un token firmado criptográficamente que el cliente lleva en cada petición y que cualquier servicio puede verificar de forma independiente, sin llamar a ningún servidor externo.

---

## Conceptos clave: Autenticación vs Autorización

Antes del código, estos dos conceptos:

**Autenticación** — ¿Quién eres?
El proceso de verificar la identidad. El usuario demuestra que es quien dice ser — normalmente con usuario y contraseña. El resultado es un token.

**Autorización** — ¿Qué puedes hacer?
Una vez autenticado, qué operaciones tiene permitidas ese usuario. John puede ver sus propios pedidos pero no los de Jane. Solo los admins pueden cancelar pedidos de otros.

En OrderFlow:
- El **Gateway** autentica — valida el token JWT en cada request
- Los **servicios internos** autorizan — comprueban que el userId del token coincide con el recurso que se pide

## OAuth 2.0: el estándar de autorización

OAuth 2.0 es el protocolo estándar para delegar autorización. Define cómo un cliente obtiene un token de acceso de un Authorization Server para acceder a recursos protegidos.

```
Cliente (browser/app)
    │
    ├──→ Authorization Server (ASP.NET Core Identity)
    │       "Quiero acceder a OrderFlow con estos scopes"
    │       Usuario hace login
    │       ←── JWT token firmado
    │
    └──→ Gateway (Resource Server)
            "Tengo este token"
            Valida firma + expiración + scopes
            ←── 200 OK con datos
```

Los cuatro actores: **Resource Owner** (el usuario), **Client** (la app), **Authorization Server** (quien emite tokens), **Resource Server** (Orders.API, Products.API...).

---

## OpenID Connect: lo que añade sobre OAuth 2.0

OAuth 2.0 es un protocolo de autorización. **OpenID Connect (OIDC)** añade una capa de identidad encima — es lo que convierte OAuth 2.0 en un sistema de autenticación completo.

Con OIDC, además del `access_token` (para acceder a recursos), se obtiene un `id_token` (información sobre el usuario: sub, email, nombre, roles).

ASP.NET Core Identity implementa OIDC completo. Con `scope=openid` en la petición de token se obtiene el `id_token` además del `access_token`.

## Flujo Authorization Code + PKCE

El flujo que usa la app web de TechShop cuando un cliente hace login:

```
1. Browser → Gateway: GET /account/login
2. Browser ← Gateway: Formulario de login (ASP.NET Core Identity UI)
3. Browser → Gateway: POST /account/login {email, password}
4. Gateway valida credenciales en Identity DB (LocalDB)
5. Gateway genera JWT firmado con la clave de firma
6. Browser ← Gateway: Set-Cookie con JWT (o redirect con token)
7. Browser → Gateway: GET /api/orders (Authorization: Bearer {token})
8. Gateway valida JWT (firma + expiración + scopes)
9. Gateway → Orders.API: forwarded request con el mismo JWT
10. Orders.API extrae userId del JWT — sin llamar a ningún servidor
```

PKCE (Proof Key for Code Exchange) protege el flujo contra ataques de interceptación. ASP.NET Core lo gestiona automáticamente.

## JWT: el formato del token

Un JWT tiene tres partes separadas por puntos: `header.payload.signature`

```json
// Header — algoritmo de firma
{
  "alg": "HS256",   // o RS256 en producción
  "typ": "JWT"
}

// Payload — los claims del usuario
{
  "sub": "7b3a2c1d-...",          // userId (subject)
  "email": "john@techshop.es",
  "role": ["customer"],
  "iss": "https://localhost:8080", // issuer (el Gateway)
  "aud": "orderflow",              // audience
  "exp": 1735689600,               // expiración (Unix timestamp)
  "iat": 1735686000                // issued at
}

// Signature — HMACSHA256(base64(header) + "." + base64(payload), secretKey)
// Solo puede verificarse con la clave de firma — imposible falsificar sin ella
```

---

## Verificación de firma JWT: cómo funciona internamente

La clave de JWT: **ningún servicio necesita llamar a ningún servidor para verificar un token.**

```
Orders.API recibe: Authorization: Bearer eyJhbGciOiJIUzI1NiJ9...

1. Decodifica header → alg=HS256
2. Lee la clave de firma de su configuración (la misma que usó el Gateway para firmar)
3. Recalcula HMACSHA256(header + "." + payload, key)
4. Compara con la signature del token → ¿Coincide? Token válido.
5. Verifica exp → ¿No ha expirado?
6. Verifica iss y aud → ¿Vienen del Gateway de OrderFlow?
7. Extrae sub (userId), role → listo para usar

Tiempo: ~1ms. Sin red. Sin base de datos.
```

En producción con RS256 (clave asimétrica): el Gateway firma con la clave privada. Los servicios verifican con la clave pública. Solo el Gateway puede emitir tokens válidos.

## Scope y Audiencia: por qué importan

**Scope** — qué puede hacer el token
```
scope: "orders.read orders.write"    // puede leer y crear pedidos
scope: "products.read"               // solo lectura de productos
scope: "admin"                       // acceso de administrador
```

**Audience** (`aud`) — para qué servicio es válido el token
```json
// Token para el frontend de TechShop
{ "aud": "orderflow-web", "scope": "orders.read orders.write" }

// Token para comunicación interna Orders→Products
{ "aud": "products-api", "scope": "products.read" }
```

Si Products.API recibe un token con `aud: "orderflow-web"`, lo rechaza aunque la firma sea válida. Esto previene que un token comprometido sirva para todos los servicios.

## Refresh Token: renovar el acceso

Los access tokens tienen vida corta (15-60 minutos) por seguridad. El refresh token permite obtener uno nuevo sin que el usuario vuelva a hacer login.

```
1. Access token expira (401 Unauthorized)
2. Cliente: POST /account/refresh { refresh_token: "..." }
3. Gateway: valida el refresh token en Identity DB
4. Gateway: emite nuevo access token (15 min) + nuevo refresh token
5. Cliente: reintenta la petición original con el nuevo access token
```

ASP.NET Core Identity gestiona los refresh tokens automáticamente. Se almacenan en la tabla `AspNetUserTokens` de LocalDB.

**El token de acceso tiene vida corta — diseño intencionado:** si un token es robado, expira en minutos. El refresh token puede revocarse en cualquier momento borrando el registro de `AspNetUserTokens`.

## ASP.NET Core Identity: nuestra solución local

**ASP.NET Core Identity** es el sistema de gestión de usuarios de Microsoft, integrado en ASP.NET Core. Incluye:

- Registro y login de usuarios
- Gestión de roles y claims
- UI scaffolded (páginas de login, registro, perfil)
- Generación y validación de JWT
- Refresh tokens
- Password hashing (PBKDF2 por defecto)
- Soporte para 2FA

**En el curso, lo embebemos en Gateway.API.** El Gateway es el punto de entrada de todos los requests — tiene sentido que también sea el punto de entrada de la autenticación.

Sin procesos externos. Sin Java. Sin Docker. Solo ASP.NET Core.

## Generación del JWT al hacer login

```csharp
// Gateway.API/Services/JwtService.cs

public class JwtService
{
    private readonly IConfiguration _config;
    private readonly UserManager<ApplicationUser> _userManager;

    public async Task<string> GenerateTokenAsync(ApplicationUser user)
    {
        var roles = await _userManager.GetRolesAsync(user);
        var claims = new List<Claim>
        {
            new(JwtRegisteredClaimNames.Sub, user.Id),
            new(JwtRegisteredClaimNames.Email, user.Email!),
            new(JwtRegisteredClaimNames.Jti, Guid.NewGuid().ToString()),
            new("name", user.FullName ?? user.Email!),
        };

        // Añadir roles como claims
        foreach (var role in roles)
            claims.Add(new Claim(ClaimTypes.Role, role));

        var key = new SymmetricSecurityKey(
            Encoding.UTF8.GetBytes(_config["Jwt:SigningKey"]!));

        var token = new JwtSecurityToken(
            issuer: _config["Jwt:Issuer"],
            audience: _config["Jwt:Audience"],
            claims: claims,
            expires: DateTime.UtcNow.AddMinutes(
                int.Parse(_config["Jwt:ExpiryMinutes"] ?? "60")),
            signingCredentials: new SigningCredentials(
                key, SecurityAlgorithms.HmacSha256)
        );

        return new JwtSecurityTokenHandler().WriteToken(token);
    }
}
```

## Users y Roles de prueba: el seed

Para no crear usuarios manualmente cada vez que arranca el sistema:

```csharp
// Gateway.API/Data/SeedData.cs

public static class SeedData
{
    public static async Task InitializeAsync(IServiceProvider services)
    {
        var userManager = services.GetRequiredService<UserManager<ApplicationUser>>();
        var roleManager = services.GetRequiredService<RoleManager<IdentityRole>>();

        // Crear roles
        foreach (var role in new[] { "admin", "customer" })
            if (!await roleManager.RoleExistsAsync(role))
                await roleManager.CreateAsync(new IdentityRole(role));

        // Usuario admin de prueba
        if (await userManager.FindByEmailAsync("admin@techshop.es") == null)
        {
            var admin = new ApplicationUser
            {
                UserName = "admin@techshop.es",
                Email = "admin@techshop.es",
                FullName = "Admin TechShop"
            };
            await userManager.CreateAsync(admin, "Admin1234!");
            await userManager.AddToRoleAsync(admin, "admin");
        }

        // Usuario cliente de prueba
        if (await userManager.FindByEmailAsync("john@techshop.es") == null)
        {
            var john = new ApplicationUser
            {
                UserName = "john@techshop.es",
                Email = "john@techshop.es",
                FullName = "John Doe"
            };
            await userManager.CreateAsync(john, "Customer1234!");
            await userManager.AddToRoleAsync(john, "customer");
        }
    }
}
```

## Usar los claims del usuario en el Controller

```csharp
// Orders.API/Controllers/OrdersController.cs

[ApiController]
[Route("api/[controller]")]
[Authorize(Policy = "CustomerOrAdmin")]
public class OrdersController : ControllerBase
{
    [HttpGet]
    public async Task<IActionResult> GetAll(CancellationToken ct)
    {
        // El userId viene del JWT — no del request
        var userId = User.FindFirstValue(ClaimTypes.NameIdentifier)
                  ?? User.FindFirstValue("sub")!;

        // Los clientes solo ven sus propios pedidos
        // Los admins ven todos
        var isAdmin = User.IsInRole("admin");
        var orders = await _mediator.Send(
            new ListOrdersQuery(userId, isAdmin), ct);

        return Ok(orders);
    }

    [HttpDelete("{id:guid}")]
    [Authorize(Policy = "AdminOnly")]
    public async Task<IActionResult> Cancel(Guid id, CancellationToken ct)
    {
        // Solo admins pueden cancelar pedidos de otros
        await _mediator.Send(new CancelOrderCommand(id), ct);
        return NoContent();
    }
}
```

## Propagación del token a los servicios internos

El Gateway propaga el JWT original cuando hace forward a los servicios:

```csharp
// Gateway.API — YARP propaga Authorization header automáticamente
// El header Authorization: Bearer {token} llega a Orders.API sin cambios

// En Orders.API, si necesita llamar a Products.API:
public class ProductsClient
{
    private readonly HttpClient _client;
    private readonly IHttpContextAccessor _httpContextAccessor;

    public async Task<ProductDto?> GetByIdAsync(Guid id, CancellationToken ct)
    {
        // Propagar el mismo JWT del request entrante
        var token = _httpContextAccessor.HttpContext?
            .Request.Headers.Authorization.ToString()
            .Replace("Bearer ", "");

        if (token != null)
            _client.DefaultRequestHeaders.Authorization =
                new AuthenticationHeaderValue("Bearer", token);

        return await _client.GetFromJsonAsync<ProductDto>(
            $"/api/products/{id}", ct);
    }
}
```

## User Secrets para la clave JWT

La clave de firma JWT no puede ir en `appsettings.json` — comprometer el repo comprometería todos los tokens.

```bash
# En Gateway.API
dotnet user-secrets init
dotnet user-secrets set "Jwt:SigningKey" "mi-clave-super-secreta-de-desarrollo-32-chars!"

# En Orders.API (necesita la misma clave para validar)
dotnet user-secrets init
dotnet user-secrets set "Jwt:SigningKey" "mi-clave-super-secreta-de-desarrollo-32-chars!"
```

**Regla de oro:** Todos los servicios que validan JWT deben tener la misma `Jwt:SigningKey` en sus User Secrets. En producción esta clave se almacena en Azure Key Vault (M5.2) y todos los servicios la leen desde allí.

## Role-Based Authorization vs Claims-Based

**Role-Based:** autorización basada en roles predefinidos.

```csharp
[Authorize(Roles = "admin")]          // solo admins
[Authorize(Roles = "admin,customer")] // admin o customer
```

**Claims-Based:** autorización basada en propiedades del token más granulares.

```csharp
// Policy con claims custom
options.AddPolicy("CanCancelOrders", policy =>
    policy.RequireClaim("permission", "orders.cancel"));

// En el Controller
[Authorize(Policy = "CanCancelOrders")]
```

**Para OrderFlow:** roles (`admin`, `customer`) son suficientes. Claims-Based es útil para sistemas con permisos granulares o multi-tenant. ASP.NET Core Identity soporta ambos.

## JwtTestHelper: generar tokens de prueba

```csharp
// Tests/Shared/JwtTestHelper.cs

public static class JwtTestHelper
{
    public static string GenerateToken(
        string? userId = null,
        string[]? roles = null,
        string signingKey = "mi-clave-super-secreta-de-desarrollo-32-chars!",
        string issuer = "orderflow-gateway",
        string audience = "orderflow",
        int expiryMinutes = 60)
    {
        var claims = new List<Claim>
        {
            new(JwtRegisteredClaimNames.Sub, userId ?? Guid.NewGuid().ToString()),
            new(JwtRegisteredClaimNames.Email, "test@techshop.es"),
            new(JwtRegisteredClaimNames.Jti, Guid.NewGuid().ToString()),
        };

        foreach (var role in roles ?? ["customer"])
            claims.Add(new Claim(ClaimTypes.Role, role));

        var key = new SymmetricSecurityKey(Encoding.UTF8.GetBytes(signingKey));
        var token = new JwtSecurityToken(
            issuer: issuer,
            audience: audience,
            claims: claims,
            expires: DateTime.UtcNow.AddMinutes(expiryMinutes),
            signingCredentials: new SigningCredentials(key, SecurityAlgorithms.HmacSha256));

        return new JwtSecurityTokenHandler().WriteToken(token);
    }
}
```

## Verificación de ownership: john no puede ver pedidos de jane

La verificación de ownership es la segunda línea de defensa después del JWT:

```csharp
// ListOrdersHandler.cs — implementación
public class ListOrdersQueryHandler : IRequestHandler<ListOrdersQuery, PagedResult<OrderSummaryDto>>
{
    public async Task<PagedResult<OrderSummaryDto>> Handle(
        ListOrdersQuery query, CancellationToken ct)
    {
        // Si es admin, ve todos los pedidos
        // Si es customer, solo ve los suyos
        var orders = query.IsAdmin
            ? await _readRepo.GetAllAsync(query.Page, query.PageSize, ct)
            : await _readRepo.GetByCustomerIdAsync(
                query.UserId, query.Page, query.PageSize, ct);

        return orders;
    }
}
```

El `userId` viene del JWT — no del request body ni de la URL. Un cliente no puede pedir pedidos de otro pasando un userId diferente en la URL, porque el handler usa el userId del token, no el de la petición.

## Preguntas frecuentes sobre JWT y ASP.NET Core Identity

**¿Por qué no usar cookies en vez de JWT?**
Las cookies son de navegador. Los servicios internos (Orders → Products) no tienen navegador. JWT funciona en ambos contextos.

**¿JWT en localStorage o en cookie?**
Para SPAs: cookie HttpOnly+SameSite es más seguro que localStorage (protege contra XSS). Para APIs server-to-server: Authorization header.

**¿Qué pasa si el signing key se compromete en desarrollo?**
En desarrollo, consecuencias limitadas — la BD es local y los datos son de prueba. En producción la clave está en Key Vault con rotación automática.

**¿Por qué HS256 y no RS256?**
HS256 (simétrico) es suficiente para el curso. RS256 (asimétrico) es mejor en producción: solo el Gateway tiene la clave privada, los servicios solo necesitan la pública. Azure AD usa RS256.

**¿ASP.NET Core Identity soporta external login (Google)?**
Sí. `AddGoogle(...)` en la configuración de Identity. Para el curso no es necesario.

## Lo que viene en M5.2

En M5.2 aseguramos los secretos y el transporte:

**Jwt:SigningKey a Azure Key Vault:**
La clave de firma es el secreto más crítico del sistema. En producción no puede vivir en variables de entorno — va a Key Vault con Managed Identity.

**HTTPS en todos los servicios:**
Certificados de desarrollo en local. SSL Termination en Azure App Services. Sin tráfico HTTP en producción.

**mTLS entre servicios:**
Autenticación mutua para que Orders.API solo acepte llamadas que vengan del Gateway — no de cualquier proceso en la misma red.

Al terminar M5.2, ningún secreto vivirá en el código ni en variables de entorno planas.
