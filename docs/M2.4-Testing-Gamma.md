## Dónde estamos

Orders.API tiene logging, observabilidad y configuración robusta. Pero todavía no tenemos ningún test. Y en microservicios, los tests no son opcionales — son la red de seguridad que permite desplegar con confianza y cambiar código sin miedo a romper cosas.

En esta sesión construimos la suite de tests de Orders.API: unitarios para el dominio, integración para los endpoints, y contract tests para verificar que el contrato con Products.API no se rompe sin enterarnos.

Al terminar M2.4, el Módulo 2 está completo. Orders.API tiene la infraestructura que necesita cualquier microservicio en producción.

---

## Por qué el testing en microservicios es diferente

En un monolito, los tests de integración son simples: levantas la aplicación en memoria con `WebApplicationFactory`, lanzas peticiones y verificas el resultado. Todo en un solo proceso.

En microservicios, la complejidad crece:

**Múltiples servicios:** Un flujo completo cruza Orders → Products → Payments → Notifications. Para testear end-to-end necesitas 4 servicios arrancados y un message broker.

**Contratos entre servicios:** Si cambias el formato de un evento en Orders, puede romper a Products sin que ningún test te avise — porque cada servicio tiene sus propios tests independientes.

**Bases de datos aisladas:** Cada servicio tiene su propia BD. Las fakes en memoria pueden no reproducir el comportamiento real de SQL Server.

**Consistencia eventual:** Los flujos asíncronos son difíciles de testear porque no sabes cuándo termina el procesamiento.

La solución: una pirámide de testing ampliada específicamente para microservicios.

---

## La pirámide de testing para microservicios

La pirámide clásica (unitarios → integración → E2E) se amplía con una capa nueva:

```
         ╔══════════════════╗
         ║    E2E Tests     ║  ← Pocos. Todo desplegado. Lentos.
         ╠══════════════════╣
         ║ Contract Tests   ║  ← NUEVA CAPA. Verifican contratos.
         ╠══════════════════╣
         ║ Integration      ║  ← Tests de endpoints con WebApplicationFactory
         ║ Tests            ║    + LocalDB (SQL Server real)
         ╠══════════════════╣
         ║  Unit Tests      ║  ← Muchos. Rápidos. Sin dependencias externas.
         ╚══════════════════╝
```

**La regla de inversión del coste:** cuanto más arriba en la pirámide, más lento, más caro y más frágil. La base ancha de unitarios te da velocidad. La capa de contract tests te da seguridad en los contratos sin necesitar servicios reales.

## Herramientas que usaremos

**xUnit** — El framework de testing estándar para .NET. Integrado con Visual Studio, dotnet test, y GitHub Actions.

**NSubstitute** — Framework de mocking. Más moderno y con sintaxis más limpia que Moq. Es nuestra elección para Orders.API.

**FluentAssertions** — Aserciones con sintaxis legible: `result.Should().Be(expected)` en vez de `Assert.Equal(expected, result)`.

**WebApplicationFactory** — Levanta el servidor ASP.NET Core en memoria. Peticiones HTTP reales sin abrir un puerto.

**LocalDB Tests** — Usa SQL Server LocalDB para tests de integración. Mismo motor que en desarrollo, sin Docker, sin instalación extra.

**Pact.NET** — Contract testing entre servicios. Verifica que los contratos de API se cumplen.

## xUnit: ciclo de vida de los tests

xUnit tiene un modelo de ciclo de vida diferente a NUnit o MSTest. Importa entenderlo para evitar state compartido:

```csharp
// xUnit crea una NUEVA instancia de la clase por cada test
public class OrderTests
{
    private readonly Order _order; // Se inicializa en cada test

    public OrderTests()
    {
        // Constructor = SetUp. Se ejecuta antes de CADA test.
        _order = Order.Create(Guid.NewGuid(), CreateTestLines());
    }
}

// Para compartir estado caro entre tests del mismo clase: IClassFixture<T>
public class OrdersEndpointTests : IClassFixture<OrdersApiFactory>
{
    private readonly OrdersApiFactory _factory; // Compartida entre todos los tests

    public OrdersEndpointTests(OrdersApiFactory factory)
        => _factory = factory; // Solo se crea una vez
}

// Para compartir entre clases diferentes: ICollectionFixture<T>
[Collection("SqlServerCollection")]
public class OrderRepositoryTests
{
    public OrderRepositoryTests(LocalDbFixture fixture) { }
}
```

## FluentAssertions: aserciones que se leen como prosa

```csharp
// Assert básico de xUnit — funciona pero no es expresivo
Assert.Equal(OrderStatus.Confirmed, order.Status);
Assert.NotNull(order.ConfirmedAt);
Assert.True(order.Lines.Count > 0);

// FluentAssertions — se lee como una frase
order.Status.Should().Be(OrderStatus.Confirmed);
order.ConfirmedAt.Should().NotBeNull();
order.Lines.Should().NotBeEmpty();

// FluentAssertions para colecciones
order.Lines.Should().HaveCount(2);
order.Lines.Should().Contain(l => l.ProductName == "MacBook Pro");
order.Lines.Should().OnlyContain(l => l.Quantity > 0);
order.Lines.Should().BeInAscendingOrder(l => l.ProductName);

// FluentAssertions para excepciones
var act = () => order.Confirm();
act.Should().Throw<DomainException>()
   .WithMessage("*already confirmed*");

// FluentAssertions para async
await act.Should().ThrowAsync<ValidationException>();
```

La ventaja real: cuando falla, el mensaje de error es mucho más descriptivo que el de `Assert.Equal`.

## Unit Tests: qué testar y qué no

La regla para decidir si algo va en unit tests:

**SÍ — unit tests para:**
- Lógica de dominio (entidades, value objects, reglas de negocio)
- Handlers de MediatR (command/query handlers)
- Validators (FluentValidation)
- Mappers y transformaciones de datos

**NO — unit tests para:**
- Controllers directamente (mejor integration tests)
- Repositorios (mejor tests con BD real)
- Configuration y DI setup

Si para testear necesitas levantar una base de datos, un servidor HTTP o un message broker — no es un unit test. Es un integration test.

## Unit Test: reglas de negocio con Theory

```csharp
// Orders.API.Tests/Unit/Domain/OrderStatusTransitionTests.cs

public class OrderStatusTransitionTests
{
    [Theory]

```

## NSubstitute: mocking con sintaxis limpia

NSubstitute es el framework de mocking que usamos en OrderFlow. Comparativa rápida con Moq:

```csharp
// Con Moq (más verboso)
var mock = new Mock<IOrderRepository>();
mock.Setup(x => x.GetByIdAsync(It.IsAny<Guid>()))
    .ReturnsAsync(new Order(...));
var repo = mock.Object;

// Con NSubstitute (más limpio)
var repo = Substitute.For<IOrderRepository>();
repo.GetByIdAsync(Arg.Any<Guid>())
    .Returns(new Order(...));
```

**¿Por qué NSubstitute?**
- Sintaxis más legible y cercana al inglés natural
- No necesitas `.Object` para usar el sustituto
- Verificación de llamadas más expresiva
- Funciona perfectamente con interfaces y clases abstractas

---

## Por qué NSubstitute sobre Moq: la decisión

Moq fue el estándar durante años. NSubstitute es la alternativa moderna. ¿Por qué cambiamos?

**Moq tuvo un incidente de seguridad en 2023.** Una versión añadió código que recopilaba información de los sistemas de los developers. Aunque se revirtió, la confianza se vio afectada.

**NSubstitute tiene sintaxis más limpia.** Sin `.Setup(x => x.Method())` verboso, sin `.Object` para obtener el doble, sin `.Verify()` separado de la aserción.

**FluentAssertions integra perfectamente con NSubstitute.** Las verificaciones de llamadas se leen como prosa: `await _repo.Received(1).GetByIdAsync(orderId)`.

**La comunidad se ha movido.** La mayoría de proyectos .NET modernos usan NSubstitute o FakeItEasy. Moq sigue siendo válido, pero NSubstitute es nuestra elección para OrderFlow.

## Fakes vs Mocks vs Stubs: la diferencia

Tres tipos de dobles de test que a menudo se confunden:

**Stub:** Devuelve datos fijos. No verifica llamadas.
```csharp
var repo = Substitute.For<IOrderRepository>();
repo.GetByIdAsync(Arg.Any<Guid>()).Returns(testOrder);
// Solo proporciona datos — no importa si se llama o no
```

**Mock:** Verifica que las llamadas se hacen correctamente.
```csharp
await publisher.Received(1).PublishAsync(Arg.Any<OrderCreated>());
// Falla si PublishAsync no se llamó exactamente una vez
```

**Fake:** Implementación simplificada real que funciona en memoria.
```csharp
public class FakeEventPublisher : IEventPublisher
{
    public List<object> PublishedEvents { get; } = new();

    public Task PublishAsync<T>(T @event, CancellationToken ct = default)
    {
        PublishedEvents.Add(@event!);
        return Task.CompletedTask;
    }
}
```

El Fake es especialmente útil en integration tests — te permite verificar qué eventos se publicaron sin necesitar RabbitMQ.

## Fake Event Publisher: capturar eventos en tests

```csharp
// Orders.API.Tests/Shared/Fakes/FakeEventPublisher.cs

public class FakeEventPublisher : IEventPublisher
{
    private readonly List<object> _events = new();

    public IReadOnlyList<object> PublishedEvents => _events.AsReadOnly();

    public Task PublishAsync<T>(T @event,
        CancellationToken cancellationToken = default)
    {
        _events.Add(@event!);
        return Task.CompletedTask;
    }

    public IEnumerable<T> GetEvents<T>() =>
        _events.OfType<T>();

    public void Clear() => _events.Clear();
}
```

Uso en un integration test:

```csharp
[Fact]
public async Task CreateOrder_ShouldPublishOrderCreatedEvent()
{
    var fakePublisher = _factory.Services
        .GetRequiredService<IEventPublisher>() as FakeEventPublisher;

    await _client.PostAsJsonAsync("/api/orders", createOrderRequest);

    var events = fakePublisher!.GetEvents<OrderCreated>().ToList();
    events.Should().HaveCount(1);
    events[0].CustomerId.Should().Be(createOrderRequest.CustomerId);
}
```

## Unit Test: handlers de MediatR con NSubstitute

```csharp
// Orders.API.Tests/Unit/Application/GetOrderByIdHandlerTests.cs

public class GetOrderByIdHandlerTests
{
    private readonly IOrderRepository _repository;
    private readonly GetOrderByIdHandler _handler;

    public GetOrderByIdHandlerTests()
    {
        _repository = Substitute.For<IOrderRepository>();
        _handler = new GetOrderByIdHandler(_repository);
    }

    [Fact]
    public async Task Handle_ExistingOrder_ReturnsOrderDto()
    {
        // Arrange
        var orderId = Guid.NewGuid();

        var result = await _handler.Handle(query, CancellationToken.None);

        // Assert
        result.Should().NotBeNull();
        result!.Id.Should().Be(order.Id);

    }

    [Fact]
    public async Task Handle_NonExistingOrder_ReturnsNull()
    {
        _repository.GetByIdAsync(Arg.Any<Guid>()).Returns((Order?)null);

        var result = await _handler.Handle(
            new GetOrderByIdQuery(Guid.NewGuid()), CancellationToken.None);

        result.Should().BeNull();
    }
}
```

## Verificar llamadas con NSubstitute

NSubstitute permite verificar no solo que el resultado es correcto, sino que las llamadas a dependencias se hacen como se espera:

```csharp
[Fact]
public async Task CreateOrder_ShouldPublishOrderCreatedEvent()
{
    // Arrange
    var publisher = Substitute.For<IEventPublisher>();
    var repository = Substitute.For<IOrderRepository>();
    var handler = new CreateOrderHandler(repository, publisher);

    var command = new CreateOrderCommand(Guid.NewGuid(), CreateTestItems());

    // Act
    await handler.Handle(command, CancellationToken.None);

    // Assert
    // Verificar que el repositorio guardó el pedido
    await repository.Received(1).SaveAsync(Arg.Any<Order>());

    // Verificar que el publisher publicó exactamente un evento
    await publisher.Received(1)
        .PublishAsync(Arg.Is<OrderCreated>(e =>
            e.CustomerId == command.CustomerId));

    // Verificar que NO se llamó a algo que no debería
    await publisher.DidNotReceive()
        .PublishAsync(Arg.Any<OrderCancelled>());
}
```

## Testing con autenticación

En M5.1 añadiremos JWT. Para que los tests de integración funcionen, necesitamos poder emitir tokens de test:

```csharp
// Helper para generar tokens JWT en tests
public static class JwtTestHelper
{
    public static string GenerateToken(
        string userId = "test-user",
        string role = "customer",
        string secret = "test-secret-key-long-enough")
    {
        var key = new SymmetricSecurityKey(
            Encoding.UTF8.GetBytes(secret));

        var token = new JwtSecurityToken(
            issuer: "test-issuer",
            audience: "orders-api",
            claims: [
                new Claim(ClaimTypes.NameIdentifier, userId),
                new Claim(ClaimTypes.Role, role)
            ],
            expires: DateTime.UtcNow.AddHours(1),
            signingCredentials: new SigningCredentials(
                key, SecurityAlgorithms.HmacSha256));

        return new JwtSecurityTokenHandler().WriteToken(token);
    }
}

// Uso en test
_client.DefaultRequestHeaders.Authorization =
    new AuthenticationHeaderValue("Bearer", JwtTestHelper.GenerateToken());
```

## Builder Pattern para datos de test

Crear datos de test repetitivos es tedioso. El Builder Pattern lo simplifica:

```csharp
// Orders.API.Tests/Shared/Builders/OrderBuilder.cs

public class OrderBuilder
{
    private Guid _customerId = Guid.NewGuid();
    private List<OrderLine> _lines = new();

    public OrderBuilder WithCustomer(Guid customerId)
    {
        _customerId = customerId;
        return this;
    }

    public OrderBuilder WithLine(
        string productName = "Test Product",
        int quantity = 1,
        decimal price = 10.00m)
    {
        _lines.Add(new OrderLine(
            productId: Guid.NewGuid(),
            productName: productName,
            quantity: quantity,
            unitPrice: new Money(price, "EUR")));
        return this;
    }

    public Order Build()
    {
        if (!_lines.Any())
            WithLine(); // Añadir al menos una línea por defecto

        return Order.Create(_customerId, _lines);
    }
}

// Uso en tests — mucho más limpio
var order = new OrderBuilder()
    .WithCustomer(specificCustomerId)
    .WithLine("MacBook Pro", quantity: 1, price: 1999.99m)
    .WithLine("Ratón Magic", quantity: 2, price: 79.99m)
    .Build();
```

## Tests que NO merece la pena escribir

Hay tests que consumen tiempo sin aportar valor:

**Tests que verifican el framework:**
```csharp
// ❌ Esto testa a Entity Framework, no tu código
[Fact]
public async Task DbContext_CanSaveAndRetrieve_Entity() { ... }
```

**Tests de getters/setters:**
```csharp
// ❌ Esto no tiene lógica que testear
[Fact]
public void Order_Id_ShouldReturnSetId()
{
    var order = new Order { Id = Guid.NewGuid() };
    order.Id.Should().NotBeEmpty(); // Esto siempre pasa
}
```

**Tests que replican la implementación:**
```csharp
// ❌ El test es una copia de la implementación
[Fact]
public void Calculate_Total_ShouldSumAllLines()
{
    // Si cambias la fórmula, el test falla aunque sea correcto
    var expected = lines.Sum(l => l.Quantity * l.UnitPrice.Amount);
    order.Total.Amount.Should().Be(expected);
}
```

El tiempo de writing y mantenimiento de estos tests no compensa. Focaliza en tests que verifican comportamiento de negocio.

## WebApplicationFactory: integration tests de endpoints

`WebApplicationFactory` levanta toda la aplicación ASP.NET Core en memoria. Sin puerto abierto, sin proceso separado. Las peticiones HTTP son reales pero van por el pipeline interno.

```csharp
// Orders.API.Tests/Shared/Fixtures/OrdersApiFactory.cs

public class OrdersApiFactory : WebApplicationFactory<Program>
{
    protected override void ConfigureWebHost(IWebHostBuilder builder)
    {
        builder.ConfigureTestServices(services =>
        {
            // Reemplazar la BD real por InMemory para tests rápidos
            var dbDescriptor = services.SingleOrDefault(
                d => d.ServiceType == typeof(DbContextOptions<OrderDbContext>));
            if (dbDescriptor != null) services.Remove(dbDescriptor);

            services.AddDbContext<OrderDbContext>(options =>
                options.UseInMemoryDatabase("TestOrdersDb"));

            // Reemplazar el event publisher por un fake
            services.AddSingleton<IEventPublisher, FakeEventPublisher>();
        });

        builder.UseEnvironment("Testing");
    }
}
```

## El problema de InMemory Database

`UseInMemoryDatabase` es cómodo, pero tiene limitaciones importantes:

**Lo que InMemory NO hace:**
- No tiene constraints de base de datos (UNIQUE, FOREIGN KEY, NOT NULL a nivel BD)
- No soporta transacciones reales
- No ejecuta queries SQL reales — no detecta queries problemáticas
- No soporta funciones de SQL Server (JSON, temporal tables)
- El comportamiento puede diferir del SQL Server real

**Ejemplo de problema:**

```csharp
// Con InMemory: este test pasa
await dbContext.Orders.AddAsync(new Order { Id = existingId });
await dbContext.SaveChangesAsync();
// No hay UNIQUE constraint en InMemory → segundo add pasa sin error

// Con SQL Server real: este test falla como esperamos
// → Duplicate key violation
```

Para tests que interactúan con la base de datos, usamos SQL Server LocalDB — el mismo motor que en desarrollo, sin instalación adicional.

## Contract Testing: el problema que resuelve

Imagina este escenario:

1. El equipo de Orders decide añadir validación extra en el evento `OrderCreated`. Cambian el campo `CustomerId` de `Guid` a `string`.
2. El equipo de Products tiene tests unitarios propios que pasan correctamente.
3. Nadie cambia los tests de Products porque no saben que Orders cambió el contrato.
4. En producción: Products no puede deserializar el evento porque espera `Guid` y recibe `string`.

**Sin contract testing:** el bug aparece en producción. A las 3 de la madrugada.

**Con contract testing:** el pipeline de CI de Orders falla porque el nuevo formato de `OrderCreated` rompe el contrato que Products espera.

---

## Pact.NET: cómo funciona

Pact implementa contract testing con un enfoque consumer-driven:

**1. El consumer (Products.API) define el contrato:**
"Cuando recibo un evento `OrderCreated`, espero que tenga estos campos con estos tipos."

**2. Pact genera un fichero de contrato (pact file):**
Un JSON que describe exactamente qué espera el consumer.

**3. El producer (Orders.API) verifica el contrato:**
"¿Cumple mi implementación con lo que el consumer espera?"

Si Orders cambia el formato y el consumer no actualiza el contrato → el producer falla en CI.

---

## Pact: el consumer escribe el contrato

```csharp
// En Products.API.Tests/Contract/OrderCreatedConsumerTest.cs

public class OrderCreatedConsumerTest : IDisposable
{
    private readonly IPactBuilderV4 _pactBuilder;

    public OrderCreatedConsumerTest()
    {
        var pact = Pact.V4("products-api", "orders-api",
            new PactConfig { PactDir = "./pacts" });
        _pactBuilder = pact.WithMessageInteractions();
    }

    [Fact]
    public async Task ShouldDeserializeOrderCreatedEvent()
    {
        await _pactBuilder
            .ExpectsToReceive("an OrderCreated event")
            .Given("an order has been created")
            .WithJsonContent(new
            {
                orderId = Match.Type(Guid.NewGuid().ToString()),
                customerId = Match.Type(Guid.NewGuid().ToString()),
                total = Match.Decimal(150.00m),
                currency = Match.Regex("EUR", "^[A-Z]{3}$"),
                items = Match.MinType(new
                {
                    productId = Match.Type(Guid.NewGuid().ToString()),
                    quantity = Match.Integer(1)
                }, 1)
            })
            .Verify<OrderCreated>(message =>
            {
                // Verificar que Products puede deserializar y procesar el evento
                message.Should().NotBeNull();
                message.OrderId.Should().NotBeEmpty();
                message.Items.Should().NotBeEmpty();
            });
    }

    public void Dispose() => _pactBuilder.Build();
}
```

## Pact en el pipeline de CI

El flujo completo de contract testing en GitHub Actions:

```yaml
# En el pipeline de Products.API
```

## Parallel test execution: cuidado con el estado compartido

xUnit puede ejecutar tests en paralelo por defecto, lo que puede causar problemas:

```csharp
    public async Task InitializeAsync()
        => _transaction = await _dbContext.Database.BeginTransactionAsync();

    public async Task DisposeAsync()
        => await _transaction.RollbackAsync(); // Siempre rollback
}
```

## Preguntas frecuentes de testing en microservicios

**¿Por qué LocalDB y no TestContainers?**
LocalDB es suficiente para el curso. TestContainers existe y es útil en CI/CD Linux donde LocalDB no está disponible, pero requiere Docker — que en este curso hemos descartado deliberadamente.

**¿Cuántos tests son suficientes?**
No hay número mágico. La pregunta correcta: ¿tengo confianza para desplegar este cambio? Si la respuesta es sí, los tests son suficientes.

**¿Los contract tests reemplazan a los E2E tests?**
No, se complementan. Los contract tests verifican los contratos individualmente (rápido). Los E2E verifican el flujo completo integrado (lento, para antes de producción).

**¿Cómo testeo la Saga en M4.3?**
Con integration tests que usan MassTransit en modo "in-memory" (MassTransit.Testing). No necesitas RabbitMQ real para los tests de la saga. Lo veremos en M4.3.

## Lo que viene en M3.1

Empezamos el Módulo 3. Primera sesión: diseño del dominio de Orders.

- Domain Events: `OrderCreatedEvent`, `OrderConfirmedEvent`
