## Bienvenidos al Módulo 4

El Módulo 3 nos dejó dos microservicios funcionando de forma independiente: Orders.API y Products.API. El Módulo 4 los conecta.

Hay dos formas de comunicación entre microservicios:

**Síncrona:** El servicio que llama espera la respuesta antes de continuar. El resultado llega en la misma petición. Ejemplo: Orders pregunta a Products el precio de un artículo antes de crear el pedido.

**Asíncrona:** El servicio publica un evento y sigue su camino. El resultado llega más tarde, si llega. Ejemplo: Cuando se crea un pedido, Orders publica `OrderCreated` y Notifications lo procesa cuando puede.

En M4.1 cubrimos la comunicación síncrona: HTTP con HttpClientFactory, clientes tipados con Refit, y una introducción a gRPC. En M4.2 y M4.3 cubrimos la asíncrona con RabbitMQ y MassTransit.

## ¿Cuándo usar comunicación síncrona?

La regla para decidir entre síncrona y asíncrona:

**Síncrona cuando:**
- El resultado es necesario para continuar
- El usuario espera la respuesta inmediata
- La operación es una consulta (lectura)

**Asíncrona cuando:**
- El resultado puede llegar más tarde
- El usuario no necesita esperar
- La operación tiene efectos secundarios en otros servicios

**En OrderFlow:**

| Llamada | Tipo | Razón |
|---|---|---|
| Orders → Products (verificar precio) | **Síncrona** | Necesitamos el precio para crear el pedido |
| Orders → Products (reservar stock) | **Asíncrona** | La Saga lo gestiona en M4.3 |
| Orders → Notifications (enviar email) | **Asíncrona** | El email puede esperar |
| Cliente → Orders (crear pedido) | **Síncrona** | El usuario espera el 201 |

## El problema de crear HttpClient directamente

Antes de HttpClientFactory, el código de comunicación HTTP era así:

```csharp
// ❌ El antipatrón — NO hagas esto
public class ProductsService
{
    public async Task<ProductDto?> GetProductAsync(Guid id)
    {
        // Nuevo HttpClient en cada llamada
        using var client = new HttpClient();
        client.BaseAddress = new Uri("http://products-api");

        return await client.GetFromJsonAsync<ProductDto>($"/api/products/{id}");
    }
}
```

**Problema 1 — Socket exhaustion:** `HttpClient` implementa `IDisposable`, pero usar `using` y crear uno nuevo por llamada agota los sockets del sistema operativo. Los sockets en `TIME_WAIT` no se liberan inmediatamente.

**Problema 2 — DNS staleness:** `HttpClient` cachea DNS. Si la IP de Products.API cambia (escala, failover), el cliente antiguo sigue usando la IP vieja.

**Problema 3 — Sin resiliencia compartida:** Cada instancia de `HttpClient` tiene su propia configuración. No hay forma de centralizar Polly.

---

## HttpClientFactory: la solución correcta

`IHttpClientFactory` gestiona el pool de `HttpMessageHandler` internamente. Reutiliza los handlers pero renueva los clientes HTTP periódicamente para evitar el problema de DNS:

```csharp
// ✅ HttpClientFactory — correcto
builder.Services.AddHttpClient("products-api", client =>
{
    client.BaseAddress = new Uri("http://products-api");
    client.DefaultRequestHeaders.Add("Accept", "application/json");
    client.Timeout = TimeSpan.FromSeconds(10);
});

// Uso en el servicio
public class ProductsService
{
    private readonly IHttpClientFactory _factory;

    public ProductsService(IHttpClientFactory factory)
        => _factory = factory;

    public async Task<ProductDto?> GetProductAsync(Guid id)
    {
        var client = _factory.CreateClient("products-api");
        return await client.GetFromJsonAsync<ProductDto>($"/api/products/{id}");
    }
}
```

`IHttpClientFactory` gestiona los sockets, el DNS y el pool. El desarrollador solo gestiona la lógica de negocio.

## Typed Clients: el patrón recomendado

Los Named Clients con strings son propensos a errores tipográficos. Los **Typed Clients** encapsulan toda la lógica HTTP de una dependencia en una clase dedicada:

```csharp
// ✅ Typed Client — un cliente por servicio externo
public class ProductsClient
{
    private readonly HttpClient _httpClient;
    private readonly ILogger<ProductsClient> _logger;

    // El constructor recibe el HttpClient gestionado por la factory
    public ProductsClient(HttpClient httpClient, ILogger<ProductsClient> logger)
    {
        _httpClient = httpClient;
        _logger = logger;
    }

    public async Task<ProductDetailDto?> GetProductAsync(
        Guid productId, CancellationToken ct = default)
    {
        var response = await _httpClient.GetAsync(
            $"/api/products/{productId}", ct);

        if (response.StatusCode == HttpStatusCode.NotFound)
            return null;

        response.EnsureSuccessStatusCode();

        return await response.Content
            .ReadFromJsonAsync<ProductDetailDto>(ct);
    }
}

// Registrar como Typed Client (Aspire resuelve la URL)
builder.Services.AddHttpClient<ProductsClient>(client =>
    client.BaseAddress = new Uri("http://products-api"));
```

## ProductDetailDto: el contrato entre servicios

Cuando Orders llama a Products, necesita un DTO que represente lo que Products devuelve:

```csharp
// Orders.API/Infrastructure/Http/DTOs/ProductDetailDto.cs

namespace Orders.API.Infrastructure.Http.DTOs;

/// <summary>
/// DTO con los datos que Orders necesita de Products.
/// Solo las propiedades que Orders usa — no el objeto completo de Products.
/// </summary>
public record ProductDetailDto(
    Guid Id,
    string Name,
    decimal Price,
    string Currency,
    int Stock,
    bool IsActive,
    Guid CategoryId);
```

**La regla del Anti-Corruption Layer:** Orders no importa clases del proyecto Products.API. Define sus propios DTOs para lo que necesita de Products. Si Products cambia el formato, solo cambia este DTO — no el dominio de Orders.

## ProductsClient: el cliente completo

```csharp
// Orders.API/Infrastructure/Http/ProductsClient.cs

namespace Orders.API.Infrastructure.Http;

public class ProductsClient
{
    private readonly HttpClient _httpClient;
    private readonly IMemoryCache _cache;
    private readonly ILogger<ProductsClient> _logger;

    public ProductsClient(
        HttpClient httpClient,
        IMemoryCache cache,
        ILogger<ProductsClient> logger)
    {
        _httpClient = httpClient;
        _cache = cache;
        _logger = logger;
    }

    public async Task<ProductDetailDto?> GetProductAsync(
        Guid productId, CancellationToken ct = default)
    {
        var cacheKey = $"product:{productId}";

        if (_cache.TryGetValue(cacheKey, out ProductDetailDto? cached))
        {
            _logger.LogDebug(
                "Cache hit for product {ProductId}", productId);
            return cached;
        }

        _logger.LogDebug(
            "Fetching product {ProductId} from Products.API", productId);

        var response = await _httpClient.GetAsync(
            $"/api/products/{productId}", ct);

        if (response.StatusCode == HttpStatusCode.NotFound)
        {
            _logger.LogWarning(
                "Product {ProductId} not found in Products.API", productId);
            return null;
        }

        response.EnsureSuccessStatusCode();

        var product = await response.Content
            .ReadFromJsonAsync<ProductDetailDto>(ct);

        if (product is not null)
        {
            _cache.Set(cacheKey, product, TimeSpan.FromMinutes(5));
            _logger.LogDebug(
                "Product {ProductId} cached for 5 minutes", productId);
        }

        return product;
    }

    public async Task<bool> ReserveStockAsync(
        Guid productId, Guid orderId, int quantity, CancellationToken ct = default)
    {
        var request = new { OrderId = orderId, Quantity = quantity };
        var response = await _httpClient.PostAsJsonAsync(
            $"/api/products/{productId}/reserve", request, ct);

        if (response.StatusCode == HttpStatusCode.UnprocessableEntity)
        {
            _logger.LogWarning(
                "Insufficient stock for product {ProductId}. Quantity: {Qty}",
                productId, quantity);
            return false;
        }

        response.EnsureSuccessStatusCode();
        return true;
    }
}
```

## Actualizar CreateOrderHandler con ProductsClient real

Ahora el handler usa el cliente real para verificar precios y stock:

```csharp
// Application/Commands/CreateOrderHandler.cs — actualizado

public class CreateOrderHandler : IRequestHandler<CreateOrderCommand, OrderDto>
{
    private readonly IOrderRepository _repository;
    private readonly ProductsClient _productsClient;
    private readonly ILogger<CreateOrderHandler> _logger;

    public async Task<OrderDto> Handle(
        CreateOrderCommand command, CancellationToken ct)
    {
        // Verificar precio y stock para cada ítem
        var validatedItems = new List<(Guid, string, int, Money)>();

        foreach (var item in command.Items)
        {
            var product = await _productsClient
                .GetProductAsync(item.ProductId, ct);

            if (product is null)
                throw new DomainException(
                    $"Product '{item.ProductId}' does not exist or is unavailable");

            if (!product.IsActive)
                throw new DomainException(
                    $"Product '{product.Name}' is not active");

            if (product.Stock < item.Quantity)
                throw new DomainException(
                    $"Insufficient stock for '{product.Name}'. " +
                    $"Available: {product.Stock}, Requested: {item.Quantity}");

            // Usar el precio oficial del catálogo
            validatedItems.Add((
                product.Id,
                product.Name,
                item.Quantity,
                new Money(product.Price, product.Currency)));
        }

        var address = new Address(
            command.ShippingAddress.Street,
            command.ShippingAddress.City,
            command.ShippingAddress.ZipCode,
            command.ShippingAddress.Country);

        var order = Order.Create(command.CustomerId, validatedItems, address);
        await _repository.SaveAsync(order, ct);

        return order.ToDto();
    }
}
```

## Refit: clientes HTTP tipados desde una interfaz

Refit genera automáticamente la implementación del cliente HTTP a partir de una interfaz:

```bash

```

```csharp
// Orders.API/Infrastructure/Http/IProductsApi.cs

using Refit;

namespace Orders.API.Infrastructure.Http;

/// <summary>
/// Interfaz Refit — genera el cliente HTTP automáticamente.
/// Los atributos definen el contrato REST.
/// </summary>
public interface IProductsApi
{
    [Get("/api/products/{id}")]
    Task<ProductDetailDto?> GetProductAsync(Guid id, CancellationToken ct = default);

    [Get("/api/products")]
    Task<PagedResult<ProductSummaryDto>> ListProductsAsync(
        [AliasAs("category")] string? category = null,
        [AliasAs("page")] int page = 1,
        [AliasAs("pageSize")] int pageSize = 20,
        CancellationToken ct = default);

    [Post("/api/products/{id}/reserve")]

        Guid id,
        [Body] ReserveStockRequest request,
        CancellationToken ct = default);

    [Put("/api/products/{id}/stock")]
    Task UpdateStockAsync(
        Guid id,
        [Body] UpdateStockRequest request,
        CancellationToken ct = default);
}
```

## Typed Client vs Refit: cuándo usar cada uno

| Aspecto | Typed Client (manual) | Refit |
|---|---|---|
| Código | Más código, más control | Menos código, más magic |
| Flexibilidad | Total — cualquier lógica | Limitada a lo que Refit soporta |
| Error handling | Manual y explícito | Puede necesitar DelegatingHandler |
| Testing | Mockear el HttpClient | Mockear la interfaz directamente |
| Rendimiento | Sin overhead | Mínimo overhead de reflection |
| Visibilidad | Sabes exactamente qué hace | Magia en runtime |

**Para OrderFlow:** Usamos `ProductsClient` manual (más control, más visible). Refit es ideal para equipos que generan clientes desde specs OpenAPI — lo vemos a continuación.

## Service Discovery: cómo Aspire resuelve las URLs

En desarrollo con Aspire, `http://products-api` no es una URL real — es un nombre de servicio que Aspire resuelve automáticamente:

```csharp
// El AppHost declara la referencia
var orders = builder.AddProject<Projects.Orders_API>("orders-api")
    .WithReference(products); // ← Aspire inyecta la URL de products-api

// En Orders.API Program.cs
builder.Services.AddHttpClient<ProductsClient>(client =>
    client.BaseAddress = new Uri("http://products-api")); // ← Aspire lo resuelve
```

Lo que hace Aspire:
1. Asigna un puerto dinámico a cada servicio
2. Inyecta en Orders.API una variable de entorno `services__products-api__http__0` con la URL real
3. El middleware de service discovery de Aspire reemplaza `http://products-api` por `http://localhost:PORT`

En **producción (Azure):** App Services tiene URLs estables (`https://products-api.azurewebsites.net`). La misma variable de entorno apunta a la URL real. El código no cambia.

## CorrelationId: propagar el contexto entre servicios

Cuando Orders llama a Products, el CorrelationId debe propagarse para que los logs estén correlacionados:

```csharp
// Orders.API/Infrastructure/Http/CorrelationIdDelegatingHandler.cs

public class CorrelationIdDelegatingHandler : DelegatingHandler
{
    private readonly IHttpContextAccessor _httpContextAccessor;

    public CorrelationIdDelegatingHandler(IHttpContextAccessor accessor)
        => _httpContextAccessor = accessor;

    protected override async Task<HttpResponseMessage> SendAsync(
        HttpRequestMessage request, CancellationToken ct)
    {
        // Leer el CorrelationId del request HTTP actual
        var correlationId = _httpContextAccessor.HttpContext?
            .Items["CorrelationId"]?.ToString()
            ?? Activity.Current?.TraceId.ToString()
            ?? Guid.NewGuid().ToString("N");

        // Añadirlo al header de la llamada saliente
        request.Headers.TryAddWithoutValidation(
            "X-Correlation-Id", correlationId);

        return await base.SendAsync(request, ct);
    }
}

// Registrar en Program.cs
builder.Services.AddTransient<CorrelationIdDelegatingHandler>();
builder.Services.AddHttpClient<ProductsClient>(...)
    .AddHttpMessageHandler<CorrelationIdDelegatingHandler>();
```

Ahora en los logs de Products.API, el mismo CorrelationId de Orders aparece — puedes trazar toda la cadena de llamadas.

## Propagar JWT entre servicios

En M5.1 añadiremos JWT. Preview de cómo se propaga el token entre servicios:

```csharp
// DelegatingHandler para propagar el JWT
public class AuthTokenDelegatingHandler : DelegatingHandler
{
    private readonly IHttpContextAccessor _accessor;
    private readonly ITokenService _tokenService;

    protected override async Task<HttpResponseMessage> SendAsync(
        HttpRequestMessage request, CancellationToken ct)
    {
        // Opción A: Propagar el token del usuario (On-Behalf-Of flow)
        var userToken = _accessor.HttpContext?
            .Request.Headers.Authorization.ToString()
            .Replace("Bearer ", "");

        // Opción B: Token de servicio a servicio (Client Credentials)
        var serviceToken = await _tokenService
            .GetServiceTokenAsync("products-api", ct);

        request.Headers.Authorization =
            new AuthenticationHeaderValue("Bearer", serviceToken);

        return await base.SendAsync(request, ct);
    }
}
```

El patrón es el mismo que para CorrelationId: un DelegatingHandler que añade el header antes de enviar la petición.

## gRPC: comunicación de alto rendimiento

gRPC es un protocolo de comunicación entre servicios desarrollado por Google. Usa Protocol Buffers (protobuf) como formato de serialización y HTTP/2 como transporte.

**Ventajas sobre REST:**
- **Rendimiento:** Protobuf es binario — 5-10x más compacto que JSON
- **Tipado fuerte:** El contrato es un fichero `.proto` — no hay interpretación
- **Streaming:** gRPC soporta streams bidireccionales — ideal para datos en tiempo real
- **Code generation:** Se generan clientes y servidores automáticamente

**Desventajas:**
- **Depuración:** Binario — no puedes leerlo con el navegador o curl
- **Soporte de clientes:** Los navegadores no soportan gRPC directamente (necesitan gRPC-Web)
- **Curva de aprendizaje:** Los ficheros `.proto` añaden un paso extra

**Cuándo usar gRPC en OrderFlow:**
- Comunicación interna entre microservicios (en vez de REST)
- Servicios con alto volumen de llamadas internas (miles/segundo)
- Cuando el tamaño del payload importa (datos grandes)

## Definir el contrato gRPC: el fichero .proto

```protobuf
// Products.API/Protos/products.proto

syntax = "proto3";

option csharp_namespace = "Products.API.Grpc";

package products;

// El servicio gRPC de Products
service ProductsGrpcService {
  // Obtener un producto por ID
  rpc GetProduct (GetProductRequest) returns (ProductResponse);

  // Listar productos con filtros
  rpc ListProducts (ListProductsRequest) returns (ListProductsResponse);

  // Reservar stock
  rpc ReserveStock (ReserveStockRequest) returns (ReserveStockResponse);
}

// Mensajes de request y response
message GetProductRequest {
  string product_id = 1; // UUID como string
}

message ProductResponse {
  string id = 1;
  string name = 2;
  double price = 3;
  string currency = 4;
  int32 stock = 5;
  bool is_active = 6;
}

message ReserveStockRequest {
  string product_id = 1;
  string order_id = 2;
  int32 quantity = 3;
}

message ReserveStockResponse {
  bool success = 1;
  int32 remaining_stock = 2;
  string error_message = 3;
}
```

## Implementar el servidor gRPC en Products.API

```bash
# En Products.API
dotnet add package Grpc.AspNetCore
```

```csharp
// Products.API/Grpc/ProductsGrpcService.cs

using Grpc.Core;
using Products.API.Grpc; // Generado desde el .proto

public class ProductsGrpcService : ProductsGrpcService.ProductsGrpcServiceBase
{
    private readonly IProductRepository _repository;

    public ProductsGrpcService(IProductRepository repository)
        => _repository = repository;

    public override async Task<ProductResponse> GetProduct(
        GetProductRequest request,
        ServerCallContext context)
    {
        var productId = Guid.Parse(request.ProductId);
        var product = await _repository.GetByIdAsync(
            productId, context.CancellationToken);

        if (product is null)
            throw new RpcException(new Status(
                StatusCode.NotFound,
                $"Product {productId} not found"));

        return new ProductResponse
        {
            Id = product.Id.ToString(),
            Name = product.Name,
            Price = (double)product.Price,
            Currency = product.Currency,
            Stock = product.Stock,
            IsActive = product.IsActive
        };
    }
}

// Registrar en Products.API Program.cs
builder.Services.AddGrpc();
app.MapGrpcService<ProductsGrpcService>();
```

## Implementar el cliente gRPC en Orders.API

```bash
# En Orders.API
dotnet add package Grpc.Net.Client
dotnet add package Google.Protobuf
dotnet add package Grpc.Tools
```

```csharp
// Orders.API/Infrastructure/Http/ProductsGrpcClient.cs

public class ProductsGrpcClient
{
    private readonly ProductsGrpcService.ProductsGrpcServiceClient _client;
    private readonly ILogger<ProductsGrpcClient> _logger;

    public ProductsGrpcClient(
        ProductsGrpcService.ProductsGrpcServiceClient client,
        ILogger<ProductsGrpcClient> logger)
    {
        _client = client;
        _logger = logger;
    }

    public async Task<ProductDetailDto?> GetProductAsync(
        Guid productId, CancellationToken ct = default)
    {
        try
        {
            var response = await _client.GetProductAsync(
                new GetProductRequest { ProductId = productId.ToString() },
                cancellationToken: ct);

            return new ProductDetailDto(
                Guid.Parse(response.Id),
                response.Name,
                (decimal)response.Price,
                response.Currency,
                response.Stock,
                response.IsActive,
                Guid.Empty); // CategoryId no está en el proto
        }
        catch (RpcException ex) when (ex.StatusCode == StatusCode.NotFound)
        {
            return null;
        }
    }
}

// Registrar en Orders.API Program.cs
builder.Services.AddGrpcClient<ProductsGrpcService.ProductsGrpcServiceClient>(options =>
    options.Address = new Uri("http://products-api"))
    .AddResilienceHandler("products-grpc", pipeline =>
    {
        pipeline.AddRetry(new RetryStrategyOptions<GrpcResponse>
        {
            ShouldHandle = new PredicateBuilder<GrpcResponse>()
                .Handle<RpcException>(ex =>
                    ex.StatusCode is StatusCode.Unavailable
                        or StatusCode.DeadlineExceeded),
            MaxRetryAttempts = 3
        });
    });
```

## REST vs gRPC: comparativa práctica

Para OrderFlow, elegimos REST. La comparativa que nos llevó a esa decisión:

| Aspecto | REST (JSON/HTTPS) | gRPC (Protobuf/HTTP2) |
|---|---|---|
| Rendimiento | Bueno | Excelente (~5x más rápido) |
| Tamaño payload | JSON: ~1KB | Protobuf: ~200 bytes |
| Herramientas debug | curl, Postman, .http files | grpcurl, BloomRPC |
| Aprendizaje | Conocido por todos | Curva de aprendizaje |
| Soporte clientes | Universal | Limitado en browsers |
| Versionado | Headers, URLs | Field IDs en proto |
| Para el curso | ✅ Más fácil de demostrar | ❌ Más complejo de mostrar |

**Nuestra decisión:** REST para OrderFlow porque el rendimiento de REST en el volumen del curso (50.000 pedidos/mes) es más que suficiente. gRPC tiene sentido cuando tienes millones de llamadas internas al día y cada ms importa.

## DelegatingHandlers: middleware para HttpClient

Los `DelegatingHandler` son el equivalente al middleware de ASP.NET Core pero para peticiones HTTP salientes. Se encadenan igual:

```csharp
// Cadena de DelegatingHandlers
HttpClient
  → CorrelationIdDelegatingHandler (añade X-Correlation-Id)
  → AuthTokenDelegatingHandler     (añade Authorization header)
```

```csharp
    .AddHttpMessageHandler<CorrelationIdDelegatingHandler>()
    .AddHttpMessageHandler<LoggingDelegatingHandler>()
```

## LoggingDelegatingHandler: loggear todas las llamadas salientes

```csharp
// Infrastructure/Http/LoggingDelegatingHandler.cs

public class LoggingDelegatingHandler : DelegatingHandler
{
    private readonly ILogger<LoggingDelegatingHandler> _logger;

    public LoggingDelegatingHandler(ILogger<LoggingDelegatingHandler> logger)
        => _logger = logger;

    protected override async Task<HttpResponseMessage> SendAsync(
        HttpRequestMessage request, CancellationToken ct)
    {
        var sw = Stopwatch.StartNew();

        _logger.LogInformation(
            "HTTP {Method} {Uri}",
            request.Method, request.RequestUri);

        try
        {
            var response = await base.SendAsync(request, ct);
            sw.Stop();

            _logger.LogInformation(
                "HTTP {Method} {Uri} → {StatusCode} ({ElapsedMs}ms)",
                request.Method, request.RequestUri,
                (int)response.StatusCode, sw.ElapsedMilliseconds);

            return response;
        }
        catch (Exception ex)
        {
            sw.Stop();
            _logger.LogError(ex,
                "HTTP {Method} {Uri} FAILED after {ElapsedMs}ms",
                request.Method, request.RequestUri, sw.ElapsedMilliseconds);
            throw;
        }
    }
}
```

## Caché HTTP: Cache-Control headers

Para optimizar las llamadas a Products, podemos usar los headers HTTP estándar de caché:

```csharp
// En ProductsClient — verificar Cache-Control de la response
public async Task<ProductDetailDto?> GetProductAsync(
    Guid productId, CancellationToken ct = default)
{
    var response = await _httpClient.GetAsync(
        $"/api/products/{productId}", ct);

    if (!response.IsSuccessStatusCode) return null;

    // Respetar el Cache-Control de Products.API
    var maxAge = response.Headers.CacheControl?.MaxAge;
    var ttl = maxAge ?? TimeSpan.FromMinutes(5); // Default 5 min si no hay header

    var product = await response.Content
        .ReadFromJsonAsync<ProductDetailDto>(ct);

    if (product is not null)
        _cache.Set($"product:{productId}", product, ttl);

    return product;
}
```

En Products.API, podemos añadir Cache-Control:
```csharp
group.MapGet("/{id:guid}", GetProductEndpoint.Handle)
     .CacheOutput(p => p.Expire(TimeSpan.FromMinutes(5)));
```

## Invalidación de caché: el problema difícil

La caché en ProductsClient tiene un problema: si el stock de un producto cambia en Products.API, Orders sigue viendo el stock antiguo durante 5 minutos.

**Soluciones:**

**Opción A: TTL corto (1-2 minutos)**
- Simple de implementar
- Ligero lag en datos de stock
- Para TechShop con 50K pedidos/mes: aceptable

**Opción B: Cache invalidation por evento**
- Products publica evento `StockUpdated`
- Orders consume el evento y limpia la caché
- Más complejo, data siempre fresca
- Requiere mensajería (M4.2)

**Opción C: Sin caché para stock, caché para precio**
- El precio cambia con poca frecuencia → caché 24h
- El stock cambia con cada pedido → sin caché
- Balance entre frescura y rendimiento

**Para el curso:** Opción A. En producción real: Opción C o B según el volumen.

## ETag y Conditional Requests

Los ETags permiten al cliente verificar si un recurso cambió sin descargarlo de nuevo:

```csharp
// En ProductsClient — usar ETags
private readonly Dictionary<Guid, string> _eTags = new();

public async Task<ProductDetailDto?> GetProductAsync(
    Guid productId, CancellationToken ct = default)
{
    var request = new HttpRequestMessage(
        HttpMethod.Get, $"/api/products/{productId}");

    // Si tenemos un ETag guardado, enviarlo
    if (_eTags.TryGetValue(productId, out var etag))
        request.Headers.IfNoneMatch.Add(new EntityTagHeaderValue(etag));

    var response = await _httpClient.SendAsync(request, ct);

    if (response.StatusCode == HttpStatusCode.NotModified)
    {
        // El producto no cambió — usar la versión en caché
        return _cache.Get<ProductDetailDto>($"product:{productId}");
    }

    // Guardar el nuevo ETag para la próxima llamada
    if (response.Headers.ETag is not null)
        _eTags[productId] = response.Headers.ETag.Tag;

    var product = await response.Content
        .ReadFromJsonAsync<ProductDetailDto>(ct);

    _cache.Set($"product:{productId}", product, TimeSpan.FromMinutes(10));
    return product;
}
```

Con ETags, si Products devuelve 304 Not Modified, no transferimos el body — ahorramos ancho de banda.

## El anti-pattern: el God Client

Un error frecuente: meter toda la lógica de comunicación en un único cliente:

```csharp
// ❌ God Client — demasiada responsabilidad
```

Cada cliente es independiente, testeable y deployable por separado.

## Anti-patterns de comunicación síncrona entre microservicios

Errores comunes que convierten una arquitectura de microservicios en un monolito distribuido:

**Anti-pattern 1: Llamadas en cadena excesivas**
```
Orders → Products → Categories → Brands → ...
```
Si Orders necesita que Products llame a otros servicios para responder, la latencia se multiplica y el acoplamiento aumenta. Diseña los servicios para responder con sus propios datos.

**Anti-pattern 2: Llamadas para todos los ítems en bucle**
```csharp
// ❌ N+1 problem — una llamada por ítem
foreach (var item in order.Items)
{
    var product = await _productsClient.GetProductAsync(item.ProductId);
}
// Si el pedido tiene 10 ítems → 10 llamadas HTTP
```

**La solución:**
```csharp
// ✅ Llamada batch — una llamada para todos
var productIds = order.Items.Select(i => i.ProductId).ToList();
var products = await _productsClient.GetProductsByIdsAsync(productIds);
// 1 llamada HTTP, 10 productos
```

Añadir el endpoint batch en Products.API:
```csharp
// GET /api/products/batch?ids=guid1,guid2,...
group.MapGet("/batch", async (
    [FromQuery] string ids,
    IProductRepository repo, CancellationToken ct) =>
{
    var productIds = ids.Split(',').Select(Guid.Parse).ToList();
    var products = await repo.GetByIdsAsync(productIds, ct);
    return TypedResults.Ok(products.Select(p => p.ToResponse()));
});
```

## Versionado de API: gestionar cambios sin romper clientes

Cuando Products.API necesita cambiar un endpoint que Orders.API consume, el versionado evita breaking changes:

**Estrategia para OrderFlow:**

```csharp
// Products.API — añadir v2 sin eliminar v1
// v1 sigue respondiendo: GET /api/products/{id} → ProductResponse v1
// v2 añade campos nuevos: GET /api/v2/products/{id} → ProductResponse v2

// En Orders.API — migrar gradualmente
// 1. Orders apunta a v1
// 2. Products despliega v2 (v1 sigue funcionando)
// 3. Orders actualiza el cliente a v2
// 4. Orders despliega
// 5. Products elimina v1 (cuando nadie lo usa)
```

Esta estrategia se llama **Expand and Contract** (la veremos en profundidad en M6.4 para datos). La misma idea aplica a APIs.

---

## Versioning del cliente: gestionar contratos cambiantes

Un desafío real en producción: Products.API v2 añade un campo nuevo. ¿Cómo maneja Orders.API la transición?

**El principio de tolerancia:**
```csharp
// ProductDetailDto en Orders.API
public record ProductDetailDto(
    Guid Id,
    string Name,
    decimal Price,
    string Currency,
    int Stock,
    bool IsActive,
    Guid CategoryId,
    // Campos nuevos de v2 (opcionales para compatibilidad hacia atrás)
    string? Brand = null,        // Nullable — no rompe si viene null de v1
    decimal? OriginalPrice = null // Nullable — opcional
);
```

Si Products.API v2 añade `Brand` y `OriginalPrice`, y Orders.API tiene estos campos como nullable, la deserialización funciona con ambas versiones de Products:
- Products v1 responde sin esos campos → `null` en Orders (aceptable)
- Products v2 responde con esos campos → valores reales en Orders

Este patrón se llama **Tolerant Reader** — el consumidor es tolerante a cambios en el productor.

## El patrón Request-Response vs Fire and Forget

Dos variantes de comunicación síncrona:

**Request-Response (lo que hemos visto):**
```csharp
// Orders espera la respuesta de Products
var product = await _productsClient.GetProductAsync(productId, ct);
// El hilo de Orders está bloqueado esperando
// Continúa solo cuando Products responde
```

**Fire and Forget (semi-síncrono):**
```csharp
// Orders envía el request pero no espera (poco común en microservicios)
_ = _productsClient.NotifyLowStockAsync(productId);
// El hilo de Orders continúa inmediatamente
// El request viaja en segundo plano
```

Fire and Forget es problemático: si el request falla, nadie lo sabe. Mejor usar mensajería asíncrona (M4.2) para estos casos.

**En OrderFlow:** siempre Request-Response para comunicación síncrona. Para los casos donde no necesitamos la respuesta inmediata → mensajería asíncrona.

## Comparativa final: REST vs gRPC en el contexto de OrderFlow

Después de ver ambas opciones, la decisión final para el proyecto:

**Por qué REST en OrderFlow:**

1. **Volumen:** 50.000 pedidos/mes = ~0.6 req/s. REST a 4M req/s tiene un headroom de x6.6M. El rendimiento de gRPC no aporta nada.

2. **Debug:** El equipo de TechShop (18 devs) puede depurar REST con el navegador, curl, Postman o .http files. gRPC requiere herramientas adicionales.

3. **Clientes externos:** YARP (el API Gateway) trabaja nativamente con HTTP REST. Añadir gRPC requeriría gRPC-Web o una capa de traducción.

4. **Aprendizaje del curso:** REST es conocido. Dedicar tiempo a gRPC en un curso de microservicios introductorio diluye el aprendizaje.

**Cuándo gRPC sería la elección correcta:**
- Servicios con millones de llamadas internas diarias (Streaming de video, trading)
- Equipos con experiencia en gRPC
- Cuando el tamaño del payload impacta el ancho de banda

**La lección:** Elegir la tecnología correcta para el contexto. No la más moderna, no la más potente — la más adecuada.

## Resumen de M4.1: comunicación síncrona lista

En M4.1 hemos establecido la comunicación síncrona entre Orders.API y Products.API:

**HttpClientFactory:**
El estándar para crear clientes HTTP en .NET. Gestiona el pool de sockets, el DNS y el ciclo de vida. Siempre usado como Typed Client o Named Client, nunca creando `HttpClient` directamente.

**ProductsClient (Typed Client):**
Clase dedicada a la comunicación con Products.API. Encapsula la URL, la serialización, el manejo de errores, la caché y la resiliencia. Orders.API solo ve una interfaz limpia.

**Polly integrado:**
El mismo pipeline de M3.5 aplicado al HttpClient. Orders.API es resiliente a fallos de Products.API.

**CorrelationId propagado:**
Cada llamada de Orders a Products lleva el mismo CorrelationId. Los logs de ambos servicios están correlacionados.

**gRPC (concepto):**
Alternativa de alto rendimiento a REST para comunicación interna. Contrato en `.proto`, código generado automáticamente. Elegimos REST para el curso por accesibilidad; gRPC para sistemas de alto throughput.

**La comunicación síncrona ya funciona.** En M4.2 añadimos la asíncrona.
