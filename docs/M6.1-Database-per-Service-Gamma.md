## Bienvenidos al Módulo 6

Los módulos anteriores construyeron la comunicación y la seguridad de OrderFlow. El Módulo 6 se centra en los datos: el reto más complejo de los microservicios.

**El problema central:** En un monolito, una sola base de datos garantiza consistencia ACID. En microservicios, cada servicio tiene su propia BD. ¿Cómo gestionamos eso?

En M6.1 cubrimos el patrón fundamental:
- Por qué Database per Service y no una BD compartida
- Cómo TechShop tiene 4 BDs SQL Server en el mismo servidor
- Múltiples `DbContext` en la misma solution de .NET 10
- Migraciones sin downtime con `Migration Bundles`
- Estrategias para datos que parecen compartidos

Al terminar M6.1, OrderFlow tiene 4 bases de datos completamente independientes, cada una gestionada por su propio servicio.

## El antipatrón: Shared Database

El error más común al migrar a microservicios: mantener una sola base de datos para todos los servicios.

```
┌─────────────────────────────────────────────┐
│              Shared Database ❌              │
│  Orders Orders   Products Products  Payments │
│  Table  Lines    Table    Categories Table   │
└─────────────────────────────────────────────┘
       ↑                ↑               ↑
  Orders.API       Products.API    Payments.API
```

**Los problemas del Shared Database:**

- **Acoplamiento de esquema:** Si Products.API cambia la tabla Products, puede romper Orders.API sin que nadie lo sepa hasta producción.
- **Sin autonomía de deploy:** Para cambiar el esquema de Products, necesitas coordinar con el equipo de Orders.
- **Cuello de botella:** La BD compartida escala peor que BDs independientes.
- **Sin aislamiento de fallos:** Un query lento de Analytics bloquea las tablas de Orders.

Es microservicios en el nombre, monolito en los datos.

---

## El patrón correcto: Database per Service

Cada servicio tiene su propia base de datos, invisible para los demás:

```
Orders.API ──── OrdersDb ────────────────────── SQL Server
Products.API ── ProductsDb ──────────────────── SQL Server
Payments.API ── PaymentsDb ──────────────────── SQL Server
Notifications── NotificationsDb ─────────────── SQL Server
```

**Las reglas:**
1. Solo Orders.API puede leer y escribir en OrdersDb
2. Products.API NO tiene acceso a OrdersDb, ni siquiera en modo lectura
3. Si Orders necesita datos de Products, los pide mediante la API (HTTP o mensajes)
4. Las migraciones de OrdersDb las gestiona el equipo de Orders

**El resultado:** Los servicios son verdaderamente independientes. Products puede cambiar su esquema sin afectar a Orders. Orders puede escalar su BD sin tocar la de Products.

## Database per Service en SQL Server: esquemas

Para TechShop, usamos SQL Server con **esquemas separados** dentro del mismo servidor físico. Más económico que cuatro servidores, pero con el mismo aislamiento lógico:

```sql
-- Un servidor SQL, cuatro esquemas (o cuatro BDs)
Server: (localdb)\MSSQLLocalDB  (desarrollo)
Server: orderflow-sql.database.windows.net  (Azure SQL)

Opción A — 4 bases de datos:
  OrdersDb     → orders.Orders, orders.OrderLines
  ProductsDb   → products.Products, products.Categories
  PaymentsDb   → payments.Payments
  NotificationsDb → notifications.Notifications

Opción B — 1 BD, 4 esquemas:
  OrderFlowDb
  → orders.Orders, orders.OrderLines
  → products.Products, products.Categories
  → payments.Payments
  → notifications.Notifications
```

**Para OrderFlow:** Opción A (4 BDs) en Azure (elastic pool) y Opción B (esquemas) en LocalDB para desarrollo. El código no cambia — solo la connection string.

## El DbContext de cada servicio

Cada servicio tiene su propio `DbContext`, completamente independiente:

```csharp
// Orders.API/Infrastructure/Persistence/OrderDbContext.cs

namespace Orders.API.Infrastructure.Persistence;

public class OrderDbContext : DbContext
{
    public OrderDbContext(DbContextOptions<OrderDbContext> options)
        : base(options) { }

    // Solo las entidades de Orders — sin acceso a Products
    protected override void OnModelCreating(ModelBuilder modelBuilder)
    {
        modelBuilder.HasDefaultSchema("orders");
        modelBuilder.ApplyConfigurationsFromAssembly(typeof(OrderDbContext).Assembly);
    }
}
```

## Repository Pattern: interfaces vs DbContext directo

Hay debate sobre si usar el Repository Pattern con EF Core o acceder al DbContext directamente en los handlers:

**Con Repository Pattern (nuestra elección):**
```csharp
public class CreateOrderHandler : IRequestHandler<CreateOrderCommand, OrderDto>
{
    private readonly IOrderRepository _repository; // ← Interfaz
    // Fácil de mockear en tests
}
```

**Sin Repository (DbContext directo):**
```csharp
public class CreateOrderHandler : IRequestHandler<CreateOrderCommand, OrderDto>
{
    private readonly OrderDbContext _dbContext; // ← Concreto
    // Más simple, menos capas, EF Core ya es un patrón Repository
}
```

**Por qué el Repository en OrderFlow:**
- Los tests de unidad del handler usan `NSubstitute.For<IOrderRepository>()` sin BD
- El dominio no depende de EF Core — Clean Architecture
- Si en el futuro migramos a Dapper para el read model (M6.3), la interfaz no cambia

Para servicios simples como Notifications, el DbContext directo es suficiente.

## Fluent API: configuraciones de entidad separadas

En vez de configurar todo en `OnModelCreating`, usar clases de configuración separadas — más limpio y escalable:

```csharp
// Orders.API/Infrastructure/Persistence/Configurations/OrderConfiguration.cs

public class OrderConfiguration : IEntityTypeConfiguration<Order>
{
    public void Configure(EntityTypeBuilder<Order> builder)
    {
        builder.ToTable("Orders", "orders");
        builder.HasKey(o => o.Id);
        builder.Property(o => o.Id).ValueGeneratedNever();

        // Status como columna string
        builder.Property(o => o.Status)
            .HasConversion(
                s => s.Value,
                v => OrderStatus.FromString(v))
            .HasColumnName("Status")
            .HasMaxLength(20)
            .IsRequired();

        // Money como owned entity (dos columnas)
        builder.OwnsOne(o => o.Total, money =>
        {
            money.Property(m => m.Amount)
                .HasColumnName("TotalAmount")
                .HasPrecision(18, 2)
                .IsRequired();
            money.Property(m => m.Currency)
                .HasColumnName("TotalCurrency")
                .HasMaxLength(3)
                .IsRequired();
        });

        // Address como owned entity
        builder.OwnsOne(o => o.ShippingAddress, addr =>
        {
            addr.Property(a => a.Street).HasColumnName("Street").HasMaxLength(200);
            addr.Property(a => a.City).HasColumnName("City").HasMaxLength(100);
            addr.Property(a => a.ZipCode).HasColumnName("ZipCode").HasMaxLength(20);
            addr.Property(a => a.Country).HasColumnName("Country").HasMaxLength(2);
        });

        // Índices para queries frecuentes
        builder.HasIndex(o => o.CustomerId);
        builder.HasIndex(o => o.Status);
        builder.HasIndex(o => o.CreatedAt);

        // Concurrencia optimista
        builder.Property<byte[]>("RowVersion")
            .IsRowVersion()
            .IsConcurrencyToken();
    }
}
```

```csharp
        builder.Property(l => l.Quantity).IsRequired();

    }
```

## Registrar los DbContext en cada servicio

```csharp
// Orders.API/Program.cs

// EF Core con LocalDB en desarrollo, Azure SQL en producción
builder.Services.AddDbContext<OrderDbContext>(options =>
{
    var connectionString = builder.Configuration
        .GetConnectionString("OrdersDb")!;

    options.UseSqlServer(connectionString, sqlOptions =>
    {
        // Retry automático para transient errors de Azure SQL
        sqlOptions.EnableRetryOnFailure(
            maxRetryCount: 5,
            maxRetryDelay: TimeSpan.FromSeconds(30),
            errorNumbersToAdd: null);

        // Timeout de comando (default: 30s)
        sqlOptions.CommandTimeout(60);
    });

    // En desarrollo: logging de SQL
    if (builder.Environment.IsDevelopment())
    {
        options.EnableSensitiveDataLogging();
        options.EnableDetailedErrors();
    }
});
```

## Crear y aplicar migraciones: el flujo diario

Con cuatro `DbContext` independientes, las migraciones se gestionan por separado:

```bash
# Migration para OrdersDB
dotnet ef migrations add InitialOrderSchema \
    --project src/Orders.API \
    --startup-project src/Orders.API \
    --context OrderDbContext \
    --output-dir Infrastructure/Persistence/Migrations

# Migration para ProductsDB
dotnet ef migrations add InitialProductSchema \
    --project src/Products.API \
    --startup-project src/Products.API \
    --context ProductDbContext \
    --output-dir Infrastructure/Persistence/Migrations

# Aplicar a LocalDB
dotnet ef database update \
    --project src/Orders.API \
    --context OrderDbContext

dotnet ef database update \
    --project src/Products.API \
    --context ProductDbContext
```

Cada equipo gestiona sus propias migraciones. El equipo de Orders no sabe ni que existe la migration de Products.

## Aplicar migraciones al arrancar: ¿buena idea?

Un patrón común es aplicar migraciones al arrancar el servicio:

```csharp
// ❌ Cuidado — puede fallar en producción con múltiples instancias

var app = builder.Build();

using (var scope = app.Services.CreateScope())
{
    var dbContext = scope.ServiceProvider
        .GetRequiredService<OrderDbContext>();
    await dbContext.Database.MigrateAsync();
}

app.Run();
```

**El problema:** Con 3 instancias arrancando simultáneamente, las 3 intentan aplicar la misma migration a la vez. Puede generar condiciones de carrera.

**La solución correcta:** Migration Bundles en el pipeline de CI/CD, no en el startup de la aplicación.

```csharp
// ✅ En producción: solo verificar que las migraciones están aplicadas
using (var scope = app.Services.CreateScope())
{
    var dbContext = scope.ServiceProvider
        .GetRequiredService<OrderDbContext>();
    var pending = await dbContext.Database.GetPendingMigrationsAsync();
    if (pending.Any())
        throw new InvalidOperationException(
            $"There are {pending.Count()} pending migrations. " +
            "Run the migration bundle before starting the service.");
}
```

Un Migration Bundle es un ejecutable que aplica las migraciones — ideal para pipelines de CI/CD:

## Estrategias de migración sin downtime

El reto real: cambiar el esquema de la BD sin que el servicio deje de funcionar.

**El escenario:** Orders.API necesita renombrar la columna `CustomerId` a `BuyerId`.

**Migración naive (CON downtime):**
```
1. Detener Orders.API
2. Aplicar migration: RENAME COLUMN CustomerId TO BuyerId
3. Desplegar nueva versión de Orders.API
4. Arrancar Orders.API
→ Downtime total
```

**Expand and Contract (SIN downtime):**
```
FASE 1 - EXPAND (la BD tiene ambas columnas):
  - Añadir columna BuyerId (nullable)
  - Desplegar versión que escribe en AMBAS columnas

FASE 2 - MIGRATE (copiar datos):
  UPDATE Orders SET BuyerId = CustomerId WHERE BuyerId IS NULL
  - Hacer BuyerId NOT NULL

FASE 3 - CONTRACT (eliminar la columna vieja):
  - Desplegar versión que solo lee/escribe BuyerId
  - Aplicar migration: DROP COLUMN CustomerId
→ Cero downtime en cada fase
```

## Expand and Contract en código EF Core

```csharp
// FASE 1 — Migration que añade BuyerId
public partial class AddBuyerIdColumn : Migration
{
    protected override void Up(MigrationBuilder migrationBuilder)
    {
        // Añadir la nueva columna como nullable
        migrationBuilder.AddColumn<Guid>(
            name: "BuyerId",
            schema: "orders",
            table: "Orders",
            nullable: true); // ← Nullable para compatibilidad

        // Copiar datos en la misma migration
        migrationBuilder.Sql(
            "UPDATE orders.Orders SET BuyerId = CustomerId");

        // Hacer NOT NULL después de copiar
        migrationBuilder.AlterColumn<Guid>(
            name: "BuyerId",
            schema: "orders",
            table: "Orders",
            nullable: false);
    }
}

// FASE 3 — Migration que elimina CustomerId (en otra release)
public partial class RemoveCustomerIdColumn : Migration
{
    protected override void Up(MigrationBuilder migrationBuilder)
    {
        migrationBuilder.DropColumn(
            name: "CustomerId",
            schema: "orders",
            table: "Orders");
    }
}
```

## El problema de los datos "compartidos": el catálogo de productos

Orders.API necesita mostrar el nombre y precio de cada producto en el detalle del pedido. Pero OrdersDB no tiene una tabla de productos — eso es de ProductsDB.

**Las opciones:**

**Opción A — Llamada HTTP en tiempo real:**
```
Cliente pide GET /api/orders/123
Orders busca el pedido → tiene solo ProductId
Orders llama a Products.API para obtener el nombre
Devuelve el pedido con el nombre
```
Problema: Si Products.API está caído → Orders no puede mostrar el detalle del pedido.

**Opción B — Desnormalizar al escribir (nuestra elección):**
```
Cuando se crea el pedido:
Orders guarda ProductId + ProductName + UnitPrice en OrderLines
Más tarde, Products cambia el nombre del producto
Orders sigue mostrando el nombre que tenía cuando se hizo el pedido
```

Para pedidos: correcto — el nombre al momento de compra es lo que importa.

## Desnormalización: guardar lo que necesitas en tu propia BD

La desnormalización es la técnica clave para el Database per Service:

```csharp
// En OrderLine — guardamos todo lo que necesitamos en el momento de la compra
public class OrderLine
{
    public Guid Id { get; private set; }
    public Guid ProductId { get; private set; }       // Referencia (solo el ID)
    public string ProductName { get; private set; }   // Desnormalizado
    public string? ProductSku { get; private set; }   // Desnormalizado
    public int Quantity { get; private set; }
    public Money UnitPrice { get; private set; }      // Precio al momento de compra
    public Money LineTotal => UnitPrice.Multiply(Quantity);
}
```

**Qué guardamos:** el nombre, el SKU, el precio al momento de la compra.
**Qué NO guardamos:** la descripción larga, las imágenes, el stock — no los necesitamos en el pedido.
**La referencia:** solo el `ProductId` — si necesitamos el producto actual, llamamos a Products.API.

---

## El dilema: ¿referencia o copia?

Decidir qué desnormalizar y qué solo referenciar:

| Dato | ¿Copia o referencia? | Razón |
|---|---|---|
| Nombre del producto en el pedido | **Copia** | El nombre puede cambiar. El pedido muestra el nombre cuando se hizo |
| Precio unitario en el pedido | **Copia** | El precio puede cambiar. El pedido tiene el precio que pagó el cliente |
| Stock actual del producto | **Referencia** | El stock cambia constantemente. Solo tiene sentido el valor actual |
| Descripción del producto | **Referencia** | No la necesitamos en el pedido. Llamamos a Products si el usuario quiere verla |
| País de envío del pedido | **Copia** | No debe cambiar si el país cambia |
| Email del cliente | **Copia** | El email puede cambiar pero el pedido ya fue enviado a ese email |

La regla: copia si el valor histórico importa, referencia si solo importa el valor actual.

## Manteniendo el ProductName actualizado: eventos de dominio

¿Qué pasa si Products.API cambia el nombre de un producto y Orders.API tiene ese nombre guardado en sus pedidos cerrados?

Para pedidos ya completados: no importa — el pedido muestra el nombre en el momento de la compra.

Para el catálogo de Orders (si tuviéramos una vista de "productos disponibles"):

```csharp
// Products.API publica un evento cuando cambia el nombre
public record ProductNameUpdated
{
    public Guid ProductId { get; init; }
    public string OldName { get; init; } = string.Empty;
    public string NewName { get; init; } = string.Empty;
}

// Orders.API tiene un consumer que actualiza su caché
public class ProductNameUpdatedConsumer : IConsumer<ProductNameUpdated>
{
    private readonly IMemoryCache _cache;

    public async Task Consume(ConsumeContext<ProductNameUpdated> context)
    {
        // Invalidar la caché del producto
        _cache.Remove($"product:{context.Message.ProductId}");
    }
}
```

## Elastic Pool: una BD por servicio en Azure sin coste multiplicado

En Azure, 4 bases de datos podrían costar 4x el precio de una. **Azure SQL Elastic Pool** resuelve esto:

```
Sin Elastic Pool:
  OrdersDb: 5 DTUs = 5€/mes
  ProductsDb: 5 DTUs = 5€/mes
  PaymentsDb: 5 DTUs = 5€/mes
  NotificationsDb: 5 DTUs = 5€/mes
  Total: 20€/mes
  (No aprovecha que no todas están al 100% a la vez)

Con Elastic Pool:
  Pool: 50 eDTUs compartidos = 9€/mes
  OrdersDb: usa los eDTUs del pool cuando los necesita
  ProductsDb: usa los eDTUs del pool cuando los necesita
  Total: 9€/mes
  (Pico de Orders no afecta a Products si hay suficientes eDTUs)
```

Para OrderFlow en producción inicial: Elastic Pool con 50 eDTUs (~9€/mes) es suficiente para las 4 BDs.

## Crear el Elastic Pool con Azure CLI

```bash
# Crear el servidor SQL

        --server orderflow-sql \
        --name $db \
        --elastic-pool orderflow-pool
done
```

## Auditar cambios con EF Core: Interceptors

```csharp
// Infrastructure/Persistence/Interceptors/AuditInterceptor.cs

public class AuditInterceptor : SaveChangesInterceptor
{
    private readonly ICurrentUser _currentUser;

    public override ValueTask<InterceptionResult<int>> SavingChangesAsync(
        DbContextEventData eventData,
        InterceptionResult<int> result,
        CancellationToken ct = default)
    {
        if (eventData.Context is null) return new(result);

        var entries = eventData.Context.ChangeTracker.Entries()
            .Where(e => e.State is EntityState.Added
                     or EntityState.Modified
                     or EntityState.Deleted);

        foreach (var entry in entries)
        {
            // Añadir audit fields automáticamente
            if (entry.Entity is IAuditableEntity auditable)
            {
                if (entry.State == EntityState.Added)
                    auditable.CreatedAt = DateTime.UtcNow;
                auditable.UpdatedAt = DateTime.UtcNow;
                auditable.UpdatedBy = _currentUser.Id;
            }
        }

        return new(result);
    }
}

// Registrar en Program.cs
builder.Services.AddDbContext<OrderDbContext>((sp, options) =>
{
    options.UseSqlServer(connectionString)
           .AddInterceptors(sp.GetRequiredService<AuditInterceptor>());
});
```

## Consultas optimizadas: Specification Pattern

Para consultas complejas con múltiples filtros y ordenaciones:

```csharp
// Application/Specifications/OrderSpecification.cs

public class OrdersByCustomerSpecification : Specification<Order>
{
    public OrdersByCustomerSpecification(
        Guid customerId, OrderStatus? status,
        int page, int pageSize)
    {
        // Filtros
        Query.Where(o => o.CustomerId == customerId);

        if (status is not null)
            Query.Where(o => o.Status == status);

        // Ordenar por más reciente primero
        Query.OrderByDescending(o => o.CreatedAt);

        // Incluir las líneas
        Query.Include(o => o.Lines);

        // Paginación
        Query.Skip((page - 1) * pageSize).Take(pageSize);
    }
}

// Uso en el Repository
public async Task<IReadOnlyList<Order>> ListAsync(
    Specification<Order> spec, CancellationToken ct)
{
    return await ApplySpecification(spec).ToListAsync(ct);
}
```

## Índices: el impacto en rendimiento

Los índices correctos pueden marcar la diferencia entre 2ms y 2 segundos:

```csharp
// En OrderConfiguration — índices basados en los queries frecuentes
```

## Concurrencia optimista: RowVersion en SQL Server

EF Core gestiona la concurrencia optimista con `RowVersion`:

```csharp
// En OrderConfiguration
builder.Property<byte[]>("RowVersion")
    .IsRowVersion()
    .IsConcurrencyToken();

// ¿Qué hace RowVersion?
// SQL Server añade una columna que cambia en cada UPDATE
// Si dos usuarios leen el mismo pedido (RowVersion = 0x001)
// y ambos intentan guardarlo:
// → El primero guarda (RowVersion pasa a 0x002)
// → El segundo falla: DbUpdateConcurrencyException
//   "el RowVersion que tengo (0x001) no coincide con el actual (0x002)"

// Manejar el conflicto
try
{
    await _dbContext.SaveChangesAsync(ct);
}
catch (DbUpdateConcurrencyException ex)
{
    // Recargar la entidad y reintentar
    await ex.Entries.Single().ReloadAsync(ct);
    throw new DomainException("The order was modified by another user. Please retry.");
}
```

## Transacciones explícitas en EF Core

Cuando un handler necesita guardar en múltiples tablas atomicamente:

```csharp
// En TransactionBehavior — ya lo teníamos, pero con más detalle

public async Task<TResponse> Handle(
    TRequest request,
    RequestHandlerDelegate<TResponse> next,
    CancellationToken ct)
{
    // Iniciar transacción explícita
    await using var transaction = await _dbContext.Database
        .BeginTransactionAsync(IsolationLevel.ReadCommitted, ct);

    try
    {
        var response = await next();

        // Commit — todos los cambios o ninguno
        await _dbContext.SaveChangesAsync(ct);
        await transaction.CommitAsync(ct);

        return response;
    }
    catch
    {
        await transaction.RollbackAsync(ct);
        throw;
    }
}
```

**Nivel de aislamiento `ReadCommitted`:** El más equilibrado. Lee solo datos commiteados. Evita dirty reads sin ser tan restrictivo como Serializable.

## Lazy Loading vs Eager Loading: la elección correcta

EF Core soporta dos estrategias de carga de entidades relacionadas:

**Lazy Loading — carga cuando se accede:**
```csharp
// ❌ En microservicios: desactivar siempre
// Cada acceso a una propiedad navegacional genera una query SQL
// En un bucle: N queries para N pedidos (N+1 problem)
var orders = await _dbContext.Orders.ToListAsync();
foreach (var order in orders)
{
    var lines = order.Lines; // ← Query SQL aquí (1 por pedido)
}
// Con 100 pedidos: 101 queries en vez de 2
```

**Eager Loading — carga en la misma query:**
```csharp
// ✅ En microservicios: siempre eager loading explícito
var orders = await _dbContext.Orders
    .Include(o => o.Lines) // ← Un JOIN, una query
    .Where(o => o.CustomerId == customerId)
    .ToListAsync();
// Con 100 pedidos: 2 queries totales (Orders + Lines)
```

En OrderFlow, lazy loading está desactivado (no se instala el paquete `Microsoft.EntityFrameworkCore.Proxies`). Todo eager loading.

## Split Queries: para colecciones grandes

Cuando un pedido tiene muchas líneas y muchas propiedades, el JOIN puede ser caro:

```csharp
// Para pedidos con muchas líneas — separar en 2 queries
var order = await _dbContext.Orders
    .Include(o => o.Lines)
    .AsSplitQuery() // ← 2 queries separadas en vez de 1 JOIN grande
    .FirstOrDefaultAsync(o => o.Id == orderId, ct);

// Genera:
// Query 1: SELECT * FROM orders.Orders WHERE Id = @orderId
// Query 2: SELECT * FROM orders.OrderLines WHERE OrderId = @orderId
// Más eficiente que un JOIN cuando hay muchas líneas
```

**Cuándo usar AsSplitQuery:** Cuando el JOIN genera muchas filas duplicadas. Para Orders con pocas líneas (< 20), el JOIN normal es más eficiente.

## Proyecciones: leer solo lo que necesitas

Para el listado de pedidos, no necesitamos cargar todo el aggregate:

```csharp
// ❌ Carga todo el aggregate (ineficiente para listados)

    .Where(o => o.CustomerId == customerId)

// ✅ Proyectar solo lo necesario para el listado
```

## Soft Delete: archivar en vez de borrar

Los pedidos nunca se borran físicamente — se archivan:

## Gestión del ciclo de vida del DbContext

En ASP.NET Core, el DbContext se registra como `Scoped` por defecto — una instancia por request HTTP. Pero con MassTransit, el ciclo de vida es diferente:

## Connection Resiliency: reintentos automáticos para Azure SQL

Azure SQL puede tener fallos transitorios (throttling, reconexiones). EF Core los maneja automáticamente:

```csharp
builder.Services.AddDbContext<OrderDbContext>(options =>
    options.UseSqlServer(connectionString, sqlOptions =>
    {
        // Política de reintento automático para errores transitorios
        sqlOptions.EnableRetryOnFailure(
            maxRetryCount: 5,
            maxRetryDelay: TimeSpan.FromSeconds(30),
            errorNumbersToAdd: null); // null = usar los errores por defecto de Azure SQL

        // Timeout de cada comando SQL (default: 30s)
        sqlOptions.CommandTimeout(60);
    }));
```

Los errores transitorios que EF Core reintenta automáticamente incluyen:
- Timeouts de conexión (error 10060)
- Throttling de Azure SQL (error 40501)
- Pérdida de conexión temporal (error -2)

Con 5 reintentos y backoff exponencial hasta 30s, la mayoría de errores transitorios se resuelven solos.

## La transición del monolito: migrando la BD de TechShop

Si TechShop tuviera un monolito con una sola BD, la migración a Database per Service sería:

```
FASE 1 — Strangler Fig (M1.2 lo explicamos):
  Monolito sigue con su BD monolítica

  Los nuevos servicios se sincronizan via eventos
```

Para TechShop en el curso, partimos de cero — no hay monolito que migrar. Pero el 80% de los proyectos reales tienen esta migración como contexto.

## Preguntas frecuentes sobre Database per Service

**¿Puedo hacer un JOIN entre Orders y Products para un informe?**
No directamente. Para informes que cruzan servicios, tienes dos opciones: (1) un servicio de Analytics que consolida datos mediante eventos, o (2) un Read Model (M6.3) que agrega datos de múltiples fuentes.

**¿Qué pasa si necesito búsqueda full-text sobre todos los pedidos de todos los clientes?**
El servicio de Orders puede ofrecer una API de búsqueda que internamente consulta su propia BD. Si necesitas búsqueda cross-service, necesitas un índice dedicado (Elasticsearch o Azure Cognitive Search) alimentado por eventos.

**¿El Elastic Pool puede ser un cuello de botella?**
Sí, si los 4 servicios hacen queries intensivos simultáneamente. La solución: escalar el pool (más vCores) o separar BDs con más recursos individuales para los servicios críticos.

**¿Cómo hago backup y restauración si necesito un point-in-time consistente entre BDs?**
Azure SQL permite restaurar todas las BDs al mismo timestamp. Pero los eventos en RabbitMQ/Service Bus en ese periodo no se pueden restaurar — es el trade-off de la consistencia eventual.

## Lo que viene en M6.2

En M6.2 abordamos el reto más difícil del Database per Service: la consistencia entre bases de datos.

**El problema:** Orders guarda el pedido en OrdersDb Y publica `OrderCreated` a RabbitMQ. ¿Qué pasa si el proceso cae entre las dos operaciones? El pedido existe pero el evento nunca se publicó — la Saga nunca arranca.

**La solución:** El **Outbox Pattern**. El evento se guarda en la misma transacción que el pedido — en una tabla `OutboxMessages`. Un background service lo publica a RabbitMQ de forma garantizada.

Al terminar M6.2, OrderFlow tiene publicación de eventos atómicamente garantizada. No es posible tener un pedido sin su evento correspondiente.
