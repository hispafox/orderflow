## Application Insights: qué es

**Azure Application Insights** es el servicio de APM (Application Performance Monitoring) de Azure. Recibe telemetría de los servicios, la almacena y permite consultarla.

```
OrderFlow (5 servicios)
    │ OpenTelemetry OTLP
    ▼
Application Insights (Azure)
    │
    ├── Logs (todas las líneas de log de Serilog)
    ├── Traces (spans distribuidos de todos los servicios)
    ├── Metrics (latencias HTTP, CPU, memoria, errores)
    ├── Exceptions (stack traces completos)
    └── Custom Events (OrderCreated, PaymentProcessed, etc.)
```

En el dashboard de Aspire (M7.1) teníamos la misma información pero en local. Application Insights es el equivalente de producción — con retención, alertas y KQL.

## Conectar OpenTelemetry con Application Insights

```bash
# Instalar el exportador de Application Insights para OTel
dotnet add package Azure.Monitor.OpenTelemetry.AspNetCore
```

```csharp
// OrderFlow.ServiceDefaults/Extensions.cs — actualizar

public static IHostApplicationBuilder AddServiceDefaults(
    this IHostApplicationBuilder builder)
{
    builder.AddOpenTelemetryDefaults();
    // ...resto de configuración

    return builder;
}

public static IHostApplicationBuilder AddOpenTelemetryDefaults(
    this IHostApplicationBuilder builder)
{
    builder.Services.AddOpenTelemetry()
        .WithTracing(tracing => tracing
            .AddAspNetCoreInstrumentation()
            .AddHttpClientInstrumentation()
            .AddEntityFrameworkCoreInstrumentation()
            .AddSource("MassTransit")
            .AddSource("Orders.API")
            .AddOtlpExporter()) // ← Dashboard de Aspire (dev)

        .WithMetrics(metrics => metrics
            .AddAspNetCoreInstrumentation()
            .AddHttpClientInstrumentation()
            .AddRuntimeInstrumentation()
            .AddMeter("Microsoft.EntityFrameworkCore")
            .AddOtlpExporter()) // ← Dashboard de Aspire (dev)

        .WithLogging(logging => logging
            .AddOtlpExporter()); // ← Dashboard de Aspire (dev)

    // En producción: añadir Application Insights
    if (builder.Environment.IsProduction())
    {
        builder.Services.AddOpenTelemetry()
            .UseAzureMonitor(); // ← Application Insights (prod)
    }

    return builder;
}
```

## Sampling adaptativo: controlar el volumen de telemetría

```csharp
// ServiceDefaults — configurar sampling para no saturar Application Insights

builder.Services.AddOpenTelemetry()
    .WithTracing(tracing => tracing
        .AddAspNetCoreInstrumentation(options =>
        {
            // No trackear health checks — son muy frecuentes y poco útiles
            options.Filter = ctx =>
                !ctx.Request.Path.StartsWithSegments("/health") &&
                !ctx.Request.Path.StartsWithSegments("/alive");
        })
        .AddHttpClientInstrumentation()
        .AddEntityFrameworkCoreInstrumentation(options =>
        {
            // No loggear el SQL completo en producción (puede contener datos)
            options.SetDbStatementForText = false;  // Solo en desarrollo
        })
        // Sampling: 10% en producción, 100% en dev
        .AddAspNetCoreInstrumentation()
        .SetSampler(builder.Environment.IsProduction()
            ? new TraceIdRatioBasedSampler(0.1)  // 10%
            : new AlwaysOnSampler()));             // 100%
```

## Enriquecer las trazas con datos de negocio

```csharp
// Añadir atributos de negocio a las trazas automáticamente

public class OrderTelemetryEnricher : IStartupFilter
{
    public Action<IApplicationBuilder> Configure(Action<IApplicationBuilder> next)
    {
        return app =>
        {
            app.Use(async (context, nextMiddleware) =>
            {
                var activity = Activity.Current;
                if (activity != null)
                {
                    // Añadir el userId del JWT a todas las trazas
                    var userId = context.User?.FindFirstValue("sub");
                    if (userId != null)
                        activity.SetTag("user.id", userId);

                    // Añadir el correlationId del header
                    if (context.Request.Headers.TryGetValue(
                        "X-Correlation-Id", out var correlationId))
                        activity.SetTag("correlation.id", correlationId.ToString());
                }

                await nextMiddleware();
            });

            next(app);
        };
    }
}
```

## KQL: el lenguaje de consulta de Application Insights

**KQL (Kusto Query Language)** es el lenguaje con el que se consultan los datos de Application Insights. Sintaxis similar a SQL pero optimizada para datos de telemetría.

```kql
// Estructura básica de una query KQL:
NombreDeTabla
| where condición
| project columnas
| summarize agregación
| order by campo desc
| take N

// Las tablas principales en Application Insights:
// requests     → cada request HTTP recibido
// dependencies → llamadas salientes (HTTP, SQL, Service Bus)
// traces       → logs (de Serilog, ILogger)
// exceptions   → excepciones
// customEvents → eventos custom (OrderCreated, etc.)
// customMetrics → métricas custom
// pageViews    → (para web, no para APIs)
```

## KQL #1: los 5 endpoints más lentos

```kql
// Los endpoints más lentos en las últimas 24 horas
requests
| where timestamp > ago(24h)
| where success == true
| summarize
    P50 = percentile(duration, 50),
    P95 = percentile(duration, 95),
    P99 = percentile(duration, 99),
    Count = count()
  by name
| where Count > 100  // Solo endpoints con suficiente tráfico

```

## KQL #2: tasa de errores por servicio

```kql
// Tasa de errores en los últimos 7 días por servicio
requests
| where timestamp > ago(7d)
| summarize
    Total = count(),
    Errors = countif(success == false),
    ErrorRate = round(100.0 * countif(success == false) / count(), 2)
  by cloud_RoleName, bin(timestamp, 1d)
| order by timestamp desc, ErrorRate desc

// Si hay un pico de errores un día específico:
// → investigar con la query de excepciones en ese timestamp
```

## KQL #4: detectar anomalías de rendimiento

```kql
// Comparar el P95 de hoy vs la semana pasada
let today = requests
    | where timestamp > ago(1d)
    | summarize P95Today = percentile(duration, 95) by name;

let lastWeek = requests
    | where timestamp between (ago(8d) .. ago(7d))
    | summarize P95LastWeek = percentile(duration, 95) by name;

today
| join kind=inner lastWeek on name
| extend DegradationPct = round(100.0 * (P95Today - P95LastWeek) / P95LastWeek, 1)
| where DegradationPct > 20  // Endpoints degradados más de un 20%
| order by DegradationPct desc

// Si POST /api/orders degradó un 45%:
// → revisar si hubo algún deploy o cambio de schema ese día
```

// Trackear eventos de negocio importantes

// En CreateOrderHandler — evento custom al crear pedido
public async Task<OrderDto> Handle(CreateOrderCommand cmd, CancellationToken ct)
{
    var order = Order.Create(...);
    await _repository.SaveAsync(order, ct);

}

## Alertas: SLOs de TechShop

Un **SLO (Service Level Objective)** es el objetivo de calidad del servicio. TechShop define estos SLOs:

```
SLO 1 — Disponibilidad:
  Objetivo: 99.5% de las requests de /api/orders exitosas
  Medición: countif(success==true) / count() >= 0.995
  Ventana: últimos 5 minutos
  Severidad: Crítica

SLO 2 — Latencia:
  Objetivo: P95 de POST /api/orders < 500ms
  Medición: percentile(duration, 95) < 500
  Ventana: últimos 15 minutos
  Severidad: Warning

SLO 3 — Errores de base de datos:
  Objetivo: 0 errores de conexión a SQL
  Medición: count() de exceptions con "connection" > 0
  Ventana: últimos 5 minutos
  Severidad: Crítica

SLO 4 — Mensajería:
  Objetivo: No más de 100 mensajes en el Outbox pendientes
  Medición: CustomMetric "OutboxPendingMessages" > 100
  Ventana: últimos 5 minutos
  Severidad: Warning
```

## Crear alertas en Azure Monitor

```bash
# Alerta SLO 1: tasa de error de Orders.API > 0.5%
az monitor scheduled-query create \
    --resource-group orderflow-rg \
    --name "Orders-ErrorRate-High" \
    --scopes "/subscriptions/{sub}/resourceGroups/orderflow-rg/providers/microsoft.insights/components/orderflow-insights" \
    --condition "count > 0" \
    --condition-query "
        requests
        | where timestamp > ago(5m)
        | where cloud_RoleName == 'orders-api'
        | summarize ErrorRate = 100.0 * countif(success == false) / count()
        | where ErrorRate > 0.5" \
    --description "Orders.API error rate above 0.5% SLO" \
    --severity 1 \
    --evaluation-frequency 1 \
    --window-size 5 \
    --action-groups "/subscriptions/{sub}/resourceGroups/orderflow-rg/providers/microsoft.insights/actionGroups/techshop-alerts"

echo "✅ Alerta SLO 1 creada"
```

## Integrar alertas con el pipeline de CI/CD

Si Application Insights detecta una degradación post-deploy, puede marcar el deploy como fallido:

```yaml
# En el pipeline de CI/CD (M7.3)

      - name: Monitorear post-deploy (10 minutos)
        run: |
          $endTime = (Get-Date).AddMinutes(10)
          $service = "${{ matrix.service }}"

          while ((Get-Date) -lt $endTime) {
            # Consultar Application Insights via REST API
            $query = "requests | where timestamp > ago(5m) | where cloud_RoleName == '$service' | summarize ErrorRate = 100.0 * countif(success == false) / count()"

            $result = Invoke-RestMethod -Uri "..." -Headers @{Authorization = "Bearer $token"} -Body @{query = $query}
            $errorRate = $result.tables[0].rows[0][0]

            if ($errorRate -gt 0.5) {
              Write-Error "❌ Post-deploy error rate $errorRate% > 0.5% SLO"
              exit 1  # → pipeline marca el deploy como fallido
            }

            Start-Sleep -Seconds 60
          }
          Write-Host "✅ 10 minutos sin degradación post-deploy"
        shell: pwsh
```

## Application Insights y GDPR

Los datos de telemetría pueden contener información personal (emails, IPs):

```csharp
// Configurar sampling para reducir datos y coste
builder.Services.Configure<TelemetryConfiguration>(config =>
{
    // Enviar solo el 10% de los requests normales
    // 100% de los requests con errores
    config.DefaultTelemetrySink.TelemetryProcessorChainBuilder
        .UseAdaptiveSampling(
            maxTelemetryItemsPerSecond: 5,
            excludedTypes: "Exception");
});

// Anonimizar datos sensibles antes de enviar
public class AnonymizingTelemetryProcessor : ITelemetryProcessor
{
    public void Process(ITelemetry item)
    {
        if (item is RequestTelemetry request)
        {
            // Eliminar query strings que pueden contener datos personales
            var uri = new Uri(request.Url);
            request.Url = $"{uri.Scheme}://{uri.Host}{uri.AbsolutePath}";
        }
        // No loggear el email en Application Insights — solo el userId
        _next.Process(item);
    }
}
```

## MTTR: el tiempo de recuperación de TechShop

**MTTR (Mean Time to Recovery)** es el tiempo medio desde que ocurre un fallo hasta que se resuelve. Con el pipeline de OrderFlow:

```
ESCENARIO 1: Bug en código (introducido en el último deploy)

  t=0:   Alert: ErrorRate > 0.5% en orders-api
  t=2m:  Live Metrics confirma el problema
  t=3m:  Application Map: el error es en orders-api, no en SQL
  t=5m:  Rollback: slot swap de staging ← production
  t=6m:  Alert se resuelve: ErrorRate < 0.1%
  MTTR: 6 minutos

ESCENARIO 2: Saturación de la BD (Elastic Pool)

  t=0:   Smart Detection: Degradation in response time
  t=5m:  KQL: latencia P95 subió de 145ms a 2300ms
  t=6m:  Azure Portal: Elastic Pool al 95% de DTUs
  t=7m:  az sql elastic-pool update --capacity 8 (scale up)
  t=10m: Latencia vuelve a normal
  MTTR: 10 minutos
```

Con monolito sin observabilidad: MTTR típico de 2-4 horas. Con OrderFlow + Application Insights: MTTR < 15 minutos en la mayoría de escenarios.

## Las decisiones clave que tomó TechShop

**Decisión 1: Clean Architecture**
Dominio aislado de la infraestructura. EF Core en Infrastructure, no en Domain. La Dependency Rule garantiza que un cambio de BD no rompe el dominio.

**Decisión 2: Sin Docker en desarrollo**
Aspire orquesta servicios nativos (LocalDB, RabbitMQ, ASP.NET Core Identity). Sin contenedores: setup más simple, debugging más directo, recursos compartidos sin overhead.

**Decisión 3: Azure App Services, no Kubernetes**
5 servicios con 1 equipo. App Services es suficiente hasta 20+ servicios. Managed Identity, deployment slots y auto-scaling incluidos sin complejidad operativa.

**Decisión 4: Outbox Pattern desde el primer día**
La consistencia eventual no es un problema si se diseña correctamente. El Outbox garantiza que ningún pedido queda huérfano.

**Decisión 5: CQRS con Read Model**
El write side (EF Core + dominio rico) y el read side (Dapper + tabla desnormalizada) son independientes. Las queries de listado son 10x más rápidas sin tocar el modelo de escritura.
