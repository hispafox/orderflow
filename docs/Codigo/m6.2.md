# M6.2 — Consistencia Eventual y Outbox Pattern

## Branch

```bash
git checkout m6.1
git checkout -b m6.2
```

## Objetivo

Eliminar el **dual-write problem** de OrderFlow. Actualmente el `CreateOrderHandler` guarda el pedido en SQL y luego publica el evento a RabbitMQ en dos operaciones separadas. Si el proceso cae entre ambas, el pedido existe pero la Saga nunca arranca. El Outbox Pattern resuelve esto con una línea de configuración en MassTransit: `_publishEndpoint.Publish` deja de ir a RabbitMQ directamente y pasa a la misma transacción SQL que el pedido.

## El problema en código

```
await _repository.SaveAsync(order, ct);     // ✅ INSERT en OrdersDb

// ↑ Si el proceso cae aquí: pedido huérfano para siempre

await _publishEndpoint.Publish(new OrderCreated { ... }, ct);  // ❌ NUNCA llega
```

Con el Outbox:

```
BEGIN TRANSACTION
  INSERT orders.Orders           ← el pedido
  INSERT orders.OutboxMessage    ← el evento pendiente de publicar
COMMIT

[Background Service — cada 5s]
  Lee OutboxMessage WHERE ProcessedAt IS NULL
  Publica a RabbitMQ
  UPDATE ProcessedAt = NOW()
```

```
src/Orders.API/
  ├── Program.cs                              ← añadir AddEntityFrameworkOutbox
```

## Paso 2 — Actualizar la configuración de MassTransit en Orders.API

Reemplazar el bloque de `AddMassTransit` en `src/Orders.API/Program.cs`:

```csharp
using MassTransit;

// ─── MassTransit + RabbitMQ + Saga + Outbox ───────────────────────────────────
builder.Services.AddMassTransit(x =>
{
    // ─── Saga ────────────────────────────────────────────────────────────────
    x.AddSagaStateMachine<OrderSaga, OrderSagaState>()
        .EntityFrameworkRepository(r =>
        {
            r.ConcurrencyMode = ConcurrencyMode.Optimistic;
            r.AddDbContext<DbContext, OrderDbContext>(
                (provider, options) => options.UseSqlServer(
                    provider.GetRequiredService<IConfiguration>()
                            .GetConnectionString("OrdersDb")));
        });

    // ─── Consumers ───────────────────────────────────────────────────────────
    x.AddConsumers(typeof(Program).Assembly);

    // ─── Outbox integrado de MassTransit ─────────────────────────────────────
    // Intercepta TODOS los _publishEndpoint.Publish / bus.Send del proceso.
    // En vez de ir a RabbitMQ directamente, el mensaje se guarda en
    // orders.OutboxMessage dentro de la misma transacción EF Core.
    // Un BackgroundService de MassTransit lo publica a RabbitMQ cada 5 segundos.
    //
    // Resultado:
    //   INSERT orders.Orders + INSERT orders.OutboxMessage → misma transacción
    //   → Si falla el COMMIT: rollback de ambos (sin pedido huérfano)
    //   → Si la app cae: al reiniciar el BackgroundService publica pendientes
    //   → Si RabbitMQ está caído: mensajes esperan en la tabla hasta que vuelva
    x.AddEntityFrameworkOutbox<OrderDbContext>(o =>
    {
        o.UseSqlServer();   // Dialect SQL Server para los locks y queries del Outbox

        // Intervalo de polling del BackgroundService
        // 5s = latencia máxima aceptable para TechShop
        // Ajustar si se necesita menos latencia (1s) o menos carga (10s)
        o.QueryDelay = TimeSpan.FromSeconds(5);

        // Máximo de mensajes por ciclo — evita bloquear la BD en picos
        o.QueryMessageLimit = 100;

        // Auto-limpieza: borrar mensajes ya publicados tras 24h
        // Sin esto, la tabla crece indefinidamente
        o.RemoveDelivered = true;

        // Integra el Outbox con el bus — intercepta Publish/Send
        o.UseBusOutbox(busOutbox =>
        {
            // Si un mensaje lleva más de 5 minutos sin poderse entregar
            // (ej: RabbitMQ completamente caído), se marca como error
            busOutbox.MessageDeliveryTimeout = TimeSpan.FromMinutes(5);
        });
    });

    // ─── RabbitMQ ────────────────────────────────────────────────────────────
    x.UsingRabbitMq((context, cfg) =>
    {
        cfg.Host(
            builder.Configuration.GetConnectionString("messaging")!,
            h => { h.Username("guest"); h.Password("guest"); });

        cfg.UseMessageRetry(r => r.Exponential(
            retryLimit:    5,
            minInterval:   TimeSpan.FromSeconds(1),
            maxInterval:   TimeSpan.FromSeconds(30),
            intervalDelta: TimeSpan.FromSeconds(2)));

        cfg.ConfigureEndpoints(context);
    });
});
```

## Paso 4 — TransactionBehavior: sin cambios

El `TransactionBehavior` de M3.4 ya es correcto. Con Outbox habilitado, el `SaveChangesAsync` del handler incluye automáticamente el INSERT del OutboxMessage:

```csharp
// TransactionBehavior.cs — NO modificar
public async Task<TResponse> Handle(
    TRequest request,
    RequestHandlerDelegate<TResponse> next,
    CancellationToken ct)
{
    await using var transaction = await _dbContext.Database
        .BeginTransactionAsync(IsolationLevel.ReadCommitted, ct);
    try
    {
        var response = await next();
        // SaveChangesAsync incluye:
        //   INSERT orders.Orders          (del handler)
        //   INSERT orders.OutboxMessage   (del Publish interceptado por MassTransit)
        await _dbContext.SaveChangesAsync(ct);
        await transaction.CommitAsync(ct);
        return response;
    }
    catch
    {
        await transaction.RollbackAsync(ct);
        throw;
    }
}
```

Esto crea en el schema `orders` (ya configurado en `OnModelCreating`):

```sql
-- orders.OutboxMessage — mensajes pendientes de publicar a RabbitMQ
CREATE TABLE [orders].[OutboxMessage] (
    [SequenceNumber] bigint IDENTITY(1,1) NOT NULL,
    [LockId]         uniqueidentifier NOT NULL,
    [RowVersion]     rowversion NOT NULL,           -- Lock optimista para multi-instancia
    [SentTime]       datetime2 NOT NULL,
    [MessageId]      uniqueidentifier NOT NULL,
    [ContentType]    nvarchar(256) NOT NULL,
    [MessageType]    nvarchar(max) NOT NULL,
    [Body]           nvarchar(max) NOT NULL,        -- JSON del evento
    [OutboxId]       uniqueidentifier NULL,          -- NULL = pendiente de publicar
    [ConversationId] uniqueidentifier NULL,
    [CorrelationId]  uniqueidentifier NULL,
    CONSTRAINT PK_OutboxMessage PRIMARY KEY ([SequenceNumber])
);

-- orders.InboxState — mensajes ya procesados (idempotencia en consumers)
CREATE TABLE [orders].[InboxState] (
    [Id]          bigint IDENTITY(1,1) NOT NULL,
    [MessageId]   uniqueidentifier NOT NULL,
    [ConsumerId]  uniqueidentifier NOT NULL,
    [Received]    datetime2 NOT NULL,
    [Delivered]   datetime2 NULL,
    [Consumed]    datetime2 NULL,
    [ExpirationTime] datetime2 NULL,
    CONSTRAINT PK_InboxState PRIMARY KEY ([Id])
);

-- Índice clave para el BackgroundService de MassTransit
-- Filtra: WHERE OutboxId IS NULL (mensajes no publicados)
CREATE INDEX IX_OutboxMessage_OutboxId_SentTime
ON [orders].[OutboxMessage] ([OutboxId], [SentTime]);
```

## Paso 6 — OutboxCleanupJob (opcional pero recomendado)

MassTransit con `RemoveDelivered = true` limpia automáticamente los mensajes publicados. Como respaldo y para mensajes con error, añadir un cleanup job explícito:

Crear `src/Orders.API/Infrastructure/Outbox/OutboxCleanupJob.cs`:

```csharp
using Microsoft.EntityFrameworkCore;
using Orders.API.Infrastructure.Persistence;

namespace Orders.API.Infrastructure.Outbox;

/// <summary>
/// Limpia periódicamente los mensajes del Outbox ya procesados.
/// MassTransit con RemoveDelivered=true ya hace limpieza automática,
/// este job actúa como respaldo y limpia mensajes con error > 48h.
/// Se ejecuta cada hora para no impactar el rendimiento.
/// </summary>
public class OutboxCleanupJob : BackgroundService
{
    private readonly IServiceScopeFactory _scopeFactory;
    private readonly ILogger<OutboxCleanupJob> _logger;

    public OutboxCleanupJob(
        IServiceScopeFactory        scopeFactory,
        ILogger<OutboxCleanupJob>   logger)
    {
        _scopeFactory = scopeFactory;
        _logger       = logger;
    }

    protected override async Task ExecuteAsync(CancellationToken stoppingToken)
    {
        // Esperar al arranque del servicio antes del primer ciclo
        await Task.Delay(TimeSpan.FromMinutes(5), stoppingToken);

        while (!stoppingToken.IsCancellationRequested)
        {
            await CleanupAsync(stoppingToken);
            await Task.Delay(TimeSpan.FromHours(1), stoppingToken);
        }
    }

    private async Task CleanupAsync(CancellationToken ct)
    {
        try
        {
            using var scope    = _scopeFactory.CreateScope();
            var dbContext      = scope.ServiceProvider.GetRequiredService<OrderDbContext>();
            var cutoff         = DateTime.UtcNow.AddHours(-24);

            // Borrar mensajes publicados (OutboxId IS NOT NULL) con más de 24h
            var deletedCount = await dbContext.Database.ExecuteSqlRawAsync(
                "DELETE FROM orders.OutboxMessage " +
                "WHERE OutboxId IS NOT NULL AND SentTime < {0}",
                new object[] { cutoff }, ct);

            if (deletedCount > 0)
                _logger.LogInformation(
                    "Outbox cleanup: deleted {Count} processed messages older than {Cutoff}",
                    deletedCount, cutoff);

            // Alerta si hay mensajes bloqueados (pendientes > 10 minutos)
            var stuckCount = await dbContext.Database.SqlQuery<int>(
                $"SELECT COUNT(*) FROM orders.OutboxMessage " +
                $"WHERE OutboxId IS NULL AND SentTime < {DateTime.UtcNow.AddMinutes(-10)}")
                .FirstOrDefaultAsync(ct);

            if (stuckCount > 0)
                _logger.LogWarning(
                    "[OUTBOX ALERT] {Count} messages stuck for > 10 minutes. " +
                    "Check RabbitMQ connectivity.",
                    stuckCount);
        }
        catch (Exception ex) when (!ct.IsCancellationRequested)
        {
            _logger.LogError(ex, "Outbox cleanup failed");
        }
    }
}
```

Registrar en `src/Orders.API/Program.cs`:

```csharp

```

## Paso 7 — Consumidores: idempotencia ya implementada

La idempotencia en los consumers ya estaba implementada desde M4.2. Con el Outbox activo que garantiza at-least-once delivery, es aún más importante verificar que está correcta.

`OrderCreatedConsumer` en Notifications.API (ya correcto desde M4.2):

```csharp
public async Task Consume(ConsumeContext<OrderCreated> context)
{
    // Verificar idempotencia: ¿ya procesamos este OrderId + Type?
    var alreadyProcessed = await _dbContext.Notifications.AnyAsync(
        n => n.OrderId == context.Message.OrderId && n.Type == "OrderConfirmation",
        context.CancellationToken);

    if (alreadyProcessed)
    {
        _logger.LogWarning(
            "Duplicate OrderCreated for {OrderId} — skipping",
            context.Message.OrderId);
        return; // ACK sin reprocesar — el Outbox puede entregar duplicados
    }

    // Procesar y marcar como procesado en la misma SaveChanges
    await _emailService.SendOrderConfirmationAsync(...);
    _dbContext.Notifications.Add(
        Notification.Create(context.Message.OrderId, "OrderConfirmation",
            context.Message.CustomerEmail, true));
    await _dbContext.SaveChangesAsync(context.CancellationToken);
}
```

**Resultado combinado:** Outbox (at-least-once) + Consumer idempotente = **exactly-once semántico** a nivel de negocio.

## Paso 9 — Monitoreo del Outbox

Consultas SQL para el dashboard de operaciones y alertas de Application Insights:

```sql
-- Ver mensajes pendientes de publicar (OutboxId IS NULL = no procesado)
SELECT TOP 10
    SequenceNumber,
    SentTime,
    MessageType,
    DATEDIFF(SECOND, SentTime, GETUTCDATE()) AS AgeSeconds
FROM orders.OutboxMessage
WHERE OutboxId IS NULL
ORDER BY SentTime ASC;

-- Alerta: mensajes con > 60 segundos sin publicar (BackgroundService posiblemente caído)
SELECT COUNT(*) AS PendingCount
FROM orders.OutboxMessage
WHERE OutboxId IS NULL
  AND DATEDIFF(SECOND, SentTime, GETUTCDATE()) > 60;

-- Audit trail: todos los eventos publicados para un pedido concreto
SELECT
    SequenceNumber,
    JSON_VALUE(Body, '$.messageType') AS EventType,
    SentTime,
    OutboxId AS ProcessedAt
FROM orders.OutboxMessage
WHERE JSON_VALUE(Body, '$.message.orderId') = '7b3a2c1d-...'
ORDER BY SequenceNumber;

-- Detectar Sagas atascadas (llevan > 10 minutos en un estado transitorio)
SELECT
    CorrelationId,
    CurrentState,
    CreatedAt,
    DATEDIFF(MINUTE, CreatedAt, GETUTCDATE()) AS MinutesInState
FROM orders.OrderSagaState
WHERE CurrentState NOT IN ('Completed', 'Failed')
  AND CreatedAt < DATEADD(MINUTE, -10, GETUTCDATE())
ORDER BY CreatedAt ASC;
```

## Antes y después: comparativa visual

```
ANTES (dual-write problem):
  t=0:   INSERT orders.Orders     ← COMMIT ✅
  t=?:   CRASH del proceso
  t=∞:   Publish(OrderCreated)    ← NUNCA OCURRE ❌
          → Saga nunca arranca
          → Stock nunca reservado
          → Cliente sin confirmación → soporte técnico
          → Script manual de compensación

DESPUÉS (Outbox Pattern):
  t=0:   BEGIN TRANSACTION
         INSERT orders.Orders          ← el pedido
         INSERT orders.OutboxMessage   ← el evento pendiente
         COMMIT ✅
  t=?:   CRASH del proceso
  t=restart: BackgroundService de MassTransit arranca
  t=+5s:     Lee OutboxMessage (OutboxId IS NULL)
             Publish(OrderCreated) → RabbitMQ ✅
             UPDATE OutboxId = BackgroundId
          → Saga arranca normalmente
          → Cliente recibe confirmación sin intervención manual
```

## Las tres tablas que garantizan el flujo

```
orders.Orders              ← El pedido (fuente de verdad)
orders.OrderSagaState      ← El estado de la Saga
orders.OutboxMessage       ← Eventos pendientes de publicar

Si el sistema cae en CUALQUIER momento:
  Al reiniciar: OutboxMessage pendiente → se publica → Saga continúa
  Sin intervención manual. Sin scripts. Sin guardia de turno.
```
