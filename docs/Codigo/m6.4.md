## Objetivo

Establecer el proceso que permite a TechShop evolucionar el schema de `orders.Orders` — con 150.000 pedidos en producción — **sin un segundo de downtime**. En este módulo aplicamos el patrón Expand-Contract a tres casos reales de OrderFlow, integramos Migration Bundles en el pipeline de CI/CD, añadimos Feature Flags para activación controlada y documentamos el proceso para el equipo.

## El patrón Expand-Contract: las tres fases

```
EXPAND    → La BD crece: acepta la estructura nueva Y la vieja simultáneamente
MIGRATE   → Los datos evolucionan en lotes (sin downtime, sin bloqueos)
CONTRACT  → La BD se contrae: eliminar la estructura vieja, schema limpio
```

Cada fase es un **deploy independiente**. La BD va **siempre por delante** del código.

```
BD vieja ────────────────────────────────────────────────────────── 
        | Expand:   BD tiene CustomerId + BuyerId (ambas)          |
        |                  ↑ código v1 escribe en CustomerId        |
        |                  ↑ código v2 escribe en ambas             |
        | Migrate:  backfill completo, verificar en staging         |
        | Contract: DROP CustomerId, código v3 solo usa BuyerId     |
BD nueva ────────────────────────────────────────────────────────── 
```

## Paso 1 — Caso 1: Añadir DiscountAmount (más simple)

### Fase 1 — Expand: columna nullable

Crear la migration:

```bash
dotnet ef migrations add ExpandAddDiscountAmount \
    --project src/Orders.API \
    --context OrderDbContext
```

Editar la migration generada `...ExpandAddDiscountAmount.cs`:

```csharp
public partial class ExpandAddDiscountAmount : Migration
{
    protected override void Up(MigrationBuilder migrationBuilder)
    {
        // NULLABLE — el código viejo (que no conoce DiscountAmount) sigue funcionando.
        // Los INSERTs existentes no fallan porque la columna acepta NULL.
        migrationBuilder.AddColumn<decimal>(
            name:          "DiscountAmount",
            schema:        "orders",
            table:         "Orders",
            type:          "decimal(18,2)",
            nullable:      true,
            defaultValue:  null);
    }

    protected override void Down(MigrationBuilder migrationBuilder)
    {
        // Rollback seguro — la columna era nueva, no había datos valiosos en ella
        migrationBuilder.DropColumn(
            name:   "DiscountAmount",
            schema: "orders",
            table:  "Orders");
    }
}
```

Aplicar:

```bash
dotnet ef database update --project src/Orders.API --context OrderDbContext
```

### Fase 2 — Backfill idempotente (script SQL separado)

Crear `scripts/backfill_discount_amount.sql`:

```sql
-- Backfill de DiscountAmount para registros históricos
-- IDEMPOTENTE: puede ejecutarse varias veces sin efecto secundario
-- Establecer 0.00 para todos los pedidos que aún no tienen descuento

DECLARE @BatchSize INT = 1000;
DECLARE @Processed INT = 0;
DECLARE @BatchRows INT;

PRINT 'Starting DiscountAmount backfill...';

WHILE 1 = 1
BEGIN
    UPDATE TOP(@BatchSize) orders.Orders
    SET DiscountAmount = 0.00
    WHERE DiscountAmount IS NULL;

    SET @BatchRows = @@ROWCOUNT;
    SET @Processed = @Processed + @BatchRows;

    IF @BatchRows = 0 BREAK;

    PRINT CONCAT('Processed ', @Processed, ' rows...');
    -- Pausa de 100ms entre lotes para no saturar en producción
    WAITFOR DELAY '00:00:00.100';
END

PRINT CONCAT('Backfill complete. Total rows updated: ', @Processed);
```

Ejecutar (una sola vez, en horas de baja actividad):

```bash
# Con sqlcmd (incluido en las dev tools)
sqlcmd -S "(localdb)\MSSQLLocalDB" -d OrdersDb \
    -i scripts/backfill_discount_amount.sql
```

public partial class ContractDiscountAmountNotNull : Migration
{
    protected override void Up(MigrationBuilder migrationBuilder)
    {
        // Solo ejecutar DESPUÉS de que el backfill haya completado
        // y verificado en staging

            name:         "DiscountAmount",

            table:        "Orders",
            type:         "decimal(18,2)",

    protected override void Down(MigrationBuilder migrationBuilder)
    {
        migrationBuilder.AlterColumn<decimal>(
            name:     "DiscountAmount",
            schema:   "orders",
            table:    "Orders",
            nullable: true,
            oldNullable: false);
    }
}

## Paso 2 — Caso 2: Renombrar CustomerId → BuyerId

Este es el caso más frecuente en proyectos reales. Requiere tres releases.

### Fase 1 — Expand: añadir BuyerId manteniendo CustomerId

```bash
dotnet ef migrations add ExpandAddBuyerIdColumn \
    --project src/Orders.API \
    --context OrderDbContext
```

```csharp
public partial class ExpandAddBuyerIdColumn : Migration
{
    protected override void Up(MigrationBuilder migrationBuilder)
    {
        // 1. Añadir la nueva columna como nullable
        migrationBuilder.AddColumn<Guid>(
            name:     "BuyerId",
            schema:   "orders",
            table:    "Orders",
            nullable: true);

        // 2. Copiar datos en la misma migration — atómico con la creación de columna
        migrationBuilder.Sql(
            "UPDATE orders.Orders SET BuyerId = CustomerId");

        // 3. Hacer NOT NULL después de copiar
        migrationBuilder.AlterColumn<Guid>(
            name:        "BuyerId",
            schema:      "orders",
            table:       "Orders",
            nullable:    false,
            oldNullable: true);

        // CustomerId NO se elimina aquí — el código viejo la sigue usando
    }

    protected override void Down(MigrationBuilder migrationBuilder)
    {
        migrationBuilder.DropColumn("BuyerId", "orders", "Orders");
    }
}
```

**Deploy 1 — Código que escribe en AMBAS columnas** (durante la transición):

```csharp
// Order.cs — fase de transición
public Guid CustomerId { get; private set; }  // mantener por compatibilidad
public Guid BuyerId    { get; private set; }  // la nueva

public static Order Create(Guid buyerId, ...)
{
    return new Order
    {
        CustomerId = buyerId,  // compatibilidad con código viejo en producción
        BuyerId    = buyerId,  // la nueva — ya se usa aquí
    };
}
```

### Fase 3 — Contract: eliminar CustomerId (release posterior)

```bash
# SOLO después de verificar que el Deploy 2 lleva al menos 1 semana
# en producción sin errores relacionados con CustomerId
dotnet ef migrations add ContractDropCustomerIdColumn \
    --project src/Orders.API \
    --context OrderDbContext
```

```csharp
// Referencia — esta migration se aplica en la siguiente release
public partial class ContractDropCustomerIdColumn : Migration
{
    protected override void Up(MigrationBuilder migrationBuilder)
    {
        // DROP: los datos ya están en BuyerId — no hay pérdida
        migrationBuilder.DropIndex("IX_Orders_CustomerId", "orders", "Orders");
        migrationBuilder.DropColumn("CustomerId", "orders", "Orders");
    }

    protected override void Down(MigrationBuilder migrationBuilder)
    {
        // Si necesitamos revertir este DROP, restaurar desde backup
        // o recrear la columna y hacer backfill desde BuyerId
        migrationBuilder.AddColumn<Guid>(
            name:     "CustomerId",
            schema:   "orders",
            table:    "Orders",
            nullable: true);
        migrationBuilder.Sql("UPDATE orders.Orders SET CustomerId = BuyerId");
    }
}
```

### Uso en un controller

```csharp
using Microsoft.FeatureManagement;

[ApiController, Route("api/orders")]
public class OrdersController : ControllerBase
{
    private readonly IFeatureManager _featureManager;
    private readonly IMediator       _mediator;

    // ...

    [HttpGet("{id:guid}")]
    [Authorize]
    public async Task<IActionResult> GetById(
        Guid id, CancellationToken ct = default)
    {
        var userId  = GetUserId();
        var isAdmin = IsAdmin();

        // Feature flag: cuando UseBuyerId está activo, usar la nueva query
        if (await _featureManager.IsEnabledAsync("UseBuyerId"))
        {
            var result = await _mediator.Send(
                new GetOrderByIdQuery(id, userId, isAdmin), ct);
            return result is null ? NotFound() : Ok(result);
        }

        // Fallback: query antigua con CustomerId
        var legacy = await _mediator.Send(
            new GetOrderByIdLegacyQuery(id, userId, isAdmin), ct);
        return legacy is null ? NotFound() : Ok(legacy);
    }
}
```

## Paso 4 — Tests de migration

Crear `tests/Orders.API.Tests/Integration/Migrations/MigrationTests.cs`:

```csharp
using FluentAssertions;
using Microsoft.EntityFrameworkCore;
using Microsoft.EntityFrameworkCore.Infrastructure;
using Microsoft.EntityFrameworkCore.Migrations;
using Microsoft.Extensions.DependencyInjection;
using Orders.API.Infrastructure.Persistence;
using Orders.API.Tests.Shared;
using Xunit;

namespace Orders.API.Tests.Integration.Migrations;

/// <summary>
/// Tests que verifican la integridad de las migrations antes de producción.
/// Detectan errores de SQL en el pipeline de CI — no en producción.
/// </summary>
[Trait("Category", "Migration")]
public class MigrationTests : IClassFixture<OrdersApiFactory>
{
    private readonly OrdersApiFactory _factory;

    public MigrationTests(OrdersApiFactory factory) => _factory = factory;

    [Fact]
    public async Task AllMigrations_Apply_Successfully()
    {
        // Verificar que todas las migrations se aplican sin errores SQL
        using var scope  = _factory.Services.CreateScope();
        var dbContext    = scope.ServiceProvider.GetRequiredService<OrderDbContext>();

        // Si hay un error en alguna migration (typo en SQL, tipo incorrecto...)
        // este test falla aquí — no en producción
        await dbContext.Database.MigrateAsync();

        // Verificar que las tablas esperadas existen
        var tables = await dbContext.Database
            .SqlQuery<string>(
                $"SELECT TABLE_NAME FROM INFORMATION_SCHEMA.TABLES " +
                $"WHERE TABLE_SCHEMA = 'orders'")
            .ToListAsync();

        tables.Should().Contain("Orders");
        tables.Should().Contain("OrderLines");
        tables.Should().Contain("OutboxMessage");
        tables.Should().Contain("InboxState");
        tables.Should().Contain("OrderSummaries");
        tables.Should().Contain("AuditLogs");
    }

    [Fact]
    public async Task NoPendingMigrations_InCurrentBranch()
    {
        // Verificar que todo el código está alineado con las migrations
        // Si alguien modifica OrderDbContext sin crear la migration correspondiente,
        // este test lo detecta en CI
        using var scope  = _factory.Services.CreateScope();
        var dbContext    = scope.ServiceProvider.GetRequiredService<OrderDbContext>();
        await dbContext.Database.MigrateAsync();

        var pending = await dbContext.Database.GetPendingMigrationsAsync();
        pending.Should().BeEmpty(
            "all code changes to the domain model require a corresponding migration");
    }

    [Fact]
    public async Task ExpandMigration_DoesNotDropColumns()
    {
        // Verificar que las migrations en Fase 1 (Expand) son SOLO aditivas
        // Si alguien añade un DropColumn a una migration de Expand → falla en CI
        using var scope  = _factory.Services.CreateScope();
        var dbContext    = scope.ServiceProvider.GetRequiredService<OrderDbContext>();
        var migrator     = dbContext.GetService<IMigrator>();

        // Obtener las migrations aplicadas recientemente
        var history      = await dbContext.Database
            .GetAppliedMigrationsAsync();

        // Buscar migrations con "Expand" en el nombre — deben ser aditivas
        var expandMigrations = history
            .Where(m => m.Contains("Expand", StringComparison.OrdinalIgnoreCase))
            .ToList();

        // Para cada migration Expand, verificar que no tiene Drop operations
        // Nota: verificación simplificada — en CI más completo se puede analizar el SQL generado
        expandMigrations.Should().NotBeEmpty(
            "at least one Expand migration should exist");
    }

    [Fact]
    public async Task ExpectedIndexes_ExistOnOrderSummaries()
    {
        using var scope  = _factory.Services.CreateScope();
        var dbContext    = scope.ServiceProvider.GetRequiredService<OrderDbContext>();
        await dbContext.Database.MigrateAsync();

        var indexes = await dbContext.Database
            .SqlQuery<string>($"""
                SELECT i.name
                FROM sys.indexes i
                JOIN sys.tables t ON t.object_id = i.object_id
                JOIN sys.schemas s ON s.schema_id = t.schema_id
                WHERE s.name = 'orders'
                  AND t.name = 'OrderSummaries'
                  AND i.type > 0
                """)
            .ToListAsync();

        indexes.Should().Contain(i =>
            i.Contains("CustomerId", StringComparison.OrdinalIgnoreCase),
            "IX_OrderSummaries_CustomerId should exist");
        indexes.Should().Contain(i =>
            i.Contains("Status", StringComparison.OrdinalIgnoreCase),
            "IX_OrderSummaries_Status should exist");
    }
}
```

## Paso 5 — Pipeline de GitHub Actions con Migration Bundle

Crear `.github/workflows/deploy-orders.yml`:

```yaml
name: Deploy Orders.API

on:
  push:
    branches: [main]
    paths:
      - 'src/Orders.API/**'
      - 'shared/**'

jobs:
  test:
    name: Test
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4

      - name: Setup .NET 10
        uses: actions/setup-dotnet@v4
        with:
          dotnet-version: '10.x'

      - name: Build
        run: dotnet build OrderFlow.sln -c Release

      - name: Run migration tests
        run: |
          dotnet test \
            --filter "Category=Migration" \
            --no-build -c Release \
            --logger "trx;LogFileName=migration-tests.xml"

      - name: Run integration tests
        run: |
          dotnet test \
            --filter "Category=Integration" \
            --no-build -c Release

  migrate:
    name: Apply Migrations
    runs-on: ubuntu-latest
    needs: test
    environment: production   # Requiere aprobación manual en GitHub

    steps:
      - uses: actions/checkout@v4
      - uses: actions/setup-dotnet@v4
        with:
          dotnet-version: '10.x'

      - name: Generate migration plan
        run: |
          dotnet ef migrations script \
            --project src/Orders.API \
            --context OrderDbContext \
            --idempotent \
            --output migration-script.sql
          cat migration-script.sql  # Log para revisión

      - name: Create Migration Bundle
        run: |
          dotnet ef migrations bundle \
            --project src/Orders.API \
            --context OrderDbContext \
            --output ./artifacts/efbundle-orders \
            --self-contained \
            --runtime linux-x64

      # ─── MIGRATION ANTES DEL CÓDIGO ──────────────────────────────────────────
      # La BD se actualiza PRIMERO. El código viejo sigue funcionando
      # porque la migration es aditiva (columna nullable o nueva tabla).
      - name: Apply Orders DB Migrations
        run: |
          chmod +x ./artifacts/efbundle-orders
          ./artifacts/efbundle-orders \
            --connection "${{ secrets.ORDERS_DB_CONNECTION }}"

      - name: Verify migration applied
        run: |
          echo "Checking pending migrations..."
          # Si hay migrations pendientes tras aplicar el bundle → falla el pipeline
          curl -f "https://orderflow-orders.azurewebsites.net/health" \
            | jq '.entries["orders-db-migrations"].status' \
            | grep -q "Healthy"

  deploy:
    name: Deploy Code
    runs-on: ubuntu-latest
    needs: migrate   # El código SIEMPRE despliega DESPUÉS de las migrations

    steps:
      - uses: actions/checkout@v4
      - uses: actions/setup-dotnet@v4
        with:
          dotnet-version: '10.x'

      - name: Publish
        run: |
          dotnet publish src/Orders.API -c Release \
            -o ./publish/orders-api

      # Blue-Green: deploy al slot staging primero
      - name: Deploy to Staging Slot
        run: |
          az webapp deploy \
            --resource-group orderflow-rg \
            --name orderflow-orders \
            --slot staging \
            --src-path ./publish/orders-api

      - name: Smoke test staging
        run: |
          sleep 10  # Esperar al arranque
          curl -f "https://orderflow-orders-staging.azurewebsites.net/health"

      # Swap: staging → production (sin downtime)
      - name: Swap staging to production
        run: |
          az webapp deployment slot swap \
            --resource-group orderflow-rg \
            --name orderflow-orders \
            --slot staging \
            --target-slot production

      - name: Verify production
        run: |
          sleep 10
          curl -f "https://orderflow-orders.azurewebsites.net/health"
```

### KQL en Application Insights (referencia para M7.4)

exceptions

  )

### Script de verificación post-deploy

Crear `scripts/verify-migration.sh`:

```bash
#!/bin/bash
# Ejecutar después de cada deploy que incluya migrations

SERVICE_URL="${1:-https://orderflow-orders.azurewebsites.net}"
MAX_RETRIES=5

echo "=== POST-MIGRATION VERIFICATION ==="
echo "Service: $SERVICE_URL"
echo "Time: $(date -u)"

for i in $(seq 1 $MAX_RETRIES); do
    echo ""
    echo "Attempt $i/$MAX_RETRIES..."

    RESPONSE=$(curl -s "$SERVICE_URL/health")
    STATUS=$(echo "$RESPONSE" | jq -r '.status // "unknown"')
    DB_STATUS=$(echo "$RESPONSE" | jq -r '.entries["orders-db-migrations"].status // "unknown"')

    echo "Health: $STATUS | DB Migrations: $DB_STATUS"

    if [ "$STATUS" = "Healthy" ] && [ "$DB_STATUS" = "Healthy" ]; then
        echo "✅ Verification PASSED"
        exit 0
    fi

    echo "⏳ Not ready yet, waiting 10 seconds..."
    sleep 10
done

echo "❌ Verification FAILED after $MAX_RETRIES attempts"
echo "Consider rollback: az webapp deployment slot swap --slot production --target-slot staging"
exit 1
```

## Los 5 principios que no se negocian

```
1. ADDITIVE FIRST    — Toda migration empieza añadiendo, nunca eliminando
2. BD ANTES QUE CÓDIGO — La migration se aplica ANTES del deploy del código
3. BACKFILL INCREMENTAL — Lotes de ≤1.000 registros con pausa entre lotes
4. VERIFICAR ANTES DE CONTRAER — ≥1 semana en producción sin errores relacionados
5. ROLLBACK SIEMPRE LISTO — Si el rollback requiere restaurar backup: dividir más
```
