# M6.1 — Database per Service con SQL Server y EF Core 10

## Branch

```bash
git checkout m5.3
git checkout -b m6.1
```

## Objetivo

Establecer el aislamiento completo de datos entre los 4 microservicios de OrderFlow. Cada servicio tiene su propio `DbContext`, su propio schema, sus propias migrations y sus propios índices. Ningún servicio tiene acceso directo a la BD de otro. Se configuran las herramientas de desarrollo, resiliencia y observabilidad de EF Core 10 correctamente para SQL Server / LocalDB.

## Paso 2 — AuditInterceptor reutilizable

Crear `src/Orders.API/Infrastructure/Persistence/Interceptors/AuditInterceptor.cs`:

```csharp
using Microsoft.EntityFrameworkCore;
using Microsoft.EntityFrameworkCore.Diagnostics;

namespace Orders.API.Infrastructure.Persistence.Interceptors;

/// <summary>
/// Rellena automáticamente CreatedAt / UpdatedAt en cada entidad auditable.
/// Sin ICurrentUser por ahora — en M5.x ya tenemos el userId del JWT en el handler.
/// El interceptor solo gestiona timestamps.
/// </summary>
public class AuditInterceptor : SaveChangesInterceptor
{
    public override ValueTask<InterceptionResult<int>> SavingChangesAsync(
        DbContextEventData    eventData,
        InterceptionResult<int> result,
        CancellationToken     ct = default)
    {
        if (eventData.Context is null) return new(result);

        var now = DateTime.UtcNow;

        foreach (var entry in eventData.Context.ChangeTracker.Entries()
            .Where(e => e.State is EntityState.Added or EntityState.Modified))
        {
            // Timestamps automáticos — el interceptor los rellena sin que
            // el código de negocio tenga que recordarlo
            if (entry.State == EntityState.Added
                && entry.Properties.Any(p => p.Metadata.Name == "CreatedAt"))
            {
                entry.Property("CreatedAt").CurrentValue = now;
            }

            if (entry.Properties.Any(p => p.Metadata.Name == "UpdatedAt"))
                entry.Property("UpdatedAt").CurrentValue = now;
        }

        return new(result);
    }
}
```

El mismo `AuditInterceptor.cs` se copia/replica en **Products.API**, **Payments.API** y **Notifications.API** ajustando el namespace.

## Paso 3 — Actualizar OrderDbContext (fichero completo)

Reemplazar `src/Orders.API/Infrastructure/Persistence/OrderDbContext.cs`:

```csharp
using Microsoft.EntityFrameworkCore;
using Orders.API.Domain.Entities;
using Orders.API.Infrastructure.Audit;
using Orders.API.Infrastructure.Persistence.Interceptors;
using Orders.API.Sagas;

namespace Orders.API.Infrastructure.Persistence;

/// <summary>
/// DbContext exclusivo de Orders.API.
/// Solo contiene entidades del dominio de pedidos.
/// Ningún otro servicio tiene referencia a este DbContext.
/// Schema: "orders" — diferencia la BD en entornos con esquemas compartidos (dev).
/// </summary>
public class OrderDbContext : DbContext
{
    public OrderDbContext(DbContextOptions<OrderDbContext> options)
        : base(options) { }

    public DbSet<Order>           Orders      => Set<Order>();
    public DbSet<AuditLogEntry>   AuditLogs   => Set<AuditLogEntry>();

    // OrderSagaState es gestionado por MassTransit — se mapea aquí para
    // que el endpoint GET /api/orders/{id}/saga-state pueda consultarlo
    public DbSet<OrderSagaState>  SagaStates  => Set<OrderSagaState>();

    protected override void OnModelCreating(ModelBuilder modelBuilder)
    {
        // Schema por defecto — todas las tablas van bajo "orders."
        // En LocalDB con una sola BD, diferencia de otros servicios.
        // En Azure SQL, cada servicio tiene su propia BD.
        modelBuilder.HasDefaultSchema("orders");

        // Carga automáticamente todas las clases IEntityTypeConfiguration<T>
        // del ensamblado — sin registrarlas manualmente una a una
        modelBuilder.ApplyConfigurationsFromAssembly(typeof(OrderDbContext).Assembly);
    }

    protected override void OnConfiguring(DbContextOptionsBuilder optionsBuilder)
    {
        // El interceptor de audit se añade aquí para que todas las instancias
        // del DbContext lo incluyan automáticamente
        optionsBuilder.AddInterceptors(new AuditInterceptor());
    }
}
```

## Paso 4 — OrderConfiguration.cs con Fluent API completa

Crear `src/Orders.API/Infrastructure/Persistence/Configurations/OrderConfiguration.cs`:

```csharp
using Microsoft.EntityFrameworkCore;
using Microsoft.EntityFrameworkCore.Metadata.Builders;
using Orders.API.Domain.Entities;

namespace Orders.API.Infrastructure.Persistence.Configurations;

public class OrderConfiguration : IEntityTypeConfiguration<Order>
{
    public void Configure(EntityTypeBuilder<Order> builder)
    {
        builder.ToTable("Orders", "orders");
        builder.HasKey(o => o.Id);
        builder.Property(o => o.Id).ValueGeneratedNever();

        // CustomerId — solo el ID como referencia. El nombre del cliente
        // no se guarda aquí: viene del JWT cuando se necesita mostrar.
        builder.Property(o => o.CustomerId)
               .IsRequired();

        // Status como Value Object → columna string
        builder.Property(o => o.Status)
               .HasConversion(
                   s => s.Value,
                   v => OrderStatus.FromString(v))
               .HasColumnName("Status")
               .HasMaxLength(20)
               .IsRequired();

        // Money (Total) como Owned Entity → dos columnas en la misma fila
        builder.OwnsOne(o => o.Total, money =>
        {
            money.Property(m => m.Amount)
                 .HasColumnName("TotalAmount")
                 .HasPrecision(18, 2)
                 .IsRequired();
            money.Property(m => m.Currency)
                 .HasColumnName("TotalCurrency")
                 .HasMaxLength(3)
                 .IsRequired();
        });

        // Address como Owned Entity
        builder.OwnsOne(o => o.ShippingAddress, addr =>
        {
            addr.Property(a => a.Street) .HasColumnName("Street") .HasMaxLength(200);
            addr.Property(a => a.City)   .HasColumnName("City")   .HasMaxLength(100);
            addr.Property(a => a.ZipCode).HasColumnName("ZipCode").HasMaxLength(20);
            addr.Property(a => a.Country).HasColumnName("Country").HasMaxLength(2);
        });

        builder.Property(o => o.CreatedAt).IsRequired();

        // Soft delete — el filtro global excluye automáticamente los archivados
        builder.Property(o => o.IsDeleted).HasDefaultValue(false);
        builder.HasQueryFilter(o => !o.IsDeleted);

        // ─── Índices ──────────────────────────────────────────────────────────
        // CustomerId: query más frecuente — "dame los pedidos de este cliente"
        builder.HasIndex(o => o.CustomerId)
               .HasDatabaseName("IX_Orders_CustomerId");

        // Status: filtrado por estado
        builder.HasIndex(o => o.Status)
               .HasDatabaseName("IX_Orders_Status");

        // Índice compuesto — "mis pedidos en estado Pending" (la query más frecuente)
        builder.HasIndex(o => new { o.CustomerId, o.Status })
               .HasDatabaseName("IX_Orders_CustomerId_Status");

        // CreatedAt: ordenación descendente habitual en listados
        builder.HasIndex(o => o.CreatedAt)
               .HasDatabaseName("IX_Orders_CreatedAt");

        // ─── Concurrencia optimista ───────────────────────────────────────────
        // RowVersion actualiza SQL Server automáticamente en cada UPDATE.
        // Si dos instancias intentan modificar el mismo pedido simultáneamente,
        // una recibirá DbUpdateConcurrencyException y deberá reintentar.
        builder.Property<byte[]>("RowVersion")
               .IsRowVersion()
               .IsConcurrencyToken();
    }
}
```

## Paso 5 — OrderLineConfiguration.cs

Crear `src/Orders.API/Infrastructure/Persistence/Configurations/OrderLineConfiguration.cs`:

```csharp
using Microsoft.EntityFrameworkCore;
using Microsoft.EntityFrameworkCore.Metadata.Builders;
using Orders.API.Domain.Entities;

namespace Orders.API.Infrastructure.Persistence.Configurations;

public class OrderLineConfiguration : IEntityTypeConfiguration<OrderLine>
{
    public void Configure(EntityTypeBuilder<OrderLine> builder)
    {
        builder.ToTable("OrderLines", "orders");
        builder.HasKey(l => l.Id);
        builder.Property(l => l.Id).ValueGeneratedNever();

        // Mapear el campo privado _lines del aggregate Order.
        // EF Core accede a la colección privada sin que sea pública —
        // fundamental para encapsular el aggregate en DDD.
        builder.HasOne<Order>()
               .WithMany("_lines")            // ← Campo privado en Order
               .HasForeignKey("OrderId")
               .OnDelete(DeleteBehavior.Cascade);

        // ProductId — solo la referencia (no el nombre ni el precio actual)
        builder.Property(l => l.ProductId).IsRequired();

        // Desnormalización estratégica: guardamos el nombre y precio
        // al momento de la compra — valores históricos.
        // Si el nombre del producto cambia, el pedido muestra el nombre correcto
        // para ese momento histórico. Esto es correcto por diseño semántico.
        builder.Property(l => l.ProductName)
               .HasMaxLength(200)
               .IsRequired();

        builder.Property(l => l.ProductSku)
               .HasMaxLength(50);

        // UnitPrice como Owned Entity (precio al momento de la compra)
        builder.OwnsOne(l => l.UnitPrice, money =>
        {
            money.Property(m => m.Amount)
                 .HasColumnName("UnitPriceAmount")
                 .HasPrecision(18, 2)
                 .IsRequired();
            money.Property(m => m.Currency)
                 .HasColumnName("UnitPriceCurrency")
                 .HasMaxLength(3)
                 .IsRequired();
        });

        builder.Property(l => l.Quantity).IsRequired();

        // FK siempre indexada — JOIN con Orders en lecturas
        builder.HasIndex("OrderId")
               .HasDatabaseName("IX_OrderLines_OrderId");
    }
}
```

## Paso 6 — Actualizar Program.cs de Orders.API: AddDbContext con resiliencia

Reemplazar el bloque de `AddDbContext` en `src/Orders.API/Program.cs`:

```csharp
using Microsoft.EntityFrameworkCore;
using Orders.API.Infrastructure.Persistence;
using Orders.API.Infrastructure.Persistence.Interceptors;

// ─── DbContext con resiliencia para Azure SQL ─────────────────────────────────
builder.Services.AddScoped<AuditInterceptor>();

builder.Services.AddDbContext<OrderDbContext>((sp, options) =>
{
    var connectionString = builder.Configuration.GetConnectionString("OrdersDb")!;

    options.UseSqlServer(connectionString, sqlOptions =>
    {
        // Reintentos automáticos para errores transitorios de Azure SQL:
        // throttling (40501), timeout (-2), reconexión (10060)
        sqlOptions.EnableRetryOnFailure(
            maxRetryCount: 5,
            maxRetryDelay: TimeSpan.FromSeconds(30),
            errorNumbersToAdd: null);   // null = errores por defecto de Azure SQL

        sqlOptions.CommandTimeout(60);  // 60s máximo por comando

        // __EFMigrationsHistory en schema "orders" — sin colisión con otros servicios
        // cuando todos usan la misma instancia LocalDB (desarrollo)
        sqlOptions.MigrationsHistoryTable("__EFMigrationsHistory", "orders");
    })
    .AddInterceptors(sp.GetRequiredService<AuditInterceptor>());

    // Desarrollo: logging completo de SQL generado
    if (builder.Environment.IsDevelopment())
    {
        options.EnableSensitiveDataLogging();  // Muestra valores de parámetros
        options.EnableDetailedErrors();        // Stack traces más detallados
    }
});
```

```csharp
// ─── Verificación de migrations al arrancar ───────────────────────────────────
// En producción: si hay migrations pendientes, el servicio NO arranca con tráfico.
// El health check detectará Unhealthy hasta que el Migration Bundle se aplique.
// En Development: aplicar migrations automáticamente para facilitar el trabajo.
if (app.Environment.IsDevelopment())
{
    using var scope = app.Services.CreateScope();
    var dbContext = scope.ServiceProvider.GetRequiredService<OrderDbContext>();

    // En dev: aplicar migrations automáticamente (cómodo para el developer)
    await dbContext.Database.MigrateAsync();

    // Seed de datos de prueba
    await SeedData.InitializeAsync(scope.ServiceProvider);
}
else
{
    // En producción: solo verificar — NO aplicar (las hace el Migration Bundle en CI)
    using var scope = app.Services.CreateScope();
    var dbContext = scope.ServiceProvider.GetRequiredService<OrderDbContext>();
    var pending = await dbContext.Database.GetPendingMigrationsAsync();
    if (pending.Any())
    {
        // El servicio arranca pero el health check lo marcará Unhealthy
        // El gateway no le enviará tráfico hasta que el bundle lo resuelva
        app.Logger.LogCritical(
            "There are {Count} pending migrations: {Migrations}. " +
            "Service will start but health check will be Unhealthy.",
            pending.Count(), string.Join(", ", pending));
    }
}
```

## Paso 11 — ProductNameUpdatedConsumer en Orders.API

Si en el futuro Products.API cambia el nombre de un producto, Orders.API puede invalidar su caché:

Crear `src/Orders.API/Consumers/ProductNameUpdatedConsumer.cs`:

```csharp
using MassTransit;
using Microsoft.Extensions.Caching.Memory;
using OrderFlow.Contracts.Events.Products;

namespace Orders.API.Consumers;

/// <summary>
/// Invalida la caché del ProductsClient cuando Products.API cambia el nombre de un producto.
/// Los pedidos ya completados NO se modifican — conservan el nombre histórico.
/// Solo afecta a la caché de consultas activas en Orders.API.
/// </summary>
public class ProductNameUpdatedConsumer : IConsumer<ProductUpdated>
{
    private readonly IMemoryCache _cache;
    private readonly ILogger<ProductNameUpdatedConsumer> _logger;

    public ProductNameUpdatedConsumer(IMemoryCache cache, ILogger<ProductNameUpdatedConsumer> logger)
    {
        _cache  = cache;
        _logger = logger;
    }

    public Task Consume(ConsumeContext<ProductUpdated> context)
    {
        var productId = context.Message.ProductId;

        // Invalidar la caché del ProductsClient para este producto
        // ProductsClient cachea durante 5 minutos — tras este evento la siguiente
        // llamada obtendrá el nombre actualizado directamente de Products.API
        _cache.Remove($"product:{productId}");

        _logger.LogInformation(
            "Cache invalidated for product {ProductId} after name update",
            productId);

        return Task.CompletedTask;
    }
}
```

```bash

dotnet ef migrations bundle \
    --project src/Orders.API \
    --context OrderDbContext \
    --output artifacts/efbundle-orders \
```

```csharp
public partial class ExpandAddBuyerId : Migration
{
    protected override void Up(MigrationBuilder mb)
    {
        // Nullable para compatibilidad con la versión anterior del servicio
        mb.AddColumn<Guid>("BuyerId", "Orders", "orders", nullable: true);

        // Copiar datos inmediatamente

    protected override void Down(MigrationBuilder mb)
        => mb.DropColumn("BuyerId", "Orders", "orders");
```

## Las reglas del Database per Service

```
REGLA 1 — Aislamiento absoluto:
  Ningún servicio tiene acceso directo a la BD de otro.
  Nunca. Ni siquiera en lectura.

REGLA 2 — Comunicación por API:
  Si Orders necesita datos de Products:
    - Llama a la API HTTP (M4.1) para datos síncronos
    - Escucha eventos (M4.2) para datos asíncronos
    - NUNCA hace JOIN cross-BD

REGLA 3 — Desnormalización estratégica:
  Copia si el valor HISTÓRICO importa (nombre, precio al momento de compra)
  Referencia si solo importa el valor ACTUAL (stock, precio actual)

REGLA 4 — Migrations independientes:
  El equipo de Orders no sabe que existe la migration de Products
  Los bundles se aplican por separado, en el orden correcto
```

## La tabla de desnormalización

| Dato en OrderLine | Tipo | Razón |
|---|---|---|
| `ProductId` | Referencia | Solo el ID — para obtener el estado actual |
| `ProductName` | Copia histórica | El nombre puede cambiar — el pedido lo conserva |
| `ProductSku` | Copia histórica | El SKU puede cambiar — el pedido lo conserva |
| `UnitPrice` | Copia histórica | El precio puede bajar — el pedido tiene el que pagó |
| `Quantity` | Dato propio | Cuántas unidades en este pedido |
| Stock actual | Sin guardar | Solo relevante en Products.API vía API |
