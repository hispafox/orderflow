# M5.2 — HTTPS, mTLS y Gestión de Secretos

## Branch

```bash
git checkout m5.1
git checkout -b m5.2
```

## Objetivo

Eliminar todos los secretos del código fuente y variables de entorno planas. En desarrollo: User Secrets (ya configurados en M5.1). En producción: Azure Key Vault + Managed Identity. Sin passwords entre servicios Azure. HTTPS en todos los entornos. mTLS conceptual con alternativa pragmática (NSG + JWT).

## Paso 4 — Añadir AddAzureKeyVault a todos los servicios

### Patrón común — añadir al comienzo de Program.cs de cada servicio

Este bloque va **antes** de cualquier acceso a la configuración, justo después de `CreateBuilder`:

```csharp
using Azure.Identity;

var builder = WebApplication.CreateBuilder(args);

// ─── Azure Key Vault (solo en producción) ─────────────────────────────────────
// En desarrollo: User Secrets se aplican automáticamente por ASP.NET Core.
// En producción: Key Vault sobreescribe cualquier valor de appsettings.json.
// DefaultAzureCredential funciona en TODOS los entornos sin cambiar código:
//   - Local con `az login`: credenciales del developer
//   - GitHub Actions con OIDC: variables del runner
//   - App Service: Managed Identity automático
if (builder.Environment.IsProduction())
{
    var keyVaultUri = new Uri(
        builder.Configuration["KeyVault:Uri"]
        ?? throw new InvalidOperationException(
            "KeyVault:Uri is required in Production"));

    builder.Configuration.AddAzureKeyVault(
        keyVaultUri,
        new DefaultAzureCredential());
}

// El resto de Program.cs continúa igual...
```

Aplicar este bloque en: **Gateway.API**, **Orders.API**, **Products.API**, **Payments.API**, **Notifications.API**.

builder.Services.AddDbContext<OrderDbContext>(options =>

## Paso 6 — MassTransit con Managed Identity en producción

Actualizar el bloque de MassTransit en **Orders.API**, **Products.API**, **Payments.API** y **Notifications.API**:

```csharp
using Azure.Identity;
using MassTransit;

// ─── MassTransit: RabbitMQ (dev) / Azure Service Bus (prod) ──────────────────
if (builder.Environment.IsProduction())
{
    builder.Services.AddMassTransit(x =>
    {
        x.AddConsumers(typeof(Program).Assembly);

        // Azure Service Bus con Managed Identity — sin connection string ni password
        x.UsingAzureServiceBus((context, cfg) =>
        {
            cfg.Host(
                new Uri("sb://orderflow.servicebus.windows.net"),
                host => { host.TokenCredential = new DefaultAzureCredential(); });

            cfg.UseMessageRetry(r => r.Exponential(
                5, TimeSpan.FromSeconds(1),
                TimeSpan.FromSeconds(30), TimeSpan.FromSeconds(2)));

            cfg.ConfigureEndpoints(context);
        });
    });
}
else
{
    // Desarrollo: RabbitMQ nativo vía Aspire
    builder.Services.AddMassTransit(x =>
    {
        x.AddConsumers(typeof(Program).Assembly);

        x.UsingRabbitMq((context, cfg) =>
        {
            cfg.Host(
                builder.Configuration.GetConnectionString("messaging")!,
                h => { h.Username("guest"); h.Password("guest"); });

            cfg.UseMessageRetry(r => r.Exponential(
                5, TimeSpan.FromSeconds(1),
                TimeSpan.FromSeconds(30), TimeSpan.FromSeconds(2)));

            cfg.ConfigureEndpoints(context);
        });
    });
}
```

> **Nota:** En M4.x se configuró MassTransit solo con RabbitMQ. Este paso añade el bloque condicional para producción. El bloque `else` es el código existente.

## Paso 8 — mTLS en .NET 10 (implementación conceptual)

mTLS añade autenticación mutua: el servidor verifica quién es el cliente. Para el curso se muestra la configuración básica; en producción de TechShop se usará NSG + JWT (ver Paso 9).

### Servidor — Kestrel requiere certificado de cliente

```csharp
// En cualquier servicio que deba requerir mTLS (ej. Products.API)
// Añadir en Program.cs ANTES de builder.Build():

builder.WebHost.ConfigureKestrel(options =>
{
    options.ConfigureHttpsDefaults(httpsOptions =>
    {
        // Requerir certificado del cliente en cada conexión
        httpsOptions.ClientCertificateMode =
            ClientCertificateMode.RequireCertificate;

        // Validar que el certificado viene de nuestra CA interna
        httpsOptions.ClientCertificateValidation = (cert, chain, errors) =>
            cert.Issuer.Contains("OrderFlow Internal CA");
    });
});
```

### Cliente — HttpClient envía certificado en cada request

```csharp
// En Orders.API — configurar ProductsClient para enviar certificado
// (en la factory de HttpClient, en los DelegatingHandlers o en la configuración base)

var certPath = builder.Configuration["Mtls:CertPath"];
var keyPath  = builder.Configuration["Mtls:KeyPath"];

if (!string.IsNullOrEmpty(certPath) && File.Exists(certPath))
{
    builder.Services.AddHttpClient<ProductsClient>()
        .ConfigurePrimaryHttpMessageHandler(() =>
        {
            var handler = new HttpClientHandler();
            var cert = X509Certificate2.CreateFromPemFile(certPath, keyPath);
            handler.ClientCertificates.Add(cert);
            return handler;
        });
}
```

## Paso 9 — Alternativa pragmática: NSG + JWT (la elección de TechShop)

Para TechShop, mTLS completo entre todos los servicios es excesivo en esta fase. La combinación **NSG + JWT** ofrece el 90% de la seguridad con el 20% de la complejidad.

```
Arquitectura de red en Azure (comandos de referencia, no ejecutar en dev):

1. Virtual Network — todos los servicios en la misma VNet privada
2. NSG (Network Security Group) — solo el gateway recibe tráfico externo
3. JWT — cada servicio valida el JWT (ya implementado en M5.1)

Resultado:
- Un atacante externo no puede llamar directamente a orders-api
  (solo el gateway tiene IP pública)
- Un atacante en la red interna (poco probable en VNet Azure)
  puede llamar a orders-api pero sin un JWT válido recibe 401

Comandos Azure CLI (referencia para M7.2):
az network nsg rule create \
    --resource-group orderflow-rg \
    --nsg-name orderflow-nsg \
    --name "DenyDirectInternalAccess" \
    --priority 100 \
    --direction Inbound \
    --access Deny \
    --source-address-prefix Internet \
    --destination-port-ranges 7153 7200 7300 7400
# Solo el puerto 8080 (gateway) queda expuesto públicamente
```

## Paso 10 — Infraestructura Azure: Key Vault y Managed Identity

Estos comandos se ejecutan **una vez** al preparar el entorno de producción (M7.2). Se documentan aquí para referencia.

### Crear Key Vault y añadir secretos

```bash
# Login
az login

# Crear Key Vault (si no existe)
az keyvault create \
    --name orderflow-kv \
    --resource-group orderflow-rg \
    --location westeurope \
    --sku standard

# Añadir secretos — NOTA: Key Vault usa '--' como separador
# Esto mapea automáticamente a 'Jwt:SigningKey' en .NET
az keyvault secret set \
    --vault-name orderflow-kv \
    --name "Jwt--SigningKey" \
    --value "CAMBIAR-POR-CLAVE-SEGURA-MINIMO-32-CHARS"

az keyvault secret set \
    --vault-name orderflow-kv \
    --name "ConnectionStrings--OrdersDb" \
    --value "Server=tcp:orderflow-sql.database.windows.net,1433;Database=OrdersDb;Authentication=Active Directory Default;"

az keyvault secret set \
    --vault-name orderflow-kv \
    --name "ConnectionStrings--ProductsDb" \
    --value "Server=tcp:orderflow-sql.database.windows.net,1433;Database=ProductsDb;Authentication=Active Directory Default;"

az keyvault secret set \
    --vault-name orderflow-kv \
    --name "ConnectionStrings--PaymentsDb" \
    --value "Server=tcp:orderflow-sql.database.windows.net,1433;Database=PaymentsDb;Authentication=Active Directory Default;"

az keyvault secret set \
    --vault-name orderflow-kv \
    --name "ConnectionStrings--NotificationsDb" \
    --value "Server=tcp:orderflow-sql.database.windows.net,1433;Database=NotificationsDb;Authentication=Active Directory Default;"

az keyvault secret set \
    --vault-name orderflow-kv \
    --name "ConnectionStrings--IdentityDb" \
    --value "Server=tcp:orderflow-sql.database.windows.net,1433;Database=IdentityDb;Authentication=Active Directory Default;"
```

### Activar Managed Identity y dar acceso a Key Vault (principio de mínimo privilegio)

```bash
# Activar Managed Identity en cada App Service
for service in orderflow-gateway orderflow-orders orderflow-products orderflow-payments orderflow-notifications; do
    az webapp identity assign \
        --resource-group orderflow-rg \
        --name $service
done

# Gateway: acceso solo a Jwt--SigningKey e IdentityDb
GATEWAY_PRINCIPAL=$(az webapp show \
    --name orderflow-gateway \
    --resource-group orderflow-rg \
    --query identity.principalId -o tsv)

az keyvault set-policy \
    --name orderflow-kv \
    --object-id $GATEWAY_PRINCIPAL \
    --secret-permissions get list

# Orders: acceso solo a sus secretos
ORDERS_PRINCIPAL=$(az webapp show \
    --name orderflow-orders \
    --resource-group orderflow-rg \
    --query identity.principalId -o tsv)

az keyvault set-policy \
    --name orderflow-kv \
    --object-id $ORDERS_PRINCIPAL \
    --secret-permissions get

# Repeat para products, payments, notifications...
```

## Paso 12 — La cadena de configuración completa: verificación

Verificar que la cadena de precedencia funciona correctamente en todos los entornos:

```bash
# Desarrollo — debe leer de User Secrets
dotnet run --project src/Gateway.API
# [DBG] JWT signing key loaded from: UserSecrets

# Producción (simulado) — añadir variable de entorno y verificar Key Vault
ASPNETCORE_ENVIRONMENT=Production dotnet run --project src/Gateway.API
# [INF] Key Vault configuration loaded from: https://orderflow-kv.vault.azure.net/
```

La cadena de precedencia en .NET (la última fuente gana):

```
1. appsettings.json                ← valores base, sin secretos (git-safe)
2. appsettings.{Environment}.json  ← valores de entorno, sin secretos (git-safe)
3. User Secrets (Development)      ← secretos locales, fuera del repo
4. Variables de entorno            ← inyectadas por Aspire o Azure
5. Azure Key Vault (Production)    ← secretos de producción, máxima precedencia
```

En código, siempre igual:

```csharp
// Transparente independientemente del entorno
var signingKey = builder.Configuration["Jwt:SigningKey"];
// Dev:  viene de User Secrets
// Prod: viene de Key Vault (Jwt--SigningKey → Jwt:SigningKey, automático)
```
