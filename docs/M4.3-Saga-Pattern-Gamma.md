## Dónde estamos

Orders publica `OrderCreated`. Notifications lo consume y simula un email. Pero el flujo real de TechShop es más complejo: hay que reservar stock en Products y cobrar en Payments antes de confirmar el pedido.

Si el cobro falla, hay que liberar el stock. Si no hay stock, no se cobra. ¿Cómo coordinamos esto sin una transacción ACID que cruce los tres servicios?

La respuesta es el **Saga Pattern**. En M4.3:
- Creamos `Payments.API`
- Implementamos la Saga completa en MassTransit
- El flujo: Orders → Products (stock) → Payments (cobro) → Orders (confirmar)
- La compensación: si algo falla, se deshace en orden inverso

---

## El problema: transacciones entre servicios

En un monolito, esto es trivial:

```csharp
using var tx = await _dbContext.Database.BeginTransactionAsync();
order.Confirm();
product.ReserveStock(quantity);
payment.Process(amount);
await _dbContext.SaveChangesAsync();
await tx.CommitAsync(); // Todo o nada
```

En microservicios, cada servicio tiene su propia base de datos. No existe una transacción que cruce las tres:

```
OrdersDb      ProductsDb     PaymentsDb
   │               │               │
INSERT Order  UPDATE Stock   INSERT Payment
   │               │               │
¿Cómo garantizas que los tres o ninguno se completan?
```

La solución: el Saga Pattern — una secuencia de transacciones locales coordinadas con eventos.

---

## Choreography vs Orchestration

Hay dos enfoques para implementar Sagas:

**Choreography (coreografía):**
Cada servicio reacciona a eventos y publica sus propios eventos. No hay coordinador central.

```
Orders → [OrderCreated]
Products escucha → reserva stock → [StockReserved]
Payments escucha → cobra → [PaymentProcessed]
Orders escucha → confirma → [OrderConfirmed]
```

**Orchestration (orquestación):**
Un coordinador central (la Saga) dirige el flujo. Sabe en qué estado está y qué paso ejecutar.

```
OrderSaga → ReserveStock → Products
OrderSaga ← StockReserved
OrderSaga → ProcessPayment → Payments
OrderSaga ← PaymentProcessed
OrderSaga → ConfirmOrder
```

**Para OrderFlow:** orquestación con MassTransit State Machine. Más fácil de depurar y gestionar el estado.

---

## Cuándo NO usar Saga

La Saga es poderosa pero no es la solución para todo. Cuándo evitarla:

**Flujos simples sin compensación:** Si la operación en un servicio no necesita deshacerse ante un fallo posterior, una simple llamada HTTP síncrona es suficiente.

**Datos que deben ser inmediatamente consistentes:** Si el negocio no puede tolerar inconsistencia temporal, la Saga no es apropiada — aunque en práctica, muy pocos requisitos de negocio son tan estrictos.

**Equipos sin experiencia en eventos:** La Saga añade complejidad operativa. Si el equipo no tiene experiencia depurando flujos asíncronos, el coste de aprendizaje puede no justificarse.

**Para OrderFlow:** la Saga es la elección correcta porque el flujo de pedido tiene compensación natural (liberar stock si el pago falla), los servicios son verdaderamente independientes, y el equipo de TechShop se está formando en este curso para gestionar esa complejidad.

---

## Saga vs Two-Phase Commit: la elección correcta

¿Por qué Saga y no 2PC (Two-Phase Commit)?

**Two-Phase Commit (2PC):**
```
Coordinator → Prepare → (Orders, Products, Payments responden "Ready")
Coordinator → Commit → (todos ejecutan)
Si alguno falla → Abort → (todos deshacen)
```

**Problemas de 2PC en microservicios:**
- Requiere que todos los servicios participen en el protocolo
- Si el coordinador cae durante Commit → estado inconsistente indefinido
- Bloquea recursos hasta que todos responden → baja disponibilidad
- No escala bien con muchos servicios

**Ventajas de Saga:**
- Cada servicio hace su transacción local y sigue
- Si falla, la compensación deshace los pasos completados
- No hay bloqueo global de recursos
- Escala horizontalmente sin problemas

**El precio de la Saga:** consistencia eventual en vez de inmediata. Entre el INSERT de la order y el UPDATE del stock, hay un momento en que la BD no es consistente — pero converge al estado correcto.

## El dominio de Payments: simple y enfocado

```csharp
// Payments.API/Domain/Payment.cs

public class Payment
{
    public Guid Id { get; private set; } = Guid.NewGuid();
    public Guid OrderId { get; private set; }
    public Guid CustomerId { get; private set; }
    public decimal Amount { get; private set; }
    public string Currency { get; private set; } = string.Empty;
    public PaymentStatus Status { get; private set; }
    public string? FailureReason { get; private set; }
    public DateTime CreatedAt { get; private set; } = DateTime.UtcNow;
    public DateTime? ProcessedAt { get; private set; }

    private Payment() { }

    public static Payment Create(
        Guid orderId, Guid customerId, decimal amount, string currency)
        => new()
        {
            OrderId = orderId,
            CustomerId = customerId,
            Amount = amount,
            Currency = currency,
            Status = PaymentStatus.Pending
        };

    public void MarkAsProcessed()
    {
        Status = PaymentStatus.Processed;
        ProcessedAt = DateTime.UtcNow;
    }

    public void MarkAsFailed(string reason)
    {
        Status = PaymentStatus.Failed;
        FailureReason = reason;
        ProcessedAt = DateTime.UtcNow;
    }

    public void MarkAsRefunded()
    {
        if (Status != PaymentStatus.Processed)
            throw new InvalidOperationException("Cannot refund unprocessed payment");
        Status = PaymentStatus.Refunded;
    }
}

public enum PaymentStatus { Pending, Processed, Failed, Refunded }
```

## El FakePaymentGateway

```csharp
// Payments.API/Services/FakePaymentGateway.cs

public class FakePaymentGateway : IPaymentGateway
{
    private readonly ILogger<FakePaymentGateway> _logger;
    private static readonly Random _random = new();

    public FakePaymentGateway(ILogger<FakePaymentGateway> logger)
        => _logger = logger;

    public async Task<PaymentResult> ProcessAsync(
        Guid orderId, decimal amount, string currency,
        CancellationToken ct = default)
    {
        // Simular latencia de un gateway real
        await Task.Delay(TimeSpan.FromMilliseconds(200), ct);

        // 90% de éxito, 10% de fallo (para poder probar compensaciones)
        var success = _random.NextDouble() > 0.1;

        if (success)
        {
            _logger.LogInformation(
                "[PAGO SIMULADO] ✅ Cobro de {Amount} {Currency} " +
                "para pedido {OrderId} autorizado",
                amount, currency, orderId);

            return new PaymentResult(Success: true,
                AuthCode: $"AUTH-{Guid.NewGuid():N}"[..12].ToUpperInvariant());
        }
        else
        {
            _logger.LogWarning(
                "[PAGO SIMULADO] ❌ Cobro de {Amount} {Currency} " +
                "para pedido {OrderId} RECHAZADO",
                amount, currency, orderId);

            return new PaymentResult(Success: false,
                FailureReason: "Insufficient funds");
        }
    }
}

public record PaymentResult(bool Success, string? AuthCode = null,
    string? FailureReason = null);
```

## El Consumer de ProcessPayment

```csharp
// Payments.API/Consumers/ProcessPaymentConsumer.cs

public class ProcessPaymentConsumer : IConsumer<ProcessPayment>
{
    private readonly PaymentDbContext _dbContext;
    private readonly IPaymentGateway _gateway;
    private readonly ILogger<ProcessPaymentConsumer> _logger;

    public async Task Consume(ConsumeContext<ProcessPayment> context)
    {
        var msg = context.Message;

        _logger.LogInformation(
            "Processing payment for order {OrderId}. Amount: {Amount} {Currency}",
            msg.OrderId, msg.Amount, msg.Currency);

        var payment = Payment.Create(
            msg.OrderId, msg.CustomerId, msg.Amount, msg.Currency);

        var result = await _gateway.ProcessAsync(
            msg.OrderId, msg.Amount, msg.Currency,
            context.CancellationToken);

        if (result.Success)
        {
            payment.MarkAsProcessed();
            _dbContext.Payments.Add(payment);
            await _dbContext.SaveChangesAsync(context.CancellationToken);

            // Publicar evento de éxito → la Saga lo recibe
            await context.Publish(new PaymentProcessed
            {
                OrderId = msg.OrderId,
                PaymentId = payment.Id,
                Amount = payment.Amount
            }, context.CancellationToken);
        }
        else
        {
            payment.MarkAsFailed(result.FailureReason!);
            _dbContext.Payments.Add(payment);
            await _dbContext.SaveChangesAsync(context.CancellationToken);

            // Publicar evento de fallo → la Saga inicia compensación
            await context.Publish(new PaymentFailed
            {
                OrderId = msg.OrderId,
                Reason = result.FailureReason!
            }, context.CancellationToken);
        }
    }
}
```

## La Saga: configuración del flujo feliz

```csharp
    public OrderSaga()
    {
        InstanceState(x => x.CurrentState);

        // Correlacionar por OrderId
        Event(() => OrderCreatedEvent,
            x => x.CorrelateById(ctx => ctx.Message.OrderId));
        Event(() => StockReservedEvent,
            x => x.CorrelateById(ctx => ctx.Message.OrderId));
        Event(() => PaymentProcessedEvent,
            x => x.CorrelateById(ctx => ctx.Message.OrderId));

        // Estado inicial → Pending
        Initially(
            When(OrderCreatedEvent)
                .Then(ctx =>
                {
                    ctx.Saga.CustomerId = ctx.Message.CustomerId;
                    ctx.Saga.CustomerEmail = ctx.Message.CustomerEmail;
                    ctx.Saga.ProductId = ctx.Message.Items.First().ProductId;
                    ctx.Saga.Quantity = ctx.Message.Items.First().Quantity;
                    ctx.Saga.Amount = ctx.Message.Total;
                    ctx.Saga.Currency = ctx.Message.Currency;
                    ctx.Saga.CreatedAt = DateTime.UtcNow;
                })
                // Pedir a Products que reserve stock
                .Publish(ctx => new ReserveStock
                {
                    OrderId = ctx.Saga.CorrelationId,
                    ProductId = ctx.Saga.ProductId,
                    Quantity = ctx.Saga.Quantity
                })
                .TransitionTo(Pending));
```

## La Saga: reserva de stock y cobro

```csharp
        // Pending → StockReserved: pedir cobro
        During(Pending,
            When(StockReservedEvent)
                .Publish(ctx => new ProcessPayment
                {
                    OrderId = ctx.Saga.CorrelationId,
                    CustomerId = ctx.Saga.CustomerId,
                    Amount = ctx.Saga.Amount,
                    Currency = ctx.Saga.Currency
                })
                .TransitionTo(PaymentProcessing),

            // Si no hay stock → fallo directo sin compensación
            When(StockInsufficientEvent)
                .Then(ctx => ctx.Saga.FailureReason =
                    $"Insufficient stock: requested {ctx.Message.RequestedQuantity}, " +
                    $"available {ctx.Message.AvailableQuantity}")
                .Publish(ctx => new OrderFailed
                {
                    OrderId = ctx.Saga.CorrelationId,
                    Reason = ctx.Saga.FailureReason!
                })
                .TransitionTo(Failed)
                .Finalize());

        // PaymentProcessing → Completed: confirmar pedido
        During(PaymentProcessing,
            When(PaymentProcessedEvent)
                .Then(ctx =>
                {
                    ctx.Saga.PaymentId = ctx.Message.PaymentId;
                    ctx.Saga.CompletedAt = DateTime.UtcNow;
                })
                .Publish(ctx => new OrderConfirmed
                {
                    OrderId = ctx.Saga.CorrelationId,
                    CustomerId = ctx.Saga.CustomerId,
                    CustomerEmail = ctx.Saga.CustomerEmail,
                    PaymentId = ctx.Saga.PaymentId!.Value,
                    Total = ctx.Saga.Amount
                })
                .TransitionTo(Completed)
                .Finalize());
```

## La Saga: compensación cuando el pago falla

```csharp
            // PaymentProcessing → Compensating: liberar stock
            When(PaymentFailedEvent)
                .Then(ctx =>
                    ctx.Saga.FailureReason = ctx.Message.Reason)
                // Pedir a Products que libere el stock reservado
                .Publish(ctx => new ReleaseStock
                {
                    OrderId = ctx.Saga.CorrelationId,
                    ProductId = ctx.Saga.ProductId,
                    Quantity = ctx.Saga.Quantity
                })
                .TransitionTo(Compensating));

        During(Compensating,
            When(StockReleasedEvent)
                // Stock liberado → publicar fallo del pedido
                .Publish(ctx => new OrderFailed
                {
                    OrderId = ctx.Saga.CorrelationId,
                    Reason = ctx.Saga.FailureReason!
                })
                .TransitionTo(Failed)
                .Finalize());

        // Limpiar el estado de la Saga cuando finaliza
        SetCompletedWhenFinalized();
    }
}
```

## El diagrama de estados de la Saga

```
Initial
  │ OrderCreated
  ▼
Pending ──────────────────────────────────────────► Failed
  │ StockReserved                StockInsufficient  │
  ▼                                                  │
PaymentProcessing                                    │
  │                                                  │
  ├─ PaymentProcessed ──────────────────────────► Completed
  │
  └─ PaymentFailed
       │
       ▼
  Compensating
  (ReleaseStock publicado)
       │ StockReleased
       ▼
     Failed
  (OrderFailed publicado)
```

El estado de la Saga se persiste en base de datos entre cada transición. Si el sistema cae y se reinicia, la Saga continúa desde el último estado guardado.

## El patrón Outbox y la Saga: coordinación perfecta

En M4.2 publicamos `OrderCreated` directamente desde el handler. Hay un problema:

```csharp
await _repository.SaveAsync(order, ct);     // 1. INSERT en BD (éxito)
await _publishEndpoint.Publish(OrderCreated); // 2. Publish a RabbitMQ
// Si el proceso cae entre 1 y 2:
// Pedido en BD: ✅
// OrderCreated publicado: ❌
// Saga nunca arranca → pedido huérfano
```

La solución robusta es el **Outbox Pattern** (M6.2), donde el evento se guarda en la misma transacción que el pedido y se publica de forma garantizada.

MassTransit tiene su propio Outbox integrado:

```csharp
// En Program.cs de Orders.API
x.AddEntityFrameworkOutbox<OrderDbContext>(o =>
{
    o.UseSqlServer();
    o.UseBusOutbox(); // Integra el outbox con el bus
});
```

Con esto, `_publishEndpoint.Publish()` dentro de una transacción EF Core guarda el mensaje en una tabla `OutboxMessages`. Un background service lo publica a RabbitMQ. Si el proceso cae, el mensaje sigue ahí.

## Persistir el estado de la Saga con EF Core

```csharp
// Orders.API/Program.cs — Registrar la Saga con EF Core

builder.Services.AddMassTransit(x =>
{
    // Registrar la Saga con persistencia EF Core
    x.AddSagaStateMachine<OrderSaga, OrderSagaState>()
        .EntityFrameworkRepository(r =>
        {
            r.ConcurrencyMode = ConcurrencyMode.Optimistic;
            r.AddDbContext<DbContext, OrderDbContext>(
                (provider, builder) =>
                    builder.UseSqlServer(
                        provider.GetRequiredService<IConfiguration>()
                            .GetConnectionString("OrdersDb")));
        });

    // Registrar el consumer de Payments.API
    x.AddConsumer<OrderCreatedConsumer>();

    x.UsingRabbitMq((context, cfg) =>
    {
        cfg.Host(builder.Configuration.GetConnectionString("messaging")!);
        cfg.UseMessageRetry(r => r.Exponential(5,
            TimeSpan.FromSeconds(1), TimeSpan.FromSeconds(30),
            TimeSpan.FromSeconds(2)));
        cfg.ConfigureEndpoints(context);
    });
});
```

## El flujo completo con la Saga

Flujo feliz cuando todo va bien:

```
Cliente → POST /api/orders → Orders.API
    Orders crea Order → SQL INSERT
    Orders publica OrderCreated → RabbitMQ
    Saga inicia: Pending
                    ↓
    Saga publica ReserveStock → Products.API
    Products reserva stock → SQL UPDATE
    Products publica StockReserved → RabbitMQ
    Saga: Pending → PaymentProcessing
                    ↓
    Saga publica ProcessPayment → Payments.API
    Payments cobra → SQL INSERT Payment
    Payments publica PaymentProcessed → RabbitMQ
    Saga: PaymentProcessing → Completed
                    ↓
    Saga publica OrderConfirmed → RabbitMQ
    Notifications consume OrderConfirmed → email de confirmación
    Orders actualiza estado a Confirmed
← 201 Created (Orders respondió mucho antes, cliente ya tiene respuesta)
```

---

## El flujo de compensación

Cuando el pago falla:

```
...
Payments cobra → RECHAZADO por el gateway
Payments publica PaymentFailed → RabbitMQ
Saga: PaymentProcessing → Compensating
    │
    ├─ Saga publica ReleaseStock → Products.API
    │  Products libera stock → SQL UPDATE (stock vuelve)
    │  Products publica StockReleased → RabbitMQ
    │
    └─ Saga: Compensating → Failed
       Saga publica OrderFailed → RabbitMQ
       Notifications consume OrderFailed → email de fallo al cliente
       Orders actualiza estado a Cancelled
```

El cliente ya tiene el 201 (pedido creado). Si falla la compensación completa, recibirá un email indicando que el pedido no pudo procesarse. La compensación garantiza que el stock no quede bloqueado indefinidamente.

## Consumer de StockReserved en Products.API

Products.API necesita un consumer para el comando `ReserveStock`:

```csharp
// Products.API/Consumers/ReserveStockConsumer.cs

public class ReserveStockConsumer : IConsumer<ReserveStock>
{
    private readonly IProductRepository _repository;

    public async Task Consume(ConsumeContext<ReserveStock> context)
    {
        var msg = context.Message;
        var product = await _repository.GetByIdAsync(msg.ProductId,
            context.CancellationToken);

        if (product is null)
        {
            await context.Publish(new StockInsufficient
            {
                OrderId = msg.OrderId,
                ProductId = msg.ProductId,
                RequestedQuantity = msg.Quantity,
                AvailableQuantity = 0
            }, context.CancellationToken);
            return;
        }

        if (!product.HasSufficientStock(msg.Quantity))
        {
            await context.Publish(new StockInsufficient
            {
                OrderId = msg.OrderId,
                ProductId = msg.ProductId,
                RequestedQuantity = msg.Quantity,
                AvailableQuantity = product.Stock
            }, context.CancellationToken);
            return;
        }

        product.ReserveStock(msg.Quantity, msg.OrderId);
        await _repository.SaveAsync(product, context.CancellationToken);

        await context.Publish(new StockReserved
        {
            OrderId = msg.OrderId,
            ProductId = msg.ProductId,
            ReservedQuantity = msg.Quantity
        }, context.CancellationToken);
    }
}
```

```csharp
// Products.API/Consumers/ReleaseStockConsumer.cs

public class ReleaseStockConsumer : IConsumer<ReleaseStock>
{
    private readonly IProductRepository _repository;
    private readonly ILogger<ReleaseStockConsumer> _logger;

    public async Task Consume(ConsumeContext<ReleaseStock> context)
    {
        var msg = context.Message;
        var product = await _repository.GetByIdAsync(msg.ProductId,
            context.CancellationToken);

        if (product is null)
        {
            _logger.LogError(
                "Cannot release stock: product {ProductId} not found",
                msg.ProductId);
            return;
        }

        // Devolver el stock al inventario
        product.UpdateStock(product.Stock + msg.Quantity);
        await _repository.SaveAsync(product, context.CancellationToken);

        _logger.LogInformation(
            "Released {Quantity} units of {ProductId} for order {OrderId}",
            msg.Quantity, msg.ProductId, msg.OrderId);

        await context.Publish(new StockReleased
        {
            OrderId = msg.OrderId,
            ProductId = msg.ProductId,
            ReleasedQuantity = msg.Quantity
        }, context.CancellationToken);
    }
}
```

## Concurrencia en la Saga: el ConcurrencyMode

Cuando múltiples mensajes llegan para la misma Saga simultáneamente, necesitamos controlar la concurrencia:

```csharp
// ConcurrencyMode.Optimistic (nuestra elección):
// Usa rowversion en SQL Server
// Si dos instancias intentan actualizar el mismo estado → una falla
// La que falla reintenta → eventualmente una gana

// ConcurrencyMode.Pessimistic:
// Bloquea la fila con SELECT FOR UPDATE
// Más lento pero garantiza exclusión mutua estricta

r.ConcurrencyMode = ConcurrencyMode.Optimistic;
```

**Para OrderFlow:** Optimistic es suficiente. La Saga de un pedido recibe mensajes en secuencia (StockReserved → PaymentProcessed), raramente concurrentes.

## Saga Timeout: qué pasa si un servicio no responde

¿Qué pasa si Products nunca responde con `StockReserved`?

MassTransit Sagas soportan timeouts:

```csharp
public class OrderSaga : MassTransitStateMachine<OrderSagaState>
{
    public Schedule<OrderSagaState, OrderSagaTimeout> ReservationTimeout { get; private set; } = null!;

    public OrderSaga()
    {
        Schedule(() => ReservationTimeout, x => x.TimeoutTokenId,
            s =>
            {
                s.Delay = TimeSpan.FromMinutes(5); // Si en 5min no hay respuesta
                s.Received = x => x.CorrelateById(ctx => ctx.Message.OrderId);
            });

        Initially(
            When(OrderCreatedEvent)
                .Publish(/* ReserveStock */)
                .Schedule(ReservationTimeout, // Activar timeout
                    ctx => new OrderSagaTimeout { OrderId = ctx.Saga.CorrelationId })
                .TransitionTo(Pending));

        During(Pending,
            When(StockReservedEvent)
                .Unschedule(ReservationTimeout) // Cancelar timeout
                .TransitionTo(PaymentProcessing),

            When(ReservationTimeout.Received)
                .Then(ctx => ctx.Saga.FailureReason = "Stock reservation timeout")
                .Publish(/* OrderFailed */)
                .Finalize());
    }
}
```

## Idempotencia en la Saga: mensajes duplicados

La Saga puede recibir el mismo mensaje dos veces (at-least-once delivery). Debemos manejar esto:

```csharp
// En ReserveStockConsumer — verificar idempotencia
public async Task Consume(ConsumeContext<ReserveStock> context)
{
    var msg = context.Message;

    // ¿Ya reservamos stock para este pedido?
    var alreadyReserved = await _repository
        .ExistsReservationAsync(msg.OrderId, msg.ProductId, ct);

    if (alreadyReserved)
    {
        // Idempotente: publicar StockReserved de nuevo (ya lo procesamos)
        await context.Publish(new StockReserved
        {
            OrderId = msg.OrderId,
            ProductId = msg.ProductId,
            ReservedQuantity = msg.Quantity
        });
        return;
    }

    // Primera vez — procesar normalmente
    // ...
}
```

La Saga también es idempotente por diseño: si recibe el mismo `StockReserved` dos veces, el segundo simplemente no cambia el estado (ya está en `PaymentProcessing`).

## Compensación parcial: el caso difícil

¿Qué pasa si la compensación también falla? Por ejemplo, `ReleaseStock` falla:

```
PaymentFailed → Saga publica ReleaseStock
ReleaseStockConsumer falla
MassTransit reintenta (5 veces, exponencial)
Todos los reintentos fallan → ReleaseStock va a la DLQ
```

**El estado:** stock reservado, pago fallido, pedido en estado `Compensating` sin resolver.

**La solución:**
1. Alerta en Application Insights: mensajes en DLQ
2. Operación manual: el equipo de soporte libera el stock manualmente
3. Herramienta de administración: endpoint interno para forzar la compensación

Este es el "happy path operativo" — la Saga tiene mecanismos de recovery para casos extremos.

## La Saga como documentación viva

Uno de los beneficios menos mencionados de la Saga: es documentación ejecutable del flujo de negocio.

```csharp
// Cualquier developer puede leer la Saga y entender el flujo completo
Initially(
    When(OrderCreatedEvent)
        // "Cuando llega un pedido, reservamos el stock"
        .Publish(ReserveStock)
        .TransitionTo(Pending));

During(Pending,
    When(StockReservedEvent)
        // "Cuando el stock está reservado, procesamos el pago"
        .Publish(ProcessPayment)
        .TransitionTo(PaymentProcessing),

    When(StockInsufficientEvent)
        // "Si no hay stock suficiente, el pedido falla directamente"
        .Publish(OrderFailed)
        .Finalize());

During(PaymentProcessing,
    When(PaymentProcessedEvent)
        // "Cuando el pago se procesa, el pedido se confirma"
        .Publish(OrderConfirmed)
        .Finalize(),

    When(PaymentFailedEvent)
        // "Si el pago falla, liberamos el stock y notificamos del fallo"
        .Publish(ReleaseStock)
        .TransitionTo(Compensating));
```

Un product manager puede leer este código y verificar que implementa correctamente el flujo de negocio.

## El valor real de la Saga: resiliencia del flujo

¿Qué pasa si Payments.API se cae durante el procesamiento?

**Sin Saga:**
El pedido queda en estado inconsistente: stock reservado, sin cobro, sin confirmación. Nadie lo sabe.

**Con Saga:**
```
OrderSaga está en estado PaymentProcessing
El mensaje ProcessPayment está en la queue
Payments.API se cae → mensaje queda en la queue
Payments.API se recupera → consume el mensaje pendiente
Payments.API procesa → publica PaymentProcessed
OrderSaga continúa desde PaymentProcessing → Completed
```

El mensaje persiste en RabbitMQ mientras Payments está caído. Cuando vuelve, continúa exactamente donde se quedó. El cliente ya tiene el 201 — el email de confirmación llega unos minutos después.

## Preguntas frecuentes sobre Sagas

**¿Cuántos pasos puede tener una Saga?**
No hay límite técnico. En la práctica, más de 6-8 pasos empiezan a ser difíciles de gestionar. Si el flujo es muy complejo, considera dividir en sub-Sagas.

**¿Puedo tener múltiples Sagas activas para el mismo pedido?**
No deberías — pero puede pasar si hay mensajes duplicados sin idempotencia. El `ConcurrencyMode.Optimistic` previene actualizaciones simultáneas del mismo estado de Saga.

**¿La Saga garantiza exactamente una ejecución?**
No — garantiza at-least-once. Por eso los consumers deben ser idempotentes.

**¿Cuándo usar Choreography en vez de Orchestration?**
Choreography es mejor cuando los servicios son verdaderamente independientes y el flujo puede cambiar sin un coordinador central. Para flujos complejos con compensación, la Orchestration (Saga state machine) es más manejable.

## Resumen de M4.3

**El patrón Saga resuelve el problema de transacciones distribuidas** sin necesitar 2PC.

**Choreography vs Orchestration:** elegimos orquestación (Saga como state machine) porque el flujo es predecible, el estado es visible y la compensación es directa.

**MassTransit State Machine:** `Initially`, `During`, `When`, `Publish`, `TransitionTo`, `Finalize`. El código de la Saga es el flujo de negocio ejecutable.

**Estado persistido en SQL Server:** `OrderSagaState` guarda el progreso entre mensajes. Si el sistema cae, la Saga continúa desde donde estaba.

**Compensación automática:** si el pago falla, `ReleaseStock` se publica automáticamente. Si la compensación falla, los mensajes van a la DLQ para revisión manual.

**Payments.API:** servicio nuevo con `FakePaymentGateway` (90% éxito). Consume `ProcessPayment`, publica `PaymentProcessed` o `PaymentFailed`.

Al terminar M4.3, el flujo de pedido de TechShop está completamente implementado. En M4.4 añadimos YARP como punto de entrada único.
