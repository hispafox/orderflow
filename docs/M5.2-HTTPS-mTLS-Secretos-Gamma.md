## Dónde estamos

En M5.1 añadimos autenticación con JWT y ASP.NET Core Identity. Los usuarios se identifican y el gateway valida los tokens. Pero quedan dos problemas:

**Problema 1 — Secretos:** La `Jwt:SigningKey` está en User Secrets en desarrollo. ¿Dónde va en producción? No en variables de entorno planas — si alguien obtiene acceso al App Service, los lee todos.

**Problema 2 — Comunicación interna:** Los servicios se comunican por HTTP internamente. Si alguien compromete la red interna de Azure, puede leer el tráfico. mTLS añade cifrado y autenticación mutua entre servicios.

En M5.2 resolvemos ambos con herramientas nativas de Azure y .NET 10.

## La pirámide de gestión de secretos

Los secretos tienen diferentes niveles de protección según el entorno:

```
DESARROLLO LOCAL:
  ✅ User Secrets (dotnet user-secrets)
  ❌ NUNCA en appsettings.json
  ❌ NUNCA en el código fuente
  ❌ NUNCA en variables de entorno del sistema

CI/CD (GitHub Actions):
  ✅ GitHub Secrets (cifrados, no visibles en logs)
  ✅ Variables de entorno del runner (solo durante la ejecución)

PRODUCCIÓN (Azure):
  ✅ Azure Key Vault (el estándar)
  ✅ Managed Identity para acceder a Key Vault (sin password)
  ❌ NUNCA en Application Settings como texto plano
```

## User Secrets: desarrollo local (ya configurado)

Los User Secrets de .NET almacenan secretos fuera del repositorio, en la máquina del developer:

```bash
# Inicializar User Secrets en un proyecto
dotnet user-secrets init --project src/Gateway.API

# Añadir un secreto
dotnet user-secrets set "Jwt:SigningKey" "clave-secreta-32-chars-minimo!" \
    --project src/Gateway.API

# Ver todos los secretos
dotnet user-secrets list --project src/Gateway.API

# Ruta de almacenamiento en Windows:
# %APPDATA%\Microsoft\UserSecrets\{userSecretsId}\secrets.json
```

Los User Secrets están fuera del directorio del proyecto — nunca entran en git aunque alguien olvide el `.gitignore`.

## Azure Key Vault: el estándar para secretos en producción

**Azure Key Vault** es el servicio de Azure para almacenar y gestionar:
- **Secretos:** Strings cifrados (client secrets, connection strings, API keys)
- **Claves:** Claves criptográficas RSA/EC (para firma, cifrado)
- **Certificados:** Certificados X.509 con gestión del ciclo de vida

**Por qué Key Vault y no variables de entorno de App Service:**
```
Variables de entorno de App Service:
❌ Cualquier persona con acceso al portal puede leerlas
❌ No hay registro de quién accedió a qué secreto
❌ Rotación manual — hay que actualizar en cada servicio
❌ Sin versionado — si algo falla, no puedes volver atrás

Azure Key Vault:
✅ Cifrado en reposo y en tránsito
✅ Audit log completo (quién accedió, cuándo)
✅ Rotación automática de secretos
✅ Versionado — puedes recuperar versiones anteriores
✅ Managed Identity — sin passwords para acceder
```

## Crear el Key Vault en Azure

```bash
# Instalar Azure CLI si no está
winget install Microsoft.AzureCLI

# Login
az login

# Crear Resource Group (si no existe)
az group create --name orderflow-rg --location westeurope

# Crear Key Vault
az keyvault create \
    --name orderflow-kv \
    --resource-group orderflow-rg \
    --location westeurope \
    --sku standard

# Añadir secretos
az keyvault secret set \
    --vault-name orderflow-kv \
    --name "Jwt--SigningKey" \
    --value "tu-client-secret-aqui"

az keyvault secret set \
    --vault-name orderflow-kv \
    --name "ConnectionStrings--OrdersDb" \
    --value "Server=tcp:orderflow-sql.database.windows.net,1433;..."
```

**Nota:** Azure Key Vault usa `--` como separador de jerarquía (en vez de `:`) para mapear a la configuración de .NET.

## Rotación de secretos: el problema de los secretos de larga vida

Los secretos que nunca rotan son una vulnerabilidad permanente. Si un secreto se filtra y nadie lo sabe, el atacante tiene acceso indefinido.

**Rotación manual (lo mínimo):**
```bash
# Generar nuevo client clave de firma JWT
# Admin Console → Clients → orderflow-gateway → Credentials → Regenerate Secret

# Actualizar en Key Vault
az keyvault secret set \
    --vault-name orderflow-kv \
    --name "Jwt--SigningKey" \
    --value "nuevo-secret-generado"

# Los App Services leen Key Vault en cada request de configuración
# El cambio se aplica sin redeploy
```

**Rotación automática con Key Vault:**
Key Vault puede rotar secretos automáticamente para servicios compatibles (Azure SQL, Azure Storage). Para la Jwt:SigningKey, la rotación requiere actualizar el secreto en Key Vault y reiniciar los servicios.

## Secrets Rotation: el proceso de rotación en producción

Cuando hay que rotar el Jwt:SigningKey (cada 6-12 meses es buena práctica):

```
PROCESO SIN DOWNTIME:

1. Generar nuevo clave de firma JWT
   (los tokens emitidos con la clave anterior siguen siendo válidos hasta su expiración natural)

2. Añadir el nuevo secret a Key Vault como versión nueva
   az keyvault secret set --name "Jwt--SigningKey" --value "nuevo-secret"

3. Los App Services leen Key Vault periódicamente (o en el próximo reinicio)
   → Algunos servicios usan el viejo, otros el nuevo (ventana de transición)

4. Verificar que todos los servicios funcionan con el nuevo secret
   (logs: sin errores de auth en Application Insights)

5. Eliminar la clave de firma antigua de Key Vault

6. Limpiar la versión vieja de Key Vault (opcional)
```

Sin downtime, sin interrupciones. Este proceso se puede documentar y ejecutar en < 30 minutos.

## Configurar Key Vault en el pipeline de CI/CD

```yaml

name: Deploy OrderFlow

on:
  push:
    branches: [main]

jobs:
  deploy:
    runs-on: ubuntu-latest

    permissions:
      id-token: write   # Para OIDC con Azure
      contents: read

    steps:

      # Login en Azure con OIDC (sin client secret en GitHub Secrets)

          tenant-id: ${{ secrets.AZURE_TENANT_ID }}

      # No hay secretos de Key Vault en el pipeline

            --resource-group orderflow-rg \
            --name orderflow-gateway \
```

Con OIDC, incluso el `AZURE_CLIENT_ID` no es un secret real — es un ID público de la aplicación registrada.

## Principio de mínimo privilegio: cada servicio solo accede a lo que necesita

Un error común: dar a todos los servicios acceso a todos los secretos.

**Correcto — acceso granular por servicio:**

```
orderflow-gateway (Managed Identity):
  ✅ Key Vault: GET Jwt--SigningKey
  ✅ Key Vault: GET Jwt--Issuer
  ❌ Key Vault: OrdersDb connection string (no la necesita)

orderflow-orders (Managed Identity):
  ✅ Key Vault: GET ConnectionStrings--OrdersDb
  ✅ Key Vault: GET Jwt--Issuer
  ❌ Key Vault: PaymentsDb connection string (no la necesita)

orderflow-payments (Managed Identity):
  ✅ Key Vault: GET ConnectionStrings--PaymentsDb
  ✅ Key Vault: GET PaymentGateway--ApiKey
  ❌ Key Vault: OrdersDb connection string (no la necesita)
```

Si un servicio es comprometido, el atacante solo tiene acceso a los secretos de ese servicio.

Para que el alumno pueda justificar la elección en su empresa:

| Enfoque | Entorno | Pro | Contra |
|---|---|---|---|

## Managed Identity: autenticación sin contraseña

**Managed Identity** es la forma en que los servicios de Azure se autentican entre sí sin passwords, sin secrets, sin nada que gestionar.

```
SIN Managed Identity:
Gateway.API necesita acceder a Key Vault
→ Necesita un client_id y client_secret para autenticarse en Azure AD
→ ¿Dónde guardas ese secret? ¿En Key Vault? ¿Pero para acceder a Key Vault necesitas el secret... 🔄

CON Managed Identity:
Azure asigna automáticamente una identidad al App Service de Gateway.API
→ Azure AD sabe que "este App Service = esta identidad"
→ Gateway.API puede autenticarse en Key Vault sin ningún secret
→ No hay nada que rotar, nada que gestionar
```

## Activar Managed Identity en App Services

```bash
# Activar System-Assigned Managed Identity en cada App Service
az webapp identity assign \
    --resource-group orderflow-rg \
    --name orderflow-gateway

az webapp identity assign \
    --resource-group orderflow-rg \
    --name orderflow-orders

az webapp identity assign \
    --resource-group orderflow-rg \
    --name orderflow-products

# Dar acceso a Key Vault a cada identidad
az keyvault set-policy \
    --name orderflow-kv \
    --object-id $(az webapp show --name orderflow-gateway \
        --resource-group orderflow-rg \
        --query identity.principalId -o tsv) \
    --secret-permissions get list
```

Cada App Service tiene su propia identidad. Solo el gateway y los servicios que necesitan los secretos tienen acceso — principio de mínimo privilegio.

## Renovación automática de Managed Identity: cómo funciona internamente

Managed Identity usa tokens de Azure AD internamente, con renovación automática transparente:

```
1. App Service arranca → Azure inyecta un endpoint local especial
   http://169.254.169.254/metadata/identity/oauth2/token

2. Cuando tu código usa DefaultAzureCredential:
   → Llama automáticamente a ese endpoint local
   → Obtiene un token de Azure AD para la identidad del App Service
   → El token dura 24 horas (se renueva automáticamente)

3. Con ese token, accede a Key Vault, SQL, Service Bus, etc.
   → Sin configuración manual
   → Sin passwords
   → Sin rotaciones
```

El developer no ve nada de esto — solo usa `new DefaultAzureCredential()` y funciona.

```csharp
var builder = WebApplication.CreateBuilder(args);

// En producción: leer secretos de Key Vault
if (builder.Environment.IsProduction())
{

        builder.Configuration["KeyVault:Uri"]

// En desarrollo: User Secrets (ya configurado automáticamente)
// builder.Configuration.AddUserSecrets<Program>();
// ASP.NET Core lo añade automáticamente en Development
```

## La cadena de configuración completa

En .NET 10, la configuración se lee de múltiples fuentes en orden de precedencia (la última gana):

```
1. appsettings.json             ← valores base (sin secretos)
2. appsettings.{Environment}.json ← valores de entorno (sin secretos)
3. User Secrets (Development)   ← secretos locales del developer
4. Variables de entorno         ← inyectadas por Aspire o Azure
5. Azure Key Vault (Production) ← secretos de producción
```

Un secreto en Key Vault como `Jwt--SigningKey` mapea automáticamente a `Jwt:SigningKey` en la configuración de .NET.

```csharp
// En código — se lee igual independientemente de la fuente
var secret = builder.Configuration["Jwt:SigningKey"];
// En desarrollo: viene de User Secrets
// En producción: viene de Key Vault (transparente)
```

```csharp
// Orders.API/Program.cs — en producción

builder.Services.AddDbContext<OrderDbContext>(options =>
```

## HTTPS en los servicios: SSL Termination

En producción con Azure App Services, el SSL se termina en Azure Front Door o Application Gateway:

```
Cliente ──HTTPS──► Azure App Service (SSL termination en la plataforma)
                        │
                        ├──HTTP──► orders-api (red interna Azure)
                        ├──HTTP──► products-api
                        └──HTTP──► payments-api
```

Azure gestiona el certificado SSL. Los servicios internos se comunican por HTTP dentro de la red privada de Azure Virtual Network.

**Para el curso:** HTTPS está configurado automáticamente por Aspire en desarrollo con certificados de desarrollo. En producción, App Services gestiona los certificados.

## CORS y su relación con HTTPS

CORS (Cross-Origin Resource Sharing) y HTTPS son conceptos relacionados pero distintos:

**HTTPS:** Cifra el tráfico entre cliente y servidor. Protege contra interceptación (man-in-the-middle).

**CORS:** Controla qué orígenes (dominios) pueden hacer requests al servidor desde un navegador. Protege contra CSRF y acceso cross-origin no autorizado.

Para el gateway de OrderFlow, ambos son necesarios:

```csharp
// HTTPS — Kestrel fuerza HTTPS
app.UseHttpsRedirection();

// CORS — solo el dominio de TechShop puede llamar al gateway
builder.Services.AddCors(options =>
    options.AddPolicy("TechShopOnly", policy =>
        policy.WithOrigins("https://techshop.es")
              .AllowAnyMethod()
              .AllowAnyHeader()));

app.UseCors("TechShopOnly");
```

## Criptografía post-cuántica en .NET 10

.NET 10 incluye soporte nativo para los nuevos algoritmos post-cuánticos del NIST:

**ML-KEM (Module Lattice-based Key Encapsulation Mechanism):**
- Sucesor de los algoritmos de intercambio de claves RSA/ECDH
- Resistente a ataques de computadores cuánticos
- Disponible en `System.Security.Cryptography.MLKem`

**ML-DSA (Module Lattice-based Digital Signature Algorithm):**
- Sucesor de RSA/ECDSA para firmas digitales
- Disponible en `System.Security.Cryptography.MLDsa`

```csharp
// Ejemplo de ML-KEM en .NET 10
using System.Security.Cryptography;

// Generar par de claves
using var kem = MLKem.GenerateKey(MLKemAlgorithm.MLKem768);
var publicKey = kem.ExportPublicKey();

// Encapsular (generar shared secret cifrado)
var (ciphertext, sharedSecret) = MLKem.Encapsulate(publicKey);

// Desencapsular (recuperar el shared secret)
var recoveredSecret = kem.Decapsulate(ciphertext);
```

Para OrderFlow: JWT usa HS256 en desarrollo y RS256 en producción. En el futuro, cuando los clientes soporten ML-DSA, podremos migrar. .NET 10 ya está preparado.

## mTLS: autenticación mutua entre servicios

En mTLS (mutual TLS), ambos lados del canal se autentican con certificados:

```
HTTP (normal):
Cliente → Servidor: "Aquí está mi request"
Servidor → Cliente: "OK, aquí está mi response"
(Solo el servidor se autentica con un certificado)

mTLS:
Cliente → Servidor: "Aquí está mi certificado de cliente"
Servidor verifica el certificado del cliente
Servidor → Cliente: "Aquí está mi certificado de servidor"
Cliente verifica el certificado del servidor
(Ambos se autentican — autentica la identidad del servicio)
```

Con mTLS, aunque alguien intercepte la red interna, no puede impersonar a Orders.API sin el certificado correcto.

## mTLS en .NET 10: configuración

```csharp
// Configurar el servidor para requerir certificado de cliente
var builder = WebApplication.CreateBuilder(args);

builder.WebHost.ConfigureKestrel(options =>
{
    options.ConfigureHttpsDefaults(httpsOptions =>
    {
        // Requerir certificado de cliente
        httpsOptions.ClientCertificateMode =
            ClientCertificateMode.RequireCertificate;

        // Validar el certificado del cliente
        httpsOptions.ClientCertificateValidation =
            (cert, chain, errors) =>
            {
                // Verificar que el certificado está firmado por nuestra CA
                return cert.Issuer.Contains("OrderFlow Internal CA");
            };
    });
});
```

```csharp
// Configurar el cliente (HttpClient) para enviar certificado
var handler = new HttpClientHandler();
var cert = X509Certificate2.CreateFromPemFile(
    "certs/orders-api.crt",
    "certs/orders-api.key");
handler.ClientCertificates.Add(cert);

var client = new HttpClient(handler);
```

## mTLS en Azure App Services

En Azure, la gestión de certificados para mTLS es más sencilla con App Service Certificates:

```bash
# Subir el certificado de cliente de Orders.API
az webapp client-cert upload \
    --resource-group orderflow-rg \
    --name orderflow-orders \
    --certificate-file orders-api.pfx \
    --certificate-password password123

# Configurar el App Service destino para requerir certificados de cliente
az webapp update \
    --resource-group orderflow-rg \
    --name orderflow-products \
    --set clientCertEnabled=true \
    --set clientCertMode=Required
```

## Alternativa pragmática: Network Security Groups

Para muchos equipos, mTLS entre todos los servicios internos es excesivo. Una alternativa más pragmática para OrderFlow en Azure:

```
Red interna de Azure Virtual Network:
- Solo el Gateway tiene IP pública (ingress)
- Los servicios internos solo aceptan tráfico desde el Gateway
- Configurado con Azure Network Security Groups (NSG)

NSG Rules para orders-api:
ALLOW: Gateway-API → orders-api (puerto 443)
DENY: Todo lo demás → orders-api

¿Alguien compromete la red interna?
→ No puede llamar a orders-api directamente
→ Debe pasar por el gateway (que valida el JWT)
```

Esta arquitectura de red + JWT es suficiente para TechShop. mTLS añade una capa extra para sistemas de alta seguridad (financiero, sanidad).

## Jwt:SigningKey: la clave más crítica

La clave de firma JWT sigue la misma pirámide de secretos:

```
Desarrollo:
  dotnet user-secrets set "Jwt:SigningKey" "abc123"
  ← Solo en la máquina del developer

CI/CD:
  GitHub Secret: JWT_SIGNING_KEY
  ← Cifrado por GitHub, no visible en logs

Producción:
  az keyvault secret set --name "Jwt--SigningKey" --value "..."
  ← En Key Vault, accedido por Managed Identity
```

En el código, siempre:
```csharp
var clientSecret = builder.Configuration["Jwt:SigningKey"];
// En dev: viene de User Secrets
// En prod: viene de Key Vault → transparente
```

## Renovación automática de tokens: el refresh en acción

En un escenario real, los tokens expiran. El gateway puede gestionar la renovación automáticamente:

```csharp
// En el Gateway — middleware para refrescar tokens próximos a expirar
public class TokenRefreshMiddleware
{
    private readonly RequestDelegate _next;

    public async Task InvokeAsync(HttpContext context)
    {
        await _next(context);

        // Si la respuesta es 401 y el cliente envió un Refresh-Token header
        if (context.Response.StatusCode == 401 &&
            context.Request.Headers.TryGetValue("X-Refresh-Token", out var refreshToken))
        {
            // Intentar renovar el token automáticamente
            var newToken = await RefreshTokenAsync(refreshToken!);

            if (newToken is not null)
            {
                // Informar al cliente del nuevo token en el header de respuesta
                context.Response.Headers["X-New-Access-Token"] = newToken;
            }
        }
    }
}
```

Para el curso, la renovación se gestiona en el cliente (.http file: obtener nuevo token). Este middleware es un patrón avanzado para aplicaciones de producción.

## La regla de oro: nunca confiar en el origen del request

Un principio de seguridad que aplica a nivel de código:

```csharp
// ❌ VULNERABLE — confiar en el header X-User-Id que cualquiera puede falsificar
[HttpGet("my-orders")]
public async Task<IActionResult> GetMyOrders(
    [FromHeader(Name = "X-User-Id")] Guid userId)
{
    // Un atacante puede poner cualquier userId en el header
    return Ok(await _repository.GetByCustomerAsync(userId));
}

// ✅ CORRECTO — el userId viene del JWT verificado criptográficamente
[HttpGet("my-orders")]
[Authorize]
public async Task<IActionResult> GetMyOrders()
{
    // El JWT fue verificado por el gateway y/o por este servicio
    // No puede ser falsificado sin la Jwt:SigningKey
    var userId = Guid.Parse(User.FindFirstValue("sub")!);
    return Ok(await _repository.GetByCustomerAsync(userId));
}
```

Esta regla aplica a TODOS los identificadores sensibles: userId, orderId (verificar que pertenece al usuario), email, roles.

## Verificación de ownership: john no puede ver pedidos de jane

Un error de autorización común (OWASP A01 — Broken Access Control):

```csharp
// ❌ VULNERABLE — el usuario puede ver pedidos de otros
[HttpGet("{id:guid}")]
[Authorize]
public async Task<ActionResult<OrderDto>> GetById(Guid id)
{
    var order = await _repository.GetByIdAsync(id);
    if (order is null) return NotFound();
    return Ok(order.ToDto()); // john podría ver el pedido de jane si conoce el ID
}

// ✅ CORRECTO — verificar que el pedido pertenece al usuario
[HttpGet("{id:guid}")]
[Authorize]
public async Task<ActionResult<OrderDto>> GetById(Guid id)
{
    var order = await _repository.GetByIdAsync(id);
    if (order is null) return NotFound();

    var currentUserId = Guid.Parse(User.FindFirstValue("sub")!);
    var isAdmin = User.IsInRole("admin");

    if (order.CustomerId != currentUserId && !isAdmin)
        return Forbid(); // 403 — el pedido existe pero no es tuyo

    return Ok(order.ToDto());
}
```

## Protección de datos en reposo: SQL TDE

Azure SQL tiene Transparent Data Encryption (TDE) activado por defecto:

```
Sin TDE:
Ficheros .mdf en disco → datos en texto plano
Si alguien accede al disco → lee los datos

Con TDE (activo por defecto en Azure SQL):
Ficheros .mdf en disco → datos cifrados con AES-256
Si alguien accede al disco → solo ve bytes cifrados
La clave DEK (Database Encryption Key) está protegida por Azure Key Vault
```

Para OrderFlow: TDE está activo automáticamente en Azure SQL Database. No necesitas configurarlo. Para LocalDB en desarrollo, los datos no están cifrados (no es producción, no tiene datos reales).

## La prueba de fuego: revocar un token comprometido

Si se filtra un JWT de un usuario, ¿cómo lo invalidas inmediatamente?

**Opción A — Esperar a que expire (5 minutos):**
Si el access token dura 5 minutos, el atacante tiene 5 minutos máximo. Para muchos casos, aceptable.

**Opción B — Revocar el refresh token (Identity DB):**
```sql
DELETE FROM AspNetUserTokens WHERE UserId = '{userId}';
```
```bash
curl -X POST https://gateway.local/account/revoke \
  -H "Authorization: Bearer $ADMIN_TOKEN"
```

Esto invalida el refresh token — el usuario no puede obtener nuevos access tokens. El access token actual sigue siendo válido hasta que expire.

**Opción C — Token introspection por cada request:**
El servicio puede mantener una blacklist de tokens revocados en memoria o Redis (breve duración hasta expiración natural). ASP.NET Core Identity gestiona la revocación de refresh tokens directamente en AspNetUserTokens. Para revocar access tokens antes de su expiración, la opción más práctica es acortar su vida (15 min) y aceptar que el token puede usarse hasta que expire natudar en tiempo real. Añade latencia (~20ms por request). Para sistemas de alta seguridad (banca).

**Para TechShop:** Opción A + B. Tokens de 5 minutos + logout de usuario si se detecta compromiso.

## Una nota sobre Zero Trust

El modelo de seguridad tradicional es "confiar en la red interna". Si estás dentro del firewall, confías en todo.

El modelo **Zero Trust** dice: "Nunca confíes, siempre verifica". Incluso dentro de la red interna.

OrderFlow implementa Zero Trust de forma pragmática:
- El gateway verifica JWT en cada request externo
- Los servicios verifican JWT en las peticiones internas (opcional pero recomendado)
- Key Vault + Managed Identity: verificación mutua sin contraseñas
- NSG: incluso dentro de la red, solo el gateway llega a los servicios
- Audit log: registro de cada acceso

No es Zero Trust completo (mTLS en todas las comunicaciones internas), pero es un 90% del valor con un 20% de la complejidad.

## Preguntas frecuentes sobre seguridad de secretos

**¿Son seguros los User Secrets si alguien accede a mi PC?**
No — los User Secrets están en texto plano en `%APPDATA%`. Si alguien tiene acceso físico o de sistema a tu PC, puede leerlos. Son solo para proteger de commits accidentales en git. Para secretos muy sensibles en local: usar Bitlocker y contraseña de Windows.

**¿Key Vault tiene coste?**
Sí, pero mínimo. ~0.004€ por 10.000 operaciones. Con OrderFlow haciendo ~100 lecturas por reinicio de servicio, el coste mensual sería de céntimos. Totalmente asumible.

**¿Puedo usar Key Vault en desarrollo local?**
Sí. Con `az login`, `DefaultAzureCredential` usa las credenciales del developer para acceder a Key Vault. Útil si quieres tener el mismo Key Vault en dev y prod — aunque User Secrets suele ser más cómodo.

**¿Managed Identity funciona fuera de Azure?**
No directamente. Para máquinas on-premise, usar un Service Principal (client_id + client_secret) o Workload Identity Federation. Para el curso (todo en Azure): Managed Identity es suficiente.
