## Dónde estamos

Orders.API tiene dominio, Controllers, CQRS con MediatR. Products.API tiene sus Minimal APIs. Cuando en M4.1 conectemos estos dos servicios, Orders necesitará llamar a Products para verificar stock y precios antes de crear un pedido.

Esa llamada puede fallar. Products puede estar caído, lento, o respondiendo con errores. Si Orders no tiene protección, cualquier problema en Products hace que crear pedidos falle también — fallo en cascada.

En M3.5 añadimos resiliencia con Polly v8: la capa que permite que Orders siga funcionando aunque Products tenga problemas temporales.

---

## El problema del fallo en cascada

Sin resiliencia, un fallo en Products destruye a Orders:

```
1. Products.API empieza a responder lento (query lenta en BD)
2. Orders llama a Products — espera... timeout a 30 segundos
3. Mientras espera, Orders acumula peticiones (thread pool)
4. Más peticiones llegan — más threads bloqueados esperando
5. El thread pool se agota
6. Orders no puede procesar NINGUNA petición
7. YARP devuelve 503 a todos los clientes
8. Todo cae — por culpa de una query lenta en Products
```

**El coste:** si Products tiene un problema durante 5 minutos y TechShop pierde 500 pedidos/hora, son más de 40 pedidos perdidos. Con un ticket medio de 200€, 8.000€ de ventas perdidas.

La resiliencia no es un lujo — es dinero.

## Polly v8: la evolución del estándar

Polly es la librería de resiliencia más usada en el ecosistema .NET. En la versión 8 (la actual), el API cambió completamente:

**Polly v7 (legacy):**
```csharp
// Cadenas de policies separadas y verbosas
Policy.Handle<HttpRequestException>()
      .OrResult<HttpResponseMessage>(r => !r.IsSuccessStatusCode)
      .WaitAndRetryAsync(3, attempt => TimeSpan.FromSeconds(Math.Pow(2, attempt)));
```

**Polly v8 (nuevo API):**
```csharp
// ResiliencePipeline — un builder fluido
new ResiliencePipelineBuilder<HttpResponseMessage>()
    .AddRetry(new RetryStrategyOptions<HttpResponseMessage>
    {
        MaxRetryAttempts = 3,
        Delay = TimeSpan.FromSeconds(1),
        BackoffType = DelayBackoffType.Exponential
    })
    .Build();
```

El nuevo API es más expresivo, más composable y tiene mejor integración con .NET 10 y HttpClientFactory.

## Polly v8 vs Polly v7: tabla de migración

Si tienes proyectos con Polly v7, la migración a v8 requiere cambios:

| Polly v7 | Polly v8 |
|---|---|
| `Policy.Handle<>().WaitAndRetryAsync()` | `ResiliencePipelineBuilder.AddRetry()` |
| `Policy.Handle<>().CircuitBreakerAsync()` | `AddCircuitBreaker()` |
| `Policy.TimeoutAsync()` | `AddTimeout()` |
| `Policy.BulkheadAsync()` | `AddRateLimiter()` (concurrencia) |
| `Policy.FallbackAsync()` | `AddFallback()` |
| `PolicyWrap` | `ResiliencePipelineBuilder` (inline) |
| `IAsyncPolicy<T>` | `ResiliencePipeline<T>` |
| `AddPolicyHandler(policy)` | `AddResilienceHandler(name, builder)` |

Para proyectos existentes con Polly v7, la migración es gradual — puedes tener ambas versiones en el mismo proyecto durante la transición.

## Los cinco estrategias de resiliencia

Polly v8 ofrece cinco estrategias que se pueden combinar en un pipeline:

**1. Retry** — Reintentar cuando falla. Con backoff exponencial y jitter.

**2. Circuit Breaker** — Cortar llamadas cuando demasiados fallan. Tres estados: Closed, Open, Half-Open.

**3. Timeout** — Cancelar si tarda demasiado. Protege de esperas indefinidas.

**4. Rate Limiter** — Limitar el ritmo de llamadas salientes. Protege a Products de sobrecarga.

**5. Fallback** — Respuesta alternativa cuando todo falla. Datos en caché, respuesta degradada.

En Orders.API, para las llamadas a Products, usamos las cinco en un pipeline.

---

## Retry: reintentar con backoff exponencial

El retry más básico: si falla, vuelve a intentarlo. Pero con cuidado — reintentar inmediatamente puede saturar al servicio caído.

**Backoff exponencial:** cada reintento espera el doble que el anterior.
- 1er reintento: espera 1 segundo
- 2º reintento: espera 2 segundos
- 3er reintento: espera 4 segundos

**Jitter:** añade variación aleatoria. Si 100 clientes reintentan todos a los 2 segundos exactos, generan un pico que puede tumbar al servicio recuperándose. Con jitter, unos reintentan a 1.7s, otros a 2.3s — la carga se distribuye.

```csharp
new RetryStrategyOptions<HttpResponseMessage>
{
    MaxRetryAttempts = 3,
    Delay = TimeSpan.FromSeconds(1),
    BackoffType = DelayBackoffType.Exponential, // 1s, 2s, 4s
    UseJitter = true, // Añade variación aleatoria ±25%
    ShouldHandle = new PredicateBuilder<HttpResponseMessage>()
        .Handle<HttpRequestException>()
        .HandleResult(r => r.StatusCode >= HttpStatusCode.InternalServerError)
}
```

## Retry: qué errores reintentar

No todos los errores merecen un retry. La regla:

**SÍ reintentar (errores transitorios):**
- Timeout de red
- 503 Service Unavailable
- 500 Internal Server Error (puede ser temporal)
- 429 Too Many Requests (Products sobrecargado momentáneamente)
- `HttpRequestException` (fallo de conexión)

**NO reintentar (errores permanentes):**
- 400 Bad Request — el request es inválido, no va a mejorar
- 401 Unauthorized — sin token válido no hay retry que funcione
- 404 Not Found — el producto no existe
- 422 Unprocessable Entity — error de negocio, no de red

```csharp
ShouldHandle = new PredicateBuilder<HttpResponseMessage>()
    .Handle<HttpRequestException>()
    .Handle<TimeoutRejectedException>()
    .HandleResult(r => r.StatusCode is
        HttpStatusCode.ServiceUnavailable or
        HttpStatusCode.InternalServerError or
        HttpStatusCode.TooManyRequests)
    // NO 400, NO 401, NO 404, NO 422
```

## Thundering Herd: por qué el Jitter es esencial

Imagina 500 instancias de Orders.API conectadas a Products. Products se cae durante 30 segundos. Cuando vuelve:

**Sin Jitter:**
```
t=30.0s: Las 500 instancias hacen su primer reintento SIMULTÁNEAMENTE
→ 500 peticiones a la vez → Products cae de nuevo
t=32.0s: Las 500 reintentan SIMULTÁNEAMENTE
→ otro pico → Products cae de nuevo
Bucle: Products nunca puede recuperarse
```

**Con Jitter:**
```
t=30.0s: Instancias reintentan en ventana de 500ms ± 25% aleatoriamente
→ ~50 peticiones cada 50ms → carga distribuida
→ Products puede recuperarse gradualmente
```

El Jitter es una línea de código (`UseJitter = true`) que puede ser la diferencia entre una recuperación suave y un bucle de fallos en cascada.

## Circuit Breaker: los tres estados

El Circuit Breaker es el patrón más importante para prevenir fallos en cascada.

**Estado CERRADO (normal):**
Las llamadas pasan. El CB cuenta los fallos. Si los fallos superan el umbral en la ventana de tiempo → **abre**.

**Estado ABIERTO (protección):**
Las llamadas se rechazan INMEDIATAMENTE sin llegar a Products. Se lanza `BrokenCircuitException`. Orders puede seguir funcionando para otras operaciones. Tras un tiempo de espera → pasa a **half-open**.

**Estado SEMI-ABIERTO (prueba):**
Permite pasar un número limitado de llamadas de prueba. Si tienen éxito → **cierra** (todo vuelve a la normalidad). Si fallan → **abre** de nuevo.

```
Closed ──(umbral de fallos)──► Open
  ▲                               │
  └──(pruebas exitosas)──── Half-Open
                               ▲
                               └──(timeout)──┘
```

---

## Circuit Breaker: implementación

```csharp
new CircuitBreakerStrategyOptions<HttpResponseMessage>
{
    // Abrir el circuito si el 50% de las peticiones fallan
    FailureRatio = 0.5,

    // En una ventana de 30 segundos
    SamplingDuration = TimeSpan.FromSeconds(30),

    // Con mínimo 5 peticiones para evaluar
    MinimumThroughput = 5,

    // Permanecer abierto 30 segundos antes de probar
    BreakDuration = TimeSpan.FromSeconds(30),

    // Qué errores cuentan como fallo
    ShouldHandle = new PredicateBuilder<HttpResponseMessage>()
        .Handle<HttpRequestException>()
        .HandleResult(r => r.StatusCode >= HttpStatusCode.InternalServerError),

    // Callbacks opcionales para logging y métricas
    OnOpened = args =>
    {
        logger.LogWarning("Circuit opened! Products.API is unavailable");
        return ValueTask.CompletedTask;
    },
    OnClosed = args =>
    {
        logger.LogInformation("Circuit closed! Products.API is back");
        return ValueTask.CompletedTask;
    }
}
```

## Timeout: no esperar para siempre

Sin timeout, si Products no responde, el thread de Orders queda bloqueado indefinidamente. Con muchos clientes, el thread pool se agota.

```csharp
new TimeoutStrategyOptions
{
    // Máximo 5 segundos esperando a Products
    Timeout = TimeSpan.FromSeconds(5),

    // Callback cuando se produce un timeout
    OnTimeout = args =>
    {
        logger.LogWarning(
            "Timeout calling Products.API after {Timeout}s",
            args.Timeout.TotalSeconds);
        return ValueTask.CompletedTask;
    }
}
```

Cuando se produce un timeout, Polly lanza `TimeoutRejectedException`. El Retry lo captura y reintenta. Si todos los reintentos hacen timeout, el Circuit Breaker lo cuenta como fallo.

**¿Cuánto timeout?** Regla práctica: 2-3 veces el tiempo de respuesta normal del 95th percentil. Si Products responde en 200ms normalmente, un timeout de 1-2 segundos es razonable.

## Rate Limiter: proteger a Products de sobrecarga

El Rate Limiter saliente (egress) limita cuántas peticiones enviamos a Products por segundo. Protege a Products de ser saturado por Orders:

```csharp
new RateLimiterStrategyOptions
{
    // Máximo 100 peticiones por segundo a Products
    Limiter = PartitionedRateLimiter.Create<ResilienceContext, string>(
        ctx => RateLimitPartition.GetFixedWindowLimiter(
            "products-api",
            _ => new FixedWindowRateLimiterOptions
            {
                Window = TimeSpan.FromSeconds(1),
                PermitLimit = 100,
                QueueLimit = 0 // Sin cola — rechaza inmediatamente si supera el límite
            }))
}
```

## Fallback: el plan B

Cuando todo falla (retries agotados, circuit abierto), el Fallback proporciona una respuesta alternativa:

```csharp
new FallbackStrategyOptions<HttpResponseMessage>
{
    ShouldHandle = new PredicateBuilder<HttpResponseMessage>()
        .Handle<BrokenCircuitException>()   // Circuit abierto
        .Handle<TimeoutRejectedException>() // Timeout total
        .Handle<HttpRequestException>(),    // Error de red

    FallbackAction = args =>
    {
        logger.LogWarning(
            "Products.API unavailable. Using cached data.");

        // Devolver respuesta alternativa con datos en caché
        var cachedResponse = cache.Get<ProductPriceDto>(
            $"product-price-{productId}");

        if (cachedResponse is not null)
        {
            var json = JsonSerializer.Serialize(cachedResponse);
            var response = new HttpResponseMessage(HttpStatusCode.OK)
            {
                Content = new StringContent(json, Encoding.UTF8, "application/json")
            };
            return ValueTask.FromResult(response);
        }

        // Sin caché — indicar que el precio no está disponible
        return ValueTask.FromResult(
            new HttpResponseMessage(HttpStatusCode.ServiceUnavailable));
    }
}
```

## Bulkhead: aislar recursos entre operaciones

El patrón Bulkhead (mamparo) aísla los recursos para que un componente saturado no afecte a otros.

Sin Bulkhead:
```
Orders.API tiene un thread pool compartido.
Si 100 peticiones de "crear pedido" esperan a Products → pool saturado.
Las peticiones de "listar pedidos" (que no necesitan Products) → también fallan.
```

Con Bulkhead (mediante Rate Limiter concurrente):
```csharp
new RateLimiterStrategyOptions
{
    // Máximo 10 peticiones concurrentes a Products
    Limiter = PartitionedRateLimiter.Create<ResilienceContext, string>(
        ctx => RateLimitPartition.GetConcurrencyLimiter(
            "products-api",
            _ => new ConcurrencyLimiterOptions
            {
                PermitLimit = 10,    // Máximo 10 llamadas simultáneas
                QueueLimit = 5       // 5 más en cola, el resto se rechazan
            }))
}
```

Si hay 20 peticiones simultáneas de crear pedido: 10 pasan, 5 esperan en cola, 5 se rechazan con error claro. El resto de Orders.API sigue funcionando.

## El ResiliencePipeline completo para Products.API

Combinamos todas las estrategias en un pipeline. El orden importa — se ejecutan de afuera hacia adentro:

```csharp
// Orders.API/Infrastructure/Resilience/ProductsResiliencePipeline.cs

public static class ProductsResiliencePipeline
{
    public static ResiliencePipeline<HttpResponseMessage> Build(
        ILogger logger,
        IMemoryCache cache)
    {
        return new ResiliencePipelineBuilder<HttpResponseMessage>()

            // 1. Fallback (más externo — captura lo que escapa)
            .AddFallback(BuildFallbackOptions(logger, cache))

            // 2. Timeout total (por todo el pipeline)
            .AddTimeout(TimeSpan.FromSeconds(10))

            // 3. Retry (reintenta antes de que el CB abra)
            .AddRetry(BuildRetryOptions(logger))

            // 4. Circuit Breaker (después del retry)
            .AddCircuitBreaker(BuildCircuitBreakerOptions(logger))

            // 5. Timeout por intento (cada retry individual)
            .AddTimeout(TimeSpan.FromSeconds(3))

            .Build();
    }
}
```

El orden: Fallback → Timeout total → Retry → Circuit Breaker → Timeout por intento.

---

## El Pipeline completo: orden de las estrategias

El orden en que se encadenan las estrategias en el pipeline es crítico. Cada estrategia envuelve a la siguiente:

```
Petición entra →
  [Fallback]            ← captura todo lo que no se recuperó
    [Timeout total]     ← límite para todo el intento incluyendo retries
      [Retry]           ← reintenta si el interior falla
        [Circuit Breaker] ← rechaza si demasiados fallos recientes
          [Timeout/intento] ← límite por cada intento individual
            → HTTP call a Products.API
```

**Por qué este orden:**
- El **Fallback** debe ser el más externo — captura cualquier excepción que el resto no resolvió
- El **Timeout total** limita el tiempo máximo incluyendo todos los reintentos
- El **Retry** reintenta cuando el Circuit Breaker o el Timeout por intento fallan
- El **Circuit Breaker** está dentro del Retry — si el CB está abierto, el Retry no lo reintenta (evita reintentos inútiles con circuito abierto)
- El **Timeout por intento** está más al interior — cada intento individual tiene su propio límite

## Polly v8 con Named Pipelines

Para reutilizar el mismo pipeline en múltiples HttpClients:

```csharp
// Registrar el pipeline con un nombre
builder.Services.AddResiliencePipeline<string, HttpResponseMessage>(
    "products-resilience",
    (pipeline, context) =>
    {
        var logger = context.ServiceProvider
            .GetRequiredService<ILogger<Program>>();

        pipeline
            .AddRetry(new HttpRetryStrategyOptions { MaxRetryAttempts = 3 })
            .AddCircuitBreaker(new HttpCircuitBreakerStrategyOptions
            {
                FailureRatio = 0.5,
                BreakDuration = TimeSpan.FromSeconds(30)
            })
            .AddTimeout(TimeSpan.FromSeconds(5));
    });

// Usar el pipeline por nombre en múltiples clientes
builder.Services
    .AddHttpClient<ProductsClient>()
    .AddResilienceHandler("products-resilience");

builder.Services
    .AddHttpClient<StockClient>()
    .AddResilienceHandler("products-resilience"); // Mismo pipeline
```

## Integración con HttpClientFactory y ServiceDefaults

La forma más limpia de usar Polly con HttpClientFactory en .NET 10:

```csharp
// Orders.API/Program.cs

// Cliente HTTP para Products con resiliencia automática
builder.Services.AddHttpClient<ProductsClient>(client =>
    client.BaseAddress = new Uri("http://products-api")) // Aspire service discovery

    // Resilience handler de Microsoft.Extensions.Http.Resilience
    .AddResilienceHandler("products-pipeline", pipeline =>
    {
        pipeline
            .AddFallback(new FallbackStrategyOptions<HttpResponseMessage>
            {
                ShouldHandle = new PredicateBuilder<HttpResponseMessage>()
                    .Handle<BrokenCircuitException>()
                    .Handle<TimeoutRejectedException>(),
                FallbackAction = _ => ValueTask.FromResult(
                    new HttpResponseMessage(HttpStatusCode.ServiceUnavailable))
            })
            .AddRetry(new HttpRetryStrategyOptions
            {
                MaxRetryAttempts = 3,
                Delay = TimeSpan.FromMilliseconds(500),
                BackoffType = DelayBackoffType.Exponential,
                UseJitter = true
            })
            .AddCircuitBreaker(new HttpCircuitBreakerStrategyOptions
            {
                FailureRatio = 0.5,
                SamplingDuration = TimeSpan.FromSeconds(30),
                MinimumThroughput = 5,
                BreakDuration = TimeSpan.FromSeconds(30)
            })
            .AddTimeout(TimeSpan.FromSeconds(5));
    });
```

## HttpRetryStrategyOptions y HttpCircuitBreakerStrategyOptions

`Microsoft.Extensions.Http.Resilience` incluye opciones específicas para HTTP que ya tienen configurado `ShouldHandle` correcto para peticiones HTTP:

```csharp
// HttpRetryStrategyOptions ya sabe qué errores HTTP reintentar:
// - HttpRequestException
// - 429 Too Many Requests
// - 503 Service Unavailable
// - 500 Internal Server Error
```

## El ProductsClient con resiliencia

El cliente HTTP que Orders.API usa para llamar a Products:

```csharp
// Orders.API/Infrastructure/Http/ProductsClient.cs

namespace Orders.API.Infrastructure.Http;

public class ProductsClient
{
    private readonly HttpClient _httpClient;
    private readonly ILogger<ProductsClient> _logger;

    public ProductsClient(HttpClient httpClient, ILogger<ProductsClient> logger)
    {
        _httpClient = httpClient;
        _logger = logger;
    }

    public async Task<ProductPriceDto?> GetProductAsync(
        Guid productId, CancellationToken ct = default)
    {
        try
        {
            var response = await _httpClient.GetAsync(
                $"/api/products/{productId}", ct);

            if (response.StatusCode == HttpStatusCode.NotFound)
                return null;

            if (!response.IsSuccessStatusCode)
            {
                _logger.LogWarning(
                    "Products.API returned {StatusCode} for product {ProductId}",
                    response.StatusCode, productId);
                return null;
            }

            return await response.Content
                .ReadFromJsonAsync<ProductPriceDto>(ct);
        }
        catch (BrokenCircuitException ex)
        {
            _logger.LogWarning(ex,
                "Circuit open — Products.API unavailable for product {ProductId}",
                productId);
            return null; // O datos cacheados
        }
    }
}
```

## Usar ProductsClient en el CreateOrderHandler

```csharp
// CreateOrderHandler actualizado con ProductsClient

public class CreateOrderHandler : IRequestHandler<CreateOrderCommand, OrderDto>
{
    private readonly IOrderRepository _repository;
    private readonly ProductsClient _productsClient;
    private readonly ILogger<CreateOrderHandler> _logger;

    public async Task<OrderDto> Handle(
        CreateOrderCommand command, CancellationToken ct)
    {
        // Verificar precios y stock en Products.API
        var validatedItems = new List<(Guid, string, int, Money)>();

        foreach (var item in command.Items)
        {
            var product = await _productsClient.GetProductAsync(item.ProductId, ct);

            if (product is null)
                throw new DomainException(
                    $"Product {item.ProductId} is not available");

            if (product.Stock < item.Quantity)
                throw new DomainException(
                    $"Insufficient stock for {product.Name}. " +
                    $"Available: {product.Stock}, Requested: {item.Quantity}");

            // Usar el precio oficial del catálogo, no el del cliente
            validatedItems.Add((
                item.ProductId,
                product.Name,
                item.Quantity,
                new Money(product.Price, product.Currency)));
        }

        var order = Order.Create(command.CustomerId, validatedItems, address);
        await _repository.SaveAsync(order, ct);

        return order.ToDto();
    }
}
```

## OnRetry: logging detallado de cada reintento

```csharp
.AddRetry(new HttpRetryStrategyOptions
{
    MaxRetryAttempts = 3,
    Delay = TimeSpan.FromMilliseconds(500),
    BackoffType = DelayBackoffType.Exponential,
    UseJitter = true,

    OnRetry = args =>
    {
        var statusCode = args.Outcome.Result?.StatusCode;
        var exception = args.Outcome.Exception?.Message;

        _logger.LogWarning(
            "Retry {Attempt}/{MaxAttempts} for Products.API. " +
            "Cause: {Cause}. Next attempt in {DelayMs}ms",
            args.AttemptNumber + 1,
            3,
            statusCode?.ToString() ?? exception ?? "Unknown",
            args.RetryDelay.TotalMilliseconds);

        // También podemos registrar una métrica
        _ordersMetrics.RecordRetry("products-api", args.AttemptNumber + 1);

        return ValueTask.CompletedTask;
    }
})
```

En los logs estructurados de Serilog, estos mensajes tienen las propiedades `Attempt`, `MaxAttempts`, `Cause` y `DelayMs`. En Application Insights puedes buscar todos los reintentos del día y ver si hay un patrón.

## Configurar los parámetros de resiliencia desde appsettings

Los parámetros de Polly no deberían estar hardcoded — deberían ser configurables:

```json
// appsettings.json
{
  "Resilience": {
    "ProductsApi": {
      "RetryAttempts": 3,
      "RetryDelayMs": 500,
      "CircuitBreakerFailureRatio": 0.5,
      "CircuitBreakerBreakDurationSec": 30,
      "TimeoutSec": 5
    }
  }
}
```

```csharp
public class ProductsResilienceOptions
{
    public const string SectionName = "Resilience:ProductsApi";
    public int RetryAttempts { get; set; } = 3;
    public int RetryDelayMs { get; set; } = 500;
    public double CircuitBreakerFailureRatio { get; set; } = 0.5;
    public int CircuitBreakerBreakDurationSec { get; set; } = 30;
    public int TimeoutSec { get; set; } = 5;
}

// Vincular y validar en startup
builder.Services.AddOptions<ProductsResilienceOptions>()
    .BindConfiguration(ProductsResilienceOptions.SectionName)
    .ValidateDataAnnotations()
    .ValidateOnStart();
```

Con los parámetros en configuración, puedes ajustar el comportamiento sin redesplegar. En producción, Azure App Service permite cambiar configuración en caliente.

## Políticas diferentes por tipo de operación

No todas las llamadas a Products necesitan la misma resiliencia:

```csharp
// Verificar precio (bloqueante para crear pedidos) — más agresivo
builder.Services.AddHttpClient<ProductsClient>()
    .AddResilienceHandler("verify-price", pipeline =>
    {
```

## Degradación graciosa: el diseño correcto

Cuando Products no está disponible, Orders.API puede degradarse graciosamente en vez de fallar:

**Operación:** Crear pedido → necesita precio de Products

```
Products disponible: Obtener precio actual → Crear pedido con precio real
Products degradado: Usar precio en caché → Crear pedido con precio cacheado
Products caído: Error claro al usuario → "El servicio de precios no está disponible"
```

**Operación:** Listar pedidos → NO necesita Products

```
Products disponible: ✅ Funciona igual
Products degradado: ✅ Funciona igual — el listado no llama a Products
Products caído: ✅ Funciona igual
```

Esta separación es el corazón de la resiliencia compartimentada. Un fallo en Products no afecta a TODAS las operaciones de Orders — solo a las que realmente necesitan a Products.

## Caché como estrategia de fallback

El caché permite dar respuestas aunque Products no esté disponible:

```csharp
// Registrar IMemoryCache
builder.Services.AddMemoryCache();

// En ProductsClient
public async Task<ProductPriceDto?> GetProductAsync(Guid productId, CancellationToken ct)
{
    var cacheKey = $"product:{productId}";

    try
    {
        var product = await _httpClient.GetFromJsonAsync<ProductPriceDto>(
            $"/api/products/{productId}", ct);

        if (product is not null)
        {
            // Guardar en caché con TTL de 5 minutos
            _cache.Set(cacheKey, product, TimeSpan.FromMinutes(5));
        }
        return product;
    }
    catch (BrokenCircuitException)
    {
        // Circuit abierto — intentar con caché
        if (_cache.TryGetValue(cacheKey, out ProductPriceDto? cached))
        {
            _logger.LogWarning(
                "Circuit open — returning cached data for {ProductId}", productId);
            return cached;
        }

        _logger.LogError(
            "Circuit open and no cache — {ProductId} unavailable", productId);
        return null;
    }
}
```

## Las 8 falacias: cómo Polly las mitiga

En M1.1 vimos las 8 falacias de la computación distribuida. Polly las mitiga directamente:

**Falacia 1 (red no fiable):** Retry con backoff — si la red falla momentáneamente, reintenta.

**Falacia 2 (latencia no es cero):** Timeout — si Products tarda más de lo esperado, cancela.

**Falacia 4 (red no segura):** Timeout también protege de ataques slow-loris.

**Sin Polly:** cada fallo de red → excepción no capturada → 500 al cliente.

**Con Polly:** cada fallo de red → retry transparente → experiencia de usuario intacta (la mayoría de veces).

## Resiliencia vs disponibilidad: la distinción

Hay confusión frecuente entre estos dos conceptos:

**Disponibilidad:** qué porcentaje del tiempo un servicio responde correctamente.
- Products.API 99.9% disponible = 8.7 horas de caída al año

**Resiliencia:** capacidad de un servicio de seguir funcionando a pesar de fallos en sus dependencias.
- Orders.API resiliente = sigue procesando pedidos aunque Products tenga una caída de 1 hora

Polly mejora la **resiliencia** de Orders.API. No mejora la **disponibilidad** de Products.API.

El resultado neto: la experiencia del usuario en Orders mejora aunque Products tenga problemas ocasionales.

## Comportamiento por escenario: la tabla de decisiones

Cuando Orders llama a Products, ¿qué pasa en cada escenario?

| Escenario | Comportamiento de Polly | Respuesta al cliente |
|---|---|---|
| Products OK | Llamada directa, sin overhead | Precio actual, 201 Created |
| Products lento (2s) | Timeout individual → Retry → OK | Ligero retraso, 201 Created |
| Products caído temporalmente | 3 Retries con backoff → OK | Retraso ~3s, 201 Created |
| Products caído largo (CB abierto) | Circuit abierto → Fallback → caché | Datos cacheados, 201 Created |
| Products caído + sin caché | Circuit abierto → Fallback → null | 422 "Product not available" |
| Products 400 Bad Request | Sin retry (error permanente) | 422 "Invalid request" |
| Products 404 Not Found | Sin retry → null | 422 "Product not found" |

## Resiliencia en el contexto de la Saga

Cuando en M4.3 implementemos la Saga de pedidos, la resiliencia cobra otra dimensión:

**Sin resiliencia:**
```
Saga: Orders crea pedido → Products reserva stock → Payments cobra
Si Payments falla temporalmente → Saga falla → Compensación necesaria
```

**Con resiliencia en cada paso:**
```
Saga: Orders → Products (con retry) → Payments (con retry)
Si Payments falla temporalmente → Polly reintenta → 3er intento OK → Saga completa
Sin compensación necesaria porque el error fue temporal
```

La resiliencia reduce el número de compensaciones necesarias en la Saga. Menos compensaciones = menos complejidad = menos bugs. Polly y la Saga son complementarios, no alternativos.

## Timeouts: el debate corto vs largo

Una discusión frecuente: ¿cuánto timeout ponerle?

**Timeout demasiado corto (ej: 500ms):**
- Products responde normalmente en 200ms → OK
- Products bajo carga responde en 600ms → timeout → fallo innecesario
- Demasiados fallos transitorios → más retries → más carga en Products

**Timeout demasiado largo (ej: 30s):**
- Si Products no responde, el thread queda bloqueado 30s
- Con 100 peticiones concurrentes → 3.000 segundos de threads bloqueados
- El thread pool se agota → Orders cae también

**La regla práctica:**
1. Mide el P95 y P99 de respuesta de Products en producción
2. Timeout = P99 × 2-3
3. Si P99 = 300ms → timeout = 600-900ms

Para el curso, usamos 5 segundos — conservador pero suficiente para LocalDB.

## Comparativa: antes y después de Polly

El mismo escenario de fallo con y sin resiliencia:

**Sin Polly — Products tiene una caída de 2 minutos:**
```
t=0: Products.API se cae
t=0 a t=2m: Todos los intentos de crear pedido devuelven 500
t=2m: Products vuelve → se recupera
Pedidos perdidos: ~40 (con 20 pedidos/min)
```

**Con Polly — Products tiene una caída de 2 minutos:**

```
t=1.5s: Circuit Breaker abre
t=1.5s a t=2m: Fallback con caché → algunos pedidos se crean con precio cacheado
t=2m: CB prueba → Products OK → CB cierra
t=2m+: Todos los pedidos con precio real
Pedidos perdidos: ~0 (excepto los sin caché en los primeros 1.5s)
```

La diferencia: Polly convirtió 40 pedidos perdidos en ~0 pedidos perdidos con un precio ligeramente desactualizado.
