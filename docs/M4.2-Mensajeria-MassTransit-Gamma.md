## Dónde estamos

En M4.1 conectamos Orders.API y Products.API de forma síncrona: Orders pregunta a Products el precio antes de crear un pedido. La respuesta llega en la misma petición.

Ahora añadimos la dimensión asíncrona. Cuando se crea un pedido, Orders publica un evento `OrderCreated`. Notifications.API lo recibe y envía un email de confirmación. Orders no sabe cuándo llega el email — ni le importa.

Al terminar M4.2:
- Orders publica eventos reales a RabbitMQ
- Notifications.API existe y consume esos eventos
- El dashboard de Aspire muestra los mensajes fluyendo entre servicios
- La solución usa RabbitMQ instalado nativamente (sin Docker)

---

## ¿Por qué mensajería asíncrona?

La comunicación síncrona tiene un límite: el servicio que llama queda bloqueado esperando respuesta. Si el destino tarda o falla, el origen también sufre.

La mensajería asíncrona rompe ese acoplamiento:

```
SIN MENSAJERÍA (síncrona encadenada):
Orders → Products → Payments → Notifications
Si Notifications tarda 2s → todo tarda 2s más
Si Notifications cae → toda la cadena falla

CON MENSAJERÍA (asíncrona):
Orders → [RabbitMQ] ← Notifications lee cuando puede
Orders responde al cliente en cuanto guarda el pedido
Notifications puede tardar 2s o 20s — Orders no espera
Notifications puede caerse y recuperarse — los mensajes esperan
```

La mensajería es el mecanismo que hace que los microservicios sean realmente independientes.

---

## El patrón Publisher-Subscriber en OrderFlow

La mensajería implementa el patrón Pub/Sub (Publisher-Subscriber):

```
PUBLISHER (Orders.API):
- No sabe quién consume el evento
- No sabe cuántos consumers hay
- No sabe cuándo se procesa
- Solo publica y sigue su camino

SUBSCRIBER (Notifications.API):
- Se suscribe a OrderCreated
- Procesa cuando puede
- Si falla, el mensaje queda en la queue
- Puede escalar independientemente

BROKER (RabbitMQ):
- Garantiza la entrega
- Almacena si el consumer está caído
- Puede enrutar a múltiples subscribers
```

Cuando en el futuro añadamos un servicio de Analytics, solo necesita suscribirse a `OrderCreated` — Orders.API no cambia nada.

## Conceptos clave de RabbitMQ

RabbitMQ tiene tres conceptos fundamentales:

**Exchange:** El punto de entrada. Los producers publican mensajes al exchange. El exchange decide a qué queues enviarlos según las reglas de routing.

**Queue:** El almacén. Los consumers leen mensajes de la queue. Si el consumer está caído, los mensajes esperan en la queue.

**Binding:** La conexión entre exchange y queue. Define qué mensajes de qué exchange van a qué queue.

```
Producer → Exchange → (binding) → Queue → Consumer
                   ↘ (binding) → Queue → Consumer 2
```

En MassTransit, toda esta configuración se genera automáticamente. No necesitas configurar exchanges ni queues manualmente.

---

## Tipos de Exchange en RabbitMQ

RabbitMQ tiene cuatro tipos de exchange:

**Direct:** Enruta por routing key exacta. Un mensaje con key `order.created` va a la queue que tiene ese binding.

**Topic:** Enruta por patrón. `order.*` captura `order.created`, `order.confirmed`, `order.cancelled`.

**Fanout:** Envía a TODAS las queues vinculadas. Broadcast total.

**Headers:** Enruta por atributos del header del mensaje.

En MassTransit, el tipo por defecto es **Fanout** para eventos y **Direct** para comandos. No necesitas configurarlo — MassTransit lo gestiona.

## Domain Events vs Integration Events: la diferencia clave

Confusión frecuente que aclararemos de una vez:

**Domain Event (dentro del aggregate):**
```csharp
// Vive en el dominio de Orders.API
// Solo tiene sentido dentro del bounded context
public sealed record OrderCreatedEvent(
    Guid OrderId, Guid CustomerId, Money Total) : IDomainEvent;

// Se usa para reacciones DENTRO de Orders.API
// Por ejemplo: disparar el Outbox (M6.2)
```

**Integration Event (entre servicios):**
```csharp
// Vive en OrderFlow.Contracts — compartido entre servicios
// Formato diseñado para cruzar la red
public record OrderCreated(
    Guid OrderId,
    Guid CustomerId,
    decimal Total,
    string Currency,
    DateTime CreatedAt,
    IList<OrderCreatedItem> Items);
```

El Domain Event es interno. El Integration Event es el contrato público entre servicios. Son dos cosas diferentes aunque compartan nombres similares.

## MassTransit: el bus de mensajes para .NET

MassTransit es la librería de mensajería más usada en .NET. Abstrae el broker subyacente — el mismo código funciona con RabbitMQ (local) y Azure Service Bus (producción).

**Por qué MassTransit y no el cliente nativo de RabbitMQ:**

```csharp
// Con cliente nativo de RabbitMQ — verbose y acoplado al broker
var factory = new ConnectionFactory { HostName = "localhost" };
var connection = factory.CreateConnection();
var channel = connection.CreateModel();
channel.ExchangeDeclare("order-events", "fanout");
var body = JsonSerializer.SerializeToUtf8Bytes(orderCreated);
var props = channel.CreateBasicProperties();
channel.BasicPublish("order-events", "", props, body);

// Con MassTransit — 1 línea, funciona con cualquier broker
await _publishEndpoint.Publish(orderCreated, ct);
```

MassTransit gestiona la serialización, la configuración del broker, los reintentos, las dead-letter queues y el tracking de mensajes.

## Al-least-once delivery: el principio fundamental

RabbitMQ y Azure Service Bus garantizan **at-least-once delivery**: el mensaje se entrega al menos una vez. En condiciones de red inestable o fallo del consumer durante el procesamiento, puede entregarse más de una vez.

**El problema:**

```
1. Consumer recibe OrderCreated
2. Llama a SendOrderConfirmationAsync → EMAIL ENVIADO
3. Consumer falla antes de hacer ACK
4. RabbitMQ vuelve a entregar el mensaje
5. Consumer procesa de nuevo → SEGUNDO EMAIL enviado ← problema
```

**La solución — idempotencia:**

```csharp
public async Task Consume(ConsumeContext<OrderCreated> context)
{
    // Verificar si ya procesamos este mensaje
    var processed = await _notificationRepo
        .ExistsAsync(context.Message.OrderId, ct);

    if (processed)
    {
        _logger.LogWarning(
            "Duplicate message for order {OrderId} — skipping",
            context.Message.OrderId);
        return; // ACK sin procesar de nuevo
    }

    // Procesar y marcar como procesado en la misma transacción
    await _emailService.SendOrderConfirmationAsync(...);
    await _notificationRepo.MarkAsProcessedAsync(context.Message.OrderId);
}
```

El Inbox Pattern (M6.2) resuelve esto de forma robusta para toda la aplicación.

## IPublishEndpoint vs ISendEndpoint

MassTransit tiene dos formas de enviar mensajes:

**`IPublishEndpoint.Publish()`** — para eventos (Pub/Sub):
```csharp
// Publica a todos los subscribers del evento
await _publishEndpoint.Publish(new OrderCreated { ... });
// Orders no sabe quién lo recibe
// Puede ser 0, 1 o 10 consumers
```

**`ISendEndpoint.Send()`** — para comandos (Point-to-Point):
```csharp
// Envía a una queue específica
var endpoint = await _bus.GetSendEndpoint(
    new Uri("queue:process-payment"));
await endpoint.Send(new ProcessPayment { ... });
// Solo un consumer recibe el comando
// Se usa en la Saga para coordinar pasos (M4.3)
```

En M4.2 usamos `Publish` para eventos. En M4.3 veremos `Send` para los comandos de la Saga.

## Consumer actualizado con idempotencia básica

```csharp
public class OrderCreatedConsumer : IConsumer<OrderCreated>
{
    private readonly IEmailService _emailService;
    private readonly NotificationDbContext _dbContext;
    private readonly ILogger<OrderCreatedConsumer> _logger;

    public async Task Consume(ConsumeContext<OrderCreated> context)
    {
        var message = context.Message;

        // Verificar idempotencia: ¿ya procesamos este pedido?
        var alreadyProcessed = await _dbContext.Notifications
            .AnyAsync(n => n.OrderId == message.OrderId
                        && n.Type == "OrderConfirmation",
                      context.CancellationToken);

        if (alreadyProcessed)
        {
            _logger.LogWarning(
                "Duplicate OrderCreated for {OrderId} — skipping",
                message.OrderId);
            return;
        }

        // Procesar
        await _emailService.SendOrderConfirmationAsync(
            message.CustomerEmail, message.OrderId,
            message.Total, message.Currency, message.Items,
            context.CancellationToken);

        // Registrar como procesado
        _dbContext.Notifications.Add(
            Notification.Create(message.OrderId,
                "OrderConfirmation", message.CustomerEmail, true));
        await _dbContext.SaveChangesAsync(context.CancellationToken);

        _logger.LogInformation(
            "Notification sent and recorded for order {OrderId}",
            message.OrderId);
    }
}
```

## Dead Letter Queue (DLQ): cuando el consumer falla

¿Qué pasa si `OrderCreatedConsumer` lanza una excepción?

```csharp
public async Task Consume(ConsumeContext<OrderCreated> context)
{
    // Si esto lanza excepción:
    await _emailService.SendOrderConfirmationAsync(...);
}
```

MassTransit tiene un mecanismo de reintento automático:

```
1. Consumer falla → MassTransit reintenta (configurable: 3-5 veces)
2. Todos los reintentos fallan → mensaje va a la Dead Letter Queue
3. DLQ: notifications-api-order-created_error
4. El mensaje no se pierde — queda en la DLQ hasta que alguien lo procese
```

Configurar los reintentos en MassTransit:

```csharp
x.UsingRabbitMq((context, cfg) =>
{
    cfg.Host(...);

    // Política de reintentos para todos los consumers
    cfg.UseMessageRetry(r =>
    {
        r.Exponential(
            retryLimit: 5,
            minInterval: TimeSpan.FromSeconds(1),
            maxInterval: TimeSpan.FromSeconds(30),
            intervalDelta: TimeSpan.FromSeconds(2));

        // No reintentar en errores de validación (son permanentes)
        r.Ignore<ValidationException>();
        r.Ignore<DomainException>();
    });

    cfg.ConfigureEndpoints(context);
});
```

## Configurar reintentos por consumer

Los reintentos se pueden configurar globalmente o por consumer específico:

```csharp
cfg.ReceiveEndpoint("notifications-api-order-created", endpoint =>
{
    endpoint.UseMessageRetry(r =>
        r.Exponential(5,
            TimeSpan.FromSeconds(1),
            TimeSpan.FromSeconds(30),
            TimeSpan.FromSeconds(2)));

    endpoint.ConfigureConsumer<OrderCreatedConsumer>(context);
});
```

MassTransit también tiene `UseDelayedRedelivery` para reintentos a largo plazo:

```csharp
endpoint.UseDelayedRedelivery(r =>
    r.Intervals(
        TimeSpan.FromMinutes(5),
        TimeSpan.FromMinutes(30),
        TimeSpan.FromHours(2)));
// Si falla inmediatamente: reintenta en 5min, 30min, 2h
// Útil para fallos que necesitan tiempo para resolverse (SMTP caído)
```

## Headers de MassTransit: trazabilidad automática

MassTransit propaga automáticamente:
- `correlationId` — ID de correlación del mensaje
- `conversationId` — ID de la conversación (puede cruzar varios mensajes)
- `initiatorId` — quién inició la cadena

Estos headers son diferentes del `X-Correlation-Id` HTTP que implementamos en el middleware. Puedes mapearlos:

```csharp
// En el Consumer: leer el CorrelationId del mensaje
public async Task Consume(ConsumeContext<OrderCreated> context)
{
    var correlationId = context.CorrelationId?.ToString()
        ?? context.Message.OrderId.ToString();

    using var scope = _logger.BeginScope(
        new Dictionary<string, object> { ["CorrelationId"] = correlationId });

    // Todos los logs dentro de este scope tienen el CorrelationId
    _logger.LogInformation("Processing order {OrderId}", context.Message.OrderId);
}
```

## Publicar múltiples Integration Events

Orders puede necesitar publicar varios eventos en la misma operación:

```csharp
// En el handler — publicar múltiples eventos
public async Task<OrderDto> Handle(CreateOrderCommand command, CancellationToken ct)
{
    var order = Order.Create(...);
    await _repository.SaveAsync(order, ct);

    // Publicar el evento de creación
    await _publishEndpoint.Publish(new OrderCreated { ... }, ct);

    // Si el pedido es de alto valor, publicar también un evento de alerta
    if (order.Total.Amount > 5000)
    {
        await _publishEndpoint.Publish(new HighValueOrderCreated
        {
            OrderId = order.Id,
            Total = order.Total.Amount
        }, ct);
    }

    return order.ToDto();
}
```

Cada evento puede tener sus propios subscribers. `HighValueOrderCreated` podría ir a un servicio de aprobación manual — sin tocar Orders.API ni Notifications.API.

## Consumer Groups: múltiples instancias del mismo consumer

¿Qué pasa cuando escalamos Notifications.API a 3 instancias?

```
RabbitMQ Queue: notifications-api-order-created
    │
    ├── Instancia 1 de Notifications.API  ← procesa mensaje A
    ├── Instancia 2 de Notifications.API  ← procesa mensaje B
    └── Instancia 3 de Notifications.API  ← procesa mensaje C
```

RabbitMQ distribuye los mensajes entre las instancias automáticamente. Cada mensaje solo lo procesa UNA instancia. Esto es el **competing consumers pattern** — la forma natural de escalar el procesamiento de mensajes.

Si una instancia cae mientras procesa un mensaje: RabbitMQ entrega el mensaje a otra instancia activa. El mensaje no se pierde.

**En Azure App Services:** cuando escalas de 1 a 3 instancias, MassTransit en todas ellas compite por los mensajes de la misma queue. Sin configuración extra.

## Ordenación de mensajes: ¿importa el orden?

Una pregunta frecuente: ¿los mensajes se procesan en el orden en que se publicaron?

**RabbitMQ:** por defecto, los mensajes dentro de una queue se entregan en orden FIFO (First In, First Out) cuando hay UN solo consumer. Con múltiples consumers (escalado), el orden no está garantizado.

**Para OrderFlow:** el orden dentro de un pedido importa (StockReserved antes de PaymentProcessed). La Saga (M4.3) gestiona esa orquestación — no depende del orden de los mensajes en la queue.

Para Notifications, el orden no importa: "email de confirmación antes que email de envío" es un requisito de negocio que Notifications gestiona con state (si ya mandó la confirmación, no manda de nuevo).

## Mensajes grandes: el claim-check pattern

Los mensajes de RabbitMQ tienen un límite de tamaño (por defecto 128MB, configurable). Para mensajes muy grandes:

```csharp
// ❌ Problema: evento con mucho payload
public record OrderCreated
{
    // Si el pedido tiene 50 ítems con imágenes → mensaje muy grande
    public IList<OrderCreatedItem> Items { get; init; } = [];
    // byte[] InvoicePdf?? → NUNCA en mensajes
}

// ✅ Claim-Check Pattern: el mensaje lleva una referencia, no los datos
public record OrderCreated
{
    public Guid OrderId { get; init; }
    // Solo el ID — quien necesita los detalles llama a la API
    // o los lee de su propia BD
}
```

Para OrderFlow, los mensajes son pequeños (IDs, importes, textos cortos). El claim-check es un patrón avanzado para casos con payloads muy grandes.

## Azure Service Bus: la alternativa de producción

En producción, RabbitMQ se reemplaza por **Azure Service Bus**. MassTransit hace el cambio con una sola línea:

```csharp
// DESARROLLO: RabbitMQ local
x.UsingRabbitMq((context, cfg) =>
{
    cfg.Host("amqp://guest:guest@localhost:5672");
    cfg.ConfigureEndpoints(context);
});

// PRODUCCIÓN: Azure Service Bus
x.UsingAzureServiceBus((context, cfg) =>
{
    cfg.Host("Endpoint=sb://orderflow.servicebus.windows.net/;...");
    cfg.ConfigureEndpoints(context);
});
```

**Los consumers, producers y contratos no cambian.** Solo cambia la configuración del transport.

En la práctica, se usa una variable de entorno o configuración por entorno:

```csharp
if (builder.Environment.IsProduction())
    x.UsingAzureServiceBus((ctx, cfg) => { cfg.Host(...); cfg.ConfigureEndpoints(ctx); });
else
    x.UsingRabbitMq((ctx, cfg) => { cfg.Host(...); cfg.ConfigureEndpoints(ctx); });
```

## Diferencias RabbitMQ vs Azure Service Bus

Para que el equipo de TechShop entienda la decisión:

| Aspecto | RabbitMQ (local) | Azure Service Bus (producción) |
|---|---|---|
| Coste | Gratuito | ~0.05€/mes (Basic tier) |
| Instalación | Local, nativa | Servicio Azure gestionado |
| Operación | Autogestionado | Microsoft lo gestiona |
| Durabilidad | Configurable | 99.9% SLA garantizado |
| Dead Letter | Sí (nativo) | Sí (nativo) |
| Particionamiento | Plugins | Nativo en Premium |
| Autenticación | user/password | Managed Identity |

Para el desarrollo local: RabbitMQ es perfecto. Para producción: Azure Service Bus elimina la carga operativa de gestionar el broker.

## MassTransit Saga: preview del M4.3

Antes de terminar M4.2, un preview del concepto central de M4.3: la Saga.

Una Saga es una secuencia de pasos locales con compensaciones. En MassTransit se implementa como una state machine:

```
OrderSaga
Estado: Pending
  ├── Recibe StockReserved → Estado: StockReserved
  ├── Recibe StockInsufficient → Compensar → Estado: Cancelled
  └── Recibe PaymentProcessed → Estado: Confirmed
```

```csharp
// Preview de la Saga (completo en M4.3)
public class OrderSaga : MassTransitStateMachine<OrderSagaState>
{
    public State Pending { get; private set; } = null!;
    public State StockReserved { get; private set; } = null!;
    public State Confirmed { get; private set; } = null!;

    public Event<OrderCreated> OrderCreatedEvent { get; private set; } = null!;
    public Event<StockReserved> StockReservedEvent { get; private set; } = null!;

    public OrderSaga()
    {
        Initially(
            When(OrderCreatedEvent)
                .TransitionTo(Pending)
                .Publish(context => new ReserveStock { ... }));

        During(Pending,
            When(StockReservedEvent)
                .TransitionTo(StockReserved)
                .Publish(context => new ProcessPayment { ... }));
    }
}
```

Lo implementamos completo en M4.3.

## Resumen de M4.2

**Lo que hemos construido:**

**OrderFlow.Contracts:**
El paquete compartido con los Integration Events. Contratos inmutables que definen el lenguaje común entre servicios.

**MassTransit configurado en Orders.API:**
`CreateOrderHandler` publica `OrderCreated` tras guardar el pedido. Orders no espera confirmación — publica y sigue.

**Notifications.API:**
Tercer microservicio de OrderFlow. Consume `OrderCreated` y simula el envío del email de confirmación.

**RabbitMQ local:**
El broker de mensajería instalado nativamente (sin Docker). MassTransit configura automáticamente exchanges, queues y bindings.

**Dead Letter Queue:**
Mensajes que fallan van a una DLQ. No se pierden. Los reintentos son exponenciales y configurables.

**At-least-once delivery:**
Los consumers deben ser idempotentes. El Inbox Pattern (M6.2) lo resuelve de forma robusta.

El sistema ya es verdaderamente asíncrono. Orders crea pedidos sin esperar a que Notifications procese. En M4.3 añadimos la Saga completa con Payments.

## Lo que viene en M4.3

En M4.3 añadimos **Payments.API** y la **Saga de pedidos**.

La Saga coordina el flujo completo:
1. Orders crea el pedido → publica `OrderCreated`
2. Products reserva stock → publica `StockReserved`
3. Payments cobra → publica `PaymentProcessed`
4. Orders confirma → publica `OrderConfirmed`
5. Si Payments falla: compensación en cadena inversa

Implementaremos la Saga con MassTransit Automatonymous (state machine). Al terminar M4.3, el flujo de pedido de TechShop estará completo de extremo a extremo.
