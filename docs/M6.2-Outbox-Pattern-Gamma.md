## El problema que resolvemos hoy

En M4.2, el `CreateOrderHandler` hace dos operaciones:

```csharp
// Operación 1: Guardar en la BD
await _repository.SaveAsync(order, ct);

// Operación 2: Publicar evento a RabbitMQ
await _publishEndpoint.Publish(new OrderCreated { ... }, ct);
```

Hay un intervalo de tiempo entre las dos. Si el proceso cae entre la línea 1 y la línea 2:

```
✅ El pedido existe en OrdersDb
❌ El evento OrderCreated NUNCA se publicó
❌ La Saga nunca arrancó
❌ El stock nunca se reservó
❌ El pago nunca se cobró
❌ El cliente tiene un pedido en estado "Pending" eternamente
```

Esto es el **problema del doble escritura** (dual-write problem). En M6.2 lo resolvemos definitivamente con el **Outbox Pattern**.

---

## Por qué no se puede usar una transacción distribuida

La solución obvia sería envolver ambas operaciones en una transacción:

```csharp
// ❌ Esto NO funciona — SQL y RabbitMQ no comparten transacciones
using var sqlTransaction = await _dbContext.Database.BeginTransactionAsync();
using var rabbitTransaction = _busControl.BeginPublishTransaction();

await _repository.SaveAsync(order, ct);
await _publishEndpoint.Publish(new OrderCreated { ... }, ct);

await sqlTransaction.CommitAsync();
await rabbitTransaction.CommitAsync();
// Si falla el commit de RabbitMQ después del de SQL → inconsistencia
```

SQL Server y RabbitMQ no participan en el mismo protocolo de transacciones. No hay manera de hacer commit o rollback de ambos atomicamente.

**La solución no es una transacción distribuida.** Es el Outbox Pattern.

---

## El Outbox Pattern: la idea central

La idea es elegante: en vez de publicar el evento directamente a RabbitMQ, lo guardamos primero en la misma base de datos, en la misma transacción que el pedido:

```
ANTES (dual-write):
  Transacción SQL: INSERT Order
  Publish → RabbitMQ (fuera de la transacción)

DESPUÉS (Outbox Pattern):
  Transacción SQL: INSERT Order + INSERT OutboxMessage
  ← La transacción garantiza que ambas ocurren o ninguna

  [Background Service]
  Lee OutboxMessages no publicados
  Publica cada mensaje a RabbitMQ
  Marca el mensaje como publicado
```

El evento no se pierde nunca:
- Si la app cae después del INSERT: el Background Service lo publica al reiniciar
- Si RabbitMQ no está disponible: el mensaje espera en la tabla hasta que RabbitMQ vuelva

---

## Las tres garantías del Outbox Pattern

Al terminar M6.2, OrderFlow tiene tres garantías fundamentales:

**Garantía 1 — Atomicidad:**
Un pedido y su evento siempre se crean juntos o no se crea ninguno. No hay pedidos sin evento asociado.

**Garantía 2 — At-Least-Once Delivery garantizada:**
El evento llega al consumer al menos una vez. Si hay fallos de red, RabbitMQ caído, o crashes de la app, el BackgroundService reintenta hasta que funciona.

**Garantía 3 — Recuperabilidad automática:**
Si el sistema cae en cualquier punto del flujo, al reiniciar continúa exactamente desde donde estaba. Sin intervención manual, sin scripts de soporte.

Estas tres garantías convierten OrderFlow en un sistema que puede fallar de forma segura — lo que los ingenieros de fiabilidad llaman **"graceful failure"**.

---

## Outbox Pattern: el patrón más importante del módulo 6

Si solo puedes quedarte con una cosa de M6.2, es esta:

```
EL PROBLEMA:
  Guardar en la BD Y publicar en el broker son dos operaciones separadas.
  Pueden fallar independientemente.
  Resultado: estado inconsistente.

EL OUTBOX:
  Guardar en la BD + guardar el evento en la misma BD
  → Una sola transacción: ambos o ninguno
  → Un BackgroundService publica el evento al broker
  → Si falla: reintenta automáticamente
  → Resultado: consistencia garantizada

EL CÓDIGO:
  x.AddEntityFrameworkOutbox<OrderDbContext>(o =>
  {
      o.UseSqlServer();
      o.UseBusOutbox();
  });
  // Una sola llamada. El resto funciona automáticamente.
```

Es el patrón que más impacto tiene en la fiabilidad del sistema con menos código añadido. Cada microservicio que publique eventos debería tenerlo.

## Guardar el evento en el Outbox

El handler ya no llama directamente a `_publishEndpoint.Publish`. En su lugar, guarda el evento en la tabla Outbox dentro de la misma transacción:

```csharp
// CreateOrderHandler actualizado con Outbox

public async Task<OrderDto> Handle(
    CreateOrderCommand command, CancellationToken ct)
{
    // ... validar precios en Products.API ...

    var order = Order.Create(command.CustomerId, validatedItems, address);

    // Crear el Integration Event
    var orderCreatedEvent = new OrderCreated
    {
        OrderId = order.Id,
        CustomerId = order.CustomerId,
        CustomerEmail = command.CustomerEmail,
        Total = order.Total.Amount,
        Currency = order.Total.Currency,
        Items = order.Lines.Select(l =>
            new OrderCreatedItem(l.ProductId, l.ProductName,
                l.Quantity, l.UnitPrice.Amount)).ToList()
    };

    // Guardar orden + outbox message en UNA sola transacción
    await _repository.SaveAsync(order, ct);
    await _outboxRepository.AddAsync(new OutboxMessage
    {
        Type = nameof(OrderCreated),
        Content = JsonSerializer.Serialize(orderCreatedEvent)
    }, ct);

    // TransactionBehavior hace el COMMIT de los dos juntos
    return order.ToDto();
}
```

## El TransactionBehavior garantiza la atomicidad

El `TransactionBehavior` que ya teníamos de M3.4 garantiza que el `SaveAsync` y el `AddAsync` van en la misma transacción:

```csharp
// TransactionBehavior — sin cambios necesarios
public async Task<TResponse> Handle(
    TRequest request,
    RequestHandlerDelegate<TResponse> next,
    CancellationToken ct)
{
    await using var transaction = await _dbContext.Database
        .BeginTransactionAsync(ct);
    try
    {
        var response = await next();
        await _dbContext.SaveChangesAsync(ct);
        await transaction.CommitAsync(ct);  // ← INSERT Order + INSERT OutboxMessage
        return response;
    }
    catch
    {
        await transaction.RollbackAsync(ct);
        throw;
    }
}
```

Ambas filas van en el mismo `SaveChangesAsync`. Si falla, el ROLLBACK elimina ambas. Atomicidad garantizada.

## El OutboxProcessor: el Background Service

Un `BackgroundService` que lee la tabla Outbox y publica los mensajes pendientes:

```csharp
// Orders.API/Infrastructure/Outbox/OutboxProcessor.cs

public class OutboxProcessor : BackgroundService
{
    private readonly IServiceScopeFactory _scopeFactory;
    private readonly ILogger<OutboxProcessor> _logger;

    protected override async Task ExecuteAsync(CancellationToken stoppingToken)
    {
        while (!stoppingToken.IsCancellationRequested)
        {
            await ProcessPendingMessagesAsync(stoppingToken);
            await Task.Delay(TimeSpan.FromSeconds(5), stoppingToken);
        }
    }

    private async Task ProcessPendingMessagesAsync(CancellationToken ct)
    {
        using var scope = _scopeFactory.CreateScope();
        var dbContext = scope.ServiceProvider
            .GetRequiredService<OrderDbContext>();
        var publishEndpoint = scope.ServiceProvider
            .GetRequiredService<IPublishEndpoint>();

        // Leer los mensajes no procesados (máximo 20 por ciclo)
        var messages = await dbContext.OutboxMessages
            .Where(m => m.ProcessedAt == null && m.RetryCount < 5)
            .OrderBy(m => m.CreatedAt)
            .Take(20)
            .ToListAsync(ct);

        foreach (var message in messages)
        {
            await PublishMessageAsync(message, publishEndpoint, dbContext, ct);
        }
    }
```

## El OutboxProcessor: publicar cada mensaje

```csharp
    // Continuación de OutboxProcessor
    private async Task PublishMessageAsync(
        OutboxMessage message,
        IPublishEndpoint publishEndpoint,
        OrderDbContext dbContext,
        CancellationToken ct)
    {
        try
        {
            // Deserializar el evento según su tipo
            var eventObject = message.Type switch
            {
                nameof(OrderCreated) =>
                    (object?)JsonSerializer.Deserialize<OrderCreated>(message.Content),
                nameof(OrderConfirmed) =>
                    JsonSerializer.Deserialize<OrderConfirmed>(message.Content),
                nameof(OrderCancelled) =>
                    JsonSerializer.Deserialize<OrderCancelled>(message.Content),
                _ => throw new InvalidOperationException(
                    $"Unknown message type: {message.Type}")
            };

            if (eventObject is not null)
                await publishEndpoint.Publish(eventObject, ct);

            // Marcar como procesado
            message.ProcessedAt = DateTime.UtcNow;
            message.Error = null;
        }
        catch (Exception ex)
        {
            _logger.LogError(ex,
                "Failed to publish outbox message {MessageId} of type {Type}",
                message.Id, message.Type);

            message.RetryCount++;
            message.Error = ex.Message;
        }

        await dbContext.SaveChangesAsync(ct);
    }
}
```

## El intervalo de polling: equilibrio entre latencia y carga

El OutboxProcessor corre cada 5 segundos por defecto. ¿Es suficiente?

```
Con polling cada 5 segundos:
  Latencia máxima del evento: ~5 segundos
  Carga en la BD: 12 queries por minuto (muy bajo)
  Para TechShop (50K pedidos/mes = ~1 pedido/minuto): más que suficiente

Con polling cada 1 segundo:
  Latencia máxima del evento: ~1 segundo
  Carga en la BD: 60 queries por minuto (aún bajo)
  Para sistemas más reactivos: razonable

Con polling cada 100ms:
  Latencia máxima del evento: ~100ms (casi inmediato)
  Carga en la BD: 600 queries por minuto (empieza a ser significativo)
  Para sistemas de tiempo real: considera alternativas
```

Para la mayoría de sistemas, **5 segundos es el punto óptimo**. El evento llega en segundos al consumer, no en minutos.

## MassTransit Outbox: el Outbox integrado

La buena noticia: MassTransit 8.x tiene un Outbox integrado que gestiona todo esto automáticamente. No necesitas implementar el `OutboxProcessor` manual.

```csharp
// Orders.API/Program.cs — habilitar el Outbox de MassTransit

builder.Services.AddMassTransit(x =>
{
    x.AddSagaStateMachine<OrderSaga, OrderSagaState>()
        .EntityFrameworkRepository(r =>
        {
            r.ConcurrencyMode = ConcurrencyMode.Optimistic;
            r.AddDbContext<DbContext, OrderDbContext>(...);
        });

    x.AddConsumers(typeof(Program).Assembly);

    // Habilitar el Outbox de MassTransit con EF Core
    x.AddEntityFrameworkOutbox<OrderDbContext>(o =>
    {
        o.UseSqlServer();           // Dialect de SQL Server
        o.UseBusOutbox();           // Integra el outbox con el bus
        o.QueryDelay = TimeSpan.FromSeconds(5); // Polling cada 5s
    });

    x.UsingRabbitMq((context, cfg) =>
    {
        cfg.Host(builder.Configuration.GetConnectionString("messaging")!);
        cfg.UseMessageRetry(r => r.Exponential(5,
            TimeSpan.FromSeconds(1), TimeSpan.FromSeconds(30),
            TimeSpan.FromSeconds(2)));
        cfg.ConfigureEndpoints(context);
    });
});
```

## At-Least-Once con Outbox: idempotencia en los consumers

El Outbox garantiza at-least-once delivery. Si el OutboxProcessor falla después de publicar pero antes de marcar como procesado, el mensaje se publica dos veces.

Esto ya lo sabíamos de M4.2 — los consumers deben ser idempotentes:

```csharp
// En OrderCreatedConsumer de Notifications.API
public async Task Consume(ConsumeContext<OrderCreated> context)
{
    // Verificar si ya procesamos este pedido
    var alreadyProcessed = await _dbContext.Notifications
        .AnyAsync(n => n.OrderId == context.Message.OrderId
                    && n.Type == "OrderConfirmation",
                  context.CancellationToken);

    if (alreadyProcessed)
    {
        _logger.LogWarning(
            "Duplicate message for order {OrderId}, skipping",
            context.Message.OrderId);
        return; // ACK — no procesar de nuevo, pero confirmar recepción
    }

    await _emailService.SendOrderConfirmationAsync(...);

    _dbContext.Notifications.Add(
        Notification.Create(context.Message.OrderId, "OrderConfirmation", ...));
    await _dbContext.SaveChangesAsync();
}
```

## Inbox Pattern: el complemento del Outbox

El Outbox garantiza que los mensajes se publican exactamente una vez desde el productor. El **Inbox Pattern** garantiza que los consumers los procesan exactamente una vez:

```csharp
// En el consumer — guardar el InboxMessage antes de procesar
public async Task Consume(ConsumeContext<OrderCreated> context)
{
    var messageId = context.MessageId ?? Guid.NewGuid();

    // Registrar el mensaje como recibido (idempotente)
    var alreadyInInbox = await _dbContext.InboxMessages
        .AnyAsync(m => m.MessageId == messageId);

    if (alreadyInInbox)
    {
        _logger.LogWarning(
            "Message {MessageId} already in inbox, skipping", messageId);
        return;
    }

    // Procesar + registrar en inbox en la misma transacción
    await _emailService.SendOrderConfirmationAsync(...);

    _dbContext.InboxMessages.Add(new InboxMessage
    {
        MessageId = messageId,
        Type = nameof(OrderCreated),
        ProcessedAt = DateTime.UtcNow
    });

    await _dbContext.SaveChangesAsync();
}
```

## Consistencia eventual: entender el compromiso

Con el Outbox Pattern, la consistencia es **eventual pero garantizada**:

```
t=0:    Cliente envía POST /api/orders
t=0ms:  El pedido se guarda en OrdersDb ← VISIBLE PARA EL CLIENTE
        El OutboxMessage se guarda en la misma transacción
        Orders responde 201 al cliente

t=0-5s: El BackgroundService procesa el OutboxMessage
        Publica OrderCreated a RabbitMQ

t=5-10s: Saga arranca, reserva stock en Products

t=10-15s: Saga procesa el pago en Payments

t=15-20s: Notifications envía el email de confirmación
          El pedido pasa a estado "Confirmed" en OrdersDb
```

El cliente ve el pedido creado inmediatamente. La confirmación del pago llega en ~15-20 segundos. Esto es consistencia eventual — no inmediata, pero garantizada.

---

## Eventual Consistency: cuándo es un problema

La consistencia eventual es apropiada para la mayoría de sistemas de e-commerce. Pero hay casos donde es problemática:

**Casos donde la consistencia eventual puede ser un problema:**

```
Caso 1: El usuario ve "pedido creado" pero el stock no se reservó aún
→ Si otro usuario compra el mismo último ítem en esos 5 segundos:
→ Ambos tienen un pedido, pero solo hay stock para uno
→ Solución: reservar el stock síncronamente antes de crear el pedido
   (no con mensajería — con una llamada HTTP síncrona a Products.API)

Caso 2: Sistema financiero con liquidación inmediata
→ El saldo debe reflejarse inmediatamente en todos los sistemas
→ Solución: para datos financieros críticos, considerar SAGA síncrona
   o 2PC con bancos de datos que lo soporten

Caso 3: Dashboard en tiempo real
→ El dashboard dice "pedidos pendientes: 150" pero la realidad es 152
→ Solución: aceptar que el dashboard tiene ~5 segundos de retraso
   O usar streaming (SignalR) para actualizaciones en tiempo real
```

Para TechShop: la consistencia eventual con ~5-20 segundos es perfectamente aceptable en todos los flujos actuales.

## Eventual Consistency desde la perspectiva del cliente

Es importante comunicar al cliente (frontend, móvil) que el sistema es eventual:

```typescript
// En el cliente (React/TypeScript) — manejar consistencia eventual

async function createOrder(orderData: CreateOrderRequest) {
    // 1. Crear el pedido → 201 Created inmediatamente
    const { data: order } = await api.post('/api/orders', orderData);

    // 2. El pedido está creado pero aún en procesamiento
    // NO mostrar "pedido confirmado" todavía — mostrar "pedido en proceso"

    // 3. Polling o WebSocket para saber cuándo se confirma
    const confirmed = await waitForConfirmation(order.id, {
        timeout: 30_000,        // Máximo 30 segundos de espera
        pollInterval: 2_000,    // Comprobar cada 2 segundos
    });

    if (confirmed) {
        showSuccess('¡Tu pedido ha sido confirmado!');
    } else {
        // Si en 30s no se confirma → mostrar estado actual
        showInfo('Tu pedido está siendo procesado. Te notificaremos por email.');
    }
}

async function waitForConfirmation(orderId: string, options: Options) {
    const deadline = Date.now() + options.timeout;
    while (Date.now() < deadline) {
        const order = await api.get(`/api/orders/${orderId}`);
        if (order.status === 'Confirmed') return true;
        if (order.status === 'Failed') return false;
        await sleep(options.pollInterval);
    }
    return false; // Timeout
}
```

## El Outbox y la Saga: interacción

La Saga también se beneficia del Outbox. Cuando la Saga hace `Publish(ReserveStock)`:

```
ANTES (sin Outbox):
  1. Saga cambia estado: Pending → recibe OrderCreated
  2. Guarda el nuevo estado en orders.OrderSagaState
  3. Publica ReserveStock a RabbitMQ (fuera de la transacción)
  Si falla entre 2 y 3: Saga en Pending, ReserveStock nunca enviado

DESPUÉS (con Outbox):
  1. Saga cambia estado
  2. Guarda el nuevo estado + ReserveStock en OutboxMessage
     (en la misma transacción)
  3. BackgroundService publica ReserveStock a RabbitMQ
  Si falla entre 2 y 3: BackgroundService lo reintenta
```

MassTransit Outbox integra automáticamente con el estado de la Saga.

## Monitorear el Outbox

El Outbox acumulando mensajes sin procesar es una señal de alarma:

```sql
-- Ver mensajes pendientes de publicar
SELECT TOP 10
    [SequenceNumber],
    [SentTime],
    [MessageType],
    DATEDIFF(SECOND, [SentTime], GETUTCDATE()) AS AgeSeconds
FROM [orders].[OutboxMessage]
WHERE [OutboxId] IS NULL  -- No procesados aún
ORDER BY [SentTime] ASC;

-- Si hay mensajes con más de 60 segundos sin procesar: alarma
SELECT COUNT(*) AS PendingCount
FROM [orders].[OutboxMessage]
WHERE [OutboxId] IS NULL
  AND DATEDIFF(SECOND, [SentTime], GETUTCDATE()) > 60;
```

En Application Insights (M7.4), puedes crear una alerta: "Si `PendingOutboxCount` > 100 durante más de 5 minutos → alertar al equipo".

## Saga sin Outbox vs con Outbox: el caso real

Escenario: el servidor de Orders.API cae inmediatamente después de guardar el pedido pero antes de publicar el evento.

**Sin Outbox:**
```
1. INSERT Order → OrdersDb ✅
2. CRASH del proceso ← aquí
3. Publish(OrderCreated) → NUNCA OCURRE
4. La Saga nunca arranca
5. El pedido queda en "Pending" para siempre
6. El cliente llama al soporte: "mi pedido no se ha procesado"
7. Soporte crea un script para forzar la compensación
→ Intervención manual necesaria
```

**Con Outbox:**
```
1. INSERT Order + INSERT OutboxMessage → OrdersDb ✅ (misma transacción)
2. CRASH del proceso ← aquí
3. Orders.API se reinicia (Azure App Service lo hace automáticamente)
4. OutboxProcessor arranca
5. Lee el OutboxMessage pendiente
6. Publica OrderCreated a RabbitMQ ✅
7. La Saga arranca y procesa normalmente
→ Sin intervención manual
```

La diferencia: un script de soporte manual vs recuperación automática en segundos.

## El flujo completo con Outbox y Saga

Revisando el flujo completo de un pedido con todas las garantías:

```
t=0s   POST /api/orders → Orders.API
       BEGIN TRANSACTION
         INSERT orders.Orders (estado: Pending)
         INSERT orders.OutboxMessage (OrderCreated, content: {...})
       COMMIT TRANSACTION
       → 201 Created al cliente

t=5s   OutboxProcessor lee OutboxMessage[OrderCreated]
       Publica → RabbitMQ exchange [OrderCreated]
       UPDATE OutboxMessage SET ProcessedAt = now

t=5s   Saga (en Orders.API) recibe OrderCreated
       BEGIN TRANSACTION
         INSERT orders.OrderSagaState (estado: Pending)
         INSERT orders.OutboxMessage (ReserveStock, content: {...})
       COMMIT TRANSACTION

t=10s  OutboxProcessor publica ReserveStock → Products.API
       Products reserva stock → SQL UPDATE
       Publica StockReserved → RabbitMQ

t=10s  Saga recibe StockReserved
       BEGIN TRANSACTION
         UPDATE orders.OrderSagaState (estado: PaymentProcessing)
         INSERT orders.OutboxMessage (ProcessPayment)
       COMMIT TRANSACTION

t=15s  OutboxProcessor publica ProcessPayment → Payments.API
       Payments cobra → SQL INSERT
       Publica PaymentProcessed → RabbitMQ

t=15s  Saga recibe PaymentProcessed
       BEGIN TRANSACTION
         UPDATE orders.OrderSagaState (estado: Completed)
         INSERT orders.OutboxMessage (OrderConfirmed)
       COMMIT TRANSACTION

t=20s  OutboxProcessor publica OrderConfirmed
       Notifications envía email
       Orders actualiza el pedido a Confirmed
```

Cada paso es atómico. Cada mensaje tiene garantía de entrega. El sistema se recupera automáticamente de cualquier fallo.

## Limpiar el Outbox: purgar mensajes procesados

La tabla `OutboxMessage` crece con el tiempo. Hay que limpiar los mensajes ya publicados:

```csharp
// BackgroundService para limpiar el Outbox periódicamente

public class OutboxCleanupJob : BackgroundService
{
    protected override async Task ExecuteAsync(CancellationToken ct)
    {
        while (!ct.IsCancellationRequested)
        {
            // Esperar 1 hora entre limpiezas
            await Task.Delay(TimeSpan.FromHours(1), ct);

            using var scope = _scopeFactory.CreateScope();
            var dbContext = scope.ServiceProvider
                .GetRequiredService<OrderDbContext>();

            // Borrar mensajes procesados hace más de 24 horas
            var cutoff = DateTime.UtcNow.AddHours(-24);
            await dbContext.Database.ExecuteSqlRawAsync(
                "DELETE FROM orders.OutboxMessage WHERE OutboxId IS NOT NULL " +
                "AND SentTime < {0}", ct, cutoff);

            _logger.LogInformation(
                "Outbox cleanup completed. Removed messages older than {Cutoff}",
                cutoff);
        }
    }
}
```

MassTransit también tiene limpieza automática configurable.

## Dead Letter Queue del Outbox

Si el OutboxProcessor no puede publicar un mensaje (RabbitMQ caído durante demasiado tiempo, o el mensaje tiene un formato incorrecto), el mensaje puede ir a una "Dead Letter" dentro del propio Outbox:

```csharp
// En el OutboxProcessor manual — tracking de errores
private async Task PublishMessageAsync(OutboxMessage message, ...)
{
    try
    {
        // Publicar...
        message.ProcessedAt = DateTime.UtcNow;
    }
    catch (Exception ex)
    {
        message.RetryCount++;
        message.Error = ex.Message[..Math.Min(500, ex.Message.Length)];

        // Si superó el máximo de reintentos → marcar para revisión manual
        if (message.RetryCount >= 5)
        {
            message.RequiresManualReview = true;
            _logger.LogError(
                "[OUTBOX ALERT] Message {Id} of type {Type} requires manual review. " +
                "Last error: {Error}",
                message.Id, message.Type, message.Error);

            // Notificar al equipo
            await _alertService.SendAlertAsync(
                $"Outbox message {message.Id} stuck after 5 retries");
        }
    }
    await dbContext.SaveChangesAsync(ct);
}
```

## Outbox con múltiples instancias del servicio

¿Qué pasa cuando hay 3 instancias de Orders.API corriendo y todas tienen su OutboxProcessor?

```
Sin protección:
  Instancia 1 lee el mensaje ID:123
  Instancia 2 lee el mensaje ID:123
  Instancia 1 publica OrderCreated para ID:123 ✅
  Instancia 2 publica OrderCreated para ID:123 ✅ (duplicado!)
```

MassTransit Outbox resuelve esto con un **lock optimista**:

```sql
-- El OutboxProcessor actualiza el LockId antes de procesar
UPDATE orders.OutboxMessage
SET LockId = @newLockId
WHERE SequenceNumber = @messageId
  AND LockId = @expectedLockId  -- Solo si nadie más lo tiene
-- Si 0 filas afectadas: otro procesador ya lo tomó → saltar
```

Solo una instancia puede tomar el lock para cada mensaje. Sin duplicados, incluso con múltiples instancias.

## Configurar el Outbox con opciones avanzadas

```csharp
x.AddEntityFrameworkOutbox<OrderDbContext>(o =>
{
    o.UseSqlServer();

    // Qué tan seguido el BackgroundService procesa mensajes
    o.QueryDelay = TimeSpan.FromSeconds(5);

    // Máximo de mensajes por ciclo de procesamiento
    o.QueryMessageLimit = 100;

    // Timeout de lock (si una instancia muere con el lock tomado)
    o.LockStatementProvider = new SqlServerLockStatementProvider();

    // Limpiar mensajes procesados automáticamente
    o.RemoveDelivered = true;  // Eliminar después de publicar
    // O: o.RemoveDeliveredMessagesAfter = TimeSpan.FromHours(24);

    // Integrar con el bus de mensajes
    o.UseBusOutbox(busOutbox =>
    {
        // Tiempo máximo antes de intentar de nuevo
        busOutbox.MessageDeliveryTimeout = TimeSpan.FromMinutes(5);
    });
});
```

## Estrategia de retry del Outbox vs retry de MassTransit

Hay dos capas de retry que conviene distinguir:

**Retry del Outbox (Orders.API → RabbitMQ):**
```
Si RabbitMQ no acepta el mensaje → OutboxProcessor reintenta cada 5s
Indefinidamente hasta que RabbitMQ vuelva
→ Protege contra RabbitMQ caído
```

**Retry de MassTransit (RabbitMQ → Consumer):**
```
Si el Consumer falla al procesar → MassTransit reintenta N veces
Con backoff exponencial (1s, 2s, 4s, 8s, 16s)
Después de N intentos → Dead Letter Queue
→ Protege contra consumers con errores transitorios
```

Son capas independientes y complementarias. El Outbox garantiza la entrega. MassTransit retry garantiza el procesamiento.

## ¿Cuándo usar el Outbox?

El Outbox añade complejidad. ¿Cuándo es necesario?

**SÍ usar Outbox:**
- Cuando la publicación del evento es crítica para el flujo de negocio
- Cuando la Saga coordina pasos que modifican múltiples servicios
- Cuando no puedes aceptar que un pedido quede "huérfano" (sin su evento)
- En sistemas financieros, logísticos o de e-commerce

**NO usar Outbox (o simplificar):**
- Para eventos puramente informativos que no afectan al flujo
- Cuando la pérdida ocasional de un evento es aceptable
- Para notificaciones que se pueden regenerar (ej: "enviar un email de marketing")

**Para OrderFlow:** Outbox para todos los eventos de la Saga (OrderCreated, StockReserved, PaymentProcessed, OrderConfirmed). Sin Outbox para eventos analíticos o de logging.

## Change Data Capture: alternativa al Outbox con CDC

Existe otra aproximación para resolver el dual-write sin Outbox Pattern: **Change Data Capture (CDC)**.

CDC captura los cambios en la BD a nivel de transaction log y los publica como eventos:

```
Sin CDC (Outbox):
  App → INSERT Order + INSERT OutboxMessage → BD
  BackgroundService → lee OutboxMessage → publica a RabbitMQ

Con CDC (Debezium):
  App → INSERT Order → BD
  Debezium lee el transaction log de SQL Server
  Debezium publica el cambio como evento a Kafka/RabbitMQ
```

**Ventajas de CDC:**
- El código de la app es más simple (sin OutboxMessage)
- Captura cualquier cambio, incluyendo los de herramientas externas
- Alta fiabilidad a nivel de plataforma

**Desventajas de CDC:**
- Requiere Debezium (Java) u otro CDC connector
- Mayor complejidad operativa
- Para el curso: demasiada infraestructura adicional

**Para TechShop:** Outbox Pattern. Simple, en .NET puro, sin dependencias adicionales. CDC para sistemas grandes con muchas fuentes de datos.

## El problema de los Long-Running Processes

La Saga gestiona flujos que tardan segundos. ¿Qué pasa con flujos que tardan horas o días?

```
Ejemplo: Pedido de un producto bajo demanda
  Día 1: Cliente crea pedido → OrderCreated → Saga arranca
  Día 1-3: Proveedor fabrica el producto (estado: Manufacturing)
  Día 4: Producto listo → StockAvailable
  Día 4-5: Envío a cliente (estado: Shipping)
  Día 6: Entregado → Delivered
```

Para estos flujos largos, la Saga de MassTransit no es la herramienta adecuada — el estado debe sobrevivir reinicios, actualizaciones y fallos durante días.

**Alternativas para Long-Running Processes:**
- **Azure Durable Functions:** Estado gestionado por Azure, pausas largas sin consumir recursos
- **Workflow engines:** Camunda, Temporal.io — diseñados para procesos de días/semanas
- **State machines simples en BD:** Una tabla con el estado del pedido, actualizada por eventos

Para OrderFlow del curso: todos los flujos completan en segundos. La Saga de MassTransit es perfecta.

## Saga Timeout en producción: monitorear procesos atascados

En producción, una Saga que lleva más de X minutos en el mismo estado es señal de problema:

```sql
-- Detectar Sagas atascadas
SELECT
    CorrelationId,
    CurrentState,
    CreatedAt,
    DATEDIFF(MINUTE, CreatedAt, GETUTCDATE()) AS MinutesInCurrentState
FROM orders.OrderSagaState
WHERE CurrentState NOT IN ('Completed', 'Failed')
  AND CreatedAt < DATEADD(MINUTE, -10, GETUTCDATE())
ORDER BY CreatedAt ASC;
```

Si una Saga lleva más de 10 minutos en estado `Pending` (esperando StockReserved), probablemente:
- Products.API está caído
- La cola de RabbitMQ está saturada
- El Consumer de ReserveStock está fallando repetidamente

Con esta query en una alerta de Application Insights: el equipo recibe notificación antes de que el cliente llame al soporte.

## Comparativa: Outbox vs Saga Choreography

Hay otra aproximación que parece resolver el mismo problema: la choreografía pura (M4.3 lo explicamos).

**Choreography pura (eventos reactivos):**
- Cada servicio reacciona a eventos y publica los suyos
- Sin coordinador central (sin Saga state)
- Más simple de entender individualmente
- Más difícil de monitorear y compensar

**Saga + Outbox (nuestra elección):**
- Un coordinador central (Saga) gestiona el flujo
- El Outbox garantiza la entrega de cada mensaje de la Saga
- Más fácil de monitorear (el estado de la Saga es visible en la BD)
- La compensación es explícita y verificable

Para OrderFlow: Saga + Outbox. La visibilidad del estado de la Saga en la BD es fundamental para debugging y soporte.

## WebSockets con SignalR para actualizaciones en tiempo real

```csharp
public class OrderStatusHub : Hub

    {
        await Groups.AddToGroupAsync(Context.ConnectionId, $"order-{orderId}");

public class OrderConfirmedConsumer : IConsumer<OrderConfirmed>

    public async Task Consume(ConsumeContext<OrderConfirmed> context)

        var msg = context.Message;

            .Group($"order-{msg.OrderId}")

    }
}
```

## Preguntas frecuentes sobre el Outbox Pattern

**¿El Outbox Pattern añade latencia?**
Sí — entre 1 y 10 segundos dependiendo del intervalo de polling. Para la mayoría de flujos de e-commerce, esto es completamente aceptable. Si necesitas latencia < 1s, considera aumentar la frecuencia de polling o usar CDC.

**¿Qué pasa si la tabla OutboxMessages se llena?**
Si el procesador no puede publicar durante horas (RabbitMQ caído), la tabla crece. SQL Server puede manejar millones de filas sin problema. Cuando RabbitMQ vuelve, el procesador las publica todas y las limpia.

**¿El Outbox funciona con Azure Service Bus?**
Sí. MassTransit Outbox es agnóstico al broker. Si cambias de RabbitMQ a Azure Service Bus (una línea de código en M7.2), el Outbox funciona igual. El mensaje se publica a Azure Service Bus en vez de RabbitMQ.

**¿Necesito el Outbox para todos los eventos?**
No. Solo para eventos críticos donde la pérdida causaría inconsistencias de negocio. Para eventos analíticos o de logging, el dual-write puede ser aceptable.

## Resumen de M6.2

El Outbox Pattern resuelve el problema del dual-write de forma elegante:

**La raíz del problema:** SQL y RabbitMQ no pueden participar en la misma transacción. Publicar un evento y guardar datos son dos operaciones que pueden fallar independientemente.

**La solución:** Guardar el evento en la BD (mismo sistema que los datos) en la misma transacción. Un BackgroundService lo publica a RabbitMQ de forma asíncrona y garantizada.

**MassTransit Outbox integrado:** `AddEntityFrameworkOutbox<OrderDbContext>` + `UseBusOutbox()`. Sin código manual de OutboxProcessor, sin tablas manuales, con lock optimista para múltiples instancias.

**Consistencia eventual garantizada:** El cliente recibe 201 inmediatamente. El evento llega al consumer en ~5 segundos. La Saga completa en ~20 segundos. Sin posibilidad de pedidos huérfanos.

**El resultado:** OrderFlow no tiene estado inconsistente posible en el flujo de creación de pedidos. Cualquier fallo en cualquier punto es recuperable automáticamente.
