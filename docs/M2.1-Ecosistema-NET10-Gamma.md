## ¿Por qué .NET 10 para microservicios?

Antes de abrir Visual Studio, una pregunta legítima: ¿hay algo específico en .NET 10 que lo haga especialmente bueno para microservicios? ¿O es lo mismo de siempre con otro número?

La respuesta: .NET 10 tiene mejoras concretas y significativas para microservicios. No es marketing.

Tres razones principales:

**Rendimiento:** .NET 10 es la versión más rápida de .NET hasta la fecha. Mejoras en JIT, NativeAOT, y stack allocation reducen el tiempo de respuesta y el consumo de memoria.

**Tooling:** .NET Aspire 9.1 convierte el desarrollo local de microservicios en una experiencia de F5. Sin Docker Desktop, sin configuraciones manuales.

**Ecosistema:** ASP.NET Core 10, EF Core 10, C# 14, OpenTelemetry integrado nativamente. Todo el stack ha madurado en la misma dirección.

El tooling ha madurado exactamente en la dirección que necesitaban los microservicios. Es el mejor momento para aprenderlos con .NET.

## .NET 10: rendimiento — los números

TechEmpower es el benchmark de referencia de la industria para frameworks web. Mide peticiones por segundo en diferentes escenarios.

.NET 10 en TechEmpower:
- **Plaintext:** ~7 millones de peticiones/segundo en un solo servidor
- **JSON serialization:** ~4 millones de peticiones/segundo
- **Multiple queries (DB):** ~300.000 peticiones/segundo

Para contexto: Python/Django a modo comparable hace ~50.000 req/seg en plaintext. Node.js/Express, ~1 millón. .NET 10 está en la élite junto a Go y Rust.

Para microservicios, esto significa dos cosas prácticas: menos instancias del App Service necesarias para el mismo tráfico (menos coste), y tiempo de respuesta bajo incluso con lógica de negocio compleja.

## .NET 10: mejoras en el runtime

Las mejoras que más impactan en microservicios:

**JIT mejorado — más inteligente.**
El compilador Just-In-Time de .NET 10 hace mejor inlining de métodos (sustituye la llamada por el código directamente), mejor devirtualización (llama directamente al método concreto sin buscar en la tabla virtual), y mejor generación de código para structs. Resultado: código que se ejecuta más rápido sin cambiar nada.

**Stack allocation para arrays pequeños.**
En .NET 10, arrays pequeños de tamaño fijo se asignan en la pila (stack) en vez de en el heap. Menos presión sobre el Garbage Collector, menos pausas de GC. En un microservicio que procesa miles de peticiones por segundo, esto se nota.

**Write-barrier GC mejorado.**
Las barreras de escritura del GC (código que se ejecuta cuando mutas una referencia) son más eficientes. Reducción de 8-20% en los tiempos de pausa del GC en benchmarks reales.

---

## .NET 10: NativeAOT — imágenes mínimas

NativeAOT (Ahead-Of-Time compilation) compila tu aplicación .NET directamente a código nativo en tiempo de build. Sin JIT en tiempo de ejecución, sin runtime de .NET separado.

Para microservicios, las ventajas son claras:

**Arranque ultra-rápido.** Un servicio .NET normal tarda ~200-500ms en arrancar. Con NativeAOT: ~10-50ms. En Kubernetes con auto-scaling, esto importa — las nuevas instancias están listas casi inmediatamente.

**Imagen de contenedor mínima.** Un microservicio ASP.NET Core normal: ~200-300MB de imagen. Con NativeAOT y Ubuntu Chiseled: ~30-80MB. Menos almacenamiento, menos tiempo de pull, menos superficie de ataque.

**Cuándo usarlo en OrderFlow:** Para servicios muy ligeros y de alto volumen, como Notifications.API. Para servicios complejos con reflection (como los que usan EF Core), NativeAOT tiene limitaciones y puede no ser viable sin ajustes.

En este curso usamos el runtime normal (framework-dependent). NativeAOT es una optimización para cuando el servicio está maduro.

## EF Core 10: novedades relevantes para OrderFlow

EF Core 10 tiene mejoras específicas para SQL Server que usaremos:

**Named query filters.**
Múltiples query filters por entidad, con posibilidad de deshabilitar solo algunos:

```csharp
modelBuilder.Entity<Order>()
    .HasQueryFilter("activeFilter", o => !o.IsDeleted);

// Consulta ignorando el filtro de borrado lógico
await dbContext.Orders
    .IgnoreQueryFilter("activeFilter")
    .ToListAsync();
```

**Temporal Tables para historial automático.**
SQL Server mantiene el historial de cambios sin código adicional:

```csharp
modelBuilder.Entity<Order>()
    .ToTable("Orders", b => b.IsTemporal());

// ¿Cómo era este pedido hace 1 hora?
var orderAnHourAgo = await dbContext.Orders
    .TemporalAsOf(DateTime.UtcNow.AddHours(-1))
    .FirstOrDefaultAsync(o => o.Id == orderId);
```

Para auditoría y compliance, esto es oro. Sin escribir código de auditoría custom, SQL Server guarda el historial completo.

## C# 14: las novedades que usaremos

C# 14 llega con .NET 10. Las novedades más relevantes para nuestro proyecto:

**Field-backed properties.**
Puedes acceder al backing field de una propiedad autoimplementada con la keyword `field`. Suaviza el paso de una propiedad auto a una con lógica custom:

```csharp
public class Order
{
    public decimal Total
    {
        get => field;
        set => field = value >= 0
            ? value
            : throw new ArgumentException("Total no puede ser negativo");
    }
}
```

Antes tenías que declarar un campo privado explícito. Ahora el compilador lo gestiona.

**Collection expression extensions.**
Inicialización de colecciones más limpia y con soporte para tipos custom.

## C# 14: records para Value Objects

Los records de C# son perfectos para Value Objects — inmutables, comparación por valor, sintaxis limpia:

```csharp
// Value Object con C# 14
public record Money(decimal Amount, string Currency)
{
    public Money
    {
        if (Amount < 0)
            throw new ArgumentException("Amount cannot be negative");

        if (string.IsNullOrEmpty(Currency) || Currency.Length != 3)
            throw new ArgumentException("Currency must be a 3-letter ISO code");
    }

    public Money Add(Money other)
    {
        if (Currency != other.Currency)
            throw new InvalidOperationException("Cannot add different currencies");
        return this with { Amount = Amount + other.Amount };
    }

    public override string ToString() => $"{Amount:F2} {Currency}";
}

// Uso
var price = new Money(99.99m, "EUR");
var tax = new Money(21.00m, "EUR");
var total = price.Add(tax);  // 120.99 EUR
```

En OrderFlow, usaremos records para Money, Address, OrderStatus y otros value objects. Lo veremos en detalle en M3.1.

## ASP.NET Core 10: el modelo de hosting

Antes de escribir código, entender el modelo de hosting — cómo arranca una aplicación ASP.NET Core:

```csharp
// Program.cs — el punto de entrada
var builder = WebApplication.CreateBuilder(args);

// 1. Configurar servicios (DI container)
builder.Services.AddControllers();
builder.Services.AddScoped<IOrderService, OrderService>();
// ... más registros

// 2. Construir la aplicación
var app = builder.Build();

// 3. Configurar el pipeline de middleware
app.UseAuthentication();
app.UseAuthorization();
app.MapControllers();

// 4. Arrancar
app.Run();
```

**WebApplicationBuilder** es el punto de entrada único. Configura servicios, configuración y logging en un solo lugar. Es el modelo moderno — el `Startup.cs` de versiones anteriores ya no existe.

## El pipeline de middleware

Una petición HTTP en ASP.NET Core pasa por una cadena de middlewares antes de llegar al endpoint:

```
HTTP Request
    ↓
ExceptionHandlerMiddleware    ← captura excepciones no manejadas
    ↓
HttpsRedirectionMiddleware   ← redirige HTTP a HTTPS
    ↓
CorrelationIdMiddleware      ← añade/propaga correlation ID
    ↓
AuthenticationMiddleware     ← valida JWT
    ↓
AuthorizationMiddleware      ← verifica permisos
    ↓
RateLimitingMiddleware       ← controla el ritmo de peticiones
    ↓
RoutingMiddleware            ← determina qué endpoint responde
    ↓
EndpointMiddleware           ← ejecuta el endpoint (Controller o Minimal API)
    ↓
HTTP Response
```

El orden importa. Authentication siempre antes que Authorization. ExceptionHandler siempre el primero para capturar errores de cualquier middleware posterior.

## Controllers vs Minimal APIs: cuál elegir

Esta es la decisión que tomaremos para cada servicio de OrderFlow. Vamos a ver las diferencias con código real.

**El mismo endpoint — con Controller:**

```csharp
[ApiController]
[Route("api/[controller]")]
public class OrdersController : ControllerBase
{
    private readonly IOrderService _service;

    public OrdersController(IOrderService service)
        => _service = service;

    [HttpGet("{id:guid}")]
    [ProducesResponseType<OrderDto>(StatusCodes.Status200OK)]
    [ProducesResponseType(StatusCodes.Status404NotFound)]
    public async Task<ActionResult<OrderDto>> GetById(Guid id)
    {
        var order = await _service.GetByIdAsync(id);
        return order is null ? NotFound() : Ok(order);
    }
}
```

---

## Controllers vs Minimal APIs: la comparativa con código

**El mismo endpoint — con Minimal API:**

```csharp
app.MapGet("/api/orders/{id:guid}", async (
    Guid id,
    IOrderService service) =>
{
    var order = await service.GetByIdAsync(id);
    return order is null
        ? Results.NotFound()
        : Results.Ok(order);
})
.WithName("GetOrderById")
.Produces<OrderDto>()
.ProducesProblem(StatusCodes.Status404NotFound);
```

**Diferencias inmediatas:**
- Minimal API: menos código, sin herencia, sin atributos de clase
- Controller: estructura más explícita, convenciones más claras en proyectos grandes
- Minimal API: mejor rendimiento en startup (no hay reflexión para descubrir Controllers)
- Controller: mejor soporte para convenciones REST estrictas y APIs complejas

## La decisión para OrderFlow

Con el análisis anterior, la decisión para nuestro proyecto:

**Orders.API → Controllers**
Es el servicio más complejo. Muchos endpoints (crear, confirmar, cancelar, listar por estado, listar por cliente, historial, detalle). Saga orchestrator. CQRS con MediatR. La estructura de Controllers aporta organización clara cuando hay complejidad.

**Products.API → Minimal APIs**
Servicio más ligero. 5-6 endpoints directos. Minimal APIs le dan rapidez de desarrollo y menos ceremony.

**Payments.API → Minimal APIs**
Pocas operaciones pero con validaciones estrictas. Minimal APIs + endpoint filters para validación funcionan perfectamente.

**Notifications.API → Minimal APIs**
Principalmente un consumer de eventos. Apenas tiene endpoints propios. Minimal APIs es natural para servicios reactivos ligeros.

Esta distinción no es dogma — es pragmatismo. El mejor enfoque para cada contexto.

## .NET Aspire 9.1: el game changer para microservicios

Antes de crear el primer proyecto, hablamos de la herramienta que cambia completamente la experiencia de desarrollo con microservicios en .NET.

.NET Aspire es un stack cloud-native para .NET. No es un framework — es una forma de componer y orquestar servicios durante el desarrollo local.

**El problema que resuelve:** Para trabajar con 4 microservicios necesitas: arrancar cada servicio en un terminal diferente, configurar connection strings para cada uno, gestionar RabbitMQ por separado, recordar los puertos de cada servicio, y abrir el dashboard de observabilidad manualmente.

Con Aspire: pulsas F5 y todo arranca. Un dashboard centralizado muestra logs, trazas y métricas de todos los servicios. La configuración se inyecta automáticamente.

## Aspire: los tres proyectos

Una solución con Aspire tiene (al menos) tres tipos de proyectos:

**AppHost.** El orquestador. Define qué servicios arrancan, en qué orden, con qué recursos. Es el "Director de Orquesta" del proyecto.

```
OrderFlow.AppHost/
└── Program.cs   ← donde defines qué arranca y cómo
```

**ServiceDefaults.** Configuración compartida para todos los servicios. OpenTelemetry, health checks, service discovery — configurados una vez, aplicados a todos.

```
OrderFlow.ServiceDefaults/
└── Extensions.cs  ← AddServiceDefaults()
```

**Los servicios:** Orders.API, Products.API, Payments.API, Notifications.API. Cada uno añade `builder.AddServiceDefaults()` y hereda la configuración compartida.

## Aspire AppHost: el Program.cs del orquestador

El AppHost define toda la infraestructura del sistema de desarrollo:

```csharp
// OrderFlow.AppHost/Program.cs

var builder = DistributedApplication.CreateBuilder(args);

// Infraestructura — AddConnectionString (sin Docker)
var sqlServer = builder.AddConnectionString("sqlserver");

// Bases de datos por servicio
var ordersDb = sqlServer.AddDatabase("OrdersDb");
var productsDb = sqlServer.AddDatabase("ProductsDb");

// Servicios
var orders = builder.AddProject<Projects.Orders_API>("orders-api")
    .WithReference(ordersDb)
    .WithReference(messaging);

var products = builder.AddProject<Projects.Products_API>("products-api")
    .WithReference(productsDb)
    .WithReference(rabbitMq);

builder.AddProject<Projects.Payments_API>("payments-api")
    .WithReference(paymentsDb)
    .WithReference(rabbitMq);

builder.AddProject<Projects.Notifications_API>("notifications-api")
    .WithReference(notificationsDb)
    .WithReference(rabbitMq);

// YARP como gateway
builder.AddProject<Projects.Gateway_API>("gateway")
    .WithReference(orders)
    .WithReference(products);

builder.Build().Run();
```

## Aspire AppHost: lo que hace automáticamente

Cuando pulsas F5 en el AppHost, Aspire:

**1. Arranca los recursos de infraestructura.**
SQL Server y RabbitMQ se levantan automáticamente si los has configurado como recursos de Aspire. Si tienes LocalDB y RabbitMQ instalados localmente, Aspire los usa directamente — sin Docker.

**2. Inyecta connection strings.**
Orders.API recibe automáticamente la connection string de OrdersDb. No necesitas escribirla en appsettings.json. Aspire la inyecta como variable de entorno.

**3. Configura service discovery.**
El servicio orders-api puede llamar a `http://products-api` y Aspire resuelve a la dirección correcta. Cero configuración manual de URLs entre servicios.

**4. Recolecta telemetría.**
Todos los servicios que usen ServiceDefaults envían sus logs, trazas y métricas al dashboard de Aspire. Sin configurar nada adicional.

**5. Abre el dashboard.**
Un navegador se abre automáticamente con el dashboard de Aspire donde puedes ver el estado de cada servicio.

¿Por qué Aspire en vez de docker-compose?

**Docker Compose:**

services:

    build: ./src/Orders.API
    ports:
      - "5001:8080"

Y con `AddConnectionString()` no necesitas Docker Desktop en absoluto.

## Aspire ServiceDefaults: configuración compartida

ServiceDefaults es el paquete que añaden todos los servicios para heredar la configuración base de Aspire:

```csharp
// OrderFlow.ServiceDefaults/Extensions.cs

public static class Extensions
{
    public static IHostApplicationBuilder AddServiceDefaults(
        this IHostApplicationBuilder builder)
    {
        // OpenTelemetry: traces, metrics, logs
        builder.AddDefaultOpenTelemetry();

        // Health checks estándar
        builder.Services.AddServiceDiscovery();

        // Service discovery automático
        builder.Services.ConfigureHttpClientDefaults(http =>
        {
            http.AddServiceDiscovery();
        });

        return builder;
    }
}
```

Cada servicio lo usa con una sola línea:

```csharp
// Orders.API/Program.cs
var builder = WebApplication.CreateBuilder(args);
builder.AddServiceDefaults(); // ← una línea para todo
```

Con esa línea, el servicio tiene OpenTelemetry, health checks y service discovery configurados. Sin repetir código en cada servicio.

## El dashboard de Aspire

El dashboard de Aspire es la herramienta de observabilidad para desarrollo local. Se abre automáticamente al pulsar F5.

**Pestaña Recursos:**
Estado de cada servicio (Running, Stopped, Error), URLs, variables de entorno inyectadas. Puedes ver de un vistazo si todos los servicios están arriba.

**Pestaña Console:**
Logs de todos los servicios en tiempo real. Filtros por servicio, por nivel (Info, Warning, Error). Búsqueda por texto.

**Pestaña Estructured:**
Logs estructurados de Serilog. Puedes filtrar por cualquier propiedad: OrderId, CustomerId, CorrelationId.

**Pestaña Traces:**
Trazas distribuidas completas. Una petición que cruza Orders → Products aparece como un timeline único con todos los spans.

**Pestaña Metrics:**
Métricas de cada servicio: peticiones por segundo, latencia P50/P95/P99, errores. Sin configurar Prometheus ni Grafana.

Es la herramienta que te va a salvar horas de debugging durante el curso.

## Aspire con LocalDB y RabbitMQ local

Una pregunta frecuente: ¿Aspire necesita Docker para SQL Server y RabbitMQ?

**No.** Hay dos formas:

**Opción A — Con Docker (si está instalado):**
Aspire levanta contenedores automáticamente. `builder.AddSqlServer()` levanta un SQL Server en Docker.

**Opción B — Con recursos locales (nuestra opción):**

## Swagger UI vs .http files vs Postman

Con .NET 10 tienes varias opciones para probar tu API. ¿Cuál elegir?

**Ficheros .http (recomendado para este curso):**

**Bruno:**
Alternativa open source a Postman. Sin cuenta, sin tracking. Si ya usas Postman, Bruno es la alternativa directa.

**Nuestra elección:** `.http` files en Visual Studio para el curso. Si ya tienes Postman instalado y te es cómodo, úsalo — el resultado es el mismo.

## Convenciones de código para OrderFlow

Antes de avanzar, establecemos las convenciones que seguiremos en todos los servicios. Consistencia entre 4 proyectos es fundamental para que el equipo pueda saltar de uno a otro sin fricción.

**Naming:**
- Clases y tipos: PascalCase (`OrdersController`, `CreateOrderCommand`)
- Métodos: PascalCase (`GetByIdAsync`, `CreateAsync`)
- Variables y parámetros: camelCase (`orderId`, `customerId`)
- Interfaces: prefijo I (`IOrderRepository`)
- Records para DTOs y eventos: PascalCase (`OrderDto`, `OrderCreated`)

**Async everywhere:** Todos los métodos que tocan I/O son `async Task<T>`. Sin bloqueos síncronos.

**Nullable habilitado:** `<Nullable>enable</Nullable>` en todos los proyectos. El compilador avisa de posibles null references.

**Namespaces siguen la estructura de carpetas:** `Orders.API.Application.Commands`, `Orders.API.Domain.Entities`.
