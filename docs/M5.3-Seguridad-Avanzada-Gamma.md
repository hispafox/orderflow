## A01 — Broken Access Control: ¿somos vulnerables?

**El riesgo:** John puede ver pedidos de Jane porque el endpoint no verifica la propiedad.

**En OrderFlow, estado actual:**

```csharp
// ❌ Vulnerable ANTES de M5.1
[HttpGet("{id:guid}")]
public async Task<ActionResult<OrderDto>> GetById(Guid id)
{
    return Ok(await _repository.GetByIdAsync(id));
    // Cualquier usuario autenticado puede ver el pedido de otro
}

// ✅ Corregido en M5.1 + M5.2
[HttpGet("{id:guid}")]
[Authorize]
public async Task<ActionResult<OrderDto>> GetById(Guid id)
{
    var order = await _repository.GetByIdAsync(id);
    if (order is null) return NotFound();

    var userId = Guid.Parse(User.FindFirstValue("sub")!);
    if (order.CustomerId != userId && !User.IsInRole("admin"))
        return Forbid();

    return Ok(order.ToDto());
}
```

**Veredicto para OrderFlow:** Mitigado con el ownership check de M5.2.

## A02 — Cryptographic Failures: algoritmos débiles a evitar

OrderFlow usa RS256 (RSA + SHA-256) para JWT. Hay algoritmos que NUNCA deben usarse:

```csharp
// ❌ NUNCA: alg=none (sin firma)
// ❌ NUNCA: HS256 con un secret débil o predecible
// ❌ NUNCA: MD5 o SHA-1 para firmas

// ✅ Para JWT: HS256 (dev) / RS256 (producción con Azure AD)
// ✅ Para hashing de passwords: BCrypt, Argon2, PBKDF2
// ✅ Para cifrado de datos: AES-256-GCM
// ✅ Para TLS: 1.2 mínimo, preferir 1.3

// En .NET 10, verificar la versión de TLS
builder.WebHost.ConfigureKestrel(options =>
{
    options.ConfigureHttpsDefaults(httpsOptions =>
    {
        httpsOptions.SslProtocols =
            SslProtocols.Tls12 | SslProtocols.Tls13;
    });
});
```

Azure App Services usa TLS 1.2 mínimo por defecto. Configurable en el portal en "TLS/SSL Settings" → "Minimum TLS Version".

## A03 — Injection: ¿EF Core nos protege?

**El riesgo:** Un atacante inyecta SQL malicioso en un input del usuario.

```
Ataque clásico:
GET /api/orders?customerId='; DROP TABLE Orders; --

Sin protección:
SELECT * FROM Orders WHERE CustomerId = ''; DROP TABLE Orders; --'
→ ¡Tabla borrada!
```

**¿EF Core nos protege automáticamente?**

```csharp
// ✅ SEGURO — EF Core usa parámetros automáticamente
var orders = await _dbContext.Orders
    .Where(o => o.CustomerId == customerId)
    .ToListAsync();
// Se genera: SELECT * FROM Orders WHERE CustomerId = @p0
// @p0 = '3fa85f64-...' (parámetro tipado — no ejecutable)

// ❌ VULNERABLE — interpolación de strings en SQL raw
var orders = await _dbContext.Database
    .SqlQueryRaw<Order>(
        $"SELECT * FROM Orders WHERE CustomerId = '{customerId}'")
    .ToListAsync();
// Si customerId viene del usuario sin validar → SQL Injection
```

**Veredicto para OrderFlow:** EF Core con LINQ nos protege automáticamente. Peligro solo si usamos `SqlQueryRaw` con interpolación de strings. Usar `SqlQuery` con parámetros si necesitamos SQL raw.

---

## A03 — Injection seguro con SQL raw

Si necesitamos SQL raw (por rendimiento o por consultas complejas), usar FormattableString o parámetros:

```csharp
// ❌ VULNERABLE
var sql = $"SELECT * FROM orders.Orders WHERE Status = '{status}'";
_dbContext.Database.SqlQueryRaw<Order>(sql);

// ✅ SEGURO — FormattableString (EF Core convierte automáticamente a parámetros)
_dbContext.Database.SqlQuery<Order>(
    $"SELECT * FROM orders.Orders WHERE Status = {status}");
// Genera: SELECT * FROM orders.Orders WHERE Status = @p0

// ✅ SEGURO — parámetros explícitos con Dapper
var orders = await _connection.QueryAsync<Order>(
    "SELECT * FROM orders.Orders WHERE Status = @Status",
    new { Status = status });
```

**Regla:** Nunca interpolar strings del usuario directamente en SQL. Siempre parámetros.

## A04 — Insecure Design: diseño seguro desde el inicio

El A04 de OWASP no es sobre bugs — es sobre decisiones de arquitectura:

**Decisiones de diseño seguras en OrderFlow:**

`✅ Gateway como único punto de entrada:` Los servicios no están expuestos directamente. Un atacante no puede saltarse el gateway accediendo directamente a Orders.API.

`✅ Database per Service:` Si la BD de Notifications es comprometida, el atacante no tiene acceso a Orders. Los datos están aislados.

`✅ Mensajería asíncrona:` Si Payments falla, la Saga compensa. No hay estados corruptos en la BD.

`✅ JWT de corta duración:` Un token robado expira en 5 minutos. No hay sesiones infinitas.

`✅ Ownership check:` Por diseño, el sistema verifica que el recurso pertenece al usuario que lo solicita. No es un afterthought.

El A04 se previene en la fase de diseño (Módulos 1-4) — no se puede añadir después fácilmente.

## A05 — Security Misconfiguration: cabeceras HTTP

**El riesgo:** El servidor revela información útil para atacantes o no tiene protecciones básicas del navegador.

Sin cabeceras de seguridad, el navegador es más vulnerable a:
- **XSS (Cross-Site Scripting):** Inyección de scripts maliciosos
- **Clickjacking:** Incrustar la web en un iframe malicioso
- **MIME Sniffing:** El navegador adivina el tipo de contenido y ejecuta cosas peligrosas

**Añadir cabeceras de seguridad en el Gateway:**

```csharp
// Gateway.API/Program.cs

app.Use(async (context, next) =>
{
    var headers = context.Response.Headers;

    // Forzar HTTPS siempre (HSTS)
    headers.Append("Strict-Transport-Security",
        "max-age=31536000; includeSubDomains");

    // Prevenir clickjacking
    headers.Append("X-Frame-Options", "DENY");

    // Prevenir MIME sniffing
    headers.Append("X-Content-Type-Options", "nosniff");

    // Política de referrer
    headers.Append("Referrer-Policy", "strict-origin-when-cross-origin");

    // Ocultar la tecnología del servidor
    headers.Remove("Server");
    headers.Remove("X-Powered-By");

    await next();
});
```

## A07 — Authentication Failures: detección de ataques

Monitorear intentos de autenticación fallidos en el gateway:

```csharp
// Gateway.API/Program.cs — detectar brute force

builder.Services.AddAuthentication(...)
    .AddJwtBearer(options =>
    {
        options.Events = new JwtBearerEvents
        {
            OnAuthenticationFailed = async ctx =>
            {
                var logger = ctx.HttpContext.RequestServices
                    .GetRequiredService<ILogger<Program>>();
                var ip = ctx.HttpContext.Connection
                    .RemoteIpAddress?.ToString();

                logger.LogWarning(
                    "[SECURITY] Authentication failed from {IpAddress}. " +
                    "Error: {Error}. Path: {Path}",
                    ip, ctx.Exception.Message,
                    ctx.HttpContext.Request.Path);

                // Incrementar contador de fallos para la IP
                // Si supera el umbral → bloquear temporalmente
                var failureCounter = ctx.HttpContext.RequestServices
                    .GetRequiredService<IAuthFailureCounter>();
                var failures = await failureCounter.IncrementAsync(ip!, ct: default);

                if (failures > 10)
                {
                    logger.LogError(
                        "[SECURITY] Possible brute force from {IpAddress}. " +
                        "{Failures} failures in last minute",
                        ip, failures);
                    // En producción: añadir IP a blocklist temporal
                }
            }
        };
    });
```

## A09 — Logging Failures: qué loggear para seguridad

El estándar mínimo de logs de seguridad:

```csharp
// ✅ Loggear siempre:
```

## Content Security Policy (CSP)

La cabecera más potente para prevenir XSS:

```csharp
// CSP para la API de OrderFlow (no una web con HTML)
headers.Append("Content-Security-Policy",
    "default-src 'none'; " +
    "frame-ancestors 'none'");
// 'none' en default-src bloquea todo
// frame-ancestors 'none' refuerza X-Frame-Options

// Si el gateway sirve una web con Scalar UI:
headers.Append("Content-Security-Policy",
    "default-src 'self'; " +
    "script-src 'self' 'unsafe-inline'; " +   // ← Necesario para Scalar
    "style-src 'self' 'unsafe-inline'; " +
    "img-src 'self' data:; " +
    "connect-src 'self'; " +
    "frame-ancestors 'none'");
```

La CSP es un header complejo que depende del contenido que sirves. Para una API pura (JSON), la política más estricta es la correcta.

---

## Permissions Policy: deshabilitar features del navegador

```csharp
// Deshabilitar acceso a cámara, micrófono, geolocalización
headers.Append("Permissions-Policy",
    "camera=(), " +
    "microphone=(), " +
    "geolocation=(), " +
    "interest-cohort=()");
```

Estas cabeceras no son críticas para una API que solo sirve JSON, pero son buenas prácticas si el gateway también sirve contenido web.

## Protección contra Clickjacking en Scalar UI

Si el gateway sirve Scalar UI o cualquier documentación interactiva, necesita protección específica:

```csharp
// Para Scalar — CSP menos restrictivo
if (app.Environment.IsDevelopment())
{
    app.Use(async (ctx, next) =>
    {
        if (ctx.Request.Path.StartsWithSegments("/openapi") ||
            ctx.Request.Path.StartsWithSegments("/scalar"))
        {
            // Scalar necesita inline scripts
            ctx.Response.Headers.Append("Content-Security-Policy",
                "default-src 'self'; " +
                "script-src 'self' 'unsafe-inline'; " +
                "style-src 'self' 'unsafe-inline'; " +
                "img-src 'self' data:;");
        }

        await next();
    });

    app.MapOpenApi();
}
```

En producción, Scalar UI no debería estar expuesto públicamente.

## Rate Limiting por usuario autenticado

En M4.4 añadimos rate limiting por IP. El problema: múltiples usuarios comparten la misma IP (NAT, oficinas, proxies). El rate limiting por IP puede bloquear a usuarios legítimos.

Con JWT disponible, podemos hacer rate limiting por usuario autenticado:

```csharp
// Gateway.API/Program.cs

builder.Services.AddRateLimiter(options =>
{
    // Rate limit por userId del JWT (usuarios autenticados)
    options.AddSlidingWindowLimiter("PerUser", config =>
    {
        config.Window = TimeSpan.FromMinutes(1);
        config.PermitLimit = 60;
        config.SegmentsPerWindow = 4;
        config.QueueLimit = 10;
    });

    // Rate limit por IP (usuarios no autenticados)
    options.AddSlidingWindowLimiter("PerIp", config =>
    {
        config.Window = TimeSpan.FromMinutes(1);
        config.PermitLimit = 30; // Más restrictivo para anónimos
        config.SegmentsPerWindow = 4;
        config.QueueLimit = 5;
    });

    // Rate limit para creación de pedidos — estricto
    options.AddFixedWindowLimiter("CreateOrder", config =>
    {
        config.Window = TimeSpan.FromMinutes(1);
        config.PermitLimit = 10;
        config.QueueLimit = 0;
    });
```

## Rate Limiting con partition key del JWT

Para usar el userId como partition key, YARP necesita un partitioned rate limiter:

```csharp
    // Continuación de AddRateLimiter...

    // Rate limiter inteligente: userId si autenticado, IP si no
    options.AddPolicy("SmartRateLimit",
        httpContext =>
        {
            // Intentar obtener el userId del JWT
            var userId = httpContext.User?.FindFirstValue("sub");

            if (!string.IsNullOrEmpty(userId))
            {
                // Usuario autenticado: limitar por userId
                return RateLimitPartition.GetSlidingWindowLimiter(
                    partitionKey: $"user:{userId}",
                    factory: _ => new SlidingWindowRateLimiterOptions
                    {
                        Window = TimeSpan.FromMinutes(1),
                        PermitLimit = 60,
                        SegmentsPerWindow = 4
                    });
            }

            // Usuario anónimo: limitar por IP
            var clientIp = httpContext.Connection.RemoteIpAddress?.ToString()
                ?? "unknown";

            return RateLimitPartition.GetSlidingWindowLimiter(
                partitionKey: $"ip:{clientIp}",
                factory: _ => new SlidingWindowRateLimiterOptions
                {
                    Window = TimeSpan.FromMinutes(1),
                    PermitLimit = 30,
                    SegmentsPerWindow = 4
                });
        });

    options.RejectionStatusCode = StatusCodes.Status429TooManyRequests;
    options.OnRejected = async (context, ct) =>
    {
        context.HttpContext.Response.Headers.RetryAfter = "60";
        await context.HttpContext.Response.WriteAsync(
            "Rate limit exceeded. Try again in 60 seconds.", ct);
    };
});
```

## Audit Logging: el registro de negocio

El **audit logging** es diferente del **logging de aplicación**:

```
LOGGING DE APLICACIÓN (lo que tenemos con Serilog):
[INF] CreateOrderHandler: Creating order {OrderId}
[INF] HTTP 201 POST /api/orders (148ms)
→ Para developers y operaciones

AUDIT LOGGING (lo que añadimos ahora):
[AUDIT] User john@techshop.es (3fa85f64) created Order 7b3a2c1d
        at 2026-04-19T10:23:45Z from IP 81.34.56.78
        Amount: 3999.99 EUR | Items: 1 (MacBook Pro)
→ Para compliance, auditoría y respuesta a incidentes
```

El audit log debe ser:
- **Inmutable:** No se puede borrar ni modificar
- **Completo:** Toda acción sensible queda registrada
- **Correlacionado:** Vinculado al usuario, al pedido y a la IP

## Implementar el Audit Log en Orders.API

```csharp
// Orders.API/Application/Services/AuditLogger.cs

public interface IAuditLogger
{
    Task LogAsync(AuditEvent auditEvent, CancellationToken ct = default);
}

public record AuditEvent
{
    public string Action { get; init; } = string.Empty;
    public string UserId { get; init; } = string.Empty;
    public string UserEmail { get; init; } = string.Empty;
    public string ResourceType { get; init; } = string.Empty;
    public string? ResourceId { get; init; }
    public string? IpAddress { get; init; }
    public Dictionary<string, object>? Details { get; init; }
    public DateTime Timestamp { get; init; } = DateTime.UtcNow;
    public bool Success { get; init; } = true;
    public string? FailureReason { get; init; }
}

public class DatabaseAuditLogger : IAuditLogger
{
    private readonly OrderDbContext _dbContext;
    private readonly ILogger<DatabaseAuditLogger> _logger;

    public async Task LogAsync(AuditEvent evt, CancellationToken ct = default)
    {
        var entry = new AuditLogEntry
        {
            Action = evt.Action,
            UserId = evt.UserId,
            UserEmail = evt.UserEmail,
            ResourceType = evt.ResourceType,
            ResourceId = evt.ResourceId,
            IpAddress = evt.IpAddress,
            Details = evt.Details is not null
                ? JsonSerializer.Serialize(evt.Details)
                : null,
            Timestamp = evt.Timestamp,
            Success = evt.Success,
            FailureReason = evt.FailureReason
        };

        _dbContext.AuditLogs.Add(entry);
        await _dbContext.SaveChangesAsync(ct);

        // También loggear estructuradamente para Application Insights
        _logger.LogInformation(
            "[AUDIT] {Action} by {UserEmail} ({UserId}) on {ResourceType} {ResourceId} " +
            "from {IpAddress} at {Timestamp}. Success: {Success}",
            evt.Action, evt.UserEmail, evt.UserId,
            evt.ResourceType, evt.ResourceId,
            evt.IpAddress, evt.Timestamp, evt.Success);
    }
}
```

## Usar el Audit Logger en el handler

```csharp
// CreateOrderHandler con audit logging

public class CreateOrderHandler : IRequestHandler<CreateOrderCommand, OrderDto>
{
    private readonly IOrderRepository _repository;
    private readonly ProductsClient _productsClient;
    private readonly IPublishEndpoint _publishEndpoint;
    private readonly IAuditLogger _auditLogger;
    private readonly IHttpContextAccessor _httpContextAccessor;

    public async Task<OrderDto> Handle(
        CreateOrderCommand command, CancellationToken ct)
    {
        Order? order = null;
        try
        {
            // ... lógica de negocio ...

            order = Order.Create(command.CustomerId, validatedItems, address);
            await _repository.SaveAsync(order, ct);
            await _publishEndpoint.Publish(new OrderCreated { ... }, ct);

            // Audit log de éxito
            await _auditLogger.LogAsync(new AuditEvent
            {
                Action = "OrderCreated",
                UserId = command.CustomerId.ToString(),
                UserEmail = command.CustomerEmail,
                ResourceType = "Order",
                ResourceId = order.Id.ToString(),
                IpAddress = _httpContextAccessor.HttpContext?
                    .Connection.RemoteIpAddress?.ToString(),
                Details = new()
                {
                    ["total"] = order.Total.Amount,
                    ["currency"] = order.Total.Currency,
                    ["itemCount"] = order.Lines.Count
                }
            }, ct);

            return order.ToDto();
        }
        catch (DomainException ex)
        {
            // Audit log de fallo de negocio
            await _auditLogger.LogAsync(new AuditEvent
            {
                Action = "OrderCreateFailed",
                UserId = command.CustomerId.ToString(),
                UserEmail = command.CustomerEmail,

                FailureReason = ex.Message

    }
```

## AuditBehavior: aplicar audit logging como pipeline de MediatR

En vez de añadir audit logging en cada handler, usar un pipeline behavior:

```csharp
// Application/Behaviors/AuditBehavior.cs

public class AuditBehavior<TRequest, TResponse>
    : IPipelineBehavior<TRequest, TResponse>
    where TRequest : IRequest<TResponse>
{
    private readonly IAuditLogger _auditLogger;
    private readonly IHttpContextAccessor _accessor;

    public async Task<TResponse> Handle(
        TRequest request,
        RequestHandlerDelegate<TResponse> next,
        CancellationToken ct)
    {
        // Solo auditar requests marcados con IAuditable
        if (request is not IAuditable auditable)
            return await next();

        TResponse? response = default;
        try
        {
            response = await next();

            await _auditLogger.LogAsync(new AuditEvent
            {
                Action = typeof(TRequest).Name,
                UserId = auditable.ActorId.ToString(),
                UserEmail = auditable.ActorEmail,
                ResourceType = auditable.ResourceType,
                ResourceId = auditable.ResourceId?.ToString(),
                IpAddress = _accessor.HttpContext?.Connection
                    .RemoteIpAddress?.ToString()
            }, ct);

            return response;
        }
        catch
        {
            await _auditLogger.LogAsync(new AuditEvent
            {
                Action = $"{typeof(TRequest).Name}Failed",
                UserId = auditable.ActorId.ToString(),
                UserEmail = auditable.ActorEmail,
                ResourceType = auditable.ResourceType,
                Success = false
            }, ct);
            throw;
        }
    }
}

// Marker interface
public interface IAuditable
{
    Guid ActorId { get; }
    string ActorEmail { get; }
    string ResourceType { get; }
    Guid? ResourceId { get; }
}
```

## Input Validation como primera línea de defensa

FluentValidation previene ataques de inputs maliciosos antes de que lleguen al dominio:

```csharp
public class CreateOrderCommandValidator
    : AbstractValidator<CreateOrderCommand>
{
    public CreateOrderCommandValidator()
    {
        // Prevenir Buffer Overflow
        RuleFor(x => x.CustomerEmail)
            .MaximumLength(256);

        // Prevenir inyección en direcciones
        RuleFor(x => x.ShippingAddress.Street)
            .MaximumLength(200)
            .Matches(@"^[\w\s\.,\-À-ÿ]+$")
            .WithMessage("Invalid characters in street address");

        // Prevenir cantidades negativas o absurdas
        RuleForEach(x => x.Items).ChildRules(item =>
        {
            item.RuleFor(i => i.Quantity)
                .InclusiveBetween(1, 999);

            // Prevenir precios negativos o inyección de precio=0
            item.RuleFor(i => i.UnitPrice)
                .GreaterThan(0.01m)
                .LessThan(999_999.99m);
        });

        // Límite de ítems por pedido
        RuleFor(x => x.Items.Count)
            .LessThanOrEqualTo(50)
            .WithMessage("Maximum 50 items per order");
    }
}
```

La validación rechaza con 422 antes de que el request llegue al handler — sin coste de BD ni de red.

## SSRF: Server-Side Request Forgery

Un ataque menos conocido pero relevante para microservicios:

**El riesgo:** Si Orders.API construye URLs a partir de input del usuario y las llama, un atacante puede hacer que Orders.API llame a URLs internas que no debería.

```csharp
// ❌ VULNERABLE — URL construida desde input del usuario
public async Task<IActionResult> GetExternalData(string url)
{
    var response = await _httpClient.GetAsync(url);
    // Atacante puede enviar: http://169.254.169.254/metadata (Azure IMDS)
    // O: http://orders-admin-internal/api/admin
}
```

**Para OrderFlow:** No construimos URLs desde inputs del usuario. Las URLs de los servicios vienen de la configuración (Aspire/appsettings). No hay vulnerabilidad de SSRF.

**Protección adicional:** Usar `HttpClient` con una `AllowedHosts` lista blanca si necesitas llamar a URLs dinámicas.

## Protección contra ataques de timing

Los ataques de timing explotan diferencias en el tiempo de respuesta para inferir información:

```csharp
// ❌ VULNERABLE a timing attacks en comparación de tokens
if (userInputToken == storedToken)
    return true;
// Si la comparación char-a-char para en el primer carácter diferente,
// el tiempo de respuesta revela cuántos caracteres son iguales

// ✅ CORRECTO — comparación en tiempo constante
if (CryptographicOperations.FixedTimeEquals(
    Encoding.UTF8.GetBytes(userInputToken),
    Encoding.UTF8.GetBytes(storedToken)))
    return true;
// Siempre tarda el mismo tiempo, sin importar dónde difieren los tokens
```

Para JWT: .NET ya usa comparación segura internamente. Para HMAC o firmas propias, usar `CryptographicOperations.FixedTimeEquals`.

## Request Size Limiting: prevenir ataques de payload grande

```csharp
// Gateway.API/Program.cs

// Limitar el tamaño de los requests para prevenir OOM
builder.WebHost.ConfigureKestrel(options =>
{
    options.Limits.MaxRequestBodySize = 1 * 1024 * 1024; // 1 MB máximo
});

// O por ruta en el gateway
app.Use(async (context, next) =>
{
    // Los pedidos no deberían ser más de 64KB
    if (context.Request.Path.StartsWithSegments("/api/orders"))
    {
        context.Request.EnableBuffering();
        if (context.Request.ContentLength > 64 * 1024) // 64KB
        {
            context.Response.StatusCode = StatusCodes.Status413PayloadTooLarge;
            await context.Response.WriteAsync("Request too large");
            return;
        }
    }

    await next();
});
```

Un request de 100MB a `POST /api/orders` podría consumir toda la memoria del servicio. Limitar el tamaño previene este vector de ataque.

## GDPR aplicado a OrderFlow

El GDPR (Reglamento General de Protección de Datos) afecta a cómo OrderFlow maneja datos personales:

**Datos personales en OrderFlow:**
- `CustomerEmail` en Orders → dato personal
- `ShippingAddress` con nombre → dato personal
- Logs que incluyen email o IP → datos personales
- El `OrderId` por sí solo → NO es dato personal (pseudónimo)

**Derechos del usuario que hay que soportar:**
```
Derecho de acceso: "John pide ver todos sus datos"
→ Endpoint GET /api/users/{id}/data — exporta todo lo de John en JSON

Derecho al olvido: "John pide borrar su cuenta"
→ Anonimizar: sustituir email por hash, dirección por "DELETED"
→ Mantener el pedido para auditoría fiscal (obligación legal)
→ NO borrar el pedido — los registros financieros son obligatorios

Derecho a la portabilidad: "John quiere exportar sus datos"
→ GET /api/users/{id}/export — CSV o JSON con todos sus datos
```

---

## Anonimizar vs borrar: la diferencia legal

En el contexto de GDPR, borrar datos no siempre es posible ni legal:

```csharp
// Anonimizar un pedido cuando el usuario ejerce su derecho al olvido
public async Task AnonymizeCustomerAsync(Guid customerId)
{
    var orders = await _repository
        .GetByCustomerAsync(customerId);

    foreach (var order in orders)
    {
        // Reemplazar datos personales con tokens anónimos
        order.AnonymizeCustomer(
            anonymousEmail: $"deleted-{Guid.NewGuid():N}@anon.local",
            anonymousAddress: new Address(
                "ELIMINADO", "ELIMINADO", "00000", "ES"));
        await _repository.SaveAsync(order);
    }

    // También anonimizar en el audit log
    await _dbContext.AuditLogs
        .Where(a => a.UserId == customerId.ToString())
        .ExecuteUpdateAsync(s =>
            s.SetProperty(a => a.UserEmail, "anon@deleted.local"));
}
```

El `OrderId`, la fecha y el importe se mantienen — son necesarios para contabilidad.

## El Módulo 5 completo: defense in depth

OrderFlow implementa seguridad en profundidad (defense in depth) con múltiples capas:

```
CAPA 0: WAF (Azure Front Door Premium — fuera del código)
  → Bloquea patrones OWASP automáticamente

CAPA 1: Gateway (YARP)
  → JWT validado, Rate limiting, CORS, Headers HTTP

CAPA 2: Autenticación (ASP.NET Core Identity / Azure AD en producción)

  → Inputs sanitizados antes de llegar al dominio

CAPA 5: Persistencia (EF Core + SQL)
  → Parámetros, TDE, Managed Identity

CAPA 6: Secretos (Key Vault)
  → Sin passwords en código ni variables de entorno

CAPA 7: Auditoría
  → Quién hizo qué, cuándo, desde dónde
```

Cada capa asume que las anteriores pueden fallar. Un atacante que supere el WAF se enfrenta al JWT. Si supera el JWT, se enfrenta al ownership check. Si supera el ownership check, los datos están cifrados en reposo.

## Lo que viene en M6.1

El Módulo 6 aborda el patrón de datos más importante en microservicios: **Database per Service**.

En M6.1 consolidamos:
- Por qué cada microservicio tiene su propia base de datos
- Cómo EF Core gestiona múltiples contextos en la misma solution
- Estrategias de migración sin downtime (Expand and Contract)
- Gestión de la consistencia entre bases de datos
- Cómo hacer backups y restauraciones en un sistema distribuido

El patrón Database per Service es lo que hace que los microservicios sean verdaderamente independientes a nivel de datos. Sin él, tenemos microservicios con una base de datos compartida — que es un monolito con más pasos.
