## Qué es .NET Aspire: la definición correcta

Aspire no es un framework de microservicios. Es una **plataforma de orquestación para desarrollo local** que resuelve el problema de arrancar y gestionar múltiples servicios .NET simultáneamente.

**Lo que hace Aspire:**

```
SIN ASPIRE:
  Terminal 1: cd src/Orders.API && dotnet run --urls https://localhost:7153
  Terminal 2: cd src/Products.API && dotnet run --urls https://localhost:7200
  Terminal 3: cd src/Payments.API && dotnet run --urls https://localhost:7400
  Terminal 4: cd src/Notifications.API && dotnet run --urls https://localhost:7300
  Terminal 5: cd src/Gateway.API && dotnet run --urls https://localhost:8080
  5 terminales abiertas, URLs hardcodeadas, sin visibilidad centralizada

CON ASPIRE:
  F5 en el AppHost
  → Todos los servicios arrancan automáticamente
  → URLs asignadas dinámicamente
  → Dashboard con logs, trazas y métricas de todos los servicios
  → Service Discovery: los servicios se conocen por nombre, no por URL
```

## El ciclo F5: el flujo de desarrollo diario con Aspire

El flujo de trabajo con Aspire en Visual Studio es radical en simplicidad:

```
1. Abrir Visual Studio → Set as Startup Project: OrderFlow.AppHost

2. F5 (o Ctrl+F5 sin debug)
   → Aspire compila todos los proyectos
   → Arranca LocalDB (si no estaba corriendo)
   → Verifica que RabbitMQ nativo está activo
   → Arranca los 5 servicios en paralelo
   → Abre el dashboard en el navegador

3. Hacer cambios en Orders.API

4. Hot Reload automático (para cambios en código de negocio)
   O Ctrl+Shift+F5 para restart completo

5. La BD y el estado de RabbitMQ se mantienen entre reinicios
```

Todo desde un solo punto. Sin 5 terminales, sin ports hardcodeados, sin `docker-compose up`.

## La estructura del AppHost de OrderFlow

```csharp
// OrderFlow.AppHost/Program.cs — el orquestador completo

var builder = DistributedApplication.CreateBuilder(args);

// ── RECURSOS EXTERNOS (LocalDB y RabbitMQ nativos) ──
var sqlServer = builder.AddConnectionString("sqlserver");
var messaging = builder.AddConnectionString("messaging");
var keycloak = builder.AddConnectionString("keycloak");

// ── BASES DE DATOS ──
var ordersDb = sqlServer.AddDatabase("OrdersDb");
var productsDb = sqlServer.AddDatabase("ProductsDb");
var paymentsDb = sqlServer.AddDatabase("PaymentsDb");
var notificationsDb = sqlServer.AddDatabase("NotificationsDb");

// ── SERVICIOS (en orden de dependencias) ──
var products = builder.AddProject<Projects.Products_API>("products-api")
    .WithReference(productsDb)
    .WithReference(messaging);

var notifications = builder.AddProject<Projects.Notifications_API>("notifications-api")
    .WithReference(notificationsDb)
    .WithReference(messaging)
    .WithReference(keycloak);

var payments = builder.AddProject<Projects.Payments_API>("payments-api")
    .WithReference(paymentsDb)
    .WithReference(messaging);

var orders = builder.AddProject<Projects.Orders_API>("orders-api")
    .WithReference(ordersDb)
    .WithReference(messaging)
    .WithReference(keycloak)
    .WithReference(products);

var gateway = builder.AddProject<Projects.Gateway_API>("gateway-api")
    .WithReference(orders)
    .WithReference(products)
    .WithReference(payments)
    .WithReference(notifications)
    .WithReference(keycloak);

builder.Build().Run();
```

## AddConnectionString: la clave de "sin Docker"

La diferencia entre usar Aspire con Docker y sin Docker es una sola función:

```csharp
// ❌ Opción con Docker (NO usamos en el curso)
var rabbitmq = builder.AddRabbitMQ("messaging");
// → Aspire levanta un contenedor Docker de RabbitMQ
// → Requiere Docker Desktop instalado

// ✅ Opción sin Docker (nuestra elección)
var messaging = builder.AddConnectionString("messaging");
// → Aspire lee la connection string de appsettings.json
// → RabbitMQ ya está instalado nativamente en Windows
// → Sin Docker, sin contenedores
```

```json
// OrderFlow.AppHost/appsettings.json
{
  "ConnectionStrings": {
    "sqlserver": "Server=(localdb)\\MSSQLLocalDB;Trusted_Connection=true;TrustServerCertificate=true",
    "messaging": "amqp://guest:guest@localhost:5672",
    "keycloak": "http://localhost:8180/realms/techshop"
  }
}
```

## Service Discovery: cómo funcionan los nombres de servicio

Cuando en Orders.API configuramos:
```csharp
builder.Services.AddHttpClient<ProductsClient>(c =>
    c.BaseAddress = new Uri("http://products-api"));
```

¿Cómo sabe `http://products-api` a qué `localhost:XXXX` conectar?

**El mecanismo:**

```
1. AppHost arranca Products.API en un puerto dinámico (ej: 7200)

2. AppHost inyecta en Orders.API estas variables de entorno:
   services__products-api__http__0 = http://localhost:7200
   services__products-api__https__0 = https://localhost:7201

3. El middleware de Service Discovery de Aspire intercepta
   las peticiones a "http://products-api"

4. Reemplaza "products-api" por "localhost:7200" en tiempo real

5. La petición llega a Products.API correctamente
```

En Azure App Services, las mismas variables se configuran con las URLs reales de los servicios desplegados.

## WithReference: el grafo de dependencias

Cada `.WithReference()` hace dos cosas:
1. Inyecta la URL/connection string del recurso como variable de entorno
2. Declara una dependencia — Aspire espera a que el dependiente esté listo antes de arrancar el dependiente

```csharp
var products = builder.AddProject<Projects.Products_API>("products-api")
    .WithReference(productsDb);   // inyecta ConnectionStrings__ProductsDb
                                  // Products arranca cuando ProductsDb está disponible

var orders = builder.AddProject<Projects.Orders_API>("orders-api")
    .WithReference(ordersDb)      // inyecta ConnectionStrings__OrdersDb
    .WithReference(messaging)     // inyecta ConnectionStrings__messaging
    .WithReference(products);     // inyecta services__products-api__http__0
                                  // Orders arranca cuando Products está sano
```

El grafo de dependencias garantiza el orden correcto de arranque.

## withExternalHttpEndpoints: exponer al navegador

Por defecto, Aspire solo expone los servicios entre sí. Para acceder desde el navegador durante el desarrollo:

```csharp
var gateway = builder.AddProject<Projects.Gateway_API>("gateway-api")
    .WithReference(orders)
    .WithReference(products)
    .WithExternalHttpEndpoints(); // ← Expone el gateway al navegador

// O solo HTTPS:
var gateway = builder.AddProject<Projects.Gateway_API>("gateway-api")
    .WithHttpsEndpoint(port: 8080) // Puerto fijo (opcional)
    .WithExternalHttpEndpoints();
```

Con `WithExternalHttpEndpoints()`, el dashboard de Aspire muestra un enlace clickable al servicio. Sin él, el servicio solo es accesible desde otros servicios de Aspire (Service Discovery interno).

## Ports y endpoints: control fino

Por defecto, Aspire asigna puertos dinámicamente. Para puertos fijos en desarrollo:

```csharp
// AppHost — puertos fijos (opcional, facilita el .http file)
var gateway = builder.AddProject<Projects.Gateway_API>("gateway-api")
    .WithHttpEndpoint(port: 8080, name: "http")
    .WithHttpsEndpoint(port: 8443, name: "https")
    .WithExternalHttpEndpoints();

var orders = builder.AddProject<Projects.Orders_API>("orders-api")
    .WithHttpEndpoint(port: 7153, name: "http")
    .WithHttpsEndpoint(port: 7154, name: "https");
```

Con puertos fijos, el fichero `.http` de desarrollo siempre funciona:
```http
@gateway = https://localhost:8443
POST {{gateway}}/api/orders
```

Sin puertos fijos, cada arranque puede usar puertos diferentes — hay que actualizar el fichero `.http`.

## ServiceDefaults: el paquete de configuración compartida

`OrderFlow.ServiceDefaults` es el proyecto que configura todos los servicios con las mismas opciones base:

```csharp
// OrderFlow.ServiceDefaults/Extensions.cs

public static class Extensions
{
    public static IHostApplicationBuilder AddServiceDefaults(
        this IHostApplicationBuilder builder)
    {
        // 1. OpenTelemetry — logs, trazas y métricas hacia el dashboard
        builder.AddOpenTelemetryDefaults();

        // 2. Health checks base (/health y /alive)
        builder.AddDefaultHealthChecks();

        // 3. Service Discovery — resolver "http://products-api" a URL real
        builder.Services.AddServiceDiscovery();

        // 4. HttpClient con Service Discovery activado
        builder.Services.ConfigureHttpClientDefaults(http =>
            http.AddServiceDiscovery());

        return builder;
    }
}
```

Cada servicio tiene una línea en su `Program.cs`:
```csharp
builder.AddServiceDefaults();
```

## Personalizar ServiceDefaults para OrderFlow

El ServiceDefaults generado por `dotnet new aspire-servicedefaults` es un punto de partida. Lo personalizamos para OrderFlow:

```csharp
// OrderFlow.ServiceDefaults/Extensions.cs — personalizado

public static IHostApplicationBuilder AddServiceDefaults(
    this IHostApplicationBuilder builder)
{
    // OTel configurado para enviar al dashboard de Aspire
    builder.AddOpenTelemetryDefaults();

    // Health checks incluyendo el propio servicio y sus dependencias
    builder.Services.AddHealthChecks()
        .AddCheck("self", () => HealthCheckResult.Healthy(),
            tags: ["live", "ready"]);

    builder.Services.AddServiceDiscovery();

    builder.Services.ConfigureHttpClientDefaults(http =>
    {
        http.AddServiceDiscovery();
        // Añadir CorrelationId a TODOS los HttpClients por defecto
        http.AddHttpMessageHandler<CorrelationIdDelegatingHandler>();
    });

    return builder;
}

public static WebApplication MapDefaultEndpoints(
    this WebApplication app)
{
    // /alive: el proceso está vivo (para el load balancer)
    app.MapHealthChecks("/alive", new HealthCheckOptions
    {
        Predicate = r => r.Tags.Contains("live")
    });

    // /health: el servicio y sus dependencias están sanos
    app.MapHealthChecks("/health", new HealthCheckOptions
    {
        Predicate = r => r.Tags.Contains("ready")
    });

    return app;
}
```

## El dashboard de Aspire: guía completa

El dashboard se abre automáticamente al arrancar el AppHost (`https://localhost:15888`). Tiene cinco pestañas principales:

**Resources:** Estado de cada servicio y recurso
- Cada servicio muestra: Running / Starting / Unhealthy / Stopped
- URLs accesibles: HTTP y HTTPS de cada servicio
- Variables de entorno inyectadas (valores ocultos para secretos)
- Logs de arranque de cada servicio

**Console:** Logs en streaming de un servicio específico
- Equivalente a la terminal del servicio
- Con colores para diferentes niveles (Info, Warning, Error)

**Structured:** Logs estructurados de todos los servicios
- Filtrar por servicio, nivel, timestamp
- Búsqueda por correlationId para trazar un pedido completo
- Propiedades estructuradas de Serilog visibles como columnas

**Traces:** Trazas distribuidas OpenTelemetry
- Cada request HTTP es un trace con sus spans
- Click en un trace → timeline de todos los spans
- Muestra el flujo completo cruzando servicios

**Metrics:** Métricas en tiempo real
- Histogramas de duración HTTP
- Métricas de EF Core (queries ejecutadas, duración)
- Métricas de runtime (.NET GC, CPU, memoria)

## La pestaña Traces: trazas distribuidas en profundidad

El dashboard de Aspire muestra las trazas distribuidas generadas por OpenTelemetry. Para un POST /api/orders:

```
[POST] /api/orders → gateway-api (8ms)
  └── [POST] /api/orders → orders-api (145ms total)
        ├── MediatR: LoggingBehavior (2ms)
        ├── MediatR: ValidationBehavior (1ms)
        ├── MediatR: TransactionBehavior (1ms)
        └── MediatR: CreateOrderHandler (138ms)
              ├── [GET] /api/products/{id} → products-api (42ms)
              │     └── SQL: SELECT Products WHERE Id=... (12ms)
              ├── SQL: INSERT orders.Orders (8ms)
              ├── SQL: INSERT orders.OrderLines (5ms)
              └── SQL: INSERT orders.OutboxMessage (3ms)

[async] OutboxProcessor: Publish OrderCreated → RabbitMQ (3ms)
[async] OrderSummaryProjector → notifications-api (12ms)
```

Cada span tiene atributos: SQL exacto, URL destino, duración, status. Si algo falla: el span en rojo con el stack trace completo.

## Aspire y la observabilidad: el stack completo

Con Aspire + ServiceDefaults + Serilog, el stack de observabilidad en desarrollo es:

```
LOGS:
  Serilog → Console sink
           → OTLP sink → Aspire Dashboard (Structured tab)
  En desarrollo: Console + Aspire Dashboard
  En producción (M7.4): Application Insights

TRAZAS:
  OpenTelemetry → OTLP exporter → Aspire Dashboard (Traces tab)
  ASP.NET Core, HttpClient, EF Core, MassTransit: instrumentados automáticamente
  En producción (M7.4): Application Insights

MÉTRICAS:
  OpenTelemetry → OTLP exporter → Aspire Dashboard (Metrics tab)
  Runtime metrics, HTTP metrics, EF Core metrics: automáticas
  Custom metrics (OrdersMetrics): con Meter API
  En producción (M7.4): Azure Monitor
```

El mismo código de instrumentación funciona en desarrollo (Aspire Dashboard) y en producción (Application Insights). Solo cambia el exporter.

## Observability-first: diseñar para ser observable

Aspire facilita el diseño "observability-first" — cada servicio está diseñado para ser monitoreado desde el primer día:

```csharp
// Cada operación importante tiene spans de OTel
public async Task<OrderDto> Handle(CreateOrderCommand command, CancellationToken ct)
{
    using var activity = _activitySource.StartActivity("CreateOrder");
    activity?.SetTag("order.customerId", command.CustomerId);
    activity?.SetTag("order.itemCount", command.Items.Count);

    try
    {
        var order = Order.Create(...);
        await _repository.SaveAsync(order, ct);

        activity?.SetTag("order.id", order.Id);
        activity?.SetTag("order.total", order.Total.Amount);
        activity?.SetStatus(ActivityStatusCode.Ok);

        return order.ToDto();
    }
    catch (Exception ex)
    {
        activity?.SetStatus(ActivityStatusCode.Error, ex.Message);
        throw;
    }
}
```

Con este patrón, cada `CreateOrder` es un span visible en el dashboard de Aspire con sus atributos de negocio.

## Docker y Kubernetes: los conceptos teóricos

Aunque el curso no usa Docker ni Kubernetes, el alumno debe conocerlos como conceptos — son el contexto del ecosistema de microservicios.

**Docker** es una plataforma de contenedorización. Un contenedor empaqueta el código, las dependencias y el entorno de ejecución en una unidad portátil:

```dockerfile
# Concepto de Dockerfile — NO lo usamos en este curso
FROM mcr.microsoft.com/dotnet/aspnet:10.0 AS base
WORKDIR /app
EXPOSE 8080

FROM mcr.microsoft.com/dotnet/sdk:10.0 AS build
COPY . .
RUN dotnet publish Orders.API -c Release -o /app/publish

FROM base AS final
COPY --from=build /app/publish .
ENTRYPOINT ["dotnet", "Orders.API.dll"]
```

**Por qué no usamos Docker en este curso:** Azure App Services gestiona el runtime .NET directamente sin necesitar imágenes Docker. Añade complejidad sin aportar beneficio para el caso de TechShop.

---

## Kubernetes: el orquestador de contenedores (concepto)

**Kubernetes** (K8s) orquesta múltiples contenedores Docker a escala:

```
Docker:    "Aquí hay un contenedor con Orders.API"
Kubernetes: "Necesito 3 réplicas de Orders.API,
             si una falla arráncala de nuevo,
             si hay mucho tráfico escala a 10 réplicas,
             gestiona el networking entre ellos"
```

**Cuándo tiene sentido Kubernetes:**
- Cientos de microservicios con equipos independientes
- Necesidad de auto-scaling complejo
- Despliegue en múltiples regiones simultáneamente
- Operación 24/7 con SLAs de 99.99%

**Por qué NO usamos Kubernetes en este curso:**
- TechShop tiene 4-5 microservicios → Azure App Services es suficiente
- K8s añade complejidad operativa enorme (cluster, networking, ingress, RBAC...)
- El tiempo de aprendizaje de K8s merece un curso dedicado propio
- Azure App Services + Aspire + GitHub Actions resuelve el 90% de los casos

## Health Checks avanzados en Aspire

El dashboard de Aspire muestra el estado de salud de cada servicio. Configurar health checks completos:

```csharp
// Orders.API/Program.cs — health checks completos

builder.Services.AddHealthChecks()
    // El propio proceso está vivo
    .AddCheck("self", () => HealthCheckResult.Healthy(), tags: ["live"])

    // La BD está disponible
    .AddSqlServer(
        connectionString: builder.Configuration.GetConnectionString("OrdersDb")!,
        name: "orders-db",
        failureStatus: HealthStatus.Unhealthy,
        tags: ["ready", "database"])

    // RabbitMQ está disponible
    .AddRabbitMQ(
        rabbitConnectionString: builder.Configuration.GetConnectionString("messaging")!,
        name: "rabbitmq",
        failureStatus: HealthStatus.Degraded,
        tags: ["ready", "messaging"])

    // Keycloak está disponible
    .AddUrlGroup(
        new Uri($"{builder.Configuration["Keycloak:Authority"]}/health/ready"),
        name: "keycloak",
        failureStatus: HealthStatus.Degraded,
        tags: ["ready", "auth"])

    // Migrations están aplicadas
    .AddDbContextCheck<OrderDbContext>(
        name: "db-migrations",
        tags: ["ready"]);
```

## Diagnóstico: solucionar problemas comunes con Aspire

**Error: "Service products-api failed to start"**
```
En el dashboard: Resources → products-api → ver logs de startup
Causa frecuente:
  - Connection string incorrecta (LocalDB no arrancado)
  - Puerto ocupado por otro proceso
  - Error en Program.cs antes del app.Run()
Solución: sqllocaldb start MSSQLLocalDB
```

**Error: "Connection refused http://products-api"**
```
Causa: Service Discovery no está configurado
Verificar en ServiceDefaults:
  builder.Services.AddServiceDiscovery()
  builder.Services.ConfigureHttpClientDefaults(http =>
      http.AddServiceDiscovery());
```

**El dashboard no muestra trazas**
```
Causa: OpenTelemetry no exporta al endpoint correcto
Verificar en ServiceDefaults:
  builder.AddOpenTelemetryDefaults()
Verificar que OTEL_EXPORTER_OTLP_ENDPOINT está configurado
(Aspire lo inyecta automáticamente en development)
```

## Monitorear el AppHost: el proceso padre

El AppHost es el proceso padre de todos los servicios. Si el AppHost muere, todos los servicios mueren también. En desarrollo esto es el comportamiento esperado:

```
Ctrl+C en el AppHost → todos los servicios se detienen gracefully
F5 de nuevo → todos los servicios arrancan de nuevo

Si un servicio individual falla (crash):
  Aspire detecta la salida del proceso
  Marca el servicio como "Exited" en el dashboard
  Muestra los últimos logs antes del crash
  NO reinicia automáticamente (en desarrollo es mejor ver el error)
```

En producción con Azure App Services, cada servicio tiene su propio proceso independiente. Si uno cae, Azure lo reinicia automáticamente sin afectar a los demás.

## Hot Reload con Aspire: límites y posibilidades

El Hot Reload funciona en Aspire para cambios en el código de negocio:

```
HOT RELOAD (sin reiniciar el servicio):
✅ Cambios en el cuerpo de un método
✅ Cambios en el valor de una constante
✅ Añadir código dentro de un método existente

REQUIERE RESTART (F5 de nuevo):
❌ Cambios en la firma de un método
❌ Añadir nuevos servicios en Program.cs
❌ Cambios en la configuración de DI
❌ Nuevas clases o interfaces
❌ Cambios en el AppHost (afecta a todos los servicios)
```

Para OrderFlow, el Hot Reload es útil para iterar rápido en la lógica de un handler. Para cambios estructurales, usar el restart completo del AppHost.

public class AspireIntegrationTests

## Aspire y la seguridad en desarrollo

El dashboard de Aspire en desarrollo no requiere autenticación por defecto. Para equipos donde esto es un problema (redes compartidas):

```csharp
// AppHost — habilitar autenticación en el dashboard
var builder = DistributedApplication.CreateBuilder(args);

// Configurar autenticación para el dashboard (solo desarrollo compartido)
builder.Services.AddAuthentication()
    .AddBearerToken(IdentityConstants.BearerScheme);

// O deshabilitar el dashboard completamente en entornos de CI
if (args.Contains("--no-dashboard"))
{
    builder.Services.Configure<DashboardOptions>(o =>
        o.DashboardEnabled = false);
}
```

En el pipeline de CI/CD, el AppHost no arranca el dashboard — solo se usa `dotnet test` directamente.

## Preguntas frecuentes sobre Aspire

**¿Aspire funciona en macOS y Linux?**
Sí. Aspire es multiplataforma. El AppHost corre en macOS y Linux. El dashboard también. La diferencia: LocalDB no existe en macOS — hay que usar Azure SQL o SQL Server Express con Wine. RabbitMQ se instala nativamente con `brew install rabbitmq` o `apt install rabbitmq-server`.

**¿Puedo usar Aspire con servicios que no son .NET?**
Sí. Puedes añadir ejecutables arbitrarios al AppHost con `AddExecutable()`. Por ejemplo, un servicio Python o Node.js puede orquestarse junto a los servicios .NET.

**¿Aspire reemplaza a Kubernetes en producción?**
No. Aspire es solo para desarrollo local. En producción usa Azure App Services (el curso), AKS, o cualquier plataforma de despliegue. No son competidores.

**¿El dashboard de Aspire está disponible en producción?**
No está pensado para producción. En producción usamos Application Insights (M7.4). El dashboard de Aspire es exclusivamente para el ciclo de desarrollo local.

## Resumen de M7.1

Aspire es el orquestador de desarrollo local de OrderFlow. Sin él, arrancar 5 servicios con sus dependencias sería manual y propenso a errores.

**Lo que hace Aspire:**
- Orquesta el arranque en el orden correcto (según las dependencias)
- Service Discovery: los servicios se conocen por nombre, no por URL
- Inyecta connection strings y URLs como variables de entorno
- Proporciona un dashboard con logs, trazas y métricas de todos los servicios

**Sin Docker:** `AddConnectionString()` en vez de `AddRabbitMQ()` o `AddSqlServer()`. RabbitMQ y LocalDB corren nativamente en Windows.

**ServiceDefaults:** el proyecto compartido que configura OpenTelemetry, health checks y Service Discovery en todos los servicios con una sola línea: `builder.AddServiceDefaults()`.

**Docker y Kubernetes:** existen, son importantes en el ecosistema, pero Azure App Services + Aspire + GitHub Actions resuelve los requisitos de TechShop sin esa complejidad adicional.

En M7.2 llevamos todo esto a Azure.
