## Dónde estamos

OrderFlow funciona perfectamente en local con Aspire. Es el momento de llevarlo a Azure y hacer que TechShop pueda usarlo en producción.

En M7.2 desplegamos la arquitectura completa en Azure:
- **5 Azure App Services** — uno por servicio (orders, products, payments, notifications, gateway)
- **Azure SQL Database en Elastic Pool** — las 4 BDs en un pool compartido
- **Azure Service Bus** — RabbitMQ en producción (1 línea de cambio en MassTransit)
- **Azure Key Vault** — ya configurado en M5.2
- **Managed Identity** — sin passwords entre servicios Azure

Todo con Azure CLI. Al terminar, OrderFlow responde en `https://orderflow-gateway.azurewebsites.net`.

## La arquitectura de Azure de OrderFlow

```
Internet
    │ HTTPS
    ▼
orderflow-gateway.azurewebsites.net (Gateway.API)
    │
    ├── orderflow-orders.azurewebsites.net (Orders.API)
    ├── orderflow-products.azurewebsites.net (Products.API)
    ├── orderflow-payments.azurewebsites.net (Payments.API)
    └── orderflow-notifications.azurewebsites.net (Notifications.API)

Datos:
    ├── orderflow-sql.database.windows.net (Azure SQL — Elastic Pool)
    │   ├── OrdersDb
    │   ├── ProductsDb
    │   ├── PaymentsDb
    │   └── NotificationsDb
    ├── orderflow.servicebus.windows.net (Azure Service Bus)
    └── orderflow-kv.vault.azure.net (Azure Key Vault)
```

Sin Kubernetes, sin contenedores. App Services gestiona el runtime directamente.

## Crear el servidor SQL y el Elastic Pool

```bash
# Crear el servidor SQL
az sql server create \
    --resource-group $RG \
    --name $PREFIX-sql \
    --location $LOCATION \
    --admin-user sqladmin \
    --admin-password "TechShop@2026!"

# Crear el Elastic Pool (las 4 BDs comparten recursos)
az sql elastic-pool create \
    --resource-group $RG \
    --server $PREFIX-sql \
    --name $PREFIX-pool \
    --edition GeneralPurpose \
    --capacity 4 \
    --max-size 32GB

# Crear las 4 bases de datos en el pool
for DB in OrdersDb ProductsDb PaymentsDb NotificationsDb; do
    az sql db create \
        --resource-group $RG \
        --server $PREFIX-sql \
        --name $DB \
        --elastic-pool $PREFIX-pool
    echo "✅ $DB creada"
done
```

```bash
    --vault-name $PREFIX-kv \
```

## Cambiar MassTransit de RabbitMQ a Azure Service Bus

Esta es la magia de MassTransit: **una línea de cambio**. El código de la Saga, los consumers y los publishers no cambian.

```csharp
// Orders.API/Program.cs — configuración según el entorno

builder.Services.AddMassTransit(x =>
{
    x.AddSagaStateMachine<OrderSaga, OrderSagaState>()
        .EntityFrameworkRepository(r => { /* sin cambios */ });

    x.AddConsumers(typeof(Program).Assembly);
    x.AddEntityFrameworkOutbox<OrderDbContext>(o =>
    {
        o.UseSqlServer();
        o.UseBusOutbox();
    });

    if (builder.Environment.IsProduction())
    {
        // PRODUCCIÓN: Azure Service Bus con Managed Identity
        x.UsingAzureServiceBus((context, cfg) =>
        {
            cfg.Host(new Uri(
                $"sb://{builder.Configuration["ServiceBus:Namespace"]}.servicebus.windows.net"),
                host => host.TokenCredential = new DefaultAzureCredential());

            cfg.UseMessageRetry(r => r.Exponential(5,
                TimeSpan.FromSeconds(1), TimeSpan.FromSeconds(30),
                TimeSpan.FromSeconds(2)));

            cfg.ConfigureEndpoints(context);
        });
    }
    else
    {
        // DESARROLLO: RabbitMQ local
        x.UsingRabbitMq((context, cfg) =>
        {
            cfg.Host(builder.Configuration.GetConnectionString("messaging")!);
            cfg.UseMessageRetry(r => r.Exponential(5,
                TimeSpan.FromSeconds(1), TimeSpan.FromSeconds(30),
                TimeSpan.FromSeconds(2)));
            cfg.ConfigureEndpoints(context);
        });
    }
});
```

## Migrar ASP.NET Core Identity a Azure AD en producción

En desarrollo local, Gateway.API usa **ASP.NET Core Identity** para emitir y validar JWT. En producción, lo reemplazamos por **Azure Active Directory**. El código de validación de Orders.API, Products.API y el resto de servicios **no cambia** — solo cambia la configuración.

**El cambio en Gateway.API:**

```csharp
// appsettings.Development.json (Identity local)
{
  "Jwt": {
    "Key": "@Microsoft.KeyVault(VaultName=orderflow-kv;SecretName=Jwt--Key)",
    "Issuer": "https://gateway.local",
    "Audience": "orderflow-api",
    "ExpiryMinutes": 60
  }
}

// appsettings.Production.json (Azure AD)
{
  "AzureAd": {
    "TenantId": "@Microsoft.KeyVault(VaultName=orderflow-kv;SecretName=AzureAd--TenantId)",
    "ClientId": "@Microsoft.KeyVault(VaultName=orderflow-kv;SecretName=AzureAd--ClientId)"
  }
}
```

```csharp
// Program.cs del Gateway — condicional por entorno
if (builder.Environment.IsDevelopment())
{
    // Identity local — desarrollo
    builder.Services.AddAuthentication(JwtBearerDefaults.AuthenticationScheme)
        .AddJwtBearer(options =>
        {
            options.TokenValidationParameters = new TokenValidationParameters
            {
                ValidIssuer = builder.Configuration["Jwt:Issuer"],
                ValidAudience = builder.Configuration["Jwt:Audience"],
                IssuerSigningKey = new SymmetricSecurityKey(
                    Encoding.UTF8.GetBytes(builder.Configuration["Jwt:Key"]!))
            };
        });
}
else
{
    // Azure AD — producción (una sola línea)
    builder.Services.AddMicrosoftIdentityWebApi(builder.Configuration);
}
```

**Configurar la app en Azure AD:**

```bash
# Registrar Gateway.API en Azure AD
az ad app create     --display-name "OrderFlow Gateway"     --sign-in-audience AzureADMyOrg

# Guardar el ClientId y TenantId en Key Vault
az keyvault secret set --vault-name orderflow-kv     --name "AzureAd--TenantId" --value "<your-tenant-id>"
az keyvault secret set --vault-name orderflow-kv     --name "AzureAd--ClientId" --value "<your-client-id>"
```

**Los servicios internos no cambian.** Orders.API, Products.API, etc. solo validan el JWT — no les importa si lo emite Identity local o Azure AD. La validación funciona igual en los dos entornos.

## Deployment Slots: Blue-Green en Azure

Los deployment slots permiten un deploy sin downtime:

```bash
# Crear el slot de staging en cada App Service
for SERVICE in orders products payments notifications gateway; do
    az webapp deployment slot create \
        --resource-group $RG \
        --name $PREFIX-$SERVICE \
        --slot staging \
        --configuration-source $PREFIX-$SERVICE

    echo "✅ Slot staging creado en $PREFIX-$SERVICE"
done
```

**Flujo de deploy con slots:**

```bash
# 1. Desplegar al slot staging (no afecta a producción)
az webapp deploy \
    --resource-group $RG \
    --name $PREFIX-orders \
    --slot staging \
    --src-path ./orders-deploy.zip \
    --type zip

# 2. Verificar que staging funciona
curl https://$PREFIX-orders-staging.azurewebsites.net/health

# 3. Swap a producción (instantáneo, sin downtime)
az webapp deployment slot swap \
    --resource-group $RG \
    --name $PREFIX-orders \
    --slot staging \
    --target-slot production

echo "✅ Deploy completado sin downtime"
```

## Auto-scaling: escalar automáticamente

```bash
# Configurar auto-scaling para Orders.API
az monitor autoscale create \
    --resource-group $RG \
    --resource $PREFIX-plan \
    --resource-type Microsoft.Web/serverFarms \
    --name $PREFIX-autoscale \
    --min-count 1 \

    --resource-group $RG \
    --autoscale-name $PREFIX-autoscale \
    --condition "CpuPercentage > 70 avg 5m" \

    --resource-group $RG \
```

## Variables de entorno vs Key Vault: cuándo usar cada uno

```bash
# En appsettings.json del App Service — visible en el portal:

Sql__Server = orderflow-sql.database.windows.net  ← OK (no es secreto)

# En Key Vault — accesible por Managed Identity:
AzureAd--ClientSecret = ****                ← SECRETO (si se usa client credentials)
AppInsights--ConnectionString = InstrumentationKey=...  ← SECRETO

# Referencia a Key Vault desde App Service:
APPLICATIONINSIGHTS_CONNECTION_STRING = @Microsoft.KeyVault(
    VaultName=orderflow-kv;
    SecretName=AppInsights--ConnectionString)
# Azure App Service resuelve la referencia automáticamente
```

## Comparativa: App Services vs Kubernetes para TechShop

Por qué Azure App Services es la decisión correcta para TechShop en esta fase:

| Criterio | Azure App Services | Kubernetes (AKS) |
|---|---|---|
| Complejidad operativa | Baja | Muy alta |
| Tiempo de setup | ~2 horas | ~2 días |
| Auto-scaling | ✅ Automático | ✅ Automático pero más complejo |
| Managed Identity | ✅ Nativo | Requiere Workload Identity |
| Blue-Green | ✅ Slots nativos | Requiere configuración manual |
| Coste base | ~200€/mes | ~300€/mes (cluster + nodos) |
| Certificados SSL | ✅ Gestionados | Requiere cert-manager |
| Idóneo para | 1-20 servicios | 20+ servicios, multi-equipo |

Para TechShop con 5 servicios y un equipo pequeño: App Services. Si TechShop crece a 50+ servicios con 10 equipos: revisar Kubernetes.

## Troubleshooting: los problemas más comunes en el primer deploy

**Error: "Login failed for user"**
```bash
# El App Service no tiene permisos en Azure SQL
# Verificar que el usuario fue creado en la BD:
az sql db execute-sql --resource-group $RG --server $PREFIX-sql \
    --database OrdersDb \
    --sql "SELECT name FROM sys.database_principals WHERE name = '$PREFIX-orders'"
# Si vacío → ejecutar de nuevo el CREATE USER del script de setup
```

**Error: "Service Bus namespace not found"**
```bash
# La variable de entorno ServiceBus__Namespace está mal configurada
az webapp config appsettings list \
    --resource-group $RG \
    --name $PREFIX-orders \
    --query "[?name=='ServiceBus__Namespace']" -o table
```

**Error: "Could not load secrets from Key Vault"**
```bash
# El App Service no tiene permisos de lectura en Key Vault
az keyvault show \
    --name $PREFIX-kv \
    --query properties.accessPolicies[].objectId -o table
# Verificar que el principalId del App Service aparece en la lista
```
