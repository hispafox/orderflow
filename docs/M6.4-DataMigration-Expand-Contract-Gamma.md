Duración estimada: 60 minutos
Curso de 25 horas · Formación técnica

---

## El reto que resolvemos en M6.4

OrderFlow lleva 6 meses en producción. TechShop tiene 150.000 pedidos en `orders.Orders`. El equipo de Orders necesita:

1. Añadir una columna `DiscountAmount` a la tabla de pedidos
2. Renombrar `CustomerId` a `BuyerId` (decisión de nomenclatura del dominio)
3. Dividir `ShippingAddress` en dos tablas separadas

En un monolito, estas operaciones significan downtime: parar el sistema, aplicar la migration, desplegar el código nuevo, arrancar. Con 150.000 clientes activos, no puedes permitirte 30 minutos de downtime.

En microservicios con **Expand-Contract**, estas tres operaciones se hacen **sin detener el servicio ni un segundo**.

---

## Por qué las migrations de EF Core no son suficientes

EF Core genera la migration correcta. El problema no es la migration — es el momento de aplicarla:

```
FLUJO INCORRECTO (con downtime):
1. Hacer COMMIT del código nuevo (espera en rama)
2. Parar el servicio (órdenes en error 503)
3. Aplicar migration: ALTER TABLE Orders ADD DiscountAmount...
4. Desplegar el código nuevo
5. Arrancar el servicio (pedidos perdidos, clientes enfadados)

FLUJO CORRECTO (Expand-Contract, sin downtime):
1. Aplicar migration ANTES que el código (columna nullable)
2. Desplegar código que escribe en AMBAS versiones
3. Migrar datos históricos sin parar el servicio
4. Desplegar código que solo usa la nueva versión
5. Eliminar la columna antigua (o la referencia vieja)
```

La migration de datos y el deploy del código son operaciones INDEPENDIENTES.

---

## El patrón Expand-Contract: las tres fases

```
FASE 1 — EXPAND (el schema crece):
  La BD acepta AMBAS versiones del schema
  El código nuevo puede leer la nueva versión
  El código viejo sigue funcionando (aún no hay código nuevo)

FASE 2 — MIGRATE (los datos evolucionan):
  Copiar/transformar datos de la estructura vieja a la nueva
  Sin downtime — el servicio sigue corriendo
  Proceso incremental: lotes de N registros

FASE 3 — CONTRACT (el schema se contrae):
  Eliminar la estructura vieja del schema
  El código ya solo usa la nueva versión
  La BD queda limpia
```

Entre cada fase: un deploy independiente. La velocidad de avance depende del equipo.

---

## Los 5 principios del Expand-Contract

Resumiendo el patrón en cinco principios que el equipo de TechShop debe interiorizar:

**1. Additive First:**
Toda migration comienza añadiendo, nunca eliminando. El sistema siempre puede volver al estado anterior.

**2. Code and Schema Independently:**
La migration de la BD y el deploy del código son operaciones separadas. La BD va siempre por delante.

**3. Backfill Incrementally:**
Los datos históricos se migran en lotes pequeños. Sin scripts que bloqueen la tabla durante horas.

**4. Verify Before Contracting:**
La columna/tabla vieja se elimina solo después de verificar que el sistema funciona correctamente durante al menos una release completa.

**5. Always Have a Rollback:**
Cualquier cambio debe poder revertirse. Si el rollback requiere restaurar un backup, el cambio es demasiado arriesgado — dividirlo en pasos más pequeños.

Estos cinco principios son la diferencia entre "migramos y rezamos" y "migramos con confianza".

## Caso 1: Añadir una columna (la más simple)

**Objetivo:** Añadir `DiscountAmount` a `orders.Orders`.

**FASE 1 — Añadir la columna nullable:**
```csharp
// Migration: AddDiscountAmountColumn
public partial class AddDiscountAmountColumn : Migration
{
    protected override void Up(MigrationBuilder m)
    {
        m.AddColumn<decimal>(
            name: "DiscountAmount",
            schema: "orders",
            table: "Orders",
            type: "decimal(18,2)",
            nullable: true,    // ← NULLABLE — el código viejo no falla
            defaultValue: null);
    }

    protected override void Down(MigrationBuilder m)
    {
        m.DropColumn("DiscountAmount", "orders", "Orders");
    }
}
```

Con esta migration aplicada, el servicio que no conoce `DiscountAmount` sigue funcionando. Los registros nuevos crean la columna con NULL.

## Caso 1: Hacer NOT NULL tras migrar datos

**FASE 2 — Backfill de datos históricos (script separado):**
```sql
-- Script que se ejecuta una vez, en lotes, sin parar el servicio
DECLARE @BatchSize INT = 1000;
DECLARE @Offset INT = 0;

WHILE 1 = 1
BEGIN
    UPDATE TOP(@BatchSize) orders.Orders
    SET DiscountAmount = 0.00
    WHERE DiscountAmount IS NULL;

    IF @@ROWCOUNT = 0 BREAK;

    WAITFOR DELAY '00:00:01'; -- Pausa entre lotes para no saturar
END
```

**FASE 3 — Hacer la columna NOT NULL (nueva migration):**
```csharp
public partial class MakeDiscountAmountNotNull : Migration
{
    protected override void Up(MigrationBuilder m)
    {
        m.AlterColumn<decimal>(
            name: "DiscountAmount",
            schema: "orders",
            table: "Orders",
            nullable: false,      // ← Ahora obligatoria
            defaultValue: 0.00m); // ← Default para futuros inserts
    }
}
```

## Caso 2: Renombrar una columna sin downtime

**Objetivo:** `CustomerId` → `BuyerId` en `orders.Orders`.

Este es el caso más común y el más peligroso si se hace mal.

**FASE 1 — EXPAND: añadir la nueva columna:**
```csharp
public partial class AddBuyerIdColumn : Migration
{
    protected override void Up(MigrationBuilder m)
    {
        // Añadir BuyerId sin eliminar CustomerId
        m.AddColumn<Guid>(
            name: "BuyerId",
            schema: "orders",
            table: "Orders",
            nullable: true); // Nullable hasta que todos los datos estén migrados

        // Copiar datos existentes en la misma migration
        m.Sql("UPDATE orders.Orders SET BuyerId = CustomerId");

        // Hacer NOT NULL después de copiar
        m.AlterColumn<Guid>(
            name: "BuyerId",
            schema: "orders",
            table: "Orders",
            nullable: false);

        // NO eliminar CustomerId todavía — el código viejo aún la usa
    }
}
```

## Caso 2: El código en transición

**DEPLOY 1 — Código que escribe en AMBAS columnas:**

```csharp
// Order.cs — durante la transición
public class Order
{
    public Guid CustomerId { get; private set; }  // Columna vieja
    public Guid BuyerId { get; private set; }      // Columna nueva

    public static Order Create(Guid buyerId, ...)
    {
        return new Order
        {
            CustomerId = buyerId,  // Escribir en ambas (compatibilidad)
            BuyerId = buyerId,     // La nueva
        };
    }
}
```

**DEPLOY 2 — Código que solo usa BuyerId:**
```csharp
// Order.cs — después de verificar que todo funciona
public class Order
{
    // public Guid CustomerId eliminado del código
    public Guid BuyerId { get; private set; }
}
```

**DEPLOY 3 (después del Deploy 2) — Migration que elimina CustomerId:**
```csharp
m.DropColumn("CustomerId", "orders", "Orders");
```

## Caso 3: Dividir una tabla en dos

**Objetivo:** Extraer `ShippingAddress` de `orders.Orders` a una tabla `orders.Addresses`.

**FASE 1 — Crear la nueva tabla:**
```csharp
public partial class CreateAddressesTable : Migration
{
    protected override void Up(MigrationBuilder m)
    {
        m.CreateTable("Addresses", "orders", table => new
        {
            Id = table.Column<Guid>(nullable: false),
            OrderId = table.Column<Guid>(nullable: false),
            Street = table.Column<string>(maxLength: 200, nullable: false),
            City = table.Column<string>(maxLength: 100, nullable: false),
            ZipCode = table.Column<string>(maxLength: 20, nullable: false),
            Country = table.Column<string>(maxLength: 2, nullable: false)
        });

        // Copiar datos existentes a la nueva tabla
        m.Sql("""
            INSERT INTO orders.Addresses (Id, OrderId, Street, City, ZipCode, Country)
            SELECT NEWID(), Id, Street, City, ZipCode, Country
            FROM orders.Orders
            WHERE Street IS NOT NULL
            """);

        // NO eliminar las columnas de Orders todavía
    }
}
```

## Migration Bundles en CI/CD

Un **Migration Bundle** es un ejecutable autónomo que aplica las migrations. Se incluye en el pipeline de GitHub Actions:

```bash
# Generar el bundle durante el build
dotnet ef migrations bundle \
    --project src/Orders.API \
    --context OrderDbContext \
    --output ./artifacts/efbundle-orders \
    --self-contained \
    --runtime linux-x64  # O windows-x64 según el runner
```

```yaml
# .github/workflows/deploy-orders.yml

jobs:
  migrate-and-deploy:
    runs-on: windows-latest
    steps:
      - uses: actions/checkout@v4

      - name: Setup .NET 10
        uses: actions/setup-dotnet@v4
        with: { dotnet-version: '10.0.x' }

      - name: Build migration bundle
        run: |
          dotnet ef migrations bundle \
            --project src/Orders.API \
            --context OrderDbContext \
            --output ./efbundle-orders.exe

      - name: Apply migrations BEFORE deploy
        run: |
          ./efbundle-orders.exe \
            --connection "${{ secrets.ORDERS_DB_CONNECTION }}"
        # ← La BD se actualiza ANTES que el código

      - name: Deploy Orders.API
        run: az webapp deploy --name orderflow-orders ...
        # ← El código llega DESPUÉS de la migration
```

## El orden importa: migration antes que código

```
CORRECTO (Expand-Contract):

  t=0:  Migration aplicada (columna nullable añadida)
        El código viejo sigue funcionando
        El código viejo no sabe de la nueva columna → la ignora (NULL)

  t=1:  Código nuevo desplegado
        Lee y escribe en la nueva columna
        Compatibilidad con datos históricos (NULL = sin descuento)

INCORRECTO (migration después del código):

  t=0:  Código nuevo desplegado
        Intenta leer columna "BuyerId" → ¡No existe!
        SQL Exception → servicio caído
        Rollback de emergencia a las 3am
```

**La regla de oro:** Las migrations son siempre **aditivas primero** (añadir sin eliminar). Las migraciones de eliminación van en releases posteriores.

## Idempotencia de migrations: ejecutar varias veces sin daño

EF Core ya garantiza idempotencia con la tabla `__EFMigrationsHistory`. Pero para scripts SQL personalizados:

```sql
-- Script de backfill idempotente
-- Se puede ejecutar múltiples veces sin efecto secundario

IF NOT EXISTS (
    SELECT 1 FROM sys.columns
    WHERE object_id = OBJECT_ID('orders.Orders')
      AND name = 'DiscountAmount'
)
BEGIN
    ALTER TABLE orders.Orders
    ADD DiscountAmount decimal(18,2) NULL DEFAULT 0.00;

    PRINT 'Column DiscountAmount added';
END
ELSE
BEGIN
    PRINT 'Column DiscountAmount already exists, skipping';
END

-- Backfill idempotente
UPDATE orders.Orders
SET DiscountAmount = 0.00
WHERE DiscountAmount IS NULL;
```

## Estrategia de naming para migrations

Un sistema de nombres consistente evita confusión:

```
CONVENCIÓN DE NOMBRES:

Para schema changes:
  {YYYYMMDD}_{Acción}{Entidad}
  20260101_AddDiscountAmountToOrders
  20260115_RenameCustomerIdToBuyerId_Phase1_Expand
  20260201_RenameCustomerIdToBuyerId_Phase3_Contract
  20260301_CreateAddressesTable

Para backfills (scripts, no migrations):
  backfill_{descripción}.sql
  backfill_buyer_id_from_customer_id.sql
  backfill_order_summaries_read_model.sql

Para eliminar columnas (siempre la última fase):
  20260315_DropCustomerIdColumn  ← Al menos 2 releases después del Expand

Ventaja: al ver el historial de migrations, entiendes la evolución del schema.
```

## Parallel Runs: el periodo de validación

Durante la transición de una columna a otra, ejecutar los dos sistemas en paralelo:

```csharp
// Durante la transición: escribir en ambas y verificar que coinciden
public async Task<Order?> GetByIdAsync(Guid id, CancellationToken ct)
{
    // Leer de la nueva estructura
    var fromNew = await _dbContext.Orders
        .Where(o => o.Id == id)
        .Select(o => new { o.BuyerId })
        .FirstOrDefaultAsync(ct);

    // En modo shadow (solo en staging): verificar que coincide con la vieja
    if (_environment.IsStaging())
    {
        var fromOld = await _dbContext.Orders
            .Where(o => o.Id == id)
            .Select(o => new { o.CustomerId })
            .FirstOrDefaultAsync(ct);

        if (fromNew?.BuyerId != fromOld?.CustomerId)
        {
            _logger.LogError(
                "INCONSISTENCY: Order {Id} BuyerId={New} != CustomerId={Old}",
                id, fromNew?.BuyerId, fromOld?.CustomerId);
        }
    }

    return /* el resultado normal */;
}
```

El Parallel Run detecta inconsistencias antes de eliminar la columna vieja.

## Feature Flags: desacoplar deploy de activación

Un **Feature Flag** permite desplegar código nuevo pero no activarlo hasta que estés listo:

```bash
dotnet add package Microsoft.FeatureManagement.AspNetCore
```

```csharp
// appsettings.json
{
  "FeatureManagement": {
    "UseBuyerId": false  // ← Desactivado inicialmente
  }
}

// Order.cs con feature flag
public class OrdersController : ControllerBase
{
    private readonly IFeatureManager _featureManager;

    [HttpGet("{id:guid}")]
    public async Task<IActionResult> GetById(Guid id)
    {
        if (await _featureManager.IsEnabledAsync("UseBuyerId"))
        {
            // Código nuevo — usa BuyerId
            return Ok(await _mediator.Send(new GetOrderByBuyerIdQuery(id)));
        }

        // Código viejo — usa CustomerId
        return Ok(await _mediator.Send(new GetOrderByCustomerIdQuery(id)));
    }
}
```

Con el feature flag, el código nuevo está desplegado pero inactivo. Se activa cambiando el `appsettings.json` o la configuración de Azure App Service — sin necesidad de un nuevo deploy.

## Rollback de datos: cuándo es posible

Cuando una migration falla en producción, el rollback del código es sencillo. El rollback de datos es más complejo:

**Migrations aditivas (añadir columnas, tablas):**
```sql
-- FÁCIL de revertir — eliminar lo añadido
ALTER TABLE orders.Orders DROP COLUMN DiscountAmount;
-- Sin pérdida de datos (la columna era nueva)
```

**Migrations destructivas (eliminar columnas, tablas):**
```sql
-- DROP COLUMN CustomerId
-- Si necesitas revertir: los datos YA NO EXISTEN
-- Necesitas restaurar desde backup
```

**La regla:** NUNCA hagas `DROP COLUMN` o `DROP TABLE` sin haber verificado que el sistema funciona correctamente en producción durante al menos 1 release completa. El Expand-Contract te da ese tiempo de seguridad.

---

## Rollback de emergencia: el plan B

Si una migration causa problemas en producción, el plan B:

```bash
# OPCIÓN A: Rollback del código (si la migration es segura de revertir)
# 1. Swap del slot de Azure App Service de vuelta a la versión anterior
az webapp deployment slot swap \
    --name orderflow-orders \
    --slot production \
    --target-slot previous-stable

# 2. La BD tiene la nueva columna pero el código viejo la ignora (es nullable)
# → No hay error de SQL porque la columna es nullable
# → El código viejo sigue funcionando

# OPCIÓN B: Rollback de la migration (solo si NO hubo DROP)
dotnet ef database update PreviousMigrationName \
    --project src/Orders.API \
    --connection "$ORDERS_DB_CONNECTION"
# EF Core revierte la migration (ejecuta el método Down())
```

## Blue-Green Deployment para datos

El **Blue-Green Deployment** tiene dos slots activos. Aplicado a datos:

```
SLOT BLUE (producción actual):
  Orders.API v1 → OrdersDb (schema v1)

SLOT GREEN (la nueva versión):
  Orders.API v2 → OrdersDb (schema v2 — ya migrado)
```

En Azure App Services, los deployment slots permiten hacer swap:

```bash
# 1. Aplicar migration a producción (schema es compatible con v1 y v2)
./efbundle-orders --connection "$ORDERS_DB_CONN"

# 2. Desplegar v2 al slot staging (no a producción todavía)
az webapp deployment source config-zip \
    --name orderflow-orders \
    --slot staging \
    --src orders-api-v2.zip

# 3. Verificar que el slot staging funciona correctamente
curl https://orderflow-orders-staging.azurewebsites.net/health

# 4. Swap: staging pasa a producción sin downtime
az webapp deployment slot swap \
    --name orderflow-orders \
    --slot staging \
    --target-slot production
```

El swap es casi instantáneo. Si algo falla: `swap` de vuelta en segundos.

## Monitoreo de migrations en producción

Después de aplicar una migration y desplegar, monitorear en Application Insights:

```kql
// KQL en Application Insights — errores post-migration
exceptions
| where timestamp > ago(30m)
| where outerMessage contains "Invalid column name"
    or outerMessage contains "Column does not exist"
    or outerMessage contains "Cannot insert NULL"
| summarize count() by bin(timestamp, 5m), outerMessage
| order by timestamp desc

// Si hay errores: la migration no está alineada con el código
// Acción: rollback del código (slot swap) mientras se investiga
```


## El ciclo de vida completo de un cambio de schema

Resumiendo el proceso end-to-end para cualquier cambio de schema en producción:

```
SEMANA 1 — EXPAND:
  Dev: Crea Migration (additive)
  CI: Genera migration bundle
  CI: Aplica migration a staging
  CI: Verifica tests de migration
  CD: Aplica migration a PRODUCCIÓN
  CD: La BD está actualizada, el código viejo sigue funcionando

  Dev: Crea código que usa la nueva estructura
  CI: Tests pasan
  CD: Deploy del código nuevo a PRODUCCIÓN

SEMANA 2 — MIGRATE:
  Ops: Ejecuta script de backfill (lotes, sin downtime)
  Monitoring: Verifica que datos migrados son correctos
  Dev: Activa feature flag (si hay) en producción

SEMANA 3 — CONTRACT:
  Dev: Elimina referencias a la estructura vieja del código
  Dev: Crea Migration (DROP COLUMN/TABLE)
  CD: Aplica migration → schema limpio
  CD: Deploy del código sin referencias a la estructura vieja

TOTAL: 3 semanas, 0 downtime, reversible en cualquier punto
```

## Particionamiento de tablas: para tablas muy grandes

Cuando `orders.Orders` supera los 50 millones de registros, el particionamiento mejora el rendimiento:

```sql
-- Particionar por año (los pedidos de 2024 van juntos, los de 2025 juntos)
CREATE PARTITION FUNCTION PF_OrdersByYear (datetime2)
AS RANGE RIGHT FOR VALUES
('2024-01-01', '2025-01-01', '2026-01-01');

CREATE PARTITION SCHEME PS_OrdersByYear
AS PARTITION PF_OrdersByYear
ALL TO ([PRIMARY]);

-- La tabla Orders usa el partition scheme
CREATE TABLE orders.Orders (
    Id uniqueidentifier NOT NULL,
    CreatedAt datetime2 NOT NULL,
    -- ...
    CONSTRAINT PK_Orders PRIMARY KEY (Id, CreatedAt) -- CreatedAt en la PK para particionamiento
) ON PS_OrdersByYear(CreatedAt);
```

**Para TechShop ahora:** Con 150.000 pedidos/año, el particionamiento no es necesario. Se menciona como patrón para cuando el volumen escale significativamente.

## Archivado de datos: los pedidos de hace 3 años

Los pedidos antiguos tienen diferentes patrones de acceso que los recientes. Archivar los antiguos mejora el rendimiento:

```csharp
// Job mensual que archiva pedidos completados de más de 3 años
public class OrderArchivalJob : BackgroundService
{
    protected override async Task ExecuteAsync(CancellationToken ct)
    {
        while (!ct.IsCancellationRequested)
        {
            // Esperar al primer día del mes
            await WaitUntilFirstOfMonthAsync(ct);

            await ArchiveOldOrdersAsync(ct);
        }
    }

    private async Task ArchiveOldOrdersAsync(CancellationToken ct)
    {
        var cutoffDate = DateTime.UtcNow.AddYears(-3);

        // Mover a tabla de archivo (misma estructura, diferente tabla)
        await _dbContext.Database.ExecuteSqlRawAsync("""
            INSERT INTO orders.ArchivedOrders
            SELECT * FROM orders.Orders
            WHERE Status = 'Delivered'
              AND CreatedAt < @Cutoff

            DELETE FROM orders.Orders
            WHERE Status = 'Delivered'
              AND CreatedAt < @Cutoff
            """, new SqlParameter("@Cutoff", cutoffDate), ct);
    }
}
```

Los pedidos archivados siguen disponibles para consulta (tabla `ArchivedOrders`) pero no saturan las queries de la tabla activa.

## Migrations en un equipo de múltiples developers

Cuando dos developers crean migrations a la vez en ramas diferentes:

```
Rama A: 20260501_AddDiscountAmount
Rama B: 20260501_AddShippingNotes

Al hacer merge en main:
orders/__Migrations/
  20260501_AddDiscountAmount.cs  ← conflicto de timestamp
  20260501_AddShippingNotes.cs   ← conflicto de timestamp

EF Core falla: "timestamp collision"
```

**La solución:**

```bash
# 1. Eliminar las dos migrations problemáticas
dotnet ef migrations remove --project src/Orders.API

# 2. Recrear la migration con el timestamp correcto (posterior)
dotnet ef migrations add AddDiscountAmount
dotnet ef migrations add AddShippingNotes

# 3. El nuevo timestamp refleja el orden correcto
# 20260502_AddDiscountAmount
# 20260502001_AddShippingNotes  (EF Core añade sufijo si mismo día)
```

**Prevención:** Usar un sufijo de equipo o ticket de Jira en vez de solo la fecha: `20260501_ORDERS123_AddDiscountAmount`.

## La deuda técnica del schema

Señales de que el schema de OrdersDb tiene deuda técnica:

```
🔴 Señales de alerta:
  - Columnas con nombres como "Field1", "Extra1", "TempValue"

  - Tablas sin índices en columnas usadas en WHERE frecuentes
  - Columnas que solo tienen NULL en producción (ya no se usan)
  - Migrations que se llaman "FixEverything" o "TempFix"

✅ Schema saludable:
  - Nombres semánticos de columnas alineados con el dominio
  - NOT NULL en columnas obligatorias del negocio
  - Índices en CustomerId, Status, CreatedAt (los más usados en WHERE)
  - 0 columnas con NULL que no deberían tenerlo
  - Historial de migrations legible: cuenta la evolución del negocio
```

El Expand-Contract, cuando se aplica consistentemente, mantiene el schema limpio porque obliga a formalizar cada cambio en tres fases explícitas.

## Preguntas frecuentes sobre migrations en producción

**¿Puedo aplicar la migration y el código en el mismo deploy?**
Solo si la migration es puramente aditiva (añadir columna nullable) y el código nuevo no rompe nada si la columna no existe todavía. En la práctica: es más seguro separarlos siempre.

**¿EF Core aplica migrations automáticamente al arrancar?**
Puede hacerlo con `Database.MigrateAsync()` en el startup, pero en producción con múltiples instancias es peligroso (condiciones de carrera). El Migration Bundle es la alternativa correcta.

**¿Qué hago si la migration tarda mucho en producción?**
Divide en migrations más pequeñas. Un `ALTER TABLE` que tarda más de 30 segundos puede causar timeouts. Para tablas grandes, crea la columna sin NOT NULL primero, luego haz el backfill por lotes, luego añade la constraint NOT NULL.

**¿Cómo sé cuándo es seguro eliminar la columna vieja?**
Cuando: (1) el código que la usa no está en ningún entorno, (2) has monitoreado producción durante al menos 1 semana sin errores relacionados, (3) tienes backup reciente por si acaso.

**¿Las migrations de EF Core son reversibles siempre?**
No. Si el método `Down()` tiene `DropColumn`, y ya aplicaste el `Up()` que la creó y llenó con datos, el `Down()` borra esos datos. Diseña las migrations pensando en el rollback.

## Resumen de M6.4

El Expand-Contract es el patrón que permite evolucionar el schema de la BD mientras el sistema está en producción:

**Las tres fases:**
`EXPAND` (añadir sin eliminar) → `MIGRATE` (datos) → `CONTRACT` (eliminar lo viejo)

**Migration Bundles en CI/CD:**
El bundle se aplica ANTES del deploy del código. La BD siempre va por delante. El código siempre compatible con el schema actual.

**Feature Flags:**
Desacoplar el deploy del código de la activación de funcionalidad. Una columna puede existir en la BD semanas antes de que el código la use activamente.

**Blue-Green con deployment slots:**
Swap instantáneo entre versiones. Rollback en < 30 segundos. La BD es compartida entre slots (el schema ya es compatible antes del swap).

**El resultado:** TechShop puede evolucionar OrderFlow continuamente, sin ventanas de mantenimiento, con confianza en el rollback y con visibilidad completa del estado del schema en cada entorno.
