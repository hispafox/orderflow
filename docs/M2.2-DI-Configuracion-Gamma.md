## ¿Por qué DI avanzada importa en microservicios?

En un microservicio bien diseñado, prácticamente todo se registra en el DI container. Repositorios, servicios de dominio, clientes HTTP, publicadores de eventos, validators.

Si registras con el lifetime incorrecto, puedes tener:
- **Memory leaks** (Singleton que retiene datos de una petición)
- **Datos corruptos entre peticiones** (Scoped que se comparte entre threads)
- **Excepciones en runtime** (el container detecta captive dependencies)
- **Problemas de concurrencia** (acceso no thread-safe a un objeto compartido)

Los errores de DI son de los más difíciles de diagnosticar: a veces se manifiestan solo en producción, bajo carga concurrente, de forma intermitente.

Entender bien los lifetimes de DI no es teoría — es prevención de bugs en producción.

## La inversión de control: el principio detrás de DI

La Inyección de Dependencias es una implementación del principio de Inversión de Control (IoC). Vale la pena entender el principio para entender por qué DI es tan potente.

**Sin IoC (control directo):**
```csharp
public class OrderService
{
    private readonly IOrderRepository _repo;

    public OrderService()
    {
        // El servicio controla la creación de sus dependencias
        _repo = new SqlOrderRepository(
            new OrderDbContext(
                new DbContextOptions<OrderDbContext>()));
    }
}
```
El servicio conoce la implementación concreta. Testear sin base de datos es imposible.

**Con IoC (inversión del control):**
```csharp
public class OrderService
{
    private readonly IOrderRepository _repo;

    public OrderService(IOrderRepository repo)
    {
        // El control se invierte: alguien externo decide qué inyectar
        _repo = repo;
    }
}
```
El servicio no sabe ni le importa qué implementación recibe. En producción recibe `SqlOrderRepository`. En tests recibe `InMemoryOrderRepository` o un mock.

## Principios SOLID y DI

La DI es la implementación práctica de varios principios SOLID:

**S — Single Responsibility:** El DI container gestiona la creación y el lifecycle de objetos. Los servicios solo se ocupan de su lógica.

**D — Dependency Inversion:** Los módulos de alto nivel (OrderService) no dependen de módulos de bajo nivel (SqlOrderRepository) — ambos dependen de abstracciones (IOrderRepository).

**O — Open/Closed:** Puedes cambiar la implementación de IOrderRepository (de SQL a NoSQL) sin tocar OrderService. Abierto para extensión, cerrado para modificación.

Estos principios no son teóricos en microservicios. Son prácticos: el día que necesitas cambiar la implementación de un repositorio, cambiar la estrategia de caching, o añadir un decorator de auditoría — el DI container te lo pone fácil.

## Los tres lifetimes: definiciones

**Transient (`AddTransient`):**
Se crea una nueva instancia cada vez que se pide al container. Si en el mismo request pides `IMyService` dos veces, obtienes dos instancias diferentes.

Usa Transient para: objetos ligeros, sin estado, que no comparten recursos.

**Scoped (`AddScoped`):**
Se crea una nueva instancia por scope. En ASP.NET Core, un scope = un request HTTP. Todas las peticiones al container dentro del mismo request obtienen la misma instancia.

Usa Scoped para: DbContext, repositorios, servicios que necesitan consistencia dentro de un request.

**Singleton (`AddSingleton`):**
Se crea una sola instancia para toda la vida de la aplicación. Todos los requests comparten la misma instancia.

Usa Singleton para: configuración, clients HTTP, loggers, caches in-memory, servicios thread-safe.

## Diagrama de lifetimes: la visualización clave

Para entender los lifetimes, visualiza la vida de los objetos:

```
Inicio de la aplicación
│
├── Singleton creado: IConfiguration, IMemoryCache, HttpClientFactory...
│   └── Vive para siempre. Compartido por TODOS los requests.
│
├── REQUEST 1 ─────────────────────────────────────────────────
│   ├── Scope creado
│   │   ├── Scoped creado: OrderDbContext, IOrderRepository...
│   │   │   └── El mismo objeto para todo el request 1
│   │   ├── Transient creado cada vez que se pide: IOrderValidator...
│   │   └── Scope destruido al final del request → Scoped disposed
│
├── REQUEST 2 ─────────────────────────────────────────────────
│   ├── Scope creado (NUEVO)
│   │   ├── Scoped creado: NUEVO OrderDbContext, NUEVO IOrderRepository
│   │   └── Scope destruido → Scoped disposed
│
└── Singleton destruido cuando la aplicación termina
```

El DbContext es Scoped por diseño: cada request tiene su propio contexto de base de datos, garantizando que los cambios de un request no interfieren con los de otro.

## DI: el container como grafo de dependencias

Una forma de entender el DI container: es un grafo dirigido de dependencias.

Cuando pides `IOrderService`, el container:
1. Mira qué implementación está registrada: `OrderService`
2. `OrderService` necesita `IOrderRepository` → resuelve `SqlOrderRepository`
3. `SqlOrderRepository` necesita `OrderDbContext` → resuelve `OrderDbContext`
4. `OrderDbContext` necesita `DbContextOptions<OrderDbContext>` → resuelve de configuración
5. Construye el grafo completo y te entrega `OrderService` con todas sus dependencias

Todo esto ocurre automáticamente. Sin factories manuales, sin `new` explícitos, sin gestión de lifecycle.

La única condición: todos los nodos del grafo deben estar registrados. Si falta alguno, el container lanza una excepción clara al arrancar (con `ValidateOnBuild`).

## Cuándo usar cada lifetime: regla práctica

La regla para no equivocarte:

**¿El objeto tiene estado que debe ser exclusivo de un request?**
→ Scoped. Ejemplo: DbContext, UnitOfWork, IOrderRepository.

**¿El objeto tiene estado compartido y es thread-safe?**
→ Singleton. Ejemplo: IMemoryCache, HttpClientFactory, IConfiguration.

**¿El objeto no tiene estado o es muy ligero?**
→ Transient. Ejemplo: validators, mappers, helpers.

**¿El objeto implementa IDisposable?**
→ Casi siempre Scoped o Transient. El container se encarga del dispose automáticamente al final del scope.

**La regla de oro:** Si tienes duda entre Transient y Scoped, elige Scoped. Si tienes duda entre Scoped y Singleton, elige Scoped. El Singleton es para objetos que explícitamente necesitas compartir entre todos los requests.

## El antipatrón: Captive Dependency

El error más común y más peligroso de DI. Ocurre cuando un Singleton captura un Scoped o Transient.

```csharp
// ❌ INCORRECTO — Captive dependency
public class OrderService  // Registrado como Singleton
{
    private readonly IOrderRepository _repo;  // Registrado como Scoped

    public OrderService(IOrderRepository repo)
    {
        _repo = repo;  // El Singleton "captura" el Scoped
    }
}
```

¿Qué pasa? El Singleton se crea una vez y mantiene la referencia al primer `IOrderRepository` que le inyectan. Todos los requests siguientes usan el mismo repositorio — incluyendo su DbContext. Los cambios de un request aparecen en otro. En el mejor caso, excepciones de threading. En el peor, datos corruptos.

.NET 5+ detecta este problema en desarrollo y lanza una `InvalidOperationException` clara. En producción (si desactivas la validación), el bug se manifiesta de forma intermitente y difícil de reproducir.

---

## Cómo detectar Captive Dependencies

.NET valida automáticamente el scope en desarrollo:

```csharp
// Program.cs
builder.Host.UseDefaultServiceProvider(options =>
{
    // Activo por defecto en Development
    options.ValidateScopes = true;
    options.ValidateOnBuild = true; // Valida en startup, no en el primer request
});
```

`ValidateOnBuild = true` hace que el container analice todos los registros en startup y lanza una excepción si detecta captive dependencies. Así descubres el problema cuando arranca la app, no en producción bajo carga.

Actívalo siempre en Development. En Production puedes dejarlo desactivado por rendimiento.

## Scoped services y thread safety

Una aclaración importante: Scoped no significa thread-safe.

En la mayoría de aplicaciones web, cada request se procesa en un único thread. El Scoped service se crea para ese thread y nadie más lo toca. No hay problema de concurrencia.

Pero en algunos casos, el código usa paralelismo dentro de un request:

```csharp
// ⚠️ CUIDADO — acceso concurrente a un Scoped service
var tasks = orderIds.Select(async id =>
{
    // Si _orderService es Scoped, esto puede ser problemático
    // porque múltiples tasks acceden al mismo DbContext simultáneamente
    var order = await _orderService.GetByIdAsync(id);
    return order;
});
await Task.WhenAll(tasks);
```

`DbContext` no es thread-safe. Si lo usas en paralelo dentro del mismo scope, obtendrás excepciones.

**La solución:** Para operaciones paralelas, crea un scope nuevo por operación:

```csharp
using var scope = _serviceScopeFactory.CreateScope();
var service = scope.ServiceProvider.GetRequiredService<IOrderService>();
var order = await service.GetByIdAsync(id);
```

## Registros con interfaz vs implementación concreta

Una pregunta frecuente: ¿siempre hay que registrar con interfaz?

**Con interfaz (recomendado para dependencias internas):**
```csharp
builder.Services.AddScoped<IOrderRepository, SqlOrderRepository>();
```
Ventajas: desacoplas la implementación, facilitas el testing (puedes mockear `IOrderRepository`), permites cambiar la implementación sin tocar el código que la usa.

**Con implementación concreta (válido para casos específicos):**
```csharp
builder.Services.AddScoped<OrderDbContext>();
builder.Services.AddSingleton<ProductCatalogCache>();
```
Válido cuando la clase concreta no tiene alternativa realista (DbContext, caches específicas). No hay nada malo en registrar concretos — es un pragmatismo razonable.

**La regla:** Si vas a mockearlo en tests, usa interfaz. Si es un tipo que nunca vas a sustituir, el concreto está bien.

## DI y testing: por qué los interfaces importan

Una de las razones más prácticas para registrar con interfaces es el testing.

Cuando registras así:
```csharp
builder.Services.AddScoped<IOrderRepository, SqlOrderRepository>();
```

En tus tests puedes sustituir la implementación:
```csharp
// En el test
var mockRepo = Substitute.For<IOrderRepository>();
mockRepo.GetByIdAsync(Arg.Any<Guid>())
    .Returns(new Order(...));

// El Controller/Service recibe el mock, no el SqlOrderRepository
services.AddScoped<IOrderRepository>(_ => mockRepo);
```

Sin la interfaz, tendrías que arrancar SQL Server en cada test unitario. Con la interfaz, el test es instantáneo y no necesita base de datos.

Esta es la razón real por la que el DI pattern es tan importante en microservicios — no solo para flexibilidad, sino para que el testing sea viable y rápido.

## Registrar múltiples implementaciones sin Keyed Services

Antes de Keyed Services (introducidos en .NET 8), el patrón común era una factory:

```csharp
// Antes de .NET 8 — factory pattern
public interface INotificationSender
{
    Task SendAsync(string message);
}

// Registrar usando factory
builder.Services.AddScoped<INotificationSender>(sp =>
{
    var env = sp.GetRequiredService<IWebHostEnvironment>();
    return env.IsDevelopment()
        ? sp.GetRequiredService<ConsoleNotificationSender>()
        : sp.GetRequiredService<EmailNotificationSender>();
});
```

Con Keyed Services (.NET 8+), es más limpio:

```csharp
// Con Keyed Services
builder.Services.AddKeyedScoped<INotificationSender,
    ConsoleNotificationSender>("console");
builder.Services.AddKeyedScoped<INotificationSender,
    EmailNotificationSender>("email");

// Resolver con clave en el constructor
public class NotificationsController(
    [FromKeyedServices("email")] INotificationSender sender)
```

---

## Keyed Services: múltiples implementaciones del mismo interface

En .NET 8+ (y .NET 10) puedes registrar múltiples implementaciones del mismo interface con una clave.

**Caso de uso real:** OrderFlow necesita publicar eventos. En desarrollo usa RabbitMQ. En pruebas, un publisher en memoria que no envía nada realmente.

```csharp
// Registrar con clave
builder.Services.AddKeyedScoped<IEventPublisher, RabbitMqEventPublisher>("production");
builder.Services.AddKeyedScoped<IEventPublisher, InMemoryEventPublisher>("testing");

// Resolver por clave
public class OrderService
{
    private readonly IEventPublisher _publisher;

    public OrderService(
        [FromKeyedServices("production")] IEventPublisher publisher)
    {
        _publisher = publisher;
    }
}
```

Otros casos de uso para Keyed Services:
- Múltiples repositorios para el mismo aggregate (primario/replica)
- Diferentes estrategias de notificación (email, SMS, push)
- Múltiples proveedores de pago (Stripe, PayPal, Bizum)

Cuando el número de registros crece, es una buena práctica agruparlos en extension methods:

```csharp
        services.AddTransient<IOrderValidator, OrderValidator>();

        return services;
    }
}
```

En Program.cs queda limpio:

```csharp

```

## Decorador: wrapping de servicios con DI

Un patrón potente que el DI container de .NET facilita: el Decorator.

Imagina que quieres añadir logging a todos los métodos de `IOrderRepository` sin tocar la implementación:

```csharp
public class LoggingOrderRepository : IOrderRepository
{
    private readonly IOrderRepository _inner;
    private readonly ILogger<LoggingOrderRepository> _logger;

    public LoggingOrderRepository(
        IOrderRepository inner,
        ILogger<LoggingOrderRepository> logger)
    {
        _inner = inner;
        _logger = logger;
    }

    public async Task<Order?> GetByIdAsync(Guid id)
    {
        _logger.LogInformation("Getting order {OrderId}", id);
        var result = await _inner.GetByIdAsync(id);
        _logger.LogInformation("Order {OrderId} {Found}",
            id, result is null ? "not found" : "found");
        return result;
    }
}

// Registrar el decorador
builder.Services.AddScoped<IOrderRepository>(sp =>
    new LoggingOrderRepository(
        new SqlOrderRepository(sp.GetRequiredService<OrderDbContext>()),
        sp.GetRequiredService<ILogger<LoggingOrderRepository>>()));
```

En la práctica, para cross-cutting concerns usaremos Pipeline Behaviors de MediatR (M3.4), que es más limpio. Pero el patrón Decorador con DI tiene su lugar en casos específicos.

## El patrón Options: el problema que resuelve

Imagina que Orders.API tiene estas configuraciones:

```json
{
  "Orders": {
    "MaxLinesPerOrder": 50,
    "MinOrderAmount": 10.00,
    "AllowBackorders": false,
    "RetentionDays": 365
  }
}
```

Opción A — el antipatrón:
```csharp
var maxLines = int.Parse(configuration["Orders:MaxLinesPerOrder"]);
```

Sin tipado. Sin validación. Si la clave no existe, devuelve null y el parse explota en runtime. No saber qué configuración usa el servicio sin leer todo el código.

```csharp
    public bool AllowBackorders { get; set; } = false;
    public int RetentionDays { get; set; } = 365;
}
```

Tipado, con valores por defecto, inyectable, validable. La configuración del servicio es un contrato explícito.

## IOptions, IOptionsSnapshot, IOptionsMonitor: cuál elegir

Tres interfaces para consumir configuración con el patrón Options. Cada una tiene su caso de uso:

**IOptions\<T\> — la más simple:**
- Se configura una vez en startup
- No detecta cambios en el archivo de configuración
- Es Singleton — la misma instancia para toda la aplicación
- Usa cuando: la configuración no cambia durante la vida del servicio

**IOptionsSnapshot\<T\> — por request:**
- Se reconfigura por cada scope (request)
- Detecta cambios en appsettings (en el siguiente request)
- Es Scoped — una instancia por request
- Usa cuando: la configuración puede cambiar y quieres que los cambios se apliquen request a request

**IOptionsMonitor\<T\> — reactivo:**
- Notifica activamente cuando la configuración cambia
- Es Singleton con suscripción a cambios
- Tiene `OnChange()` para reaccionar inmediatamente
- Usa cuando: necesitas reaccionar al cambio sin esperar al siguiente request (ej: rotar secretos sin reiniciar)

---

## Tabla de referencia: IOptions vs IOptionsSnapshot vs IOptionsMonitor

Una referencia rápida para decidir cuál usar:

| Característica | IOptions | IOptionsSnapshot | IOptionsMonitor |
|---|---|---|---|
| Lifetime | Singleton | Scoped | Singleton |
| Detecta cambios | No | Sí (por request) | Sí (inmediatamente) |
| OnChange callback | No | No | Sí |
| Uso recomendado | Configuración estática | Config que cambia rara vez | Secretos, feature flags |
| Ejemplo en OrderFlow | OrdersSettings | — | Rate limits dinámicos |

**Regla práctica:**
- La mayoría de configuraciones → `IOptions<T>`
- Si puede cambiar y quieres el valor actualizado → `IOptionsSnapshot<T>`
- Si necesitas reaccionar al cambio inmediatamente → `IOptionsMonitor<T>`

## Validación de Options con DataAnnotations

El patrón Options brilla cuando añades validación. Si la configuración es incorrecta, quieres saberlo en startup — no cuando procesas el primer pedido.

```csharp
// Añadir atributos de validación
public class OrdersSettings
{
    public const string SectionName = "Orders";

    [Range(1, 100, ErrorMessage = "MaxLinesPerOrder debe estar entre 1 y 100")]
    public int MaxLinesPerOrder { get; set; } = 50;

    [Range(0.01, double.MaxValue,
        ErrorMessage = "MinOrderAmount debe ser mayor que 0")]
    public decimal MinOrderAmount { get; set; } = 10.00m;

    public bool AllowBackorders { get; set; } = false;

    [Range(1, 3650,
        ErrorMessage = "RetentionDays debe estar entre 1 y 3650 días")]
    public int RetentionDays { get; set; } = 365;

    [Required]
    [StringLength(3, MinimumLength = 3,
        ErrorMessage = "Currency debe ser un código ISO de 3 letras")]
    public string Currency { get; set; } = "EUR";
}
```

## Activar la validación en startup

Con los atributos definidos, activamos la validación:

```csharp
// Program.cs
builder.Services
    .AddOptions<OrdersSettings>()
    .BindConfiguration(OrdersSettings.SectionName)
    .ValidateDataAnnotations()     // Valida con DataAnnotations
    .ValidateOnStart();            // Lo hace en startup, no en el primer uso
```

Ahora, si el `appsettings.json` tiene `MaxLinesPerOrder: -5`, la aplicación no arranca y lanza una excepción clara:

```
System.OptionsValidationException:
  DataAnnotation validation failed for 'OrdersSettings' members:
  'MaxLinesPerOrder' with the error: 'MaxLinesPerOrder debe estar entre 1 y 100'.
```

Fallo rápido, mensaje claro. Mucho mejor que descubrir el error cuando procesas un pedido.

---

## Por qué ValidateOnStart es imprescindible

Sin `ValidateOnStart()`, los problemas de configuración se descubren tarde:

**Escenario sin ValidateOnStart:**
1. Orders.API arranca correctamente ✅
2. Primer request llega — crea un pedido
3. El handler intenta acceder a `settings.MaxLinesPerOrder`
4. El valor no existe en la configuración — devuelve 0
5. La validación de dominio ("máximo 0 líneas") falla para todos los pedidos
6. El servicio está "roto" pero el proceso sigue corriendo

**Escenario con ValidateOnStart:**
1. Orders.API intenta arrancar
2. `ValidateOnStart()` verifica todos los valores al inicio
3. Detecta que `MaxLinesPerOrder` tiene un valor inválido
4. El proceso NO arranca — lanza una excepción clara
5. Aspire muestra el servicio en rojo con el mensaje de error
6. El developer corrige la configuración antes de que llegue ningún request

Fallo rápido y con contexto. Siempre mejor que un fallo silencioso y tardío.

## Validación custom con IValidateOptions

Para lógica de validación más compleja que no cabe en un DataAnnotation:

```csharp
// Validador custom
public class OrdersSettingsValidator : IValidateOptions<OrdersSettings>
{
    public ValidateOptionsResult Validate(string? name, OrdersSettings options)
    {
        var errors = new List<string>();

        // Regla de negocio: si no se permiten backorders,
        // el importe mínimo debe ser al menos 5€
        if (!options.AllowBackorders && options.MinOrderAmount < 5.00m)
        {
            errors.Add("Cuando AllowBackorders es false, " +
                       "MinOrderAmount debe ser al menos 5.00");
        }

        // Regla de negocio: retención mínima de 90 días para compliance
        if (options.RetentionDays < 90)
        {
            errors.Add("RetentionDays debe ser al menos 90 días " +
                       "por requisitos de compliance");
        }

        return errors.Any()
            ? ValidateOptionsResult.Fail(errors)
            : ValidateOptionsResult.Success;
    }
}

// Registrar el validador custom
builder.Services.AddSingleton<IValidateOptions<OrdersSettings>,
    OrdersSettingsValidator>();
```

## IOptionsMonitor: reaccionar a cambios en tiempo real

Para configuración que puede cambiar sin reiniciar el servicio:

```csharp
public class OrderService
{
    private readonly IOptionsMonitor<OrdersSettings> _settingsMonitor;

    public OrderService(IOptionsMonitor<OrdersSettings> settingsMonitor)
    {
        _settingsMonitor = settingsMonitor;

        // Suscribirse a cambios
        _settingsMonitor.OnChange(settings =>
        {
            _logger.LogInformation(
                "OrdersSettings changed. New MaxLinesPerOrder: {Max}",
                settings.MaxLinesPerOrder);
        });
    }

    public async Task<Order> CreateAsync(CreateOrderCommand cmd)
    {
        // Siempre accede al valor actual
        var settings = _settingsMonitor.CurrentValue;

        if (cmd.Items.Count > settings.MaxLinesPerOrder)
            throw new DomainException(
                $"Order cannot have more than {settings.MaxLinesPerOrder} lines");

        // ...
    }
}
```

`CurrentValue` siempre tiene el valor más reciente. Si el fichero cambia en disco, el cambio se propaga sin reiniciar.

## Configuración por entorno

Orders.API necesita diferentes configuraciones según el entorno: desarrollo local, staging, producción. ASP.NET Core tiene un sistema jerárquico para esto.

**Ficheros de configuración por entorno:**

## Gestión de secretos: por qué no en appsettings.json

Este es uno de los errores más frecuentes y más peligrosos:

```json
// ❌ NUNCA HAGAS ESTO
{
  "ConnectionStrings": {
    "OrdersDb": "Server=prod.database.windows.net;User=admin;Password=SuperSecret123!"
  }
}
```

Si este fichero se commitea al repositorio (aunque sea privado):
- Cualquier developer con acceso al repo tiene la password de producción
- Los secrets aparecen en el historial de git — aunque borres el fichero, siguen ahí
- Si el repo se hace público por error, las credenciales quedan expuestas
- Los pipelines de CI/CD clonan el repo — las credenciales viajan por varios sistemas

**La regla:** appsettings.json solo contiene configuración no sensible. Los secretos van en:

- **Desarrollo:** User Secrets (fuera del repo)
- **CI/CD:** Variables de entorno cifradas en GitHub Actions / Azure DevOps
- **Producción:** Azure Key Vault (managed, auditado, con control de acceso)

User Secrets es el mecanismo de ASP.NET Core para secretos en desarrollo. Se guardan fuera del repositorio, en la carpeta del usuario.

## Azure Key Vault: introducción

Azure Key Vault es el servicio de Azure para gestionar secretos, claves de cifrado y certificados. En producción, Orders.API (y todos los servicios de OrderFlow) obtienen sus secretos de Key Vault.

**Integración básica con .NET:**

```csharp
// Program.cs
if (!builder.Environment.IsDevelopment())
```

## Health Checks: qué son y por qué importan

Un health check es un endpoint que el orquestador (Aspire en desarrollo, Azure App Services en producción) consulta para saber si tu servicio está sano.

**¿Por qué importan?**

Sin health checks: si Orders.API tiene un bug que le impide conectar a la base de datos, sigue respondiendo a peticiones (con errores 500). El orquestador no sabe que está roto.

Con health checks: el health check detecta que la BD no responde, devuelve Unhealthy. Aspire muestra el servicio en rojo. Azure App Services puede restartear el proceso automáticamente.

Son los ojos del sistema sobre sus propios componentes.

---

## Liveness vs Readiness vs Startup probes

Tres tipos de health check con propósitos diferentes:

**Liveness probe — ¿El proceso está vivo?**
Pregunta: "¿El proceso sigue funcionando o hay que reiniciarlo?"
Responde: Yes/No. Si No, el orquestador reinicia el proceso.
Endpoint: `/alive`
Implementación: Siempre devuelve 200 si el proceso está vivo. No verifica dependencias externas.

**Readiness probe — ¿El servicio está listo para recibir tráfico?**
Pregunta: "¿Puedo enviarle peticiones o está arrancando/sobrecargado?"
Responde: Ready/Not Ready. Si Not Ready, el orquestador no le envía tráfico pero no lo reinicia.
Endpoint: `/health/ready`
Implementación: Verifica dependencias críticas (BD, message broker).

**Startup probe — ¿Ha terminado de arrancar?**
Pregunta: "¿Ha completado la inicialización?"
Solo relevante para servicios con arranque lento.
Endpoint: `/health/startup`

En Kubernetes, los tres tienen comportamientos específicos (lo veremos en M7.2). En Azure App Services, se usa principalmente la readiness probe para el deployment slot swap.

## Health Checks en ASP.NET Core: configuración base

ASP.NET Core incluye el middleware de health checks. ServiceDefaults ya lo configura. Pero necesitamos health checks específicos del servicio:

```csharp
// Program.cs — ampliar la configuración de health checks

builder.Services.AddHealthChecks()
    // Verificar SQL Server (OrdersDb)
    .AddSqlServer(
        connectionString: builder.Configuration
            .GetConnectionString("OrdersDb")!,
        healthQuery: "SELECT 1",
        name: "orders-db",
        failureStatus: HealthStatus.Unhealthy,
        tags: ["db", "sql"])

    // Verificar RabbitMQ
    .AddRabbitMQ(
        rabbitConnectionString: builder.Configuration
            .GetConnectionString("messaging")!,
        name: "rabbitmq",
        failureStatus: HealthStatus.Degraded,  // Degraded, no Unhealthy
        tags: ["messaging"]);
```

Nota: RabbitMQ se marca como `Degraded` (no `Unhealthy`). Si RabbitMQ no está disponible, el servicio sigue funcionando para consultas — solo la publicación de eventos falla. El servicio está degradado, no caído.

## Health Checks: estados y su significado

ASP.NET Core tiene tres estados de health:

**Healthy:**
Todo funciona correctamente. El servicio puede recibir tráfico.
```json
{ "status": "Healthy" }
```
HTTP 200

**Degraded:**
El servicio funciona pero con alguna limitación. Puede recibir tráfico pero con funcionalidad reducida.
```json
{ "status": "Degraded", "entries": { "rabbitmq": { "status": "Degraded" } } }
```
HTTP 200 (sigue recibiendo tráfico)

**Unhealthy:**
El servicio no puede funcionar. El orquestador puede decidir no enviarle tráfico o reiniciarlo.
```json
{ "status": "Unhealthy", "entries": { "orders-db": { "status": "Unhealthy" } } }
```
HTTP 503

En OrderFlow:
- `orders-db` Unhealthy → el servicio no puede procesar pedidos → Unhealthy
- `rabbitmq` Unhealthy → el servicio puede leer pedidos pero no publicar eventos → Degraded

## Health Checks con tags: granularidad

Los tags permiten agrupar checks y exponerlos en endpoints separados:

```csharp
builder.Services.AddHealthChecks()
    .AddSqlServer(

        name: "orders-db",

    .AddRabbitMQ(

```

## Health Check custom: lógica de negocio

Para lógica que no cubre la librería, implementas un health check custom:

```csharp
// Orders.API — health check que verifica que el dominio está operativo

public class OrderDomainHealthCheck : IHealthCheck
{
    private readonly OrderDbContext _dbContext;

    public OrderDomainHealthCheck(OrderDbContext dbContext)
    {
        _dbContext = dbContext;
    }

    public async Task<HealthCheckResult> CheckHealthAsync(
        HealthCheckContext context,
        CancellationToken cancellationToken = default)
    {
        try
        {
            // Verificar que podemos leer de la tabla principal
            var count = await _dbContext.Orders
                .CountAsync(cancellationToken);

            return HealthCheckResult.Healthy(
                $"Orders domain healthy. Total orders: {count}");
        }
        catch (Exception ex)
        {
            return HealthCheckResult.Unhealthy(
                "Orders domain check failed",
                exception: ex);
        }
    }
}

// Registrar
builder.Services.AddHealthChecks()
    .AddCheck<OrderDomainHealthCheck>("orders-domain",
        failureStatus: HealthStatus.Unhealthy);
```

## Preguntas frecuentes de esta sesión

**¿Puedo usar `IConfiguration` directamente en vez del patrón Options?**
Puedes, pero el patrón Options es superior: tipado, validación, inyectable, mockeable. `IConfiguration` directamente es un antipatrón para valores de configuración específicos del servicio.

**¿Por qué `IOptionsSnapshot` y no siempre `IOptionsMonitor`?**
IOptionsMonitor es Singleton y mantiene suscripciones activas. Para la mayoría de configuraciones que rara vez cambian, IOptionsSnapshot (reconfigura por request) o IOptions (fijo) son más apropiados y más simples. IOptionsMonitor es para casos donde necesitas reaccionar inmediatamente al cambio, como rotación de secretos.

**¿Los Health Checks tienen algún impacto en rendimiento?**
Mínimo. Solo se ejecutan cuando el orquestador los consulta (cada 30 segundos en Aspire por defecto). No se ejecutan en cada petición de usuario.

**¿User Secrets funciona en CI/CD?**
No. User Secrets son para desarrollo local. En CI/CD, los secretos se inyectan como variables de entorno desde GitHub Actions Secrets o Azure DevOps Variable Groups. El mecanismo de carga es el mismo — cambia la fuente.

## Lo que viene en M2.3

En el siguiente submódulo añadimos observabilidad completa a Orders.API.

**Serilog:** Structured logging con propiedades tipadas, sinks configurables (Console, Seq, Application Insights), log levels correctos por entorno.

**Correlation IDs:** Middleware que genera y propaga un ID único por petición. La clave para rastrear una petición a través de múltiples servicios.

**OpenTelemetry:** Traces distribuidas, métricas de negocio, configuración del pipeline de telemetría. Exportar al dashboard de Aspire en desarrollo.

**Métricas custom:** Counters y histograms para medir pedidos creados, tiempo de procesamiento, errores por tipo.

Al terminar M2.3, cada petición a Orders.API quedará registrada con logs estructurados, correlación entre servicios, y métricas visibles en el dashboard de Aspire.
