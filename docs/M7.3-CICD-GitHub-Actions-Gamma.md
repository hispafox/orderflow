## El problema que resolvemos en M7.3

En M7.2 desplegamos OrderFlow manualmente con comandos Azure CLI. El proceso tardó ~2 horas, implicó 8 pasos en orden exacto y requirió acceso a secretos de producción desde el portátil del developer.

**En producción real esto es inaceptable:**
- Un deploy manual a las 3am para una corrección crítica es arriesgado
- Si el developer olvida aplicar las migrations antes del código → incidente
- Si el smoke test falla nadie lo detecta hasta que un cliente llama
- 5 servicios × 8 pasos = 40 pasos manuales por deploy

**GitHub Actions resuelve esto:**
- Un push a `main` → pipeline automático → producción en 10-15 minutos
- Las migrations se aplican siempre antes que el código
- El smoke test falla → rollback automático
- Nadie toca secretos de producción desde su portátil

## La arquitectura del pipeline de OrderFlow

```
Push a main
    │
    ├── JOB: build-and-test (windows-latest)
    │   ├── Checkout código
    │   ├── Setup .NET 10
    │   ├── Restore + Build
    │   ├── Tests unitarios
    │   ├── Tests de integración (LocalDB)
    │   ├── Security scan (dependencias)
    │   └── Publicar artefactos
    │
    ├── JOB: migrate (depende de build-and-test)
    │   ├── Generar migration bundles
    │   └── Aplicar migrations a Azure SQL ← ANTES del código
    │
    └── JOB: deploy (matrix: 5 servicios, depende de migrate)
        ├── Deploy al slot staging
        ├── Smoke test del slot staging
        └── Swap a producción (Blue-Green)
```

## Configurar los secretos en GitHub

Antes de crear el workflow, configurar los secretos en GitHub:

```bash
# Desde Azure CLI — obtener los valores necesarios

# Credenciales de Azure para el pipeline
az ad sp create-for-rbac \
    --name "orderflow-github-actions" \
    --role Contributor \
    --scopes /subscriptions/{subscription-id}/resourceGroups/orderflow-rg \
    --sdk-auth

# El output JSON va al secreto AZURE_CREDENTIALS en GitHub
# Settings → Secrets and variables → Actions → New repository secret
```

**Secretos a configurar en GitHub:**
```
AZURE_CREDENTIALS     ← JSON del Service Principal
ORDERS_DB_CONN        ← Connection string para migration (con password de admin)
PRODUCTS_DB_CONN      ← Ídem para ProductsDb
PAYMENTS_DB_CONN      ← Ídem para PaymentsDb
NOTIFICATIONS_DB_CONN ← Ídem para NotificationsDb
```

## OIDC: autenticación sin secretos de larga duración

Una mejora avanzada: usar OIDC (OpenID Connect) en vez de Service Principal con credenciales:

```yaml
# Con OIDC: el pipeline obtiene tokens de Azure de forma efímera
# Sin AZURE_CREDENTIALS permanentes en los secretos

jobs:
  deploy:
    permissions:
      id-token: write  # ← Necesario para OIDC
      contents: read

    steps:
      - uses: azure/login@v1
        with:
          client-id: ${{ vars.AZURE_CLIENT_ID }}
          tenant-id: ${{ vars.AZURE_TENANT_ID }}
          subscription-id: ${{ vars.AZURE_SUBSCRIPTION_ID }}
          # Sin client-secret — usa OIDC
```

Con OIDC no hay credenciales de larga duración que puedan filtrarse. El token es válido solo para la ejecución del pipeline. Configuración adicional en Azure AD, pero más seguro.

## El workflow completo: CI (build y tests)

```yaml
# .github/workflows/ci-cd.yml

name: CI/CD — OrderFlow

on:
  push:
    branches: [main]
  pull_request:
    branches: [main]

env:
  DOTNET_VERSION: '10.0.x'
  PREFIX: orderflow

jobs:
  build-and-test:
    name: Build y Tests
    runs-on: windows-latest  # LocalDB requiere Windows

    steps:
      - name: Checkout
        uses: actions/checkout@v4

      - name: Setup .NET ${{ env.DOTNET_VERSION }}
        uses: actions/setup-dotnet@v4
        with:
          dotnet-version: ${{ env.DOTNET_VERSION }}

      - name: Restore dependencias
        run: dotnet restore OrderFlow.sln

      - name: Build
        run: dotnet build OrderFlow.sln --no-restore -c Release

      - name: Tests unitarios
        run: dotnet test OrderFlow.sln \
          --no-build -c Release \
          --filter "Category=Unit" \
          --logger "trx;LogFileName=unit-results.trx" \
          --results-directory ./TestResults
```

## El workflow: migration bundles

```yaml
  migrate:
    name: Aplicar Migrations
    runs-on: windows-latest
    needs: build-and-test  # Solo si los tests pasan
    if: github.ref == 'refs/heads/main'  # Solo en main, no en PRs

    steps:
      - uses: actions/checkout@v4

      - uses: actions/setup-dotnet@v4
        with: { dotnet-version: ${{ env.DOTNET_VERSION }} }

      - name: Instalar EF Core tools
        run: dotnet tool install --global dotnet-ef

      - name: Generar migration bundles
        run: |
          $contexts = @(
            @{ Project="Orders.API"; Context="OrderDbContext" },
            @{ Project="Products.API"; Context="ProductDbContext" },
            @{ Project="Payments.API"; Context="PaymentDbContext" },
            @{ Project="Notifications.API"; Context="NotificationDbContext" }
          )
          foreach ($ctx in $contexts) {
            dotnet ef migrations bundle `
              --project "src/$($ctx.Project)" `
              --context $ctx.Context `
              --output "efbundle-$($ctx.Project.Replace('.API','')).exe" `
              --runtime win-x64
          }
        shell: pwsh
```

## El workflow: deploy con Blue-Green (matrix)

```yaml
  deploy:
    name: Deploy — ${{ matrix.service }}
    runs-on: ubuntu-latest  # Azure CLI funciona en Linux
    needs: migrate
    if: github.ref == 'refs/heads/main'

    strategy:
      matrix:
        service: [orders, products, payments, notifications, gateway]
      max-parallel: 5  # Desplegar todos en paralelo

    steps:
      - name: Login en Azure
        uses: azure/login@v1
        with:
          creds: ${{ secrets.AZURE_CREDENTIALS }}

      - name: Descargar artefactos
        uses: actions/download-artifact@v4
        with:
          name: deploy-packages
          path: ./packages

      - name: Deploy al slot staging
        uses: azure/webapps-deploy@v3
        with:
          app-name: ${{ env.PREFIX }}-${{ matrix.service }}
          slot-name: staging
          package: ./packages/${{ matrix.service }}.zip
```

## El workflow: smoke tests y swap

```yaml
      - name: Smoke test del slot staging
        run: |
          $url = "https://${{ env.PREFIX }}-${{ matrix.service }}-staging.azurewebsites.net/health"
          $maxRetries = 10
          $retry = 0

          do {
            Start-Sleep -Seconds 10
            try {
              $response = Invoke-WebRequest -Uri $url -UseBasicParsing
              if ($response.StatusCode -eq 200) {
                Write-Host "✅ Staging health check OK: $url"
                exit 0
              }
            } catch {
              Write-Host "⏳ Esperando... intento $retry de $maxRetries"
            }
            $retry++
          } while ($retry -lt $maxRetries)

          Write-Error "❌ Staging no respondió en 100 segundos"
          exit 1
        shell: pwsh

      - name: Swap staging → producción
        run: |
          az webapp deployment slot swap \
            --resource-group orderflow-rg \
            --name ${{ env.PREFIX }}-${{ matrix.service }} \
            --slot staging \
            --target-slot production
          Write-Host "✅ ${{ matrix.service }} en producción"
        shell: pwsh
```

## El workflow: rollback automático

```yaml
      - name: Smoke test de producción
        id: prod-smoke-test
        run: |
          $url = "https://${{ env.PREFIX }}-${{ matrix.service }}.azurewebsites.net/health"
          Start-Sleep -Seconds 15
          $response = Invoke-WebRequest -Uri $url -UseBasicParsing
          if ($response.StatusCode -ne 200) {
            Write-Error "❌ Producción no saludable tras el swap"
            exit 1
          }
          Write-Host "✅ Producción OK: $url"
        shell: pwsh
        continue-on-error: true

      - name: Rollback si smoke test falla
        if: steps.prod-smoke-test.outcome == 'failure'
        run: |
          Write-Host "🔄 Iniciando rollback de ${{ matrix.service }}..."
          az webapp deployment slot swap \
            --resource-group orderflow-rg \
            --name ${{ env.PREFIX }}-${{ matrix.service }} \
            --slot production \
            --target-slot staging
          Write-Host "✅ Rollback completado — versión anterior restaurada"
          # Notificar al equipo (Teams, Slack, email)
          exit 1  # Marcar el job como fallido
        shell: pwsh
```

## Branch protection rules: calidad obligatoria

```
GitHub → Settings → Branches → main → Add rule:

✅ Require a pull request before merging
   - Required approvals: 1

✅ Require status checks to pass before merging
   - build-and-test → REQUIRED
   - (migrate y deploy no son requeridos — solo se ejecutan en main)

✅ Require branches to be up to date before merging

✅ Restrict who can push to matching branches
   - Solo el pipeline de CI/CD puede hacer push directo a main
```

Con estas reglas: nadie puede pushear código roto a main. El pipeline es el guardián de la calidad.

## Workflow de rollback manual

```yaml
# .github/workflows/rollback.yml — solo si el automático falla

name: Rollback Manual

on:
  workflow_dispatch:
    inputs:
      service:
        description: 'Servicio a revertir'
        required: true
        type: choice
        options: [orders, products, payments, notifications, gateway, all]

jobs:
  rollback:
    runs-on: ubuntu-latest
    steps:
      - uses: azure/login@v1
        with:
          creds: ${{ secrets.AZURE_CREDENTIALS }}

      - name: Swap de vuelta (staging ← producción)
        run: |
          if [ "${{ inputs.service }}" = "all" ]; then
            for svc in orders products payments notifications gateway; do
              az webapp deployment slot swap \
                --resource-group orderflow-rg \
                --name orderflow-$svc \
                --slot production \
                --target-slot staging
              echo "✅ $svc revertido"
            done
          else
            az webapp deployment slot swap \
              --resource-group orderflow-rg \
              --name orderflow-${{ inputs.service }} \
              --slot production \
              --target-slot staging
          fi
```

## Pruebas de regresión: detectar problemas antes del swap

```yaml
      - name: Pruebas de regresión en staging
        run: |
          $stagingUrl = "https://orderflow-${{ matrix.service }}-staging.azurewebsites.net"

          # Para el gateway: ejecutar el conjunto completo de smoke tests
          if ("${{ matrix.service }}" -eq "gateway") {
            # Test 1: catálogo de productos disponible
            $response = Invoke-WebRequest "$stagingUrl/api/products"
            if ($response.StatusCode -ne 200) { exit 1 }

            # Test 2: health check completo (BD, Service Bus, Identity)
            $health = Invoke-WebRequest "$stagingUrl/health" | ConvertFrom-Json
            if ($health.status -ne "Healthy") {
              Write-Error "❌ Health check no saludable: $($health | ConvertTo-Json)"
              exit 1
            }
          }

          Write-Host "✅ Pruebas de regresión completadas"
        shell: pwsh
```

## Canary deployment: alternativa al Blue-Green

El Blue-Green cambia el 100% del tráfico de golpe. El Canary lo hace gradualmente:

```
BLUE-GREEN (nuestro pipeline actual):
  t=0:  100% tráfico → versión vieja (staging tiene versión nueva)
  t=1:  SWAP → 100% tráfico → versión nueva
  Si hay un error: afecta al 100% de los usuarios hasta el rollback

CANARY (alternativa avanzada):
  t=0:  95% tráfico → versión vieja, 5% → versión nueva
  t=5m: verificar métricas — si OK:
  t=5:  70% vieja, 30% nueva
  t=10: 30% vieja, 70% nueva
  t=15: 0% vieja, 100% nueva
  Si hay un error: afecta solo al 5% de usuarios
```

Azure App Services no soporta Canary nativo. Para Canary real con App Services se usa Azure Front Door o Azure Traffic Manager. Para TechShop en esta fase: Blue-Green es suficiente.

## Resumen de M7.3

El pipeline de CI/CD de OrderFlow hace en 10 minutos lo que tardó 2 horas de forma manual en M7.2 — y con más garantías:

**Build + Test automático:** Cada push compila y ejecuta los tests. Los tests de integración usan LocalDB en el runner `windows-latest` — sin Docker, sin contenedores, sin configuración especial.

**Migrations primero:** Los migration bundles se aplican antes del código. Imposible desplegar código que no coincide con el schema de la BD.

**Blue-Green sin downtime:** Deploy al slot staging → smoke test → swap. Si algo falla: rollback en 30 segundos. Sin que el cliente note nada.

**Rollback automático:** Si el smoke test post-swap falla, el pipeline hace el swap de vuelta solo. Sin intervención manual en producción a las 3am.

**Gobernanza:** Branch protection obliga al PR y a que los tests pasen. Nadie puede pushear código roto a main directamente.

## Preguntas frecuentes sobre CI/CD con GitHub Actions

**¿Por qué `windows-latest` para los tests de integración?**
LocalDB (SQL Server Express LocalDB) solo existe en Windows. En los runners de Linux no está disponible. Usar `windows-latest` garantiza que los tests de integración tienen acceso a LocalDB sin necesitar Docker ni contenedores.

**¿Puedo usar el runner `ubuntu-latest` para algo?**
Sí — para los jobs de deploy, que solo usan Azure CLI (disponible en Linux). El deploy es más rápido en Linux porque el runner es más ligero.

**¿Qué pasa si dos developers hacen merge al mismo tiempo?**
GitHub Actions ejecuta ambos pipelines en paralelo. Ambos se despliegan al slot staging. El segundo swap sobrescribe al primero en producción. No hay corrupción — solo el segundo deploy queda en producción.

**¿Cómo gestiono las migrations si dos PRs cambian el schema simultáneamente?**
El Expand-Contract (M6.4) garantiza que cada migration es aditiva. Si dos migrations llegan simultáneamente: se aplican en orden de merge. Al ser aditivas, no hay conflicto entre ellas.
