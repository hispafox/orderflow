## Bienvenidos al Módulo 3

El Módulo 2 nos dio la infraestructura: Aspire, DI, configuración, observabilidad, tests. Orders.API arranca, responde y se puede observar — pero no hace nada útil todavía.

En el Módulo 3 construimos lo que hace que Orders.API sea único: su lógica de negocio. Empezamos por el centro — el dominio.

El dominio es el corazón de un microservicio bien diseñado. Es donde viven las reglas de negocio de TechShop: qué es un pedido, cuándo se puede cancelar, cómo se calcula el total, qué pasa cuando se confirma.

Si el dominio está bien diseñado, el resto del servicio es infraestructura que lo rodea. Si está mal diseñado, ninguna capa de infraestructura puede salvarlo.

---

## DDD táctico: los building blocks

En M1.3 vimos DDD estratégico: Bounded Contexts, Event Storming, Context Mapping. Esas herramientas nos dieron el mapa de los 4 servicios.

Ahora DDD táctico: las herramientas para diseñar el interior de un servicio.

**Entity:** Tiene identidad propia. Dos entidades son iguales si tienen el mismo Id, independientemente de sus atributos.

**Value Object:** No tiene identidad. Dos value objects son iguales si tienen los mismos atributos. Inmutables por definición.

**Aggregate:** Grupo de entidades y value objects que se tratan como una unidad de consistencia. El Aggregate Root es el único punto de entrada.

**Domain Event:** Algo que ocurrió en el dominio. Hechos pasados, inmutables. "OrderCreated", "OrderConfirmed".

**Repository:** Abstracción de la persistencia. El dominio define la interfaz, la infraestructura la implementa.

```
│   └── DomainException.cs

    └── IOrderRepository.cs   ← Solo la interfaz, no la implementación
```

El dominio no conoce EF Core, no conoce SQL Server, no conoce MassTransit. Solo conoce las reglas de negocio de TechShop.

## La regla de oro del dominio

**El dominio no depende de nadie. Todos dependen del dominio.**

```
API (Controllers)
    ↓ depende de
Application (Handlers, Commands, Queries)
    ↓ depende de
Domain (Entities, Value Objects, Interfaces)
    ↑ no depende de
Infrastructure (EF Core, SQL Server, RabbitMQ)
```

Si en tu entidad `Order` hay un `using Microsoft.EntityFrameworkCore` — el diseño está roto. El dominio no sabe que existe EF Core. No sabe que existe una base de datos.

Esta separación es la que permite testear el dominio con tests unitarios puros, sin necesitar ninguna infraestructura.

## DomainException: la base de las excepciones

Antes de las entidades, creamos la excepción base del dominio:

```csharp
// Orders.API/Domain/Exceptions/DomainException.cs

namespace Orders.API.Domain.Exceptions;

/// <summary>
/// Excepción lanzada cuando se viola una regla de negocio del dominio.
/// No debe confundirse con errores de infraestructura o validación de entrada.
/// </summary>
public class DomainException : Exception
{
    public DomainException(string message)
        : base(message) { }

    public DomainException(string message, Exception innerException)
        : base(message, innerException) { }
}
```

Toda violación de regla de negocio lanza `DomainException`. Los handlers de arriba la capturan y convierten en respuestas HTTP adecuadas (422 Unprocessable Entity).

La diferencia con `ArgumentException` o `InvalidOperationException`: una `DomainException` indica que el negocio no permite la operación — no que el código tenga un bug.

## Value Object: Money

El primer value object. `Money` representa un importe con su moneda. Es inmutable — una vez creado, no cambia.

```csharp
// Orders.API/Domain/ValueObjects/Money.cs

namespace Orders.API.Domain.ValueObjects;

public sealed record Money
{
    public decimal Amount { get; }
    public string Currency { get; }

    public Money(decimal amount, string currency)
    {
        if (amount < 0)
            throw new DomainException(
                $"Money amount cannot be negative. Got: {amount}");

        if (string.IsNullOrWhiteSpace(currency) || currency.Length != 3)
            throw new DomainException(
                $"Currency must be a 3-letter ISO code. Got: '{currency}'");

        Amount = amount;
        Currency = currency.ToUpperInvariant();
    }

    public Money Add(Money other)
    {
        EnsureSameCurrency(other);
        return new Money(Amount + other.Amount, Currency);
    }

    public Money Subtract(Money other)
    {
        EnsureSameCurrency(other);
        var result = Amount - other.Amount;
        if (result < 0)
            throw new DomainException("Money cannot be negative after subtraction");
        return new Money(result, Currency);
    }

    public Money Multiply(int quantity)
    {
        if (quantity <= 0)
            throw new DomainException("Quantity must be positive");
        return new Money(Amount * quantity, Currency);
    }

    private void EnsureSameCurrency(Money other)
    {
        if (Currency != other.Currency)
            throw new DomainException(
                $"Cannot operate on different currencies: {Currency} and {other.Currency}");
    }

    public static Money Zero(string currency) => new(0, currency);

    public override string ToString() => $"{Amount:F2} {Currency}";
}
```

## Por qué `record` para Value Objects

C# 14 records son perfectos para value objects por tres razones:

**Igualdad por valor automática.** Dos records con los mismos valores son iguales:
```csharp
var price1 = new Money(100, "EUR");
var price2 = new Money(100, "EUR");
price1 == price2 // true — sin override manual de Equals
```

**Inmutabilidad por defecto.** Las propiedades `init` solo se pueden establecer en el constructor.

**`with` expression para transformaciones:**
```csharp
// Crear una variación sin mutar el original
var discounted = price1 with { Amount = price1.Amount * 0.9m };
```

**Deconstrucción automática:**
```csharp
var (amount, currency) = price1;
```

Con `sealed record`, además evitas herencia accidental — los value objects no deben heredarse.

## Value Object: Address

La dirección de envío. Otro value object inmutable:

```csharp
// Orders.API/Domain/ValueObjects/Address.cs

namespace Orders.API.Domain.ValueObjects;

public sealed record Address
{
    public string Street { get; }
    public string City { get; }
    public string ZipCode { get; }
    public string Country { get; }
    public string? State { get; }

    public Address(
        string street,
        string city,
        string zipCode,
        string country,
        string? state = null)
    {
        Street = Guard.NotNullOrEmpty(street, nameof(street));
        City = Guard.NotNullOrEmpty(city, nameof(city));
        ZipCode = Guard.NotNullOrEmpty(zipCode, nameof(zipCode));
        Country = ValidateCountryCode(country);
        State = state;
    }

    private static string ValidateCountryCode(string country)
    {
        if (string.IsNullOrWhiteSpace(country) || country.Length != 2)
            throw new DomainException(
                $"Country must be a 2-letter ISO code. Got: '{country}'");
        return country.ToUpperInvariant();
    }

    public override string ToString()
        => $"{Street}, {City} {ZipCode}, {Country}";
}
```

## Guard Clauses: el helper de validaciones

Las validaciones se repiten. Un helper evita duplicar código:

```csharp
// Orders.API/Domain/Guards.cs

namespace Orders.API.Domain;

public static class Guard
{
    public static string NotNullOrEmpty(string value, string paramName)
    {
        if (string.IsNullOrWhiteSpace(value))
            throw new DomainException($"{paramName} cannot be null or empty");
        return value;
    }

    public static T NotNull<T>(T? value, string paramName)
        where T : class
    {
        if (value is null)
            throw new DomainException($"{paramName} cannot be null");
        return value;
    }

    public static int Positive(int value, string paramName)
    {
        if (value <= 0)
            throw new DomainException(
                $"{paramName} must be positive. Got: {value}");
        return value;
    }

    public static decimal NonNegative(decimal value, string paramName)
    {
        if (value < 0)
            throw new DomainException(
                $"{paramName} cannot be negative. Got: {value}");
        return value;
    }
}
```

## Value Object: OrderStatus

El estado de un pedido no es un simple `enum`. Tiene lógica asociada — qué transiciones son válidas. Con C# 14 podemos expresarlo de forma rica:

```csharp
// Orders.API/Domain/ValueObjects/OrderStatus.cs

namespace Orders.API.Domain.ValueObjects;

public sealed record OrderStatus
{
    public static readonly OrderStatus Pending    = new("Pending");
    public static readonly OrderStatus Confirmed  = new("Confirmed");
    public static readonly OrderStatus Shipped    = new("Shipped");
    public static readonly OrderStatus Delivered  = new("Delivered");
    public static readonly OrderStatus Cancelled  = new("Cancelled");

    public string Value { get; }

    private OrderStatus(string value) => Value = value;

    // Las transiciones válidas expresadas como regla de negocio
    private static readonly Dictionary<OrderStatus, OrderStatus[]> ValidTransitions = new()
    {
        [Pending]   = [Confirmed, Cancelled],
        [Confirmed] = [Shipped, Cancelled],
        [Shipped]   = [Delivered],
        [Delivered] = [],
        [Cancelled] = []
    };

    public bool CanTransitionTo(OrderStatus next)
        => ValidTransitions[this].Contains(next);

    public OrderStatus TransitionTo(OrderStatus next)
    {
        if (!CanTransitionTo(next))
            throw new DomainException(
                $"Cannot transition from '{Value}' to '{next.Value}'");
        return next;
    }

    public override string ToString() => Value;

    // Para EF Core mapping
    public static OrderStatus FromString(string value) => value switch
    {
        "Pending"   => Pending,
        "Confirmed" => Confirmed,
        "Shipped"   => Shipped,
        "Delivered" => Delivered,
        "Cancelled" => Cancelled,
        _ => throw new DomainException($"Unknown OrderStatus: '{value}'")
    };
}
```

## Entity: OrderLine

`OrderLine` es una entidad dentro del aggregate Order. Tiene su propia identidad (Id) y encapsula la lógica de una línea de pedido:

```csharp
// Orders.API/Domain/Entities/OrderLine.cs

namespace Orders.API.Domain.Entities;

public class OrderLine
{
    public Guid Id { get; private set; }
    public Guid ProductId { get; private set; }
    public string ProductName { get; private set; }
    public int Quantity { get; private set; }
    public Money UnitPrice { get; private set; }
    public Money LineTotal => UnitPrice.Multiply(Quantity);

    // Constructor privado — solo Order puede crear OrderLines
    private OrderLine() { }

    internal static OrderLine Create(
        Guid productId,
        string productName,
        int quantity,
        Money unitPrice)
    {
        if (quantity <= 0 || quantity > 1000)
            throw new DomainException(
                $"Quantity must be between 1 and 1000. Got: {quantity}");

        return new OrderLine
        {
            Id = Guid.NewGuid(),
            ProductId = productId,
            ProductName = Guard.NotNullOrEmpty(productName, nameof(productName)),
            Quantity = quantity,
            UnitPrice = Guard.NotNull(unitPrice, nameof(unitPrice))
        };
    }

    // Solo el Aggregate Root puede modificar sus entidades hijas
    internal void UpdateQuantity(int newQuantity)
    {
        if (newQuantity <= 0)
            throw new DomainException("Quantity must be positive");
        Quantity = newQuantity;
    }
}
```

## Aggregate Root: Order

El corazón del dominio. `Order` es el Aggregate Root — el único punto de acceso al aggregate, el guardián de las invariantes:

```csharp
// Orders.API/Domain/Entities/Order.cs

namespace Orders.API.Domain.Entities;

public class Order
{
    // Propiedades — todas con setter privado
    public Guid Id { get; private set; }
    public Guid CustomerId { get; private set; }
    public OrderStatus Status { get; private set; }
    public Address ShippingAddress { get; private set; } = null!;
    public DateTime CreatedAt { get; private set; }
    public DateTime? ConfirmedAt { get; private set; }
    public DateTime? CancelledAt { get; private set; }
    public string? CancellationReason { get; private set; }

    // Las líneas: solo lectura desde fuera
    private readonly List<OrderLine> _lines = [];
    public IReadOnlyList<OrderLine> Lines => _lines.AsReadOnly();

    // Total calculado — siempre consistente con las líneas
    public Money Total => Lines.Any()
        ? Lines.Skip(1).Aggregate(
            Lines[0].LineTotal,
            (acc, line) => acc.Add(line.LineTotal))
        : Money.Zero("EUR");

    // Domain Events — lo que pasó en este aggregate
    private readonly List<IDomainEvent> _domainEvents = [];
    public IReadOnlyList<IDomainEvent> DomainEvents => _domainEvents.AsReadOnly();

    // Constructor privado — solo el factory method puede crear Orders
    private Order() { }
```

## Order: el factory method Create

El factory method es el único punto de entrada para crear un pedido. Aplica todas las invariantes de creación:

```csharp
    // Continuación de Order.cs

    public static Order Create(
        Guid customerId,
        IEnumerable<(Guid ProductId, string ProductName,
                      int Quantity, Money UnitPrice)> items,
        Address shippingAddress)
    {
        var itemList = items.ToList();

        if (!itemList.Any())
            throw new DomainException("Order must have at least one item");

        if (itemList.Count > 50)
            throw new DomainException("Order cannot have more than 50 lines");

        var order = new Order
        {
            Id = Guid.NewGuid(),
            CustomerId = customerId,
            Status = OrderStatus.Pending,
            ShippingAddress = Guard.NotNull(shippingAddress, nameof(shippingAddress)),
            CreatedAt = DateTime.UtcNow
        };

        foreach (var item in itemList)
        {
            order._lines.Add(OrderLine.Create(
                item.ProductId, item.ProductName,
                item.Quantity, item.UnitPrice));
        }

        // Registrar el domain event
        order._domainEvents.Add(new OrderCreatedEvent(
            order.Id, order.CustomerId, order.Total));

        return order;
    }
```

## Order: los métodos de negocio

El comportamiento del aggregate — lo que puede hacer un pedido:

```csharp
    // Continuación de Order.cs

    public void Confirm()
    {
        Status = Status.TransitionTo(OrderStatus.Confirmed);
        ConfirmedAt = DateTime.UtcNow;
        _domainEvents.Add(new OrderConfirmedEvent(Id, CustomerId, Total));
    }

    public void Cancel(string reason)
    {
        Guard.NotNullOrEmpty(reason, nameof(reason));
        Status = Status.TransitionTo(OrderStatus.Cancelled);

        CancellationReason = reason;
        _domainEvents.Add(new OrderCancelledEvent(Id, CustomerId, reason));
    }

    public void Ship()
    {
        Status = Status.TransitionTo(OrderStatus.Shipped);
        _domainEvents.Add(new OrderShippedEvent(Id));
    }

    public void Deliver()
    {
        Status = Status.TransitionTo(OrderStatus.Delivered);
        _domainEvents.Add(new OrderDeliveredEvent(Id));
    }

    // Limpiar events después de publicarlos
    public void ClearDomainEvents()
        => _domainEvents.Clear();
}
```

## La diferencia entre Domain Events e Integration Events

Un concepto que confunde a muchos:

**Domain Event:** Ocurre dentro del bounded context. Se procesa en el mismo proceso, en la misma transacción. "Algo pasó en el aggregate."

```csharp
// Dentro del aggregate — nadie fuera del BC lo ve directamente
order._domainEvents.Add(new OrderCreatedEvent(order.Id, ...));
```

**Integration Event:** Se publica al exterior. Cruza bounded contexts a través de RabbitMQ o Azure Service Bus. Puede tener un formato diferente al domain event.

```csharp
// Publicado al message broker — Products y Notifications lo reciben
await _publisher.PublishAsync(new OrderCreated { // ← OrderFlow.Contracts
    OrderId = order.Id,
    CustomerId = order.CustomerId,
    Items = order.Lines.Select(l => ...).ToList()
});
```

En Orders.API: los domain events los procesa el handler de MediatR (para efectos locales y para publicar integration events). Los integration events los reciben otros servicios.

## Domain Events: cómo se procesan

Los domain events se registran en el aggregate durante la operación. Se procesan DESPUÉS de que la transacción de base de datos se confirma:

```csharp
// En el handler (Application Layer) — lo veremos en M3.4
public async Task<OrderDto> Handle(CreateOrderCommand cmd, CancellationToken ct)
{
    // 1. Crear el aggregate — registra los domain events internamente
    var order = Order.Create(cmd.CustomerId, cmd.Items, cmd.ShippingAddress);

    // 2. Persistir el aggregate
    await _repository.SaveAsync(order, ct);

    // 3. Publicar los domain events DESPUÉS de la transacción
    foreach (var domainEvent in order.DomainEvents)
    {
        await _mediator.Publish(domainEvent, ct);
    }

    // 4. Limpiar los events del aggregate
    order.ClearDomainEvents();

    return order.ToDto();
}
```

Los domain events se publican después de `SaveAsync` — la transacción de BD ya está confirmada. Si falla la publicación, el Outbox Pattern (M6.2) garantiza que se publicarán.

## El Repository: la interfaz en el dominio

El dominio define cómo quiere acceder a los datos. La implementación es trabajo de la infraestructura:

```csharp
// Orders.API/Domain/Interfaces/IOrderRepository.cs

namespace Orders.API.Domain.Interfaces;

public interface IOrderRepository
{
    Task<Order?> GetByIdAsync(Guid id, CancellationToken ct = default);
    Task<IReadOnlyList<Order>> GetByCustomerIdAsync(
        Guid customerId,
        CancellationToken ct = default);
    Task<IReadOnlyList<Order>> GetPendingAsync(CancellationToken ct = default);
    Task SaveAsync(Order order, CancellationToken ct = default);
}
```

El dominio no sabe si los datos están en SQL Server, en memoria, o en un fichero. Solo sabe que necesita guardar y recuperar `Order`.

`SaveAsync` sirve tanto para crear como para actualizar — el repositorio detecta si es nuevo o existente usando EF Core change tracking.

## El IOrderRepository: diseño de la interfaz

La interfaz del repositorio expresa exactamente qué necesita el dominio, sin filtraciones de infraestructura:

```csharp
// Orders.API/Domain/Interfaces/IOrderRepository.cs

namespace Orders.API.Domain.Interfaces;

/// <summary>
/// Puerto de salida para la persistencia de Orders.
/// El dominio define la interfaz. La infraestructura la implementa.
/// </summary>
public interface IOrderRepository
{
    /// <summary>Obtiene un pedido por su Id. Null si no existe.</summary>
    Task<Order?> GetByIdAsync(Guid id, CancellationToken ct = default);

    /// <summary>Lista pedidos de un cliente, ordenados por fecha descendente.</summary>
    Task<IReadOnlyList<Order>> GetByCustomerIdAsync(
        Guid customerId,
        CancellationToken ct = default);

    /// <summary>Lista pedidos pendientes de procesamiento.</summary>
    Task<IReadOnlyList<Order>> GetPendingAsync(CancellationToken ct = default);

    /// <summary>Persiste el aggregate. Detecta si es insert o update.</summary>
    Task SaveAsync(Order order, CancellationToken ct = default);
}
```

No hay `IQueryable`, no hay `Expression<Func<T, bool>>`, no hay `DbSet`. Esas son abstracciones de infraestructura. El dominio no las conoce.

## IOrderRepository: la implementación en infraestructura

La interfaz vive en el dominio. La implementación en infraestructura:

```csharp
// Orders.API/Infrastructure/Persistence/SqlOrderRepository.cs

namespace Orders.API.Infrastructure.Persistence;

public class SqlOrderRepository : IOrderRepository
{
    private readonly OrderDbContext _dbContext;

    public SqlOrderRepository(OrderDbContext dbContext)
        => _dbContext = dbContext;

    public async Task<Order?> GetByIdAsync(
        Guid id, CancellationToken ct = default)
        => await _dbContext.Orders
            .Include("_lines") // Campo privado
            .FirstOrDefaultAsync(o => o.Id == id, ct);

    public async Task<IReadOnlyList<Order>> GetByCustomerIdAsync(
        Guid customerId, CancellationToken ct = default)
        => await _dbContext.Orders
            .Include("_lines")
            .Where(o => o.CustomerId == customerId)
            .OrderByDescending(o => o.CreatedAt)
            .ToListAsync(ct);

    public async Task SaveAsync(Order order, CancellationToken ct = default)
    {
        var entry = _dbContext.Entry(order);
        if (entry.State == EntityState.Detached)
            _dbContext.Orders.Add(order); // Insert
        // Si está tracked por EF Core: Update automático
        await _dbContext.SaveChangesAsync(ct);
    }
}
```

## Clean Architecture: la regla de dependencias

Clean Architecture (Uncle Bob) es el patrón que separa el dominio de la infraestructura. La Dependency Rule establece que el código solo puede apuntar hacia adentro — el dominio no conoce la infraestructura:

**Puertos de entrada (Driving Ports):** Cómo el exterior llama al dominio. Las interfaces que definen los casos de uso: `ICreateOrderUseCase`, `ICancelOrderUseCase`.

En nuestra implementación: los Commands de MediatR son los puertos de entrada.

**Puertos de salida (Driven Ports):** Cómo el dominio llama al exterior. Las interfaces que el dominio define y la infraestructura implementa: `IOrderRepository`, `IEventPublisher`.

**Adaptadores primarios:** Los Controllers. Traducen peticiones HTTP en Commands de MediatR.

**Adaptadores secundarios:** EF Core (implementa `IOrderRepository`), MassTransit (implementa `IEventPublisher`).

## Invariantes del aggregate: las reglas sagradas

Las invariantes son las condiciones que el aggregate debe cumplir SIEMPRE. El aggregate root es responsable de garantizarlas:

**Invariante 1:** Un pedido debe tener al menos una línea.
```csharp
if (!itemList.Any()) throw new DomainException("Order must have at least one item");
```

**Invariante 2:** Un pedido no puede tener más de 50 líneas.
```csharp
if (itemList.Count > 50) throw new DomainException("Order cannot have more than 50 lines");
```

**Invariante 3:** Las transiciones de estado deben ser válidas.
```csharp
Status = Status.TransitionTo(OrderStatus.Confirmed); // Lanza si no es válida
```

**Invariante 4:** El total siempre es la suma de las líneas.
```csharp
public Money Total => Lines.Aggregate(...); // Calculado, nunca almacenado
```

Las invariantes NO se pueden romper. Si lo intentas, el aggregate lanza `DomainException`. No hay excepción: el código externo siempre recibe un aggregate en estado válido.

## Ubiquitous Language en el código

Comparemos dos versiones del mismo código. La primera habla "técnico". La segunda habla el lenguaje del negocio:

```csharp
// ❌ Versión técnica — nadie de negocio entiende esto
public void UpdateStatus(int statusCode)
{
    if (statusCode < 0 || statusCode > 4) throw new Exception("Invalid status");
    this.StatusCode = statusCode;
    this.UpdatedAt = DateTime.Now;
}

// ✅ Versión con Ubiquitous Language — negocio y código hablan igual
public void Confirm()
{
    Status = Status.TransitionTo(OrderStatus.Confirmed);
    ConfirmedAt = DateTime.UtcNow;
    _domainEvents.Add(new OrderConfirmedEvent(Id, CustomerId, Total));
}
```

La segunda versión lee como el lenguaje que usa negocio: "confirmar un pedido". Si alguien de producto pregunta "¿qué pasa cuando se confirma un pedido?", puedes mostrarle este código y lo entiende.

## Anemic Domain Model: el antipatrón

El antipatrón más común en .NET: el modelo anémico. Entidades sin comportamiento, toda la lógica en servicios.

```csharp
// ❌ Modelo anémico — Order es solo un contenedor de datos
public class Order
{
    public Guid Id { get; set; }
    public int StatusCode { get; set; }  // set público — cualquiera puede cambiar
    public List<OrderLine> Lines { get; set; } = new();
    public decimal Total { get; set; }   // Se puede poner cualquier valor
}

// La lógica vive en un servicio enorme
public class OrderService
{
    public void ConfirmOrder(Order order) { order.StatusCode = 2; order.Total = ...}
    public void CancelOrder(Order order) { order.StatusCode = 5; }
    // Todo el comportamiento de Order aquí, mezclado
}
```

**Consecuencias:**
- Las invariantes no se pueden garantizar — cualquiera puede poner `StatusCode = 99`
- La lógica de negocio se dispersa — difícil de encontrar, duplicar es fácil
- Los tests son complicados — para testear una regla de negocio necesitas el servicio completo

Con el modelo rico, el aggregate garantiza sus propias invariantes. La lógica de negocio vive donde pertenece.

## Cómo funciona EF Core con este diseño

EF Core puede trabajar con el aggregate aunque tenga todos los setters privados. Solo necesita configuración en el DbContext:

```csharp
// Orders.API/Infrastructure/Persistence/OrderDbContext.cs

protected override void OnModelCreating(ModelBuilder modelBuilder)
{
    modelBuilder.Entity<Order>(entity =>
    {
        entity.ToTable("Orders", schema: "orders");
        entity.HasKey(o => o.Id);

        // EF Core puede mapear propiedades privadas
        entity.Property(o => o.Status)
            .HasConversion(
                s => s.Value,          // Para guardar: OrderStatus → string
                v => OrderStatus.FromString(v))  // Para leer: string → OrderStatus
            .HasColumnName("Status");

        // Money como owned entity (columnas Amount + Currency)
        entity.OwnsOne(o => o.ShippingAddress, addr =>
        {
            addr.Property(a => a.Street).HasColumnName("ShippingStreet");
            addr.Property(a => a.City).HasColumnName("ShippingCity");
            addr.Property(a => a.ZipCode).HasColumnName("ShippingZipCode");
            addr.Property(a => a.Country).HasColumnName("ShippingCountry");
        });

        // Las líneas como entidades hijas
        entity.HasMany("_lines")  // Accede al campo privado
            .WithOne()
            .HasForeignKey("OrderId");
    });
}
```

## EF Core y el aggregate: el problema de los constructores

EF Core necesita instanciar las entidades cuando lee de la base de datos. Con el constructor privado de Order, necesitamos una forma de que EF Core funcione:

```csharp
// Opción 1: Constructor privado sin parámetros (para EF Core)
public class Order
{
    // EF Core puede usar constructores privados sin parámetros
    private Order() { }

    public static Order Create(...) { ... } // Factory method
}

// Opción 2: Constructor protected para EF Core
public class Order
{
    protected Order() { } // EF Core puede instanciar subclases también
}
```

EF Core 10 detecta automáticamente el constructor más adecuado. Un constructor sin parámetros (aunque sea privado) le permite crear instancias vacías para rellenar con los datos de la BD.

**La clave:** el constructor privado no es parte de la API pública. Solo el factory method `Create()` es la forma correcta de crear Orders en el código de negocio.

## EF Core: owned entities para Value Objects

Los Value Objects como `Money` y `Address` se mapean como "owned entities" en EF Core — sin tabla propia, sus columnas van en la tabla del Owner:

```csharp
modelBuilder.Entity<Order>(entity =>
{
    // Money → dos columnas: TotalAmount + TotalCurrency
    entity.OwnsOne(o => o.Total, money =>
    {
        money.Property(m => m.Amount).HasColumnName("TotalAmount")
             .HasPrecision(18, 2);
        money.Property(m => m.Currency).HasColumnName("TotalCurrency")
             .HasMaxLength(3);

    // Address → columnas separadas con prefijo Shipping
    entity.OwnsOne(o => o.ShippingAddress, addr =>
    {
        addr.Property(a => a.Street).HasColumnName("ShippingStreet")
            .HasMaxLength(200);
        addr.Property(a => a.City).HasColumnName("ShippingCity")
            .HasMaxLength(100);
        addr.Property(a => a.ZipCode).HasColumnName("ShippingZipCode")
            .HasMaxLength(20);
        addr.Property(a => a.Country).HasColumnName("ShippingCountry")
            .HasMaxLength(2);

```

En la tabla `Orders` habrá columnas `ShippingStreet`, `ShippingCity`, etc. Sin tabla separada para Address. La fila de Order tiene todos los datos del aggregate.

## Total calculado vs almacenado

Una decisión de diseño importante: ¿almacenamos el total en la base de datos o lo calculamos siempre?

**Total calculado (nuestra elección):**
```csharp
public Money Total => Lines.Any()
    ? Lines.Skip(1).Aggregate(Lines[0].LineTotal,
        (acc, line) => acc.Add(line.LineTotal))
    : Money.Zero("EUR");
```

**Ventaja:** Siempre consistente. No puede haber desfase entre las líneas y el total.

**Desventaja:** Requiere cargar las líneas para calcular el total. Para queries de listado, es ineficiente.

**La solución para el read model (M6.3):** el total calculado es el modelo de escritura (el aggregate). Para el modelo de lectura (listados), usamos una tabla desnormalizada con el total pre-calculado. CQRS resuelve este problema.

## Specification Pattern: queries complejas

Cuando las queries del repositorio se complican, el Specification Pattern mantiene la lógica en el dominio:

```csharp
// Specification base
public abstract class Specification<T>
{
    public abstract Expression<Func<T, bool>> ToExpression();

    public bool IsSatisfiedBy(T entity)
        => ToExpression().Compile()(entity);
}

// Specifications de Orders
public class PendingOrdersSpecification : Specification<Order>
{
    public override Expression<Func<Order, bool>> ToExpression()
        => order => order.Status == OrderStatus.Pending;
}

public class RecentOrdersSpecification : Specification<Order>
{
    private readonly int _days;
    public RecentOrdersSpecification(int days) => _days = days;

    public override Expression<Func<Order, bool>> ToExpression()
        => order => order.CreatedAt >= DateTime.UtcNow.AddDays(-_days);
}

// Uso en el repositorio
var spec = new PendingOrdersSpecification()
    .And(new RecentOrdersSpecification(7));

var pendingRecent = await _repository.GetBySpecAsync(spec);
```

Lo implementaremos si las queries se complican. Para Orders.API en el curso, las queries del repositorio son suficientemente simples.

## Un último patrón: Result vs Exception

Hay debate en la comunidad sobre si los errores de dominio deben lanzar excepciones o devolver un Result:

```csharp
// Enfoque con Exception (nuestro enfoque)
public void Confirm()
{
    Status = Status.TransitionTo(OrderStatus.Confirmed); // Lanza si inválido
    ConfirmedAt = DateTime.UtcNow;
}

// Enfoque con Result (alternativa)
public Result Confirm()
{
    if (!Status.CanTransitionTo(OrderStatus.Confirmed))
        return Result.Failure("Cannot confirm order in current status");

    Status = OrderStatus.Confirmed;
    ConfirmedAt = DateTime.UtcNow;
    return Result.Success();
}
```

**¿Cuándo usar excepciones?** Cuando la violación es definitivamente un error — el código llamante no debería llegar ahí en condiciones normales.

**¿Cuándo usar Result?** Cuando el fallo es parte del flujo normal — por ejemplo, si la validación de entrada puede producir errores que el usuario debe ver.

En Orders.API usamos excepciones para invariantes del dominio. El handler las captura y las convierte en Problem Details HTTP. En M3.2 vemos cómo.

## Por qué este diseño importa en microservicios

En un sistema distribuido, el dominio bien diseñado tiene un impacto especial:

**Reduce el chattiness entre servicios.** Un aggregate bien diseñado tiene toda la lógica que necesita para sus operaciones. No necesita preguntar a otro servicio para aplicar sus reglas.

**Domain Events como API natural.** Los eventos que el aggregate genera son exactamente los Integration Events que otros servicios necesitan escuchar. OrderCreatedEvent → OrderCreated (contrato de MassTransit).

**Facilita el Outbox Pattern.** Con domain events bien definidos, el Outbox sabe exactamente qué publicar. No hay lógica de "qué eventos generar" dispersa.

**Aísla los cambios de negocio.** Si la regla "máximo 50 líneas por pedido" cambia a 100, solo se modifica una línea en el aggregate. Ningún servicio externo, ninguna query de BD, ninguna regla de validación — solo una constante en el dominio.

## Naming conventions en el dominio

Convenciones que seguimos en todo el dominio de OrderFlow:

**Entidades y Aggregates:** PascalCase, nombre de negocio. `Order`, `OrderLine`. No `OrderEntity` ni `OrderModel`.

**Value Objects:** PascalCase, nombre del concepto. `Money`, `Address`, `OrderStatus`. No `MoneyVO` ni `ValueObjectMoney`.

**Domain Events:** PascalCase + sufijo `Event`. `OrderCreatedEvent`, `OrderConfirmedEvent`. En pasado — hechos que ya ocurrieron.

**Interfaces de repositorio:** Prefijo `I` + nombre del aggregate + `Repository`. `IOrderRepository`.

**Factory methods:** `Create`, `From`, `Of`. Nunca `new` en código externo al aggregate.

**Métodos de negocio:** Verbos del negocio. `Confirm()`, `Cancel(reason)`, `Ship()`. Nunca `SetStatus(...)`.

**Excepciones:** Solo `DomainException`. No `InvalidOperationException` ni `ArgumentException` para reglas de negocio.

## Comparativa: modelo anémico vs modelo rico

Un ejemplo concreto de cómo el mismo cambio se ve en ambos enfoques:

**Con modelo anémico:**

```csharp
public void Cancel(string reason)

    _domainEvents.Add(new OrderCancelledEvent(Id, CustomerId, reason));

    if (Total.Amount > 1000)
        _domainEvents.Add(new HighValueOrderCancelledEvent(Id, Total));
```

## Refactoring del dominio: cuándo y cómo

El dominio evoluciona con el negocio. Estas son las señales de que necesita refactoring:

**Señal 1: Métodos que necesitan datos de otro aggregate.**
Si `Order.Confirm()` necesita datos del cliente para validar, probablemente la validación está en el lugar incorrecto. Muévela a la capa de aplicación.

**Señal 2: El aggregate crece demasiado.**
Si `Order` tiene 30 métodos y 50 propiedades, probablemente tiene demasiada responsabilidad. Considera si hay sub-aggregates o si parte de la lógica pertenece a otro bounded context.

**Señal 3: Misma lógica en múltiples aggregates.**
Un `Guard.NotNullOrEmpty` se puede compartir. Pero si la lógica de negocio es la misma en dos aggregates, quizá pertenece a un Value Object compartido.

**Señal 4: Los tests se hacen difíciles de escribir.**
Si para testear un método del dominio necesitas 10 líneas de setup, el dominio tiene demasiadas dependencias internas o el método hace demasiado.

**Señal 5: Conversaciones con negocio generan confusión.**
Si al describir el código de `Order.Confirm()` al equipo de producto y no lo reconocen como "confirmar un pedido", el Ubiquitous Language está roto.

## Preguntas frecuentes sobre DDD táctico en .NET

**¿Puedo usar structs en vez de records para Value Objects?**
Puedes, pero los records son más convenientes: igualdad por valor automática, `with` expression, deconstrucción. Los structs añaden complejidad (boxing, comparaciones) sin ventajas claras para Value Objects en el dominio.

**¿EF Core puede mapearse a un dominio sin setters públicos?**
Sí. EF Core usa reflexión para acceder a campos y propiedades privadas. Con la configuración correcta en `OnModelCreating`, EF Core puede leer y escribir todas las propiedades necesarias.

**¿El dominio debe tener los DTOs de respuesta?**
No. Los DTOs viven en la capa de Application. El dominio devuelve entidades del dominio. Los handlers los transforman en DTOs.

**¿Un Bounded Context puede tener múltiples Aggregates?**
Sí. Orders podría tener también un aggregate `Cart` o `OrderTemplate`. Pero un aggregate nunca cruza la frontera de un Bounded Context.

## El siguiente paso: de dominio a API

El dominio de Orders está completo. Tenemos:
- Reglas de negocio encapsuladas en el aggregate
- Value Objects inmutables y seguros
- Domain Events que registran lo que pasó
- Repositorio con interfaz limpia
- Tests de dominio puros

En M3.2 exponemos este dominio como API REST con Controllers:
- Endpoints que traducen HTTP a Commands
- Model binding y validación de entrada
- ActionResult<T> con Problem Details para errores
- Exception middleware que convierte DomainException en 422

El Controller será delgado — solo adaptador HTTP. La lógica está en el dominio.

## Lo que añadiremos en M3.2

Con el dominio completo, en M3.2 construimos los Controllers que exponen el dominio como API REST.

Crearemos el `OrdersController` completo con:
- `GET /api/orders` — listar pedidos con filtros
- `GET /api/orders/{id}` — detalle de un pedido
- `POST /api/orders` — crear pedido
- `POST /api/orders/{id}/confirm` — confirmar pedido
- `POST /api/orders/{id}/cancel` — cancelar pedido

Veremos routing avanzado, model binding, ActionResult<T>, Problem Details, y exception middleware. Los Controllers serán delgados — solo traducen HTTP a Commands. Toda la lógica está en el dominio y en los handlers.
