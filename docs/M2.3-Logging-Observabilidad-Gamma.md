## ¿Por qué no basta con Console.WriteLine?

La respuesta instintiva ante un bug en producción: "añado un Console.WriteLine para ver qué pasa."

En un monolito local, funciona. En microservicios en producción, no:

**Sin estructura:** `Console.WriteLine($"Processing order {orderId}")` es texto plano. No puedes buscar todos los logs de un pedido concreto. No puedes filtrar por nivel. No puedes correlacionar con otros servicios.

**Sin contexto:** ¿Qué usuario hizo la petición? ¿Desde qué IP? ¿Cuánto tardó? ¿Qué servicio la procesó? Un WriteLine no lo sabe.

**Sin niveles:** No puedes distinguir entre información rutinaria y un error crítico. Necesitas saber cuándo algo va mal sin leer miles de líneas de texto.

**Sin persistencia configurable:** En producción, los logs necesitan ir a Application Insights, Seq, o Elasticsearch — no a la consola que nadie lee.

Serilog resuelve todo esto.

## Structured logging: el concepto clave

La diferencia fundamental entre logging tradicional y structured logging:

**Logging tradicional (texto plano):**
```
[2026-04-18 10:23:45] Processing order a4b5c6d7 for customer john@example.com, total: 150.00 EUR
```

Para buscar todos los pedidos de john@example.com, necesitas regex o grep. Para calcular el total medio de pedidos, imposible.

**Structured logging (propiedades nombradas):**
```
{
  "Timestamp": "2026-04-18T10:23:45Z",
  "Level": "Information",
  "Message": "Processing order",
  "OrderId": "a4b5c6d7",
  "CustomerEmail": "john@example.com",
  "Total": 150.00,
  "Currency": "EUR",
  "ServiceName": "orders-api"
}
```

Ahora puedes: filtrar por `CustomerEmail`, agrupar por `Currency`, calcular el `Total` medio, buscar todos los logs de un `OrderId` concreto. Los logs son datos, no texto.

## Serilog: el estándar de facto para .NET

Serilog es la librería de logging más usada en el ecosistema .NET. No es la incluida en el framework (Microsoft.Extensions.Logging) — pero se integra con ella perfectamente.

**Por qué Serilog sobre el logger nativo:**
- Sinks ricos: Console, Seq, Application Insights, Elasticsearch, fichero, etc.
- Enrichers: añadir propiedades automáticamente a todos los logs (CorrelationId, MachineName, Environment)
- Destructuring: registrar objetos complejos como propiedades estructuradas
- Filtros: descartar logs por nivel, propiedad, o expresión
- Output templates: controlar el formato de salida

**Sinks relevantes para OrderFlow:**
- `Serilog.Sinks.Console` — logs en la consola de desarrollo (y en Aspire dashboard)
- `Serilog.Sinks.ApplicationInsights` — logs en Azure Monitor para producción

---

## Por qué no usar ILogger nativo directamente

.NET incluye `ILogger<T>` como parte del framework. ¿Por qué usar Serilog encima?

**ILogger nativo:**
- Interfaz estándar — puede inyectarse y mockearse en tests
- Sinks: Console y EventSource de serie. El resto requieren proveedores adicionales
- Limitado en configuración dinámica
- Sin destructuring nativo

**Serilog sobre ILogger:**
- Se registra como proveedor de `ILogger` → misma interfaz, más potencia
- Decenas de sinks disponibles como NuGet
- Configuración desde appsettings con recarga en caliente
- Destructuring, filtros, enrichers potentes
- Output templates ricos

La clave: Serilog **reemplaza la implementación** de logging pero **mantiene la interfaz** de `ILogger<T>`. El código sigue usando `_logger.LogInformation(...)`. Solo cambia el backend.

## Serilog: Enrichers — contexto automático

Los Enrichers añaden propiedades a todos los logs automáticamente, sin que el código que escribe el log lo sepa:

```csharp
.Enrich.FromLogContext()        // Propiedades añadidas con LogContext.PushProperty
.Enrich.WithMachineName()       // Nombre del servidor → útil si escalas a múltiples instancias
.Enrich.WithEnvironmentName()   // Development, Production...
.Enrich.WithProcessId()         // ID del proceso
.Enrich.WithThreadId()          // ID del thread — útil para debugging de concurrencia
.Enrich.WithProperty("ServiceName", "orders-api")  // Propiedad fija
.Enrich.WithProperty("ServiceVersion", "1.0.0")
```

El resultado: cada log incluye todas estas propiedades sin que el desarrollador tenga que añadirlas manualmente. En Application Insights, puedes filtrar por `MachineName` para ver solo los logs de una instancia concreta, o por `ServiceVersion` para comparar comportamiento entre versiones.

---

## Serilog: LogContext — propiedades por scope

`LogContext.PushProperty` permite añadir propiedades solo durante un bloque de código:

```csharp
public async Task<IActionResult> CreateOrder([FromBody] CreateOrderRequest request)
{
    // Añadir OrderId al contexto — todos los logs dentro de este scope lo incluirán
    using var scope = LogContext.PushProperty("RequestedOrderId",
        Guid.NewGuid().ToString());

    _logger.LogInformation("Starting order creation");
    // Log incluye RequestedOrderId

    var order = await _orderService.CreateAsync(request);
    // Actualizar el contexto con el ID real asignado
    using var orderScope = LogContext.PushProperty("OrderId", order.Id);

    _logger.LogInformation("Order created successfully");
    // Log incluye OrderId

    return Created($"/api/orders/{order.Id}", order);
} // Al salir del using, las propiedades se eliminan del contexto
```

Esto es lo que el `CorrelationIdMiddleware` usa internamente. El CorrelationId se añade al LogContext al inicio del request y todos los logs del request lo incluyen automáticamente.

## Configurar Serilog en Program.cs

La configuración de Serilog va antes de todo lo demás — necesita estar activo desde el primer momento para capturar errores de startup:

```csharp
// Program.cs — Orders.API

using Serilog;

// Configurar Serilog desde el inicio
Log.Logger = new LoggerConfiguration()
    .WriteTo.Console()
    .CreateBootstrapLogger();

try
{
    var builder = WebApplication.CreateBuilder(args);

    // Reemplazar el logger por defecto con Serilog
    builder.Host.UseSerilog((context, services, configuration) =>
        configuration
            .ReadFrom.Configuration(context.Configuration)
            .ReadFrom.Services(services)
            .Enrich.FromLogContext()
            .Enrich.WithMachineName()
            .Enrich.WithEnvironmentName()
            .WriteTo.Console(outputTemplate:
                "[{Timestamp:HH:mm:ss} {Level:u3}] " +
                "{Message:lj} {Properties:j}{NewLine}{Exception}"));

    // ... resto de la configuración

    var app = builder.Build();
    app.UseSerilogRequestLogging(); // Log de cada request HTTP
    // ...
    app.Run();
}
catch (Exception ex)
{
    Log.Fatal(ex, "Application startup failed");
}
finally
{
    Log.CloseAndFlush();
}
```

## Log levels: cuándo usar cada uno

Un error frecuente: usar `Information` para todo. El resultado es ruido que hace imposible encontrar los problemas reales.

**Verbose/Trace:** Información extremadamente detallada. Cada línea de una query, cada byte de una petición. Solo en debugging profundo y muy localizado. Nunca en producción.

**Debug:** Información útil para debugging. Valores de variables, flujo de ejecución detallado. En desarrollo sí. En producción solo cuando diagnosticas un problema específico.

**Information:** El flujo normal del negocio. "Pedido creado", "Pago procesado", "Email enviado". Estas son las entradas importantes. Ni demasiadas ni demasiado pocas.

**Warning:** Algo inesperado pero recuperable. "Reintento 2 de 3 a Products.API", "Timeout en la conexión a la caché — usando datos del fallback". No es un error, pero merece atención.

**Error:** Algo falló y requiere atención. Excepciones no manejadas, operaciones que no pudieron completarse. Siempre con el `Exception` adjunto.

**Fatal:** El servicio no puede continuar. Error en startup, pérdida de conexión a la base de datos sin recovery. El proceso va a terminar.

## Cómo escribir logs correctamente

La diferencia entre un log que ayuda y uno que no:

**Log inútil:**
```csharp
_logger.LogInformation("Processing order");
_logger.LogError("Error occurred");
```

¿Qué pedido? ¿Qué error? Sin contexto, sin propiedades estructuradas, inutilizable.

**Log útil:**
```csharp
_logger.LogInformation(
    "Processing order {OrderId} for customer {CustomerId}. " +
    "Total: {Total} {Currency}. Items: {ItemCount}",
    order.Id, order.CustomerId, order.Total, order.Currency, order.Items.Count);

_logger.LogError(ex,
    "Failed to confirm order {OrderId}. " +
    "Payment status: {PaymentStatus}",
    order.Id, payment.Status);
```

Nota: las llaves `{PropertyName}` no son interpolación de strings — son propiedades estructuradas de Serilog. El valor se registra como una propiedad JSON buscable, no como texto embebido.

## Destructuring: registrar objetos complejos

Con Serilog, puedes registrar objetos enteros como propiedades estructuradas:

```csharp
// Registrar el objeto completo — Serilog llama a ToString()
_logger.LogInformation("Order created: {Order}", order);

// Registrar como estructura (todas las propiedades)
_logger.LogInformation("Order created: {@Order}", order);  // @ = destructuring

// Registrar solo las propiedades relevantes
_logger.LogInformation("Order created: {OrderId} {Status} {Total}",
    order.Id, order.Status, order.Total);
```

El operador `@` antes del nombre de la propiedad le dice a Serilog que serialice el objeto completo. Útil para debugging — pero úsalo con cuidado en producción porque puede registrar datos sensibles.

Para evitar registrar datos sensibles, Serilog tiene el operador `$` para destructuring destructivo y la capacidad de configurar destructuradores custom que excluyen campos.

## Request logging con Serilog

`UseSerilogRequestLogging()` añade automáticamente un log por cada petición HTTP con toda la información relevante:

```csharp
app.UseSerilogRequestLogging(options =>
{
    // Personalizar qué se registra
    options.MessageTemplate =
        "HTTP {RequestMethod} {RequestPath} responded {StatusCode} " +
        "in {Elapsed:0.0000} ms";

    // Añadir propiedades adicionales al log de la petición
    options.EnrichDiagnosticContext = (diagnosticContext, httpContext) =>
    {
        diagnosticContext.Set("RequestHost", httpContext.Request.Host.Value);
        diagnosticContext.Set("RequestScheme", httpContext.Request.Scheme);
        diagnosticContext.Set("UserAgent",
            httpContext.Request.Headers["User-Agent"].FirstOrDefault());
    };

    // Filtrar peticiones que no queremos loggear (health checks)
    options.GetLevel = (ctx, elapsed, ex) =>
        ctx.Request.Path.StartsWithSegments("/health")
            ? LogEventLevel.Verbose
            : LogEventLevel.Information;
});
```

El resultado: cada petición a Orders.API genera automáticamente un log con método HTTP, path, status code, duración, host, y cualquier propiedad adicional que configures.

## Correlation IDs: el hilo que conecta los servicios

Imagina que un usuario crea un pedido y algo falla. Para diagnosticar el problema, necesitas:
- Los logs de Orders.API mientras procesaba el pedido
- Los logs de Products.API mientras verificaba stock
- Los logs de Payments.API mientras cobraba

Sin Correlation ID, tienes tres montones de logs independientes sin forma de saber cuáles corresponden al mismo pedido.

Con Correlation ID: un ID único (`X-Correlation-Id`) se genera en el primer servicio que recibe la petición (YARP o Orders) y se propaga a todos los demás. Todos los logs incluyen ese ID. Una búsqueda por ese ID te da el timeline completo.

---

## Implementar Correlation ID middleware

```csharp
// Orders.API/API/Middleware/CorrelationIdMiddleware.cs

public class CorrelationIdMiddleware
{
    private const string CorrelationIdHeader = "X-Correlation-Id";
    private readonly RequestDelegate _next;

    public CorrelationIdMiddleware(RequestDelegate next)
        => _next = next;

    public async Task InvokeAsync(HttpContext context)
    {
        // Usar el CorrelationId que viene del upstream
        // o generar uno nuevo si no viene
        var correlationId = context.Request.Headers[CorrelationIdHeader]
            .FirstOrDefault()
            ?? Activity.Current?.TraceId.ToString()
            ?? Guid.NewGuid().ToString("N");

        // Añadir al response para que el cliente lo vea
        context.Response.Headers[CorrelationIdHeader] = correlationId;

        // Añadir al LogContext para que aparezca en todos los logs
        using (LogContext.PushProperty("CorrelationId", correlationId))
        {
            // Almacenar en HttpContext para usarlo en los handlers
            context.Items["CorrelationId"] = correlationId;
            await _next(context);
        }
    }
}
```

---

## Registrar el middleware y propagarlo

```csharp
// Program.cs — añadir el middleware
app.UseMiddleware<CorrelationIdMiddleware>();
```

Para que los logs de todos los requests incluyan el CorrelationId automáticamente, Serilog usa `LogContext.PushProperty` del middleware. Ahora, en los logs de Serilog, verás:

```json
{
  "Timestamp": "2026-04-18T10:23:45Z",
  "Level": "Information",
  "Message": "Processing order a4b5c6d7",
  "OrderId": "a4b5c6d7",
  "CorrelationId": "3fa85f6457174562b3fc2c963f66afa6",
  "ServiceName": "orders-api"
}
```

Y cuando Orders.API llama a Products.API, propaga el CorrelationId en el header:

```csharp
// En el HttpClient de Products.API
httpClient.DefaultRequestHeaders.Add(
    "X-Correlation-Id",
    httpContext.Items["CorrelationId"]?.ToString());
```

Con esto, si buscas `3fa85f64...` en Application Insights, ves todos los logs de todos los servicios que procesaron esa petición.

## OpenTelemetry: el estándar de observabilidad

OpenTelemetry (OTel) es un estándar open source para instrumentación de sistemas distribuidos. Define cómo se recopilan y exportan las tres señales de observabilidad:

**Traces:** Una petición que cruza múltiples servicios. Cada salto es un "span" con duración, nombre y atributos. El conjunto de spans forma una "trace" completa.

**Metrics:** Mediciones numéricas en el tiempo. Peticiones por segundo, latencia media, errores por minuto, pedidos creados por hora.

**Logs:** Los logs estructurados que ya configuramos con Serilog. OTel puede correlacionarlos con traces.

La clave de OTel: el código de instrumentación es el mismo para todos los backends. Si exportas a Jaeger hoy y mañana decides ir a Application Insights, solo cambias el exporter — no el código de instrumentación.

ServiceDefaults de Aspire ya configura OpenTelemetry básico. Vamos a enriquecerlo.

---

## OpenTelemetry: el pipeline de telemetría

OTel funciona como un pipeline: instrumentación → procesamiento → exportación.

```
Tu código (o instrumentación automática)
        ↓
ActivitySource / MeterFactory
        ↓
SDK de OpenTelemetry (sampling, procesamiento)
        ↓
Exporters:
  - OTLP Exporter → Dashboard de Aspire (desarrollo)
  - Azure Monitor Exporter → Application Insights (producción)
  - Console Exporter → stdout (debugging)
```

En Aspire, el AppHost levanta un recolector OTLP. Todos los servicios exportan sus trazas y métricas a ese recolector, que las muestra en el dashboard.

En Azure App Services, configuras el Azure Monitor Exporter y los datos van directamente a Application Insights. Sin cambiar el código de instrumentación — solo el exporter.

---

## Activity y ActivitySource: OTel en .NET

.NET tiene soporte nativo de OpenTelemetry a través de `System.Diagnostics.Activity` y `ActivitySource`.

```csharp
// Crear una fuente de actividades para Orders.API
public static class OrdersTelemetry
{
    public static readonly ActivitySource ActivitySource =
        new ActivitySource("Orders.API", "1.0.0");
}

// Crear spans manuales en código de negocio
public async Task<Order> CreateOrderAsync(CreateOrderCommand cmd)
{
    using var activity = OrdersTelemetry.ActivitySource
        .StartActivity("CreateOrder");

    // Añadir atributos al span
    activity?.SetTag("order.customerId", cmd.CustomerId.ToString());
    activity?.SetTag("order.itemCount", cmd.Items.Count);
    activity?.SetTag("order.total", cmd.TotalAmount.ToString());

    try
    {
        var order = await _repository.CreateAsync(cmd);

        activity?.SetStatus(ActivityStatusCode.Ok);
        return order;
    }
    catch (Exception ex)
    {
        activity?.SetStatus(ActivityStatusCode.Error, ex.Message);
        activity?.RecordException(ex);
        throw;
    }
}
```

## Configurar OpenTelemetry en Orders.API

ServiceDefaults ya añade OTel básico. Extendemos con instrumentación específica:

```csharp
// Program.cs — configuración extendida de OTel

builder.Services.AddOpenTelemetry()
    .WithTracing(tracing =>
    {
        tracing
            // Instrumentación automática — no necesitas tocar el código
            .AddAspNetCoreInstrumentation()    // Cada request HTTP
            .AddHttpClientInstrumentation()    // Cada llamada HTTP saliente
            .AddEntityFrameworkCoreInstrumentation() // Cada query EF Core
            .AddSource("Orders.API")           // Spans manuales que creemos

            // Exporters
            .AddOtlpExporter()   // OTLP → dashboard de Aspire
            .AddConsoleExporter(); // Solo en desarrollo para debugging
    })
    .WithMetrics(metrics =>
    {
        metrics
            .AddAspNetCoreInstrumentation()
            .AddHttpClientInstrumentation()
            .AddRuntimeInstrumentation()    // CPU, GC, threads
            .AddMeter("Orders.API")         // Métricas personalizadas
            .AddOtlpExporter();
    });
```

## Instrumentación automática: qué obtienes gratis

Con `AddAspNetCoreInstrumentation()` y `AddHttpClientInstrumentation()`, OTel crea automáticamente:

**Para cada request HTTP entrante:**
- Span con nombre del endpoint (`GET /api/orders/{id}`)
- Duración del request
- Status code
- Método HTTP
- URL completa
- Errores si los hay

**Para cada llamada HTTP saliente (a Products.API por ejemplo):**
- Span hijo del request actual
- URL de destino
- Status code recibido
- Duración de la llamada
- Errores de red

**Para cada query de EF Core:**
- Span hijo con el SQL ejecutado
- Duración de la query
- Base de datos y tabla afectada

Sin escribir una sola línea de código de instrumentación, tienes visibilidad completa de lo que hace el servicio. Lo ves todo en el dashboard de Aspire.

## Trazas distribuidas: cómo funciona

Cuando Orders.API llama a Products.API, OTel propaga automáticamente el contexto de traza:

```
HTTP Request → Orders.API
│
├── Span: "GET /api/orders" (20ms total)
│   │
│   ├── Span: "Orders.CreateOrder" (15ms)
│   │   │
│   │   ├── Span: "HTTP GET products-api/api/products/{id}" (5ms)
│   │   │   └── (En Products.API)
│   │   │       └── Span: "GET /api/products/{id}" (4ms)
│   │   │           └── Span: "SQL SELECT Products WHERE Id = @p0" (2ms)
│   │   │
│   │   └── Span: "SQL INSERT Orders" (3ms)
│   │
│   └── Span: "Publish OrderCreated event" (1ms)
```

Esta es la traza completa de una petición de creación de pedido. Visible en el dashboard de Aspire como un timeline.

Cuando algo va lento, ves exactamente en qué span se pierde el tiempo. ¿La query de Products tarda 200ms? Ahí está el problema.

## Métricas: medir lo que importa

Las métricas te dan visibilidad sobre el comportamiento del servicio a lo largo del tiempo. OTel tiene tres tipos de instrumentos:

**Counter:** Solo sube. Ideal para contar eventos.
```csharp
// Número de pedidos creados
var ordersCreated = meter.CreateCounter<long>(
    "orders.created",
    unit: "orders",
    description: "Total number of orders created");

// Usar en el handler
ordersCreated.Add(1, new TagList
{
    { "currency", order.Currency },
    { "customer_type", order.CustomerType }
});
```

**Histogram:** Distribución de valores. Ideal para latencias.
```csharp
// Tiempo de procesamiento de un pedido
var orderProcessingTime = meter.CreateHistogram<double>(
    "orders.processing_duration",
    unit: "ms",
    description: "Order processing duration in milliseconds");

// Usar al terminar el procesamiento
orderProcessingTime.Record(
    stopwatch.ElapsedMilliseconds,
    new TagList { { "result", "success" } });
```

**Gauge:** Valor actual. Ideal para stocks, colas.
```csharp
// Número de pedidos pendientes
// El callback de ObservableGauge debe ser síncrono — usar valor cacheado
private int _cachedPendingOrders = 0;

// Actualizar la caché periódicamente (ej. cada 30s con un BackgroundService)
public async Task RefreshCacheAsync(CancellationToken ct)
    => _cachedPendingOrders = await _repository.CountPendingAsync(ct);

// El gauge lee el valor cacheado de forma síncrona
var pendingOrders = meter.CreateObservableGauge<int>(
    "orders.pending",
    () => _cachedPendingOrders);
```

## Métricas de negocio en Orders.API

Las métricas técnicas (latencia, errores) las genera OTel automáticamente. Las métricas de negocio las defines tú:

```csharp
// Orders.API/Infrastructure/Telemetry/OrdersMetrics.cs

public class OrdersMetrics
{
    private readonly Counter<long> _ordersCreated;
    private readonly Counter<long> _ordersFailed;
    private readonly Histogram<double> _processingTime;

    public OrdersMetrics(IMeterFactory meterFactory)
    {
        var meter = meterFactory.Create("Orders.API");

        _ordersCreated = meter.CreateCounter<long>(
            "orders.created.total");

        _ordersFailed = meter.CreateCounter<long>(
            "orders.failed.total");

        _processingTime = meter.CreateHistogram<double>(
            "orders.processing.duration.ms");
    }

    public void RecordOrderCreated(string currency, string customerType)
        => _ordersCreated.Add(1,
            new TagList { { "currency", currency },
                          { "customer_type", customerType } });

    public void RecordOrderFailed(string reason)
        => _ordersFailed.Add(1,
            new TagList { { "reason", reason } });

    public void RecordProcessingTime(double milliseconds)
        => _processingTime.Record(milliseconds);
}

// Registrar como Singleton
builder.Services.AddSingleton<OrdersMetrics>();
```

## Serilog y OpenTelemetry: combinados

Serilog y OpenTelemetry se complementan. Para correlacionar logs con trazas, añadimos el Correlation ID de OTel al LogContext de Serilog automáticamente:

```csharp
// Enricher que añade TraceId y SpanId de OTel al log de Serilog
public class OpenTelemetryEnricher : ILogEventEnricher
{
    public void Enrich(LogEvent logEvent, ILogEventPropertyFactory factory)
    {
        var activity = Activity.Current;
        if (activity is null) return;

        logEvent.AddPropertyIfAbsent(
            factory.CreateProperty("TraceId", activity.TraceId.ToString()));

        logEvent.AddPropertyIfAbsent(
            factory.CreateProperty("SpanId", activity.SpanId.ToString()));
    }
}

// Añadir al builder de Serilog
.Enrich.With<OpenTelemetryEnricher>()
```

Ahora los logs de Serilog incluyen `TraceId` y `SpanId`. En Application Insights, puedes buscar el TraceId y ver tanto los logs como las trazas en el mismo contexto.

## Filtrar logs de health checks

Sin filtros, el log de request de Serilog registra cada ping al endpoint `/health` — que ocurre cada 30 segundos. En producción, eso son 2.880 logs al día solo de health checks.

Filtramos para ignorarlos:

```csharp
// En la configuración de UseSerilogRequestLogging
app.UseSerilogRequestLogging(options =>
{
    options.GetLevel = (ctx, elapsed, ex) =>
    {
        if (ex is not null) return LogEventLevel.Error;

        if (ctx.Response.StatusCode >= 500)
            return LogEventLevel.Error;

        if (ctx.Response.StatusCode >= 400)
            return LogEventLevel.Warning;

        // Los health checks en Verbose (no aparecen en producción)
        if (ctx.Request.Path.StartsWithSegments("/health") ||
            ctx.Request.Path.StartsWithSegments("/alive"))
            return LogEventLevel.Verbose;

        return LogEventLevel.Information;
    };
});
```

En producción, el MinimumLevel es Information. Los health checks (Verbose) no se registran. Cero ruido.

## Diagnóstico en producción: cómo usar los logs

Con la observabilidad configurada, cuando algo falla en producción:

**Paso 1 — Detectar:** Una alerta de Application Insights notifica que la tasa de errores ha subido por encima del 5%.

**Paso 2 — Identificar:** En Application Insights, filtras los logs por `Level: Error` en el rango de tiempo de la alerta. Encuentras 50 logs de error en 5 minutos.

**Paso 3 — Correlacionar:** Copias el `CorrelationId` del primer error. Buscas ese ID en todos los logs. Ves la secuencia completa: Request recibido → llamada a Products.API → timeout → error.

**Paso 4 — Trazar:** Buscas el `TraceId` en la vista de trazas. Ves el timeline: 3 segundos de espera en el span de `HTTP GET products-api` antes del timeout.

**Paso 5 — Diagnosticar:** Products.API estaba respondiendo lento por una query no optimizada.

Sin observabilidad, este diagnóstico llevaría horas. Con ella, minutos.

## Logging responsable: datos sensibles

Cuidado con lo que registras. Un log mal escrito puede convertirse en un problema de compliance:

**Nunca registrar:**
- Passwords o tokens
- Números de tarjeta de crédito
- Números completos de DNI/NIF
- Datos de salud (si aplica HIPAA)
- Datos que bajo GDPR no deben persistirse

**Registrar de forma segura:**
```csharp
// ❌ NO — registra el email completo
_logger.LogInformation("User {Email} logged in", user.Email);

// ✅ SÍ — enmascara o usa un identificador no sensible
_logger.LogInformation("User {UserId} logged in", user.Id);

// ❌ NO — registra datos del pago
_logger.LogInformation("Processing payment with card {@Card}", card);

// ✅ SÍ — solo el ID de la transacción
_logger.LogInformation("Processing payment {PaymentId}", payment.Id);
```

Serilog tiene destructuring custom que puede enmascarar automáticamente campos marcados con atributos. Para OrderFlow, la regla es: registra IDs y estados, nunca datos personales ni financieros.

## Errores comunes en logging de microservicios

Los que veo repetirse en proyectos reales:

**Error 1: Loggear en el nivel incorrecto.**
Todo en Information o todo en Debug. Resultado: no puedes distinguir lo importante de lo rutinario, o no ves nada útil porque el log level es demasiado restrictivo.

**Error 2: Loggear sin propiedades estructuradas.**
`_logger.LogError($"Error processing order {orderId}")` — interpolación de string. No es structured logging. No puedes filtrar por `orderId` en Application Insights. Usa siempre `{PropertyName}` con valores separados.

**Error 3: Olvidar el CorrelationId.**
Logs sin CorrelationId son imposibles de rastrear en un sistema distribuido. El middleware debe estar activo y el LogContext debe incluirlo.

**Error 4: Loggear datos sensibles.**
Emails, passwords, números de tarjeta en logs. Una brecha de seguridad esperando a ocurrir. Regla: registra IDs y estados, nunca datos personales.

**Error 5: No filtrar health checks.**
2.880 logs al día de health checks en producción. Ruido que oculta lo importante. Siempre filtra a Verbose.

**Error 6: No cerrar Serilog al terminar.**
`Log.CloseAndFlush()` en el bloque `finally` es obligatorio. Sin él, pueden perderse logs en el buffer.

## Application Insights vs Grafana + Loki + Tempo

En el curso usamos Application Insights para producción. Vale la pena conocer las alternativas:

**Application Insights (nuestra elección):**
- Managed: cero infraestructura que gestionar

- Cost: basado en datos ingestados (~3€ por GB)

- Coste: coste de infraestructura (VM, discos)

- Multi-cloud o on-premise (no quieres depender de Azure)
- Necesitas configuración muy específica que Application Insights no ofrece
- Grandes volúmenes donde el coste por GB de App Insights sale caro

Para OrderFlow en Azure: Application Insights. Si el proyecto fuera multi-cloud, Grafana.

## Distribución de logs por nivel: cómo debería verse

En producción, la distribución saludable de logs para Orders.API:

**~90% Information:**
Flujo normal de negocio. "Pedido creado", "Pago procesado", "Email enviado". Si tienes menos, estás perdiendo visibilidad. Si tienes más, hay demasiado ruido.

**~8-9% Warning:**
Situaciones anómalas pero controladas. Reintentos, timeouts resueltos, fallbacks activados. Si tienes muchos, algo no está funcionando bien pero el servicio lo está manejando.

**~1-2% Error:**
Errores reales que requieren atención. Si tienes más del 5%, hay un problema sistémico.

Solo para fallos catastróficos. Si aparece con frecuencia, el servicio no está en producción — está cayendo.

Si tu distribución no se parece a esto, revisa qué estás loggeando y a qué nivel.

## OpenTelemetry: sampling en producción

En producción, no puedes exportar el 100% de las trazas — es demasiado volumen y demasiado coste.

**Head-based sampling:** Se decide si trazar una petición al inicio, antes de procesarla. Más simple, pero no puedes capturar selectivamente trazas de peticiones lentas.

```csharp
.AddAspNetCoreInstrumentation(options =>
{
    options.Filter = ctx =>
        !ctx.Request.Path.StartsWithSegments("/health");
})
```

**Tail-based sampling:** Se decide si exportar una traza después de procesarla, basándose en el resultado. Puedes capturar el 100% de los errores y solo el 1% de las peticiones exitosas.

Para OrderFlow en producción con Application Insights, el sampling por defecto es adaptativo — ajusta automáticamente el porcentaje basándose en el volumen.

Para el curso (desarrollo con Aspire), exportamos el 100% de las trazas.

## Preguntas frecuentes de M2.3

**¿Necesito OpenTelemetry si ya tengo Serilog?**
Sí, son complementarios. Serilog maneja los logs (texto con propiedades). OpenTelemetry maneja las trazas (timeline de spans) y las métricas (números en el tiempo). Juntos dan visibilidad completa.

**¿El CorrelationId es lo mismo que el TraceId de OTel?**
Parecido pero no igual. El CorrelationId lo genera tu middleware y viaja en el header `X-Correlation-Id`. El TraceId lo genera OTel y viaja en el header `traceparent` del estándar W3C. En ordenflow los sincronizamos añadiendo el TraceId actual al CorrelationId cuando no hay uno previo.

**¿Puedo usar `ILogger<T>` normal en los Controllers si uso Serilog?**
Sí. Serilog se registra como proveedor de `Microsoft.Extensions.Logging`. Cuando inyectas `ILogger<T>` y llamas a `LogInformation`, internamente usa Serilog. La interfaz no cambia.

**¿Los logs de health checks afectan al rendimiento?**
No al rendimiento del request — el health check sigue respondiendo al mismo ritmo. El filtro solo evita que esos logs se escriban en el sink, ahorrando I/O y coste de almacenamiento.
