## Dónde estamos

En M3.4 implementamos CQRS básico con MediatR: Commands que modifican el estado y Queries que lo leen. Ambos usan el mismo `OrderDbContext` y las mismas tablas.

El problema que resolvemos en M6.3: para queries complejas, el modelo de dominio no es eficiente. Cargar un `Order` aggregate con sus `OrderLines` para mostrar solo un resumen es costoso. Y si quieres filtrar por múltiples campos, ordenar, paginar y hacer joins con datos desnormalizados, EF Core genera SQL cada vez más complicado.

La solución: un **Read Model** separado — tablas optimizadas para lectura, consultadas con Dapper directamente. El write model (EF Core + aggregate) no cambia. Las queries se vuelven 10x más rápidas.

## El problema actual del ListOrdersHandler

El handler de listado actual hace esto:

```csharp
// ListOrdersHandler — carga el aggregate completo para mostrar un resumen
public async Task<PagedResult<OrderSummaryDto>> Handle(
    ListOrdersQuery query, CancellationToken ct)
{
    return await _dbContext.Orders
        .Include(o => o.Lines)     // ← JOIN a OrderLines innecesario para el resumen
        .Where(o => o.CustomerId == query.CustomerId)
        .Select(o => new OrderSummaryDto
        {
            Id = o.Id,
            Status = o.Status.Value,
            Total = o.Total.Amount,
            LinesCount = o.Lines.Count()  // ← Se calcula en SQL
        })
        .OrderByDescending(o => o.CreatedAt)
        .Skip((query.Page - 1) * query.PageSize)
        .Take(query.PageSize)
        .ToListAsync(ct);
}
```

Para 10.000 pedidos con 5 líneas cada uno, EF Core genera un JOIN de 50.000 filas. La proyección reduce los datos devueltos, pero el JOIN se hace igualmente.

## CQRS completo: dos modelos de datos distintos

El CQRS avanzado separa radicalmente los dos lados:

```
WRITE SIDE (ya lo tenemos):
  CreateOrderCommand → CreateOrderHandler
  → Order aggregate (dominio rico)
  → EF Core con todas las validaciones
  → orders.Orders + orders.OrderLines

READ SIDE (lo que construimos en M6.3):
  ListOrdersQuery → ListOrdersHandler
  → No pasa por el dominio
  → Dapper directamente sobre SQL
  → orders.OrderSummaries (tabla desnormalizada)
```

El Write Side garantiza consistencia. El Read Side garantiza velocidad. Son completamente independientes — puedes cambiar el esquema del Read Model sin tocar el Write Model.

## Dapper: SQL directo de alto rendimiento

**Dapper** es un micro-ORM desarrollado por el equipo de Stack Overflow. Mapea resultados SQL a objetos .NET sin la capa de abstracción de EF Core:

```bash
dotnet add package Dapper
```

```csharp
// Sin Dapper (EF Core):
var summaries = await _dbContext.OrderSummaries
    .Where(s => s.CustomerId == customerId && s.Status == status)
    .OrderByDescending(s => s.CreatedAt)
    .Skip(skip).Take(take)
    .ToListAsync();
// EF Core genera SQL + overhead de tracking + materialización de entidades

// Con Dapper:
var summaries = await connection.QueryAsync<OrderSummaryDto>(
    "SELECT * FROM orders.OrderSummaries " +
    "WHERE CustomerId = @CustomerId AND Status = @Status " +
    "ORDER BY CreatedAt DESC " +
    "OFFSET @Skip ROWS FETCH NEXT @Take ROWS ONLY",
    new { CustomerId = customerId, Status = status, Skip = skip, Take = take });
// SQL directo, sin tracking, mapeo automático por nombre de columna
```

## El repositorio de lectura con Dapper

```csharp
// Orders.API/Infrastructure/Persistence/ReadModel/OrderReadRepository.cs

public class OrderReadRepository : IOrderReadRepository
{
    private readonly string _connectionString;
    private readonly ILogger<OrderReadRepository> _logger;

    public OrderReadRepository(IConfiguration configuration,
        ILogger<OrderReadRepository> logger)
    {
        _connectionString = configuration.GetConnectionString("OrdersDb")!;
        _logger = logger;
    }

    public async Task<PagedResult<OrderSummaryDto>> ListAsync(
        Guid? customerId, string? status,
        int page, int pageSize,
        CancellationToken ct = default)
    {
        await using var connection =
            new SqlConnection(_connectionString);

        // Build WHERE dinámico
        var where = new List<string>();
        if (customerId.HasValue) where.Add("CustomerId = @CustomerId");
        if (!string.IsNullOrEmpty(status)) where.Add("Status = @Status");
        var whereClause = where.Any()
            ? $"WHERE {string.Join(" AND ", where)}"
            : string.Empty;

        var skip = (page - 1) * pageSize;

        // Query de datos y count en paralelo
        var countSql = $"SELECT COUNT(1) FROM orders.OrderSummaries {whereClause}";
        var dataSql  = $"""
            SELECT OrderId, CustomerId, CustomerEmail, Status,
                   TotalAmount, Currency, LinesCount, FirstItemName,
                   CreatedAt, ConfirmedAt, CancelledAt, ShippingCity
            FROM orders.OrderSummaries
            {whereClause}
            ORDER BY CreatedAt DESC
            OFFSET {skip} ROWS FETCH NEXT {pageSize} ROWS ONLY
            """;

        var parameters = new { CustomerId = customerId, Status = status };

        var total = await connection.ExecuteScalarAsync<int>(countSql, parameters);
        var items = await connection.QueryAsync<OrderSummaryDto>(dataSql, parameters);

        return new PagedResult<OrderSummaryDto>(
            items.ToList(), total, page, pageSize);
    }
}
```

## Proyecciones: mantener el Read Model actualizado

El Read Model de `OrderSummaries` se queda obsoleto si no se actualiza cuando cambia un pedido. La estrategia: un consumer que escucha los eventos del Outbox y actualiza la tabla.

```
FLUJO COMPLETO:
1. Cliente → POST /api/orders
2. CreateOrderHandler:
   → INSERT orders.Orders (write model)
   → INSERT orders.OutboxMessage (OrderCreated)
3. OutboxProcessor publica OrderCreated → RabbitMQ
4. OrderSummaryProjector (consumer en Orders.API) recibe OrderCreated:
   → INSERT orders.OrderSummaries ← Read Model actualizado
5. Cliente → GET /api/orders/my
   → ListOrdersHandler → Dapper → orders.OrderSummaries (ya actualizado)
```

## El OrderSummaryProjector

```csharp
// Orders.API/Consumers/OrderSummaryProjector.cs

public class OrderSummaryProjector : IConsumer<OrderCreated>
{
    private readonly OrderDbContext _dbContext;
    private readonly ILogger<OrderSummaryProjector> _logger;

    public async Task Consume(ConsumeContext<OrderCreated> context)
    {
        var msg = context.Message;

        // Verificar idempotencia
        var exists = await _dbContext.Set<OrderSummary>()
            .AnyAsync(s => s.OrderId == msg.OrderId);

        if (exists)
        {
            _logger.LogWarning(
                "OrderSummary for {OrderId} already exists, skipping",
                msg.OrderId);
            return;
        }

        var summary = new OrderSummary
        {
            OrderId = msg.OrderId,
            CustomerId = msg.CustomerId,
            CustomerEmail = msg.CustomerEmail,
            Status = "Pending",
            TotalAmount = msg.Total,
            Currency = msg.Currency,
            LinesCount = msg.Items.Count,
            FirstItemName = msg.Items.FirstOrDefault()?.ProductName,
            CreatedAt = msg.CreatedAt,
            ShippingCity = msg.ShippingAddress?.City
        };

        _dbContext.Set<OrderSummary>().Add(summary);
        await _dbContext.SaveChangesAsync(context.CancellationToken);

        _logger.LogInformation(
            "OrderSummary created for order {OrderId}", msg.OrderId);
    }
}
```

## Proyector de OrderConfirmed

```csharp
// Orders.API/Consumers/OrderConfirmedProjector.cs

public class OrderConfirmedProjector : IConsumer<OrderConfirmed>
{
    private readonly OrderDbContext _dbContext;

    public async Task Consume(ConsumeContext<OrderConfirmed> context)
    {
        var msg = context.Message;

        var summary = await _dbContext.Set<OrderSummary>()
            .FirstOrDefaultAsync(s => s.OrderId == msg.OrderId);

        if (summary is null)
        {
            // Race condition: aún no existe el summary → reintentar más tarde
            throw new InvalidOperationException(
                $"OrderSummary for {msg.OrderId} not found. Will retry.");
        }

        summary.Status = "Confirmed";
        summary.ConfirmedAt = DateTime.UtcNow;

        await _dbContext.SaveChangesAsync(context.CancellationToken);
    }
}

// También: OrderCancelledProjector, OrderFailedProjector
// Cada evento de la Saga actualiza el Read Model
```

## Consistencia eventual del Read Model

El Read Model es **eventualmente consistente** con el Write Model. Hay un desfase de ~5-10 segundos (tiempo del Outbox + tiempo del consumer):

```
t=0:   POST /api/orders → 201 Created
       orders.Orders → Status: Pending ✅
       orders.OrderSummaries → NO EXISTE AÚN

t=5s:  Outbox publica OrderCreated
       OrderSummaryProjector procesa
       orders.OrderSummaries → Status: Pending ✅

t=5s+: GET /api/orders/my → devuelve el pedido ✅
```

Para el listado de pedidos, 5 segundos de desfase es completamente aceptable. Si el usuario crea un pedido y refresca inmediatamente el listado, puede que no vea su pedido. En la práctica, el frontend lo gestiona mostrando el pedido localmente hasta que el servidor lo confirme.

## Reconstruir el Read Model desde cero

¿Qué pasa si el Read Model se corrompe o necesitas añadir un campo nuevo?

```csharp
// Comando de administración: reconstruir el Read Model completo
public class RebuildOrderSummariesHandler
    : IRequestHandler<RebuildOrderSummariesCommand>
{
    private readonly OrderDbContext _dbContext;

    public async Task Handle(
        RebuildOrderSummariesCommand command, CancellationToken ct)
    {
        // 1. Borrar el Read Model actual
        await _dbContext.Database.ExecuteSqlRawAsync(
            "DELETE FROM orders.OrderSummaries", ct);

        // 2. Reconstruir desde el Write Model (fuente de verdad)
        var orders = await _dbContext.Orders
            .Include("_lines")
            .ToListAsync(ct);

        foreach (var order in orders)
        {
            _dbContext.Set<OrderSummary>().Add(new OrderSummary
            {
                OrderId = order.Id,
                CustomerId = order.CustomerId,
                Status = order.Status.Value,
                TotalAmount = order.Total.Amount,
                Currency = order.Total.Currency,
                LinesCount = order.Lines.Count,
                FirstItemName = order.Lines.FirstOrDefault()?.ProductName,
                CreatedAt = order.CreatedAt,
                ConfirmedAt = order.ConfirmedAt,
                CancelledAt = order.CancelledAt,
            });
        }

        await _dbContext.SaveChangesAsync(ct);
    }
}
```

Esto es el poder del CQRS avanzado: el Write Model es la única fuente de verdad. El Read Model puede reconstruirse en cualquier momento.

## Sincronización inicial: poblar el Read Model

Cuando el sistema se despliega por primera vez o se añade el Read Model a un sistema existente, hay que poblar la tabla `OrderSummaries` con los datos actuales:

```csharp
// Script de migración de datos (no es una EF Core migration)
// Se ejecuta UNA VEZ al desplegar la funcionalidad del Read Model

public class BackfillOrderSummariesJob
{
    private readonly OrderDbContext _dbContext;

    public async Task RunAsync(CancellationToken ct = default)
    {
        // Procesar en lotes de 100 para no saturar la BD
        var batchSize = 100;
        var offset = 0;

        while (true)
        {
            var orders = await _dbContext.Orders
                .Include("_lines")
                .OrderBy(o => o.CreatedAt)
                .Skip(offset)
                .Take(batchSize)
                .ToListAsync(ct);

            if (!orders.Any()) break;

            foreach (var order in orders)
            {
                var exists = await _dbContext.Set<OrderSummary>()
                    .AnyAsync(s => s.OrderId == order.Id, ct);

                if (!exists)
                {
                    _dbContext.Set<OrderSummary>().Add(MapToSummary(order));
                }
            }

            await _dbContext.SaveChangesAsync(ct);
            offset += batchSize;

            _logger.LogInformation(
                "Backfilled {Offset} orders", offset);
        }
    }
}
```

## GetOrderById: Dapper para el detalle también

Para el detalle de un pedido, también podemos usar Dapper con un JOIN optimizado:

```csharp
// IOrderReadRepository — detalle de pedido
public async Task<OrderDetailDto?> GetByIdAsync(
    Guid orderId, Guid requestingUserId, bool isAdmin,
    CancellationToken ct = default)
{
    await using var connection = new SqlConnection(_connectionString);

    // JOIN de Orders + OrderLines en una sola query
    var sql = """
        SELECT
            o.Id, o.CustomerId, o.CustomerEmail, o.Status,
            o.TotalAmount, o.TotalCurrency,
            o.CreatedAt, o.ConfirmedAt, o.CancelledAt, o.CancellationReason,
            o.Street, o.City, o.ZipCode, o.Country,
            l.Id AS LineId, l.ProductId, l.ProductName,
            l.Quantity, l.UnitPriceAmount, l.UnitPriceCurrency
        FROM orders.Orders o
        LEFT JOIN orders.OrderLines l ON l.OrderId = o.Id
        WHERE o.Id = @OrderId
          AND (@IsAdmin = 1 OR o.CustomerId = @UserId)
        """;

    // Dapper multi-mapping: una fila por línea, agrupa en el pedido
    OrderDetailDto? order = null;

    await connection.QueryAsync<OrderDetailDto, OrderLineDto, OrderDetailDto>(
        sql,
        (o, line) =>
        {
            order ??= o;
            if (line is not null)
                order.Lines.Add(line);
            return order;
        },
        splitOn: "LineId",
        param: new { OrderId = orderId, UserId = requestingUserId, IsAdmin = isAdmin ? 1 : 0 });

    return order;
}
```

## Dapper multi-mapping: la técnica para JOINs

El multi-mapping de Dapper permite mapear una sola query con JOIN a un objeto con colección:

```
SQL devuelve:
OrderId | Status  | TotalAmount | LineId | ProductName | Qty
7b3a2c  | Pending | 3999.99     | a1b2c3 | MacBook Pro | 1
7b3a2c  | Pending | 3999.99     | d4e5f6 | iPad case   | 2

Dapper mapea a:
OrderDetailDto {
  Id: 7b3a2c,
  Status: Pending,
  Total: 3999.99,
  Lines: [
    { Id: a1b2c3, Name: MacBook Pro, Qty: 1 },
    { Id: d4e5f6, Name: iPad case,   Qty: 2 }
  ]
}
```

Sin múltiples queries, sin N+1 problem. Un JOIN bien hecho.

## Queries complejas con Dapper: filtros dinámicos

```csharp
public async Task<PagedResult<OrderSummaryDto>> SearchAsync(
    OrderSearchQuery query, CancellationToken ct = default)
{
    await using var connection = new SqlConnection(_connectionString);

    var sb = new StringBuilder(
        "SELECT * FROM orders.OrderSummaries WHERE 1=1");
    var parameters = new DynamicParameters();

    if (query.CustomerId.HasValue)
    {
        sb.Append(" AND CustomerId = @CustomerId");
        parameters.Add("CustomerId", query.CustomerId.Value);
    }

    if (!string.IsNullOrEmpty(query.Status))
    {
        sb.Append(" AND Status = @Status");
        parameters.Add("Status", query.Status);
    }

    if (query.From.HasValue)
    {
        sb.Append(" AND CreatedAt >= @From");
        parameters.Add("From", query.From.Value);
    }

    if (query.To.HasValue)
    {
        sb.Append(" AND CreatedAt <= @To");
        parameters.Add("To", query.To.Value);
    }

    if (!string.IsNullOrEmpty(query.ShippingCity))
    {
        sb.Append(" AND ShippingCity = @City");
        parameters.Add("City", query.ShippingCity);
    }

    // Paginación
    var skip = (query.Page - 1) * query.PageSize;
    sb.Append($" ORDER BY CreatedAt DESC " +
              $"OFFSET {skip} ROWS FETCH NEXT {query.PageSize} ROWS ONLY");

    var items = await connection.QueryAsync<OrderSummaryDto>(
        sb.ToString(), parameters);

    return new PagedResult<OrderSummaryDto>(
        items.ToList(), total: 0, query.Page, query.PageSize);
}
```

## Dapper con stored procedures: cuándo tiene sentido

Para queries extremadamente complejas o heredadas, Dapper puede llamar a stored procedures:

```csharp
// Stored procedure para un reporte complejo de pedidos
public async Task<IEnumerable<OrderReportDto>> GetMonthlyReportAsync(
    int year, int month, CancellationToken ct = default)
{
    await using var connection = new SqlConnection(_connectionString);

    return await connection.QueryAsync<OrderReportDto>(
        "orders.GetMonthlyOrderReport",    // nombre del SP
        new { Year = year, Month = month },
        commandType: CommandType.StoredProcedure);
}
```

**Cuándo usar stored procedures con Dapper:**
- Lógica compleja de BD que ya existe (migración de monolito)
- Reportes que usan CTEs, window functions o funciones específicas de SQL Server
- Cuando el DBA del equipo prefiere gestionar las queries desde el lado de BD

**Para OrderFlow:** Dapper con SQL inline. Los stored procedures añaden una dependencia del esquema de BD que dificulta las migrations.

## Connection pooling: gestionar las conexiones de Dapper

EF Core usa un pool de conexiones automáticamente. Con Dapper, las conexiones se abren y cierran por query — pero el pool de ADO.NET las reutiliza:

```csharp
// En OrderReadRepository — patrón correcto
public async Task<PagedResult<OrderSummaryDto>> ListAsync(...)
{
    // await using: cierra la conexión al terminar (vuelve al pool)
    await using var connection = new SqlConnection(_connectionString);
    // NO necesitas llamar a connection.Open() — Dapper lo hace automáticamente

    var result = await connection.QueryAsync<OrderSummaryDto>(...);
    return new PagedResult<OrderSummaryDto>(result.ToList(), ...);
}
// La conexión vuelve al pool — la siguiente query la reutiliza
```

**El pool de conexiones ADO.NET:**
- Por defecto: mínimo 0, máximo 100 conexiones
- En Azure SQL Basic tier: máximo 30 sesiones simultáneas
- Configurar en la connection string: `Max Pool Size=50`

Para TechShop en producción inicial (50K pedidos/mes), el pool por defecto es más que suficiente.

## Transacciones con Dapper: cuándo son necesarias

Para el Read Model, normalmente no necesitamos transacciones (solo leemos). Pero en los proyectores (que escriben), a veces sí:

```csharp
// Proyector que necesita transacción: actualizar summary + insertar log
public async Task Consume(ConsumeContext<OrderConfirmed> context)
{
    await using var connection = new SqlConnection(_connectionString);
    await connection.OpenAsync();
    await using var transaction = await connection.BeginTransactionAsync();

    try
    {
        // 1. Actualizar el Read Model
        await connection.ExecuteAsync(
            "UPDATE orders.OrderSummaries SET Status = 'Confirmed' " +
            "WHERE OrderId = @OrderId",
            new { OrderId = context.Message.OrderId },
            transaction);

        // 2. Insertar en el log de proyecciones
        await connection.ExecuteAsync(
            "INSERT INTO orders.ProjectionLog (OrderId, EventType, ProcessedAt) " +
            "VALUES (@OrderId, @EventType, @Now)",
            new { OrderId = context.Message.OrderId,
                  EventType = "OrderConfirmed",
                  Now = DateTime.UtcNow },
            transaction);

        await transaction.CommitAsync();
    }
    catch
    {
        await transaction.RollbackAsync();
        throw;
    }
}
```

## Pagination patterns: cursor vs offset

El `ListAsync` actual usa **offset pagination** (`OFFSET N ROWS FETCH NEXT M ROWS ONLY`). Para grandes volúmenes, el cursor pagination es más eficiente:

**Offset Pagination (lo que tenemos):**
```sql
SELECT * FROM orders.OrderSummaries
ORDER BY CreatedAt DESC
OFFSET 1000 ROWS FETCH NEXT 20 ROWS ONLY
-- Problema: para la página 50, SQL Server tiene que "saltarse" 1000 filas
-- Con muchos datos, se vuelve lento
```

**Cursor Pagination (para volúmenes grandes):**
```sql
-- El cursor es el último valor de la página anterior
SELECT * FROM orders.OrderSummaries
WHERE CreatedAt < @LastCreatedAt  -- el cursor
ORDER BY CreatedAt DESC
FETCH NEXT 20 ROWS ONLY
-- No importa cuántas filas hay antes — siempre rápido
```

```csharp
// Response con cursor para el cliente
public record PagedResultWithCursor<T>
{
    public IList<T> Items { get; init; } = [];
    public string? NextCursor { get; init; }  // null si es la última página
    public bool HasMore => NextCursor is not null;
}
```

**Para TechShop:** Offset pagination es suficiente (máximo miles de pedidos por cliente). Cursor pagination para sistemas con millones de registros.

## Read Model en Commands: no mezclar

Un error frecuente: acceder al Read Model desde un Command handler.

```csharp
// ❌ INCORRECTO — Command accediendo al Read Model
public class ConfirmOrderHandler : IRequestHandler<ConfirmOrderCommand>
{
    private readonly IOrderReadRepository _readRepo; // ← INCORRECTO en un Command

    public async Task Handle(ConfirmOrderCommand cmd, CancellationToken ct)
    {
        // El Read Model puede estar 5 segundos desactualizado
        var summary = await _readRepo.GetByIdAsync(cmd.OrderId);
        // ¿Qué pasa si el summary no existe aún? Race condition.
    }
}

// ✅ CORRECTO — Command usando el Write Model (EF Core + aggregate)
public class ConfirmOrderHandler : IRequestHandler<ConfirmOrderCommand>
{
    private readonly IOrderRepository _repository; // ← Correcto — Write Model

    public async Task Handle(ConfirmOrderCommand cmd, CancellationToken ct)
    {
        var order = await _repository.GetByIdAsync(cmd.OrderId, ct);
        // El write model siempre está actualizado
        order!.Confirm();
        await _repository.SaveAsync(order, ct);
    }
}
```

La regla: **Commands usan el Write Model. Queries usan el Read Model.** Nunca al revés.

## Rendimiento: EF Core vs Dapper en el listado

Con 10.000 pedidos en la BD, la diferencia es significativa:

```
ANTES (EF Core sobre el write model):
Query generada por EF Core:
  SELECT o.Id, o.Status, o.TotalAmount, ...,
         l.Id, l.ProductId, l.ProductName, ...
  FROM orders.Orders o
  LEFT JOIN orders.OrderLines l ON l.OrderId = o.Id
  WHERE o.CustomerId = @p0

Resultado: 100 filas (20 pedidos × 5 líneas) para mostrar 20 resúmenes
Tiempo: ~45ms (con índice en CustomerId)

DESPUÉS (Dapper sobre OrderSummaries):
SELECT *
FROM orders.OrderSummaries
WHERE CustomerId = @CustomerId
ORDER BY CreatedAt DESC
OFFSET 0 ROWS FETCH NEXT 20 ROWS ONLY

Resultado: 20 filas (ya desnormalizado)
Tiempo: ~4ms (10x más rápido)
```

La diferencia se amplía con más datos y queries más complejas.

## Cuándo usar EF Core y cuándo Dapper

La guía de decisión para OrderFlow:

```
EF Core:
  ✅ Operaciones de escritura (Commands) — validaciones, tracking, transacciones
  ✅ Queries simples de una sola entidad: GetById del write model
  ✅ Migrations — gestión del esquema
  ✅ Relaciones complejas con navegación

Dapper:
  ✅ Queries de lectura (Queries) sobre el Read Model
  ✅ Reportes y dashboards — múltiples JOINs, agregaciones
  ✅ Queries con SQL específico de SQL Server (CTEs, Window Functions)
  ✅ Cuando el rendimiento es crítico y tienes control del SQL

Ambos en el mismo servicio: ✅ CORRECTO
  EF Core para writes + Dapper para reads
  No son excluyentes — se complementan perfectamente
```

## Cuándo NO usar Read Models

El Read Model añade complejidad (proyectores, consistencia eventual). No siempre merece la pena:

**NO usar Read Model cuando:**
- El servicio tiene pocas queries y baja carga
- Los datos cambian muy poco (catálogo estático)
- La latencia de 5-10 segundos no es aceptable para el negocio
- El equipo es pequeño y la complejidad extra tiene coste alto

**SÍ usar Read Model cuando:**
- Las queries de listado son lentas (>50ms con índices bien configurados)
- El mismo dato se consulta frecuentemente con diferentes filtros
- Tienes SLOs de rendimiento que EF Core no puede cumplir
- El negocio crece y el volumen de datos aumenta

**Para TechShop ahora:** Con 50K pedidos/mes, EF Core es suficiente. El Read Model se añade anticipando el crecimiento a 500K-1M pedidos/mes, cuando la diferencia de rendimiento se vuelve crítica.

## Read Model vs Caché: cuándo usar cada uno

Una duda frecuente: ¿no es lo mismo usar un Read Model que cachear el resultado?

**Caché (IMemoryCache / IDistributedCache):**
```csharp
// La caché guarda el resultado de una query
if (_cache.TryGetValue("orders:customer:john", out var cached))
    return cached;
var result = await _dbContext.Orders.Where(...).ToListAsync();
_cache.Set("orders:customer:john", result, TimeSpan.FromMinutes(5));
```
- Resultado ya procesado almacenado en memoria
- Se invalida por tiempo (TTL)
- Si el TTL es 5 minutos y hay un pedido nuevo, el usuario no lo ve
- Adecuado para datos que cambian poco (catálogo, configuración)

**Read Model (OrderSummaries):**
```csharp
// El Read Model es una tabla en BD, actualizada por eventos
var summaries = await _dapper.QueryAsync<OrderSummaryDto>(...);
```
- Datos en BD persistentes, actualizados por proyectores
- Consistencia eventual (~5-10 segundos)
- Si hay un pedido nuevo y el proyector lo procesó: el usuario lo ve
- Adecuado para datos que cambian frecuentemente

**Para TechShop:** Caché para el catálogo de productos (cambia poco). Read Model para listado de pedidos (cada nuevo pedido debe aparecer pronto).

## Versionar el Read Model: añadir campos sin downtime

Cuando se añade un campo nuevo a `OrderSummaries` (ej: `DiscountAmount`):

**FASE 1 — Añadir la columna (nullable):**
```sql
-- Migration
ALTER TABLE orders.OrderSummaries
ADD DiscountAmount decimal(18,2) NULL;
```

**FASE 2 — Actualizar el proyector:**
```csharp
// OrderSummaryProjector actualizado
var summary = new OrderSummary
{
    // ...campos existentes...
    DiscountAmount = msg.DiscountAmount  // ← nuevo campo
};
```

**FASE 3 — Backfill de datos históricos:**
```sql
-- Actualizar los pedidos existentes
UPDATE orders.OrderSummaries
SET DiscountAmount = (
    SELECT SUM(DiscountAmount) FROM orders.Orders
    WHERE Id = OrderSummaries.OrderId
)
WHERE DiscountAmount IS NULL;
```

**FASE 4 — Hacer NOT NULL (si aplica):**
```sql
ALTER TABLE orders.OrderSummaries
ALTER COLUMN DiscountAmount decimal(18,2) NOT NULL;
```

Este es el patrón Expand-Contract aplicado al Read Model — lo veremos más en profundidad en M6.4.

## Event Sourcing: introducción al concepto

El CQRS avanzado a veces lleva a **Event Sourcing**: en vez de guardar el estado actual de un pedido, guardas todos los eventos que lo modificaron.

```
ESTADO ACTUAL (lo que tenemos):
orders.Orders → { Id, Status: Confirmed, TotalAmount: 3999.99, ... }
Si quieres saber por qué está Confirmed: tienes que mirar los logs

EVENT SOURCING (concepto avanzado):
orders.Events → [
  { OrderId, Type: OrderCreated, Data: {...}, Timestamp: t0 },
  { OrderId, Type: StockReserved, Data: {...}, Timestamp: t5s },
  { OrderId, Type: PaymentProcessed, Data: {...}, Timestamp: t10s },
  { OrderId, Type: OrderConfirmed, Data: {...}, Timestamp: t15s }
]
El estado actual se RECONSTRUYE aplicando los eventos en orden
```

**Ventajas:** Audit trail perfecto, capacidad de time-travel (¿cómo estaba el pedido hace 2 horas?), fácil de añadir nuevas proyecciones.

**Desventajas:** Mayor complejidad, queries de estado actual lentas (hay que replay todos los eventos).

**Para TechShop:** Mantenemos el modelo de estado actual + Outbox Pattern. Event Sourcing es un paso más avanzado cuando el audit trail se vuelve crítico.

## Test E2E del CQRS completo

```csharp
// Test end-to-end: crear pedido → proyección → listado

[Fact]
public async Task CreateOrder_EventuallyAppearsInListing()
{
    // 1. Crear el pedido (write side)
    var createResponse = await _client.PostAsJsonAsync(
        "/api/orders", ValidOrderRequest);
    createResponse.StatusCode.Should().Be(HttpStatusCode.Created);
    var created = await createResponse.Content.ReadFromJsonAsync<OrderDto>();

    // 2. Esperar a que el proyector actualice el Read Model (~10s máximo)
    var listResponse = await RetryUntilAsync(
        async () =>
        {
            var r = await _client.GetAsync("/api/orders/my");
            var paged = await r.Content
                .ReadFromJsonAsync<PagedResult<OrderSummaryDto>>();
            return paged?.Items.Any(o => o.OrderId == created!.Id) == true;
        },
        maxRetries: 10,
        delayMs: 1000);

    // 3. Verificar que aparece en el listado (Read Model)
    listResponse.Should().BeTrue(
        "el pedido debería aparecer en el listado en menos de 10 segundos");
}

// Helper de retry
private static async Task<bool> RetryUntilAsync(
    Func<Task<bool>> condition, int maxRetries, int delayMs)
{
    for (int i = 0; i < maxRetries; i++)
    {
        if (await condition()) return true;
        await Task.Delay(delayMs);
    }
    return false;
}
```

## Preguntas frecuentes sobre CQRS avanzado

**¿El Read Model siempre tiene que ser una tabla SQL?**
No. Puede ser Redis (caché), Elasticsearch (búsqueda full-text), Azure Cosmos DB (flexibilidad de esquema) o cualquier almacén de datos optimizado para lectura. Para TechShop: SQL Server es suficiente y evita añadir más tecnologías.

**¿Qué pasa si el proyector falla y el Read Model no se actualiza?**
El mensaje va a la Dead Letter Queue de RabbitMQ (MassTransit lo gestiona). El pedido existe en el Write Model (es la fuente de verdad). El Read Model se puede reconstruir manualmente con el comando de backfill. La ausencia de un pedido en el listado es molesta pero no crítica — el pedido existe y será procesado.

**¿Puedo tener varios Read Models para el mismo dato?**
Sí. `OrderSummaries` para el listado del cliente. `AdminOrderView` para el panel de administración. `OrderAnalytics` para reportes. Cada uno optimizado para su caso de uso. Todos alimentados por los mismos eventos.

**¿El Read Model necesita migrations de EF Core?**
Sí — necesita una migration para crear la tabla `OrderSummaries`. Pero la tabla solo es gestionada por EF Core para el schema. Los datos los gestiona Dapper (lecturas) y los proyectores (escrituras al Read Model).
